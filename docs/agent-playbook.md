# エージェント・プレイブック — 予算データ探索とレポート作成の標準手順

本書は、AIエージェントが本サイト（まるみえRSシステム）のデータを探索し調査レポートを作成する際の標準手順書。**探索を始める前に必ず全文を読むこと。** モデルや実行環境が変わっても一定品質のレポートを出すことを目的とする（背景: `docs/tasks/20260703_0431_エージェント視点の欲しい能力と品質格差の考察.md`）。

対象環境: ローカル（`npm run dev` → `http://localhost:3000`）。API仕様の正典は [docs/api-guide.md](api-guide.md)。

---

## 1. 禁じ手（最初に読む）

1. **生データファイルを直接開かない。必ずAPIを使う。** `public/data/*.json` や `data/**/*.csv` がローカルにあっても読まないこと。生データは名寄せ・集約・検証の前処理を経ておらず、直接集計すると「その他」の重複計上・Multi-block の合算漏れ・単位の誤読を高確率で起こす。APIの絞り込みと `metadata.notes` は探索品質の土台である
2. **直接受注と再委託受注を合算しない。** 二重計上になる。常に分離して並記する
3. **推測で数値を書かない。** レポート中のすべての金額・件数はAPI応答から転記し、どのAPIで得たかを自分で追跡できる状態を保つ

## 2. データの罠一覧（誤読すると結論が変わるもの）

| 罠 | 正しい理解 |
|----|-----------|
| 「その他」と「その他の支出先」 | **別物**。「その他」= 支出先名が文字通り「その他」と報告されたもの（追跡不能な支出）。「その他の支出先」= 表示件数制限による集約ノード（データは存在する） |
| 年度のズレ | 事業年度YEARのデータは**予算年度YEAR-1の執行実績**（例: 2024年度データ = 2023年度予算の実績） |
| 金額の単位 | すべて**1円単位**（千円単位ではない） |
| データの範囲 | 行政事業レビュー対象事業のみ = **国の全予算の約27%**。国債費・地方交付税等は含まれない。「国の予算全体」を語らないこと |
| 品質スコアの意味 | **報告の形式品質**（特定可能性・説明性・成果設計の明確さ）であり、資金経路の透明性や事業の実効性ではない。支出先が執行団体1件のみの事業はスコアが高く出るが、その先の資金の流れはデータに現れない |
| `status=unknown` | 「辞書に未収録」であって「実在しない」ではない（辞書鮮度の問題。AI軸スコアが最終指標） |
| 執行団体の壁 | 基金・補助金執行団体（一般社団法人等）への支出は、その先（個々の採択事業者）がレビューデータに現れないことが多い。「ここから先は見えない」と明記する |
| 予算と支出の差 | 翌年度繰越・基金積み残し・支出先未報告分を含む。「未執行」と断定しない |
| ルートとベンダーの二重計上 | 支出先明細で執行団体（`r=true` のルートブロック）は**事業費全額**、その先のベンダーは**内訳**。ルートと内訳を単純合算すると約2倍になる。ベンダーのシェアを出すときは**ルート額を分母**にする |
| 支出先の名寄せ（表記揺れ） | 同一企業でも ①表記揺れ（例「株式会社NTTデータ」vs「エヌ・ティ・ティ・データ株式会社」）②別法人の子会社（例「NTTデータSMS」「…カスタマサービス」）③**法人番号の誤記載**（同一実体が別番号で重複登録）が混在する。名前の**部分一致で名寄せしない**（子会社を巻き込み・別表記を取りこぼす）。集計は**法人番号キー**で行い、見出しは**親会社の特定法人番号単独**に統一、グループ計は別掲。誤記載疑いの別番号は注記。※支出先インデックスの品質は既知の未解決課題 |

## 3. 探索の標準ルート

```text
① 全体像の数値化: GET /api/sankey/query?q={SankeyQuery}   （summary: 件数・金額・上位）
     ├─ 0件/多すぎ → filter を調整して再実行（正規表現を緩める・金額下限を足す）
     └─ 語彙がわからない → GET /api/search/projects?q= / /api/search/recipients?q=
② 対象確定後の深掘り（pid ごと）:
     ├─ 品質スコア: GET /api/quality-scores?pids=1900,4752,...   （バッチ・数KB）
     ├─ 事業の中身: GET /api/project-details/[pid]               （目的・概要・開始年度）
     ├─ 再委託構造: GET /api/subcontracts/[pid]                  （hasRedelegation の事業のみで可）
     └─ 支出先明細: GET /api/quality-scores/recipients?pid=      （「その他」等の内訳確認）
③ 横断視点: GET /api/search/recipients → GET /api/recipients/[法人番号]（支出先の府省庁横断受注）
④ 図の提示: ①の応答の links.webView をレポートに記載（人間がブラウザで開ける）
```

- `detail=full` は数値を二次加工する場合のみ。通常は summary で足りる
- 全件系API（`/api/quality-scores` の pids 無し等）は引かない。pid・キーで絞る（複数事業のスコアは `?pids=1,2,3` で一括取得できる）
- 各応答の `metadata.notes` を必ず読む（罠の注意書きが埋めてある）
- **企業の府省庁横断集計は明細を全事業ぶん引いて自前で合算しない**（トークン浪費・名寄せ誤りの温床）。`/api/search/recipients?q=` で**法人番号**を得て `/api/recipients/[法人番号]` を引けば、**1コール・数KB**で `byMinistry` 内訳・`aliases`（表記揺れ名寄せ済み）・`totals` が返る。ここが横断集計の最安ルート。ただし法人番号の**誤記載で別番号に割れる**分は取りこぼすため、疑わしい別番号は `/api/search/recipients` で確認し注記する（§2「支出先の名寄せ」）
- 事業別の webView リンクでピンを使う場合、`pp` の値は `project-spending-<pid>` 形式（`pp=5389` のような pid 単体は正規形式ではない）
- この標準ルートは**出発点**であり、価値ある逸脱（例: マッチ全事業の品質統計、低スコア事業の横断リスト）を妨げない。逸脱した場合はレポートにその方法を明記する

## 4. レポート様式

構成（模範例: `docs/tasks/20260702_1552_実験1_自律探索レポート_再エネ大型事業.md`）:

1. **調査方法**: 使ったフィルタ条件・年度・「金額は1円単位を億円換算」の明記
2. **サンキー図**: `links.webView` の URL（可能ならスクリーンショット）
3. **全体像**: 対象事業数・予算/支出合計・支出先数のテーブル + 事業一覧（pid・府省庁・予算・支出・スコア）
4. **発見（2〜4個）**: 各発見に見出しを付け、**数値の根拠**（どの値からそう言えるか）を本文に含める。単なる数値の羅列ではなく「何が問題／注目に値するか」を一文で言い切る
5. **データ留意事項**: §2 の罠のうちレポートの解釈に影響するものを明記

文体: 断定できることは断定し、データの限界による不確実性は「〜の可能性が高い」「データに現れない」と正確に区別する。

## 5. 自己検証の作法（レポート提出前に実施）

- [ ] summary の合計と、深掘りで得た明細の合算が整合するか（例: 事業の支出合計 ≒ 支出先明細の合算）
- [ ] 件数の相互確認（query API の projects.count と、深掘りした pid の数が一致するか）
- [ ] 「その他」を集約ノードと取り違えていないか（§2 の1行目を再確認）
- [ ] 直接/再委託を合算した数値がないか
- [ ] ルート（執行団体）とベンダー内訳を二重計上していないか（ベンダーのシェアはルート額が分母か）
- [ ] 支出先を名前の部分一致で名寄せしていないか（法人番号キーか・親/子会社を分離したか・§2「支出先の名寄せ」を再確認）
- [ ] レポート中の全金額に単位（億円/兆円）と換算元（1円単位）の明記があるか
- [ ] webView URL を実際に開ける形式で記載したか（ローカルなら `/sankey-svg?...` の相対パスで可）
- [ ] **使用したAPI（エンドポイントと概算呼び出し回数）をレポートに記録したか**（探索の再現性と効率測定のため必須）

## 6. API呼び出しの目安

探索レポート1本あたり **10〜15回** が標準（実験①実測）。数十回を超える場合は探索が発散している——①に戻って条件を絞り直すこと。
