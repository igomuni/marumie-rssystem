# 一般会計と特別会計の混在状況調査レポート

**作成日**: 2026年2月1日
**対象データ**: RS2024構造化データ（5,003事業）
**調査目的**: サンキー図における一般会計と特別会計の混在状況を把握し、分離表示の必要性を検討する

---

## 1. エグゼクティブサマリ

### 主な発見事項

1. **ほぼ半々の規模**: 一般会計71兆円、特別会計77兆円とほぼ同等の規模
2. **混在府省庁**: 37省庁中12省庁が両方の会計を持つ
3. **特別会計の種類**: 37種類の特別会計が存在（年金関連が最大）
4. **データ品質問題**: 49事業で「一般会計, 特別会計」のような混在表記がある

### 推奨アクション

- **優先度 高**: 一般会計と特別会計を分離して可視化
- **優先度 中**: 混在表記（カンマ区切り）の事業をどちらかに分類
- **優先度 低**: 特別会計を種類別にさらに細分化表示

## 7. データ品質の問題：混在表記の事業

### 問題の内容

`accountCategory` フィールドに以下のような混在表記がある：

1. **「特別会計, 一般会計」**: 15事業、1.42兆円
2. **「一般会計, 特別会計」**: 34事業、1.10兆円

合計49事業、2.52兆円が混在表記

### 混在表記の府省庁

| 表記パターン | 府省庁数 | 主な府省庁 |
|-------------|---------|-----------|
| 特別会計, 一般会計 | 3省庁 | 厚生労働省、経済産業省、環境省 |
| 一般会計, 特別会計 | 5省庁 | 国土交通省、厚生労働省、経済産業省など |

### 対処方針の提案

**Option 1**: カンマで分割して複数の会計区分として扱う
- メリット: データの実態を正確に反映
- デメリット: 可視化が複雑になる

**Option 2**: 最初の会計区分のみを採用
- 例: 「特別会計, 一般会計」→「特別会計」
- メリット: シンプルな分類
- デメリット: 一部の情報が失われる

**Option 3**: 予算額の大きい方の会計区分に統一
- メリット: より実態に近い分類
- デメリット: 実装が複雑

**推奨**: Option 2（最初の会計区分を採用）
- 49事業は全体の1%未満なので影響は限定的
- シンプルで実装が容易

## 8. サンキー図での分離表示案

### 案1: 会計区分を第1列目に追加（6列構成）

```
予算総計 → 会計区分 → 府省庁 → 事業(予算) → 事業(支出) → 支出先
```

**メリット**:
- 一般会計と特別会計が明確に分離される
- 全体像が把握しやすい

**デメリット**:
- 列が1つ増えて横幅が広がる
- モバイル表示がさらに困難に

### 案2: 府省庁ノードを会計区分で分割（5列のまま）

```
予算総計 → 府省庁（一般）/府省庁（特別） → 事業(予算) → 事業(支出) → 支出先
```

例:
- 「厚生労働省（一般）」: 23兆円
- 「厚生労働省（特別）」: 69兆円

**メリット**:
- 列数は変わらない（5列維持）
- 府省庁ごとの一般/特別の比率が視覚的に分かる

**デメリット**:
- 府省庁ノード数が増える（12省庁 × 2 = 24ノード追加）
- 混在していない省庁も分割すべきか判断が必要

### 案3: フィルタ機能で切り替え

```
UI上で「一般会計のみ」「特別会計のみ」「両方」を選択可能
```

**メリット**:
- 既存の5列構造を維持
- ユーザーが見たいものを選べる

**デメリット**:
- 全体像が一目で分からない
- UI実装が複雑

### 推奨案

**短期（Phase 1）**: 案2を採用
- 府省庁ノードを会計区分で分割
- 混在している12省庁のみ分割、残りは「（一般）」のみ表示
- 実装が比較的シンプル

**長期（Phase 2）**: 案3も追加
- フィルタ機能でユーザーが選択可能に
- より柔軟な分析が可能
