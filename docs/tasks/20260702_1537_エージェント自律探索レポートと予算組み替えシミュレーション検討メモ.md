## 発案の内容

1. **自律探索レポート**: 品質スコアや事業概要等の中身をエージェントが自律的に探索し、サンキー図と調査結果をレポートする
2. **予算組み替えシミュレーション**: とある事業の予算の組み替えシミュレーションのようなことができるのではないか

## 1. 自律探索レポート — 既存APIの組み合わせで今日から試せる

必要な部品はすべて実装済みで、**新規開発なしにローカル実験のシナリオとして実行できる**。

| 探索ステップ | 使うAPI（実装済み） |
|------------|-------------------|
| 語彙・対象の発見 | `/api/search/projects`、`/api/search/recipients` |
| 絞り込みと全体像の数値化 | `/api/sankey/query`（summary: 件数・金額・上位） |
| 事業の中身 | `/api/project-details/[pid]`（事業概要・実施方法） |
| データ品質の評価 | `/api/quality-scores`、`/api/quality-scores/recipients` |
| 再委託構造・企業横断 | `/api/subcontracts/[pid]`、`/api/recipients/[key]` |
| 図の提示 | `/api/sankey/query` の `links.webView` |

例（エージェントへの依頼イメージ）: 「再エネ関連の大型事業を調べて、データ品質の懸念も含めてレポートして」
→ query API で15事業に絞る → 各事業の品質スコアと概要を取得 → 再委託が深い事業を特定 → webView URL を貼ったMarkdownレポートを出力。

**ギャップ（実験で確かめること）**:

- レポートの様式が未定義 — エージェント任せで十分か、テンプレート（プレイブック的な指示書）を用意すべきか
- API横断の探索でエージェントが「その他」問題や直接/再委託の二重計上を正しく扱えるか（metadata.notes の効果測定になる）
- 探索の深さと呼び出し回数のバランス（公開時のレートリミット設計の材料）

→ **次のローカル実験は、この探索レポートを1本実走してみるのが良い**。クエリAPIの表現力検証（実験フェーズの観察項目）と同時にできる。

## 2. 予算組み替えシミュレーション — 性格が異なる機能。設計判断が先

「もし事業Aの予算を減らして事業Bに付け替えたら図がどう変わるか」のような what-if は、既存機能と**根本的に性格が異なる**点に注意が必要。

### 本プロジェクトの方針との緊張

- 本サイトは**実績データの正確な可視化**を最優先方針としてきた。シミュレーションは**仮想データの表示**であり、実績と仮想が混ざって見えることは方針への最大のリスク
- 実装するなら「これは仮想値である」ことを図・API応答の両方で構造的に区別する仕組み（例: シミュレーションモードの明示、ノード・金額への仮想フラグ）が前提条件になる

### 技術的には土台がある

- `detail=full` が返す nodes/edges は「図に描画される内容そのもの」なので、**エージェント側で金額を書き換えた仮想グラフを作り、比較レポートを書く**ことは今でも可能（サイト側は無改修）
- サイト側で描画までやる場合は、`SankeyQuery` に「上書き指示」（例: `overrides: [{ projectId, budget }]`）を足してグラフ加工層を1段挟む形が考えられるが、上記の方針整理が先

### 先に決めるべき問い

1. 「組み替え」の単位は何か — 事業間の付け替え？ 府省庁内の再配分？ 支出先への影響波及まで見るか（支出構造は予算比例で縮尺する仮定を置くのか）
2. 表示先 — エージェントのレポート内（テキスト+数値比較）で足りるか、サンキー図として描画したいか
3. 仮想データの明示方法 — 実績との混同を防ぐUI/API上の区別

### 推奨の順序

まず 1.（自律探索レポート）の実験でエージェントの探索品質とAPIの過不足を確認し、その中で「シミュレーションの入力として何が要るか」を観察してから 2. の設計に入る。2. はエージェント側の計算（サイト無改修）で試作 → 価値が確認できたらサイト側の描画対応を検討、の2段階が安全。
