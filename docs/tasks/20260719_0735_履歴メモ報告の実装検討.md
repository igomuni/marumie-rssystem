# 履歴・発見メモ・データ報告 — 構想から今すぐ実現できる部分の掘り下げ

作成日: 2026-07-19（JST）

アイデンティティ段階論）のうち、**匿名・local-first・現行インフラのみ**で実現できる
範囲を実装可能な粒度に落とす。テレメトリ（20260719_0623）は発動待ちのまま触らない。

## 0. 選定の基準と結論

基準: (a) アカウント不要・プライバシー文言を変えない、(b) 外部サービス・課金なし、
(c) ユーザー価値が主目的でシグナルは副産物、(d) URL=状態 の資産を活かす。

| 候補 | 判定 | 理由 |
|---|---|---|
| A. 探索履歴（ローカル） | **実装A（第1弾）** | 純クライアント・URL=状態で激安・毎回の利用で効く |
| B. 発見メモ（ローカル保存・URL共有） | **実装A に同梱** | Aと同じUI面・同じストア。差分が小さい |
| C. データ報告（Issue 起票） | **実装B（第2弾・lite から）** | 価値は最大級だがスパム面の考慮が要る。まず0コスト版 |
| D. 事業別人気・共起レコメンド | 見送り | テレメトリ蓄積が前提（発動待ちに従属） |

## 1. 実装A: 探索履歴 + 発見メモ（純クライアント・1PR規模）

### ユーザー価値

- 「さっき見たあの図に戻る」— 現状、戻る手段はブラウザ履歴の連打か URL の手動管理のみ
- 「発見をメモして保存・共有」— 図の状態（URL）+ 一言を残す。共有はURLコピーで完結

### データモデル（IndexedDB。api-key-store と同じ流儀で新ストア）

```ts
interface ExplorationEntry {
  id: string;            // crypto.randomUUID()
  qs: string;            // URLクエリ文字列（= 完全な再現状態）
  label: string;         // 自動要約（下記）
  note?: string;         // ユーザーのメモ（あれば「メモ」扱い）
  pinned: boolean;       // true=保存（メモ/ピン）。false=自動履歴
  ts: number;            // 最終訪問時刻
  year: string;
}
```

- **自動履歴**: URL 状態が確定的に変わったとき（popstate/pushState 経路の適用時）に
  debounce して追記。同一 qs は ts 更新のみ。自動履歴は上限50件でFIFO
- **メモ/ピン**: ユーザー操作で `pinned=true` + note 入力。上限なし（実質的に小さい）
- **label の自動生成**: SankeyQuery から「経済産業省・予算100億以上・393事業」の形式で
  合成（resolveSankeyQuery 済みの値 + 適用時の件数を利用。Pure 関数で単体テスト可能）

### UI

- 既存の AI パネルと対称の**左側スライドパネル**…は過剰。まず軽く:
  ヘッダ（年度ピル付近）に「履歴」ボタン → ドロップダウンリスト
  （自動履歴とメモをタブ or セクション分け。クリックで pushState 適用 =
  AIチャット結果適用と同じ復元経路を再利用）
- 現在の図を「メモとして保存」するボタン + 一言入力
- 各行: ラベル・相対時刻・（メモなら）note、削除、URLコピー
- 文言: 「履歴・メモはこのブラウザにのみ保存されます」（BYOK と同じ流儀）

### 実装メモ

- 保存・復元は `sankeyQueryFromUrlParams` / `sankeyQueryToUrlParams` と
  `applyUrlState`（AIチャット結果適用と同じ）を再利用。新規ロジックはほぼストアとUIのみ
- レイヤー: ストア `client/lib/exploration-store.ts`、ラベル合成は Pure で
  `app/lib/`（vitest 対象）、UI は `client/components/SankeySvg/`
- 規模感: 半日〜1日・1PR

## 2. 実装B: データ報告 — Supabase 受付方式（2026-07-19 改訂）

> 当初案の「GitHub Issue テンプレートへの deep link（lite）」は**却下**（2026-07-19）:
> リンク先にリポジトリ名（marumie-rssystem）が露出し、「サイトUIに開発名を出さない・
> 公開版は別名」の方針と衝突するため。報告の受付は Supabase（無料枠・カード不要）で行う。

### 構成: ブラウザ → `/api/feedback`（自サイト）→ Supabase

ブラウザから Supabase 直接（anon key + RLS）ではなく**サーバ経由**を採る理由:

1. **CSP を緩めない**: `connect-src` は `'self' + openrouter.ai` のまま（PR #269 の防御を維持）
2. **スパム対策をサーバで握れる**: 文字数上限・honeypot・簡易レートリミットを
   `/api/feedback` で実施。荒らし時は env でルートごと無効化できる
3. Supabase の存在をユーザーから隠せる（将来の載せ替え自由・URLも露出しない）

### データモデル（reports テーブル）

```sql
create table reports (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz default now(),
  kind text not null check (kind in ('data', 'bug', 'other')),
  year text,
  pid text,
  url text,          -- 報告時のクエリ文字列（=再現状態）。自動添付
  message text not null check (char_length(message) <= 2000),
  status text not null default 'new'  -- new / triaged / done
);
```

- 個人情報・連絡先は**収集しない**（匿名報告。UI にも「個人情報を書かないでください」を明記）
- 書き込みはサーバ（service key・env）のみ。anon からの読み書きは RLS で全面拒否

### 運用（トリアージ面）

- **管理画面は作らず Supabase ダッシュボード（Table Editor）をそのまま使う**
- 将来の接続: 分析エージェントが reports を読み（ローカル・service key）、要約・
  クラスタリングして GitHub Issue を**内部で**起票する — リポジトリ名を一切露出せずに
  構想（20260719_0653 3節）の自動ループへ接続できる
- 発動待ちテレメトリ（20260719_0623）の蓄積先候補も Upstash から **Supabase に統一可能**
  （Postgres でカウンタも張れる）。発動時に再評価

### 無料枠の注意（正直な制約）

- DB 500MB・API 無制限ではないが報告用途には十分
- **無料プロジェクトは7日間非アクティブで一時停止**され、復帰まで insert が失敗する。
  対策: 週次の keep-alive ping（GitHub Actions cron で軽い select）を同時に仕込む。
  それでも停止した場合に備え、`/api/feedback` は失敗時に「時間をおいて再度」の
  ユーザー向け文言を返す（報告が黙って消えることはない設計にする）

### UI 導線

- サイドパネル・subcontracts 詳細に「データについて報告」ボタン → 小さなフォーム
  （種別・本文のみ。URL・pid・年度は自動添付を明示表示）→ 送信 → 受付完了表示
