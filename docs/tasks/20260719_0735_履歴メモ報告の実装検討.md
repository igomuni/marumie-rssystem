## 0. 選定の基準と結論

基準: (a) アカウント不要・プライバシー文言を変えない、(b) 外部サービス・課金なし、
(c) ユーザー価値が主目的でシグナルは副産物、(d) URL=状態 の資産を活かす。

| 候補 | 判定 | 理由 |
|---|---|---|
| A. 探索履歴（ローカル） | **実装A（第1弾）** | 純クライアント・URL=状態で激安・毎回の利用で効く |
| B. 発見メモ（ローカル保存・URL共有） | **実装A に同梱** | Aと同じUI面・同じストア。差分が小さい |
| C. データ報告（Issue 起票） | **実装B（第2弾・lite から）** | 価値は最大級だがスパム面の考慮が要る。まず0コスト版 |
| D. 事業別人気・共起レコメンド | 見送り | テレメトリ蓄積が前提（発動待ちに従属） |

## 1. 実装A: 探索履歴 + 発見メモ（純クライアント・1PR規模）

### ユーザー価値

- 「さっき見たあの図に戻る」— 現状、戻る手段はブラウザ履歴の連打か URL の手動管理のみ
- 「発見をメモして保存・共有」— 図の状態（URL）+ 一言を残す。共有はURLコピーで完結

### データモデル（IndexedDB。api-key-store と同じ流儀で新ストア）

```ts
interface ExplorationEntry {
  id: string;            // crypto.randomUUID()
  qs: string;            // URLクエリ文字列（= 完全な再現状態）
  label: string;         // 自動要約（下記）
  note?: string;         // ユーザーのメモ（あれば「メモ」扱い）
  pinned: boolean;       // true=保存（メモ/ピン）。false=自動履歴
  ts: number;            // 最終訪問時刻
  year: string;
}
```

- **自動履歴**: URL 状態が確定的に変わったとき（popstate/pushState 経路の適用時）に
  debounce して追記。同一 qs は ts 更新のみ。自動履歴は上限50件でFIFO
- **メモ/ピン**: ユーザー操作で `pinned=true` + note 入力。上限なし（実質的に小さい）
- **label の自動生成**: SankeyQuery から「経済産業省・予算100億以上・393事業」の形式で
  合成（resolveSankeyQuery 済みの値 + 適用時の件数を利用。Pure 関数で単体テスト可能）

### UI

- 既存の AI パネルと対称の**左側スライドパネル**…は過剰。まず軽く:
  ヘッダ（年度ピル付近）に「履歴」ボタン → ドロップダウンリスト
  （自動履歴とメモをタブ or セクション分け。クリックで pushState 適用 =
  AIチャット結果適用と同じ復元経路を再利用）
- 現在の図を「メモとして保存」するボタン + 一言入力
- 各行: ラベル・相対時刻・（メモなら）note、削除、URLコピー
- 文言: 「履歴・メモはこのブラウザにのみ保存されます」（BYOK と同じ流儀）

### 実装メモ

- 保存・復元は `sankeyQueryFromUrlParams` / `sankeyQueryToUrlParams` と
  `applyUrlState`（AIチャット結果適用と同じ）を再利用。新規ロジックはほぼストアとUIのみ
- レイヤー: ストア `client/lib/exploration-store.ts`、ラベル合成は Pure で
  `app/lib/`（vitest 対象）、UI は `client/components/SankeySvg/`
- 規模感: 半日〜1日・1PR

## 2. 実装B: データ報告（lite → 本実装の2段階）

### lite（第2弾・ほぼ0コスト・サーバ変更なし）

- サイドパネル・詳細ダイアログに「この事業のデータについて報告」リンクを置き、
  **GitHub Issue テンプレートへの deep link**（`.github/ISSUE_TEMPLATE/` +
  `?title=&body=` プリフィル）で飛ばす。body に現在の URL・pid・年度を自動埋め込み
- 利点: スパム対策・保存基盤・API すべて GitHub に委譲。実装は数時間
- 制約: GitHub アカウントが要る（報告者層を狭めるが、公共データの指摘者層とは重なる）

### 本実装(将来・needs 判断)

- `/api/feedback` で匿名受付 → GitHub API で Issue 自動起票（トークンは env）
- 必須になる考慮: レートリミット・honeypot 等のスパム対策、荒し時の停止手段。
  **lite で報告が実際に来るか観測してから**投資判断する（利用分析の使い方として自己整合）
