# sankey-svg-next2 タッチ優先レスポンシブ修正計画

- 対象: `/sankey-svg-next2`
- 元実装: `/sankey-svg`
- 関連 Issue: [#195](https://github.com/igomuni/marumie-rssystem/issues/195)
- 関連 PR: [#219](https://github.com/igomuni/marumie-rssystem/pull/219)
- 方針: 既存 `/sankey-svg-next` は実験ブランチとして扱い、後ほど削除する。新しい検証ルート `/sankey-svg-next2` を `/sankey-svg` から作り直す。

## 背景

これまでの `/sankey-svg-next` では、レスポンシブトークン、スマホ専用 world、固定キャンバス、地図型 camera、サイドパネル変形などをまとめて扱った。その結果、検証すべき論点が多くなり、どの変更が効いているのか判断しづらくなった。

次のアプローチでは、既存本番 `/sankey-svg` を母体にして、最小変更から段階的に積み上げる。

1. まずタッチデバイスで既存 Sankey を zoom / pan できるようにする
2. 次にスマホで邪魔になる設定項目を整理する
3. その後、狭幅時のサイドパネルを bottom-sheet / fullscreen に切り替える
4. 最後に `/sankey-svg` へ戻すか、`/sankey-svg-next2` を本番置換する

## 基本方針

### やること

- `/sankey-svg` をコピーして `/sankey-svg-next2` を作る
- 最初の差分はタッチ zoom / pan のみ
- 既存の desktop レイアウト、fontScale、TopN、ミニマップ、サイドパネルは維持する
- スマホでも最初は同じ図面を使い、画面幅に合わせた大きな再レイアウトはしない
- タッチ操作は既存の `zoom` / `pan` state に接続する
- オーバーレイ、検索、設定、サイドパネルのスクロールを壊さない

### やらないこと

- 最初から DisplayMode / LayoutTokens を導入しない
- 最初から固定 world size を作らない
- 最初からスマホ専用フォントや列間隔を再設計しない
- 最初から bottom-sheet を作らない
- `/sankey-svg` 本体を直接変更しない

## `/sankey-svg-next` との関係

`/sankey-svg-next` は、これまでのレスポンシブ実験と mobile map 実験を含む作業場として残す。ただし本線にはしない。

扱い:

- 新規実装は `/sankey-svg-next2` に入れる
- `/sankey-svg-next` のコードは参照してよいが、コピーしない
- PR #219 の pointer event 知見は参考にする
- `/sankey-svg-next2` の方向が固まった段階で `/sankey-svg-next` は削除候補にする

## 現状 `/sankey-svg` の操作系

`app/sankey-svg/page.tsx` の既存操作は以下。

- `handleWheel`: wheel zoom / wheel pan
- `getZoomAnchoredPanY`: Y 軸アンカー付き zoom
- `handleMouseDown` / `handleMouseMove` / `handleMouseUp`: mouse drag pan
- `isOverlayControlTarget`: `[data-pan-disabled], button, input, select, textarea, label` 上の操作を canvas pan/zoom から除外
- `didPanRef`: pan と click 選択の誤判定防止

最初の実装では、この既存構造に Pointer Events を足す。既存 mouse / wheel はそのまま残す。

## Phase 0: next2 ルート作成

目的: 本番 `/sankey-svg` と `/sankey-svg-next` のどちらも壊さず、検証用ルートを作る。

作業:

- `app/sankey-svg/page.tsx` を `app/sankey-svg-next2/page.tsx` にコピー
- ページ内の loading 文言など、必要最低限だけ `next2` に変更
- まずは `/sankey-svg` と `/sankey-svg-next2` が同じ見た目で表示されることを確認

検証:

- `/sankey-svg-next2` が HTTP 200
- 1440x900 で `/sankey-svg` と見た目が大きく変わらない
- `npx tsc --noEmit`

コミット例:

```text
feat(sankey-svg-next2): copy sankey-svg baseline route
```

## Phase 1: タッチ zoom / pan の最小実装

目的: スマホで既存 Sankey を触って動かせる状態にする。

### 追加する state / refs

`page.tsx` の既存 pan/zoom 周辺に追加する。

```ts
const activePointersRef = useRef<Map<number, { x: number; y: number }>>(new Map());
const panRef = useRef(pan);
const zoomRef = useRef(zoom);
const touchPanStartRef = useRef<{ x: number; y: number } | null>(null);
const touchPanOriginRef = useRef<{ x: number; y: number } | null>(null);
const pinchLastDistanceRef = useRef<number | null>(null);
```

`panRef` / `zoomRef` は `pointermove` 中の stale closure 対策として同期する。

```ts
useEffect(() => { panRef.current = pan; }, [pan]);
useEffect(() => { zoomRef.current = zoom; }, [zoom]);
```

### Pointer Events

root container ではなく、Sankey SVG 操作面に限定して pointer handlers を付ける。

```tsx
<svg
  ...
  onPointerDown={handleTouchPointerDown}
  onPointerMove={handleTouchPointerMove}
  onPointerUp={handleTouchPointerEnd}
  onPointerCancel={handleTouchPointerEnd}
  style={{ ..., touchAction: 'none' }}
>
```

`pointerType === 'mouse'` は早期 return し、desktop mouse は既存 `onMouse*` に任せる。

### 1本指 pan

- pointerdown で `touchPanStartRef` と `touchPanOriginRef` を保存
- pointermove で `setPan({ x: origin.x + dx, y: origin.y + dy })`
- `TOUCH_PAN_SLOP_PX = 10` 程度を超えたら `didPanRef.current = true`
- pointerup で実際に pan していた場合だけ hover 抑制 cooldown

### 2本指 pinch zoom

既存 `/sankey-svg` の横方向は screen-fixed に近い実装なので、最初は **Y 軸アンカー中心**で実装する。

- 2本指距離比で `nextZoom` を計算
- 2本指 midpoint の `y` を使って `getZoomAnchoredPanY(midY, nextZoom)`
- `pan.x` はまず維持する
- X 軸アンカーは Phase 1 では入れない

理由:

- `/sankey-svg` は横方向を画面に収める設計が既に入っている
- いきなり X/Y 両方を world camera 化すると `/sankey-svg-next` と同じ複雑さに戻る
- スマホの最初の問題は「触って縦方向に読めない、ピンチで文字を大きくできない」なので、Y zoom + pan だけでも価値がある

### touch-action

`touchAction: 'none'` は SVG にだけ付ける。

オーバーレイ・検索結果・設定・サイドパネルには `data-pan-disabled` を付け、touch gesture を拾わせない。

### Phase 1 の受け入れ条件

- iOS Safari / Android Chrome で 1本指 pan が動く
- 2本指 pinch で文字とノードが拡大される
- tap 選択が pan と誤判定されにくい
- 検索結果や設定UIのスクロールが壊れない
- desktop mouse / wheel 挙動が変わらない

コミット例:

```text
feat(sankey-svg-next2): add touch pan and pinch zoom
```

## Phase 2: タッチ操作の品質調整

目的: タッチ zoom / pan の気持ち悪さを減らす。

候補:

- pinch 中は hover tooltip を出さない
- 2本指から1本指に戻ったときの pan 起点を取り直す
- double tap zoom は必要性を見て追加
- reset viewport ボタンをタッチで押しやすくする
- zoom slider はタッチでは邪魔なら折りたたむ

この Phase でもレイアウト設計には踏み込まない。あくまで既存画面をタッチで操作可能にする。

## Phase 3: 設定項目の選別

目的: スマホで常時表示する UI を減らす。

順番:

1. 既存 UI を分類する
   - 常時必要: 検索、年度、設定入口、全体表示
   - あればよい: TopN、フォント、ミニマップ、表示切替
   - スマホでは隠す候補: offset slider、scroll mode toggle、縦ズーム slider
2. `compact toolbar` を導入する
   - 上部は検索 + 年度 + 設定ボタン
   - 詳細設定は sheet / popover に逃がす
3. 44px hit area を確保する

この Phase で初めて「スマホ UI」を触る。

## Phase 4: 狭幅時のサイドパネル下配置

目的: スマホ縦で左 docked panel が図面を潰す問題を解消する。

方針:

- 幅が狭いときだけ side panel を bottom-sheet にする
- 最初は 2段階で十分
  - half: 画面下 45%
  - full: 画面下 90%
- 360px 級では fullscreen overlay も検討

注意:

- bottom-sheet の drag と Sankey pan が競合しやすい
- sheet drag は handle 部分だけ
- sheet 本文は scroll
- 背面 SVG は pan

## Phase 5: 表示密度の調整

目的: タッチ操作だけでは解決できない文字サイズ・列見出し・ラベル密度を調整する。

ただしこの Phase でも、最初から別 world / 別 layout にはしない。

候補:

- スマホ時だけ初期 zoom を少し大きめにする
- column header を `name-only` にする
- mini map を初期非表示にする
- フォント設定を sheet 内に入れる
- TopN の初期値を狭幅用に変えるか検討

## Phase 6: next2 採用判断と整理

判断基準:

- `/sankey-svg-next2` がタッチ操作で自然に使える
- desktop 回帰がない
- スマホで検索・年度・設定・選択詳細に到達できる
- bottom-sheet が操作競合しない

採用する場合:

- `/sankey-svg` を `/sankey-svg-next2` の内容で置換
- `/sankey-svg-next` を削除
- `/sankey-svg-next2` も削除
- e2e を `/sankey-svg` に統合

採用しない場合:

- `/sankey-svg-next2` は実験ルートとして閉じる
- `/sankey-svg` には Phase 1 の touch zoom/pan だけを移植するか再判断

## 実装上の注意

- データ単位は 1円単位のまま触らない
- 「その他」と「その他の支出先」は触らない
- `app/lib/` の `computeLayout` や `filterTopN` は Phase 1〜4 では変更しない
- `client/components/` に切り出すのは、UI が固まってからでよい
- 最初は `page.tsx` 内の小さな pointer handler 追加でよい
- PR はユーザー明示指示があるまで作成しない

## 推奨する最初の PR 範囲

最初の PR は Phase 0 + Phase 1 までに絞る。

含める:

- `/sankey-svg-next2` 作成
- touch 1本指 pan
- touch 2本指 pinch zoom
- `touchAction: none` の範囲整理
- desktop 回帰確認

含めない:

- 設定項目の再編
- bottom-sheet
- DisplayMode / LayoutTokens
- fixed world / mobile camera
- `/sankey-svg-next` 削除

この順序なら、まず「既存 Sankey をスマホで触れるか」という一番大事な仮説だけを小さく検証できる。
