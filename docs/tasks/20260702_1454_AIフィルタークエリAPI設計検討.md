## 目的

**AIエージェント（とその先の記者・研究者・市民）が、「再エネ関連で予算100億円以上の事業だけ見たい」のような自然言語の要求から、フィルタパネルを手動操作せずに目的のサンキー図へ到達できない状態を解消するため。**

現状、`/sankey-svg` のフィルタ機能（事業名・支出先名の文字列/正規表現、金額範囲、府省庁、会計区分）は人間がパネルを操作する前提で作られている。AIが同じ絞り込みを代行するには URL パラメータを組み立てるしかないが、後述の通り AI には仕様を知る術がなく、結果を検証する術もない。これを「フィルター条件を API でクエリできる」形に拡張し、AI がユーザーの要求を解釈 → 条件を組み立て → 実行・検証 → 完成したサンキー図ビューを人間に返す、という自律ループを成立させる。

## 現状整理

### フィルター処理は既に2段構成になっている

| 段 | 場所 | 内容 |
|----|------|------|
| ① プレフィルタ | `app/sankey-svg/page.tsx` の `filterExcludedIds` useMemo（page.tsx:1206 付近、約110行） | 名称（部分一致/正規表現）・予算/支出金額範囲・府省庁・会計区分で除外ノード集合を構築。孤立事業・孤立省庁のカスケード除外（Pass 2/3）も実施 |
| ② TopN集約 | `app/lib/sankey-svg-filter.ts` の `filterTopN`（Pure関数） | TopN 選抜・「その他の支出先」等の集約・ピン/フォーカスモード・オフセットウィンドウ |

①がクライアント（page.tsx）専用実装である点が本件の最大のギャップ。②は既に `app/lib/` の Pure 関数でありサーバーからも呼べる。

### フィルター状態はすべて URL に載っている

`parseSearchParams`（page.tsx:147）が既に全状態を URL と相互変換している。主要パラメータ:

| 区分 | パラメータ |
|------|-----------|
| プレフィルタ | `fnp`/`fnpr`（事業名/正規表現）、`fnr`/`fnrr`（支出先名/正規表現）、`fm`（府省庁・複数可）、`fmb`/`fxb`（予算下限/上限）、`fms`/`fxs`（支出下限/上限）、`ac`（会計区分） |
| ビュー | `tm`/`tp`/`tr`（TopN）、`ro`/`po`/`ot`（オフセット）、`pp`/`pr`/`pm`（ピン）、`fr`（フォーカス）、`ps`（ソート）、`ar`/`ap`（集約表示）、`sb`（予算スケーリング）、`yr`（年度） |

つまり「フィルタ条件を与えれば図が再現される」性質は既にあり、**欠けているのは (a) AI が読める仕様、(b) 機械可読な形式（短縮キーの羅列は AI が生成・検証しにくい）、(c) 実行結果を図を開かずに検証する手段**の3点。

### AIフレンドリー基盤の実装状況

- 実装済み: `/api/search/projects`、`/api/search/recipients`、`/api/recipients/[key]`、API共通規約（`metadata.notes`、`app/lib/api/links.ts`）、`app/robots.ts`、`app/sitemap.ts`
- 未実装: `public/llms.txt`（SG-1 の一部）— AI がサイト仕様を知る入口が現状ない
- グラフデータ: `public/data/sankey-svg-{year}-graph.json`（2024年度 約13.5MB、5,664事業 + 12,521支出先ノード）。クライアントが全量読み込みする方式で、サーバー側にも同一ファイルがある（既存 loader 群と同様のメモリキャッシュが可能なサイズ）

## 課題（なぜ URL 生成だけでは足りないか）

AI がサンキー図をリメイクする手段は現状「URL を組み立てて人間に渡す」のみ。これには次の問題がある。

1. **仕様の非公開**: URL パラメータは短縮形（`fnp` 等）で、ドキュメント化されていない。AI はソースコードを読まない限り生成できない
2. **検証不能**: URL は開くまで結果が分からない。「マッチ0件」「絞り込みが緩すぎて図が変わらない」を AI が検知できず、条件の反復調整（自律的な試行錯誤）が成立しない
3. **再現不能**: プレフィルタが page.tsx 内にあるため、サーバー側で「この条件だと何件・何円になるか」を同じロジックで計算できない

## 設計方針

核となる考え方: **「フィルタークエリ」を第一級の構造化オブジェクト（`SankeyQuery`）として定義し、(1) UI フィルタパネル状態、(2) URL パラメータ、(3) クエリ API の3者が同一の `SankeyQuery` を共有する。**

これにより「AI が API で試行錯誤したクエリ」と「人間がブラウザで見る図」が常に同じ結果になることを構造的に保証する（別実装の二重管理を作らない）。

```text
自然言語の要求
   │ AIが解釈（外部エージェント: Claude 等）
   ▼
SankeyQuery(JSON) ──→ GET /api/sankey/query ──→ summary（件数・金額）で検証
   │                        │                        │ 絞り込み過不足なら
   │                        │                        └─ クエリ修正して再試行
   │                        ▼
   │                  links.webView（/sankey-svg?fnp=...&fmb=...）
   ▼                        ▼
 人間がURLを開く ──→ 同じ SankeyQuery が復元され、同じ図が描画される
```

### SankeyQuery スキーマ（案）

`types/sankey-query.ts` に新設。現行フィルタパネルの表現力と1対1対応から始める（拡張は未決事項参照）。

```json
{
  "year": "2024",
  "filter": {
    "projectName": { "query": "再生可能エネルギー|再エネ", "regex": true },
    "recipientName": { "query": "電力", "regex": false },
    "ministries": ["経済産業省", "環境省"],
    "budget": { "min": 10000000000, "max": null },
    "spending": { "min": null, "max": null },
    "accountCategories": ["general", "special", "both", "none"]
  },
  "view": {
    "topMinistry": 5, "topProject": 10, "topRecipient": 20,
    "pin": { "projectId": null, "recipientId": null, "ministryName": null },
    "focusRelated": false,
    "offset": { "target": "recipient", "recipient": 0, "project": 0 },
    "projectSortBy": "budget",
    "showAggProject": true, "showAggRecipient": true,
    "scaleBudgetToVisible": true
  }
}
```

- 金額は既存規約通り **1円単位**（`fmb` 等の「億円テキスト」との換算は変換層が担う）
- `filter` と `view` を分離する: `filter` は「どのノードを残すか」（プレフィルタ①）、`view` は「どう見せるか」（TopN集約②）。AI は通常 `filter` を組み立て、`view` は既定値で足りる
- 全フィールド optional。省略時は現行 UI の既定値と一致させる

### コンポーネント構成（レイヤー準拠）

| Layer | ファイル | 役割 |
|-------|---------|------|
| Types | `types/sankey-query.ts`（新規） | `SankeyQuery` 型定義 |
| Domain | `app/lib/sankey-query.ts`（新規） | (a) クエリの検証・正規化・既定値補完、(b) `buildFilterExcludedIds(nodes, edges, filter)` — page.tsx の `filterExcludedIds` ロジックを **Pure 関数として移設**（Pass 1〜3 の挙動を維持）、(c) `SankeyQuery ⇔ URLSearchParams` の相互変換（`parseSearchParams` の該当部分と1対1） |
| Domain | `app/lib/api/sankey-graph-loader.ts`（新規） | グラフ JSON のサーバー側読み込み + メモリキャッシュ（既存 `quality-scores-loader.ts` と同格。fs は使うが HTTP/React なし） |
| API | `app/api/sankey/query/route.ts`（新規） | パラメータ受領 → 検証 → loader → `buildFilterExcludedIds` → `filterTopN` → 応答整形。ロジックは持たない |
| Pages | `app/sankey-svg/page.tsx` | `filterExcludedIds` useMemo の中身を `app/lib/sankey-query.ts` の Pure 関数呼び出しに置換（**挙動不変のリファクタ**）。URL 変換も共通関数に寄せる |

`filterTopN` 本体は変更しない（SG-6 検討時の方針を踏襲: 既に18引数・900行超のため手を入れない。クエリ API は既存シグネチャのまま呼ぶ）。

### API 仕様（案）

```text
GET /api/sankey/query?year=2024&q={URLエンコードしたSankeyQuery JSON}
GET /api/sankey/query?year=2024&fnp=再エネ&fmb=100&...   # 既存URLパラメータ形式も受理
```

- `q`（JSON）と個別パラメータ（既存短縮形）の両対応とする。JSON は AI が生成・自己検証しやすく、短縮形は `/sankey-svg` の URL をそのまま貼り替えて検証できる（デバッグ・ディープリンク互換）
- 出力モードを `detail` パラメータで制御:

| モード | 内容 | 想定用途 |
|--------|------|---------|
| `summary`（既定） | metadata のみ。マッチ事業数・支出先数・府省庁数、予算/支出合計、上位N件の名前と金額（事業・支出先各10件程度） | AI の条件反復調整。数KB で応答 |
| `full` | summary + `filterTopN` 適用後の `nodes`/`edges`（= 図に描画される内容そのもの） | AI が数値を引用・二次加工する場合 |

- 応答共通部:
  - `metadata.appliedQuery`: 正規化後の `SankeyQuery` をエコーバック（省略時既定値が何になったかを AI が確認できる）
  - `metadata.notes`: 既存 API 共通規約に従いデータ留意事項を同梱（単位1円、「その他」vs「その他の支出先」、予算年度=事業年度-1、グラフの支出先は名寄せ集約済みで全26,823件より少ない、等）
  - `links.webView`: 同一クエリを開く `/sankey-svg` の完全 URL（**AI→人間への受け渡しの核**。相互変換関数が保証する）
  - `links.docs`: クエリ仕様ドキュメント（llms.txt 内 or 専用ページ）
- `Cache-Control: public, s-maxage=86400, stale-while-revalidate=604800`（既存共通規約。データはデプロイ時のみ更新）
- エラー応答: 不正な正規表現・範囲逆転（min>max）・未知フィールドは 400 で「何が不正か」を機械可読に返す（AI が自己修正できる文言にする）

### AI 側の利用シナリオ（外部エージェント想定）

1. AI が `llms.txt`（本件で整備、後述）からクエリ API の存在とスキーマを知る
2. ユーザーの要求を `SankeyQuery` に翻訳し、`detail=summary` で実行
3. 応答の件数・金額を見て条件を調整（0件 → 正規表現を緩める、多すぎ → 金額下限を足す、等）。必要に応じ `/api/search/projects` 等の既存 API で語彙を探索
4. 確定したら `links.webView` の URL をユーザーに提示 →「AI がリメイクしたサンキー図」が開く

サイト内蔵のチャット UI（in-app assistant）を作る場合も、この API をそのまま消費する構図になる（実施可否は未決事項）。

### セキュリティ・性能上の論点

- **正規表現のサーバー実行（ReDoS）**: 現行の正規表現フィルタはユーザー自身のブラウザで実行されるため被害が本人に閉じるが、API 化するとサーバーの攻撃面になる。対策案:
  - パターン長上限（例: 128字）+ `try/catch` コンパイル（最低限、初版で必須）
  - 対象文字列が短い（事業名・支出先名）ため災害級の破綻は起きにくいが、量指定子のネスト等の危険パターン検出、または RE2 系ライブラリ採用は選択肢（依存追加のトレードオフあり）→ 未決事項
- **応答サイズ**: `detail=full` は絞り込み条件次第で大きくなり得る。`filterTopN` 適用後（TopN 集約済み）を返すため実際は表示ノード数に比例し、既定 TopN なら数十KB オーダー。`topProject`/`topRecipient` の上限は UI と同じ 300 を強制
- **サーバーメモリ**: グラフ JSON は年度あたり約13.5MB。既存 loader 群と同様のプロセス内キャッシュで許容範囲（2年度で約27MB）
- **レートリミット**: 既存検討（20260612 DoS 設計）の枠組みに従う。本 API 固有の追加はしない前提だが公開時に再確認 → 未決事項

### llms.txt / ドキュメント整備（本件のスコープ内）

AI がクエリ仕様を知る入口がないと本機能は成立しないため、以下を本件に含める:

- `docs/api-guide.md` に `/api/sankey/query` 仕様を追記（正を api-guide に置く既存方針を踏襲）。なお api-guide.md には現存しないルート（旧 `/api/sankey`、`/api/entities` 等）の記載が残っており、本件追記時に実在ルートと突き合わせて整理する
- `public/llms.txt` の新規設置（SG-1 で未実装のまま残っている項目。本件はこれが前提になるため同時に整備する）— サイト概要・API一覧・`SankeyQuery` スキーマ・データ留意事項・URL パラメータ対応表

## 段階分割（実装順序）

| 段階 | 内容 | 単体での価値 |
|------|------|-------------|
| Step 1 | プレフィルタの `app/lib/` への移設（page.tsx リファクタ、挙動不変）+ `SankeyQuery` 型と URL 相互変換の共通化 | ロジックの一元化。page.tsx の縮小 |
| Step 2 | `/api/sankey/query`（summary モードのみ）+ graph loader | AI の条件検証ループが成立 |
| Step 3 | `detail=full` + `links.webView` + api-guide/llms.txt 整備 | 自律リメイクのフルシナリオが成立 |

Step 1 は既存挙動のスナップショット（代表的なフィルタ URL 数パターンの描画結果）で回帰確認する。

## レイヤー設計ルールとの整合チェック

| ルール | 本設計での扱い |
|--------|---------------|
| `scripts/` は CSV 処理のみ | 本件は scripts に触れない（新規データ生成なし。既存グラフ JSON をそのまま使う） |
| `app/lib/` は Pure・HTTP/React 禁止 | `sankey-query.ts` は Pure 関数のみ。`sankey-graph-loader.ts` は fs+キャッシュで既存 loader 群（`quality-scores-loader.ts` 等）と同格。`NextResponse` 等は不使用 |
| `app/api/` はハンドラのみ | route.ts は検証→委譲→整形のみ。除外集合構築・TopN は `app/lib/` |
| `client/components/` は直接 API コール禁止 | 本件でコンポーネントは変更しない（page.tsx の内部リファクタのみ） |
| Multi-block は filter().reduce() で合算 | プレフィルタ移設はノード単位の除外判定であり集計を伴わないが、summary の金額集計実装時はこの規約に従う |
| 金額は1円単位 | `SankeyQuery` の金額フィールドは1円単位。「億円」換算は UI テキスト入力との変換層のみが持つ |

## 未決事項（ユーザー判断が必要）

1. **AI の実行主体**: 初版は外部エージェント向け API 公開（+llms.txt）で完結させる想定でよいか。サイト内蔵チャット UI（LLM 呼び出しをサーバーが持つ形）まで視野に入れるなら、API キー管理・コスト・レートリミットの設計が別途必要になる
2. **クエリの表現力**: 初版は現行フィルタパネルと1対1（AND 条件のみ）で開始する想定。事業 ID 直接指定・複数条件の OR・bureau（局庁）指定などの拡張を初版に含めるか
3. **正規表現のサーバー実行ポリシー**: 長さ上限+try/catch の最小対策で開始するか、RE2 系ライブラリを導入するか
4. **レートリミット**: 既存 DoS 設計の枠内で足りるか、本 API の公開と同時に Vercel WAF 等の設定を入れるか
5. **summary の上位 N 件の粒度**: AI の反復調整に必要十分な情報量（件数だけで足りるか、上位事業・支出先の名前まで返すか）
