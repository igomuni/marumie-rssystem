# 悪意あるユーザー対策（DoS）設計

作成日: 2026-06-12（JST）
対象: 公開 API（`/api/search/*`, `/api/recipients/[key]` 等）・静的データ（`/data/*.json(.gz)`）
前提記録: `docs/tasks/20260611_2038_APIセキュリティ基準調査_動作確認.md`（F-1〜F-5）

---

## 0. 前提とスコープ

- 本サイトのデータは**公開オープンデータ**（行政事業レビューシステム由来）。守るべきは「秘匿」ではなく **リソース乱用（DoS）／コスト爆発** である。
- `robots.txt` は**アクセス制御ではない**。善良なクローラのマナー表示にすぎず、悪意あるユーザー・行儀の悪いボットは無視する。したがって防御は別レイヤ（レート制限・WAF・キャッシュ）で組む。
- 観測事実:
  - ブラウザから `/data/sankey-svg-2025-graph.json`（生JSON）が直接取得可能（robots は人間/攻撃者をブロックしない）。
  - ChatGPT クローラは `robots.txt` の `Disallow: /api/` を尊重して API を取りに来ない（＝AI 連携は MCP 等の別経路で行う方針と整合）。

## 1. 脅威モデル（守る対象と攻撃面）

| ID | 攻撃面 | 想定攻撃 | 影響 |
|---|---|---|---|
| T-1 | 検索 API（`/api/search/*`） | クエリを毎回変える cache-busting で O(N) 全件スキャンを連打 | サーバレス関数の CPU/実行時間消費＝コスト増・レイテンシ悪化（**主DoS面 = F-3**） |
| T-2 | 生JSON 直リンク（`/data/*.json`） | 最大29MBファイルの並列大量DL | 帯域・転送量コスト爆発 |
| T-3 | `.gz` スナップショット（`/data/*.json.gz`） | 同上（圧縮済みだが ~11MB×複数年度） | 転送量コスト |
| T-4 | 逆引き/詳細 API（`/api/recipients/[key]` 等） | 大量の異なるキーで連打 | ファイル読込・JSON parse の繰り返し（ローダーのメモリ/CPU） |

### 現状の実装上の弱点（コード根拠）

- `app/lib/search/recipient-search.ts`: `Object.values(recipients)` を毎リクエスト全走査し、各 alias に `normalizeRecipientName`（NFKC + 複数 regex 置換）を実行。約 25,811 エントリ。
- `app/api/search/projects/route.ts` ほか: `Cache-Control: public, s-maxage=86400, stale-while-revalidate=604800`（`app/lib/api/api-notes.ts`）。**CDN キャッシュは同一クエリにのみ有効**で、cache-busting には無力。
- `middleware.ts` は**存在しない**（＝現状レート制限/Bot 判定の共通入口がない）。
- `vercel.json`: リージョン `hnd1` のみ。Firewall/WAF 設定は未記載。

## 2. 多層防御の設計（推奨構成）

```text
[Client/Bot]
   │
   ├─(L1) Vercel Firewall / Bot 対策 … 悪性UA・異常レート・既知ボットの遮断
   │
   ├─(L2) Edge Middleware (middleware.ts) … IP単位レート制限（トークンバケット）
   │                                          + クエリ最小長などの early reject
   │
   ├─(L3) CDN キャッシュ … s-maxage で同一クエリをエッジ応答（オリジン到達を減らす）
   │
   └─(L4) アプリ層 … 入力バリデーション + 転置インデックスで O(N)→O(log N)/O(1) 化
                      + レスポンスサイズのバウンド
```

### L1: Vercel Firewall / Bot 対策

- Vercel の **Firewall（WAF）/ Attack Challenge Mode / Bot 管理**で、異常トラフィック・既知の悪性ボットをエッジで遮断。
- `/data/*.json`（生JSON）への直アクセスにレート/チャレンジを課す、または**生JSON公開を廃止**して `.gz` か API/MCP 経由に限定（T-2 を根本から縮小）。

### L2: Edge Middleware によるレート制限（T-1/T-4 の主対策）

- `middleware.ts` を新設し、`/api/*` に **IP 単位トークンバケット**を適用。
- ストアは Vercel KV / Upstash Redis（Edge 互換）を想定。状態を持てない場合は Vercel Firewall の Rate Limiting ルールで代替。
- 基準値（調査記録より）: 検索1リクエスト ≈ 22〜37ms（ウォーム）。出発点の閾値案:
  - `/api/search/*`: 60 req/min/IP（バースト 20）
  - `/api/recipients/[key]` 等の単発取得系: 120 req/min/IP
  - 超過時は `429 Too Many Requests` + `Retry-After`。
- early reject: `q` の最小長（例: 2文字以上、normalize 後）を Middleware で先に弾き、全件スキャンに到達させない。

### L3: キャッシュ強化

- 検索 API は既に `s-maxage` 付き。**クエリ正規化キー**でキャッシュ命中率を上げる（前後空白除去・NFKC 正規化済みの `q` を canonical URL に正規化）。
- `.gz` 静的ファイルは長期 immutable キャッシュ（ファイル名にバージョン/年度が入るため安全）。

### L4: アプリ層の堅牢化（F-3 本丸 / F-4）

- **転置インデックス化**: 起動時（またはビルド時）に支出先名・事業名の n-gram or トークン転置インデックスを構築し、検索を O(N) 全走査から脱却。
  - ビルド時生成案: `scripts/` で `search-index-{year}.json(.gz)` を生成（パイプライン層の責務に合致。UI/APIロジックは持ち込まない）。
  - 実行時案: 初回リクエストでインデックスを遅延構築しモジュールスコープにメモ化（Lambda のウォーム間で再利用）。
- 正規化結果のメモ化（`normalizeRecipientName` を alias 単位でキャッシュ）。
- レスポンスサイズのバウンド（F-4）: `...entry` スプレッドで `aliases` 全量を返さず、検索結果では必要フィールドのみに射影。巨大エントリ（「国」等）での肥大を防ぐ。

## 3. 優先度と段階的着手

| 優先 | 施策 | 効果 | コスト | 備考 |
|---|---|---|---|---|
| **P1** | Vercel Firewall でレート制限ルール（L1/L2 簡易版） | 即効でT-1/T-4を緩和 | 低（設定のみ・コード不要） | まず最小で導入し挙動観測 |
| **P1** | 生JSON `/data/*.json` 直リンクの公開方針決定（廃止 or チャレンジ） | T-2 を根本縮小 | 低〜中 | UI が直参照していないか要確認 |
| **P2** | `middleware.ts` + KV でトークンバケット | きめ細かいレート制限 | 中 | 状態ストア（KV/Upstash）導入判断 |
| **P2** | 検索の転置インデックス化（F-3） | O(N) 解消・CPU 枯渇耐性 | 中〜高 | 性能と DoS 耐性の両取り |
| **P3** | レスポンス射影でサイズバウンド（F-4） | 肥大エントリ対策 | 低 | 検索結果の射影を見直し |
| **P3** | `requestId` をエラー応答に付与 | 運用時の突合 | 低 | F-2 の残課題 |

## 4. 検証方針

- 負荷試験: cache-busting クエリ（毎回ランダム `q`）で N req/s をかけ、`429` が想定閾値で返り、オリジン CPU 時間が頭打ちになることを確認。
- キャッシュ命中: 同一 `q` 連打で 2 回目以降がエッジ応答（`x-vercel-cache: HIT`）になることを確認。
- 機能非回帰: 既存のエッジケース（`year` ホワイトリスト・limit/offset クランプ・空クエリ 400・パストラバーサル拒否）が維持されること。
- `npx tsc --noEmit` / `npm run lint` / `npm run build`。

## 5. 留意点・非対象

- データは公開前提のため「秘匿」目的のアクセス制御（認証必須化）は基本不要。ただし**乱用抑止のための任意トークン制**（高レート利用者向け）は将来検討余地あり。
- robots.txt は現状維持（`Disallow: /api/`, `/data/*.json`）。これは善良クローラの帯域抑制が目的であり、DoS 対策とは別レイヤである旨を本書で明文化する。
- 本書は方針設計であり、実装は P1 から小さく着手し、観測値に基づいて閾値・転置インデックスの要否を判断する。

## 6. 付録: Vercel Firewall レート制限 設定手順（P1 実施手順）

ダッシュボード設定はコードに残らないため、運用手順としてここに記録する。
公式: <https://vercel.com/docs/security/vercel-waf/rate-limiting>（UIラベル・プラン要件は変わりうるため最終確認すること）

> 注意: **Rate Limiting アクションは有料プラン（Pro / Enterprise）機能**。Hobby プランでは利用不可の場合がある。

### 実装状況（アプリ層・既存）

- `middleware.ts`（Edge Middleware）で `/api/*` に IP 単位トークンバケットを実装済み（L2）。
  - `/api/search/*`: 60 req/min・burst 20 / その他 `/api/*`: 120 req/min・burst 40 / 超過は 429 + Retry-After。
  - ストアはインスタンス内メモリ（ベストエフォート）。分散環境での厳密化は KV 差し替え or 下記 Firewall に委譲。
- Firewall（L1）はエッジで関数到達前に遮断するため、関数課金・CPU を消費しない。**本命は Firewall、middleware はフォールバック**という位置づけ。

### 方法A: ダッシュボードのカスタムルール（推奨・コード不要）

1. Vercel ダッシュボード → 対象プロジェクト → **Firewall** タブ
2. **Configure** → **Custom Rules** → **Add Rule**
3. **Condition**: `Request Path` `starts with` `/api/search/`（重い検索API向け。別途 `/api/`・`/data/` 用ルールも）
4. **Action**: **Rate Limit** を選択
   - Requests / Window（例: 60 / 60s）、Keyed by: `IP Address`
   - 超過アクション: `Deny`（429）/ `Challenge`（Bot検証）/ `Log`（観測のみ）
   - Duration（ブロック継続）: 例 60s
5. **Save → Publish**（即時反映）

推奨初期値（本書の基準値と整合）:

```text
/api/search/*  : 60 req / 60s / per IP → Deny(429), block 60s
/api/*(その他) : 120 req / 60s / per IP → Deny(429), block 60s
/data/*.json   : 30 req / 60s / per IP → Challenge（生JSONの大量DL抑制。※アプリが直fetchするため Deny は不可）
```

> `/data/*.json` はメインページ等がブラウザから直接 fetch している（`app/sankey-svg/page.tsx` 等）。
> Deny にするとアプリが壊れるため、Challenge / 緩めの閾値に留めること。根本対策は別タスク（`.gz`+クライアント解凍への移行）。

### 方法B: `@vercel/firewall` の `rateLimit()`（コードから呼ぶ）

ダッシュボードで定義した Rate Limit ID をコードから参照する方式。条件分岐をコード側に持てるが、関数起動後の判定のため遮断効率は方法Aに劣る。

```ts
import { rateLimit } from '@vercel/firewall';

export async function GET(request: Request) {
  const { rateLimited } = await rateLimit('search-api');
  if (rateLimited) return new Response('Too many requests', { status: 429 });
  // ...通常処理
}
```

### 多層防御の関係

```text
[Bot/攻撃者]
   ├─(方法A) Vercel Firewall Rate Limit … エッジで関数到達前に遮断（最優先・コスト最小）
   ├─(既存)  middleware.ts のメモリ制限   … アプリ層ベストエフォート（無料・即時）
   └─ API 本体
```
