日時: 2026-06-11
前提ドキュメント:

## 目的

1. **AIエージェント（とその先の記者・研究者・市民）が、本サイトのデータを検索→詳細→再委託→関連企業と自走で深掘りできない状態を解消する**。現状は入口（llms.txt等）・検索API・支出先逆引きがすべて欠けており、数MB〜29MBの生JSONダウンロードでしか探索できない。
2. **再委託構造の見える化が事業単位ビューで頭打ちになっている状態を解消する**。視点を「事業中心」から「支出先（企業）中心」に反転させた新しいViewの土台を作り、府省庁横断の受注・再委託構造を見えるようにする。

## 実地調査で確認済みの前提

- `subcontracts-{YEAR}.json` には `flows: BlockEdge[]`（`sourceBlock → targetBlock`）が**明示的なエッジとして格納済み**（[types/subcontract.ts](../../types/subcontract.ts)）。委託グラフの復元に chain 文字列の解析は不要
- 支出先には法人番号（`corporateNumber`）が付与されている。2024年度実測: **支出先行 74,639行 / ユニーク法人番号 18,885 / 法人番号空欄 24.1%**（個人・海外・集約行など）
- 既存APIは5ルート。`/api/sankey/mof-overview` のみ `metadata.notes`（データ留意事項）を同梱しており、これを全APIの標準形にする
- `/api/quality-scores` は全5,003事業の pid・事業名・府省庁・予算・再委託有無を含む実質インデックス（メモリキャッシュ済み）

## SG-2: 事業検索API + API共通規約

### 新規エンドポイント

```
GET /api/search/projects?q=マイナンバー&year=2024&limit=20&offset=0
```

応答（数KB想定）:

```
{
  metadata: { year, query, totalHits, limit, offset, unit: "JPY", notes: [...], generatedAt },
  items: [{
    pid, name, ministry, bureau,
    budgetAmount, execAmount,
    hasRedelegation, redelegationDepth, totalScore,
    links: { detail: "/api/project-details/{pid}", subcontracts: "/api/subcontracts/{pid}", web: "/subcontracts/{pid}" }
  }],
  links: { next: "...&offset=20" | null }
}
```

### アーキテクチャ（レイヤー準拠）

| レイヤー | ファイル | 役割 |
|---|---|---|
| Domain | `app/lib/search/project-search.ts` | Pure関数: `searchProjects(items, query, opts)`。正規化（NFKC・大小文字・カタカナ/ひらがなは初版では NFKC+lowercase のみ）と部分一致、ソート（予算降順デフォルト） |
| Domain | `app/lib/api/quality-scores-loader.ts` | 現在 `app/api/quality-scores/route.ts` 内にあるJSON読込+メモリキャッシュを**共有モジュールに抽出**（fsは使うがHTTP/Reactなしのため app/lib 配置で可。検索APIと quality-scores API の両方が利用） |
| API | `app/api/search/projects/route.ts` | パラメータ検証（`limit` 上限100強制）→ loader → search → 応答整形 |

### API共通規約（既存5ルートにも適用）

1. **`Cache-Control: public, s-maxage=86400, stale-while-revalidate=604800`** を全ルートに付与（データはデプロイ時のみ更新のため安全。現状は project-details のみ）
2. **`metadata.notes`** を全応答に同梱（mof-overview 方式）。共通文言は `app/lib/api/api-notes.ts` に定数として一元化
3. `year` パラメータの統一（デフォルト2024、対応年度外は400）
4. 共通ヘルパー `app/lib/api/links.ts`: pid/法人番号 → 各API・WebページURLの組み立て（Pure関数。SG-4 でも使用）

### 完了条件

- `?q=マイナンバー` が13件を数KBで返す（実地調査②が3.9MBダウンロードなしで通る)
- 既存5ルートすべての応答ヘッダに Cache-Control が付く
- `/quality-check` パス

## SG-3: 法人番号逆引きインデックス + 支出先API

### データパイプライン（`scripts/`、CSV処理のみの規約準拠）

新規スクリプト `scripts/generate-recipient-index.ts`（npm script: `generate-recipient-index` / `-2025`）

- **入力**: `public/data/subcontracts-{YEAR}.json`（生成済み成果物からの導出。CSV再パースはしない。`generate-subcontracts` の後段に位置づけ、`/data-update` スキルのフロー文書に追記）
- **出力**: `public/data/recipient-index-{YEAR}.json`（+ `.gz`、`compress-data` に追加）

#### インデックス構造

```
{
  metadata: { year, generatedAt, sourceFile, recipientCount, notes },
  recipients: {
    "<key>": {
      name: string,              // 代表表記（最頻出）
      corporateNumber: string,   // 空文字あり
      aliases: string[],         // 同一キーに紐づいた表記ゆれ
      totals: {
        directAmount, directCount,        // originKind === 'direct' のブロックでの受領
        subcontractAmount, subcontractCount,  // それ以外（subcontract / separate-origin）
        // 注意: direct + subcontract の単純合算は二重計上の恐れがあるため total フィールドは持たせない
      },
      byMinistry: [{ ministry, directAmount, subcontractAmount, projectCount }],
      appearances: [{
        pid, projectName, ministry,
        blockId, originKind, amount,      // Multi-block対応: 同一事業複数ブロックは行を分けて全件保持
        upstream: { blockName, recipientKey | null },   // flows の sourceBlock から解決
        downstream: [{ blockName, amount, recipientKeys: [] }]  // flows の targetBlock から解決
      }]
    }
  }
}
```

#### キー設計（法人番号空欄24.1%への対応）

- 法人番号あり → キー = `cn:<13桁>`
- 法人番号なし → キー = `name:<正規化名>`（NFKC・空白除去・法人格表記（株式会社/(株) 等）の統一）
- 支出先名「**その他**」および明白な集約行は**インデックス対象外**とし、metadata.notes にその旨を明記（誤読防止。CLAUDE.md の「その他」問題に直結）

#### 検証（データ正確性最優先の方針に基づく）

- 生成時アサーション: 全 appearances の amount 合計 = 元JSONの recipients amount 総和と一致すること
- 実地調査の実測値（NTTコム cn:7010001064648 = 直接94件451.0億円 / 再委託54件255.9億円）を回帰テスト値としてスクリプト内 or 検証スクリプトに記録

### 新規エンドポイント

```
GET /api/recipients/[key]?year=2024        # key = 法人番号13桁 or name:キー
GET /api/search/recipients?q=エヌ・ティ・ティ&year=2024&limit=20
```

- Domain: `app/lib/search/recipient-search.ts` + `app/lib/api/recipient-index-loader.ts`（メモリキャッシュ）
- `appearances` は予算降順で `limit`（デフォルト50・上限200）。応答サイズ上限の目安100KB
- 応答の各 appearance に `links.subcontracts` / `links.web` を付与

### 完了条件

- `GET /api/recipients/7010001064648` 1発で直接/再委託の内訳・府省庁別集計・事業リストが返る（実地調査⑥の29MB全走査が不要になる）
- 回帰テスト値の一致

## SG-4: API応答の関連URL埋め込み（HATEOAS化）

SG-2/3 で新設するAPIは最初からリンク付きのため、ここでは**既存ルートへの後付け**を行う。

| ルート | 追加するリンク |
|---|---|
| `/api/project-details/[id]` | `links: { subcontracts, qualityRecipients, web }` |
| `/api/subcontracts/[id]` | 各 `BlockRecipient` に `recipientKey`（法人番号 or name:キー）と `links.recipient`（逆引きAPI）。トップレベルに `links.projectDetail`, `links.web` |
| `/api/quality-scores` | 各 item に `links.detail`, `links.subcontracts` |
| `/api/quality-scores/recipients` | 応答トップに**短縮フィールドの凡例**（`fieldLegend: { n: "name", b: "blockNo", ... }`）と `links.docs` |
| `/api/sankey/mof-overview` | `links.docs`（mof データ解説ページ） |

- リンク組み立ては SG-2 の `app/lib/api/links.ts` に集約（相対パスで返し、ホスト依存を排除）
- 既存レスポンスへの**追加のみ**（フィールド削除・改名はしない）。`/subcontracts` ページ等の既存UIには影響しないことを型チェックで担保
- 法人番号→外部サイト（gBizINFO）へのリンクは `links.external` として recipient API 応答にのみ付与

### 完了条件

- llms.txt を起点に、**URLの事前知識ゼロで** 検索→事業詳細→再委託ツリー→支出先逆引き→別事業 と辿れる（実地調査の探索ループがAPIのみで閉じる）

## SG-5: 支出先プロフィールページ + 企業中心Sankey

再委託構造見える化の新アプローチ。視点を事業中心から**企業中心**に反転する。

### ルーティング

- `app/recipients/[key]/page.tsx` — 公開はURL直打ち（`/quality` と同等の扱い）。`/subcontracts/[projectId]` の支出先名と SG-4 のリンクから到達できる
- データ取得は `/api/recipients/[key]` のみ（ページからの直接ファイル読みはしない）

### ページ構成

```
┌─ ヘッダ: 企業名・法人番号・gBizINFOリンク
├─ サマリカード: 直接受注（件数/総額） / 再委託受注（件数/総額）  ※並記し、合算値は表示しない
├─ 企業中心Sankey（下記）
├─ 府省庁別内訳（棒 or テーブル）
└─ 事業リスト（appearances、各行→ /subcontracts/[pid] リンク）
```

### 企業中心Sankey の設計

```
[左層: 発注経路]                [中央]              [右層: 再委託先]
府省庁 ─ 事業 ─ (元請ブロック) ─→ 対象企業 ─→ 下流ブロック（再委託先企業）
```

- **左**: appearance ごとに `府省庁 → 事業 → (originKind が subcontract の場合のみ) 元請ブロック → 対象企業`。direct 受注は事業から直結
- **右**: `appearances[].downstream` から対象企業のブロックが委託した先
- ノード数制御: 事業 TopN（金額降順、既定10）+「その他の発注元」集約ノード。**名称は既存の「その他の支出先」と同じ集約規約に従い、支出先名の「その他」と混同しない表記にする**
- レイヤー配置:
  - Domain: `app/lib/recipient-sankey-generator.ts` — Pure関数 `buildRecipientSankey(recipientEntry, opts): { nodes, links }`。HTTP/React禁止
  - UI: `client/components/RecipientSankey/` — 描画のみ。propsでデータ受領、直接APIコールなし
  - Page: `app/recipients/[key]/page.tsx` — fetch・状態管理・レイアウト
- 描画方式: `/sankey-svg` の SVG 実装（`client/components/SankeySvg`）の部品再利用を第一候補として実装時に評価し、流用コストが高ければ依存済みの `@nivo/sankey` で簡易実装する（このページはノード数が少なくズーム等の高機能が不要なため）

### 二重計上の扱い（表示原則）

- 直接受注と再委託受注は**常に分離表示**。例: JLIS が国から61.2億受領しNTTコムへ59.5億を再委託した場合、NTTコム側では「再委託受注 59.5億」とのみ表示し、国費総額への合算はしない
- Sankey 上もリンク値は appearance の amount をそのまま使い、ノード値はリンク合計とする（左右で意味が異なることをキャプションに明記）

### 完了条件

- `/recipients/7010001064648` で NTTコムの受注構造（直接94件/再委託54件、府省庁横断）が1画面で見える
- `/subcontracts/[pid]` の支出先名からプロフィールへ遷移できる

## SG-6（追加）: 検索結果のSankey表示（既存フィルタ×正規表現スコープ）

ユーザー発案。`/sankey-svg` は全グラフJSON（全5,664事業 + 12,521支出先ノード、検証済み）をクライアントに読み込み済みのため、**事業名・支出先名の正規表現マッチで部分グラフに絞ってから既存 `filterTopN` に渡せば、検索結果をそのままSankeyとして描画できる**。新規データ生成・APIは不要。

### アーキテクチャ

- Domain: `app/lib/sankey-svg-scope.ts`（新規・Pure関数）

  ```
  scopeByQuery(allNodes, allEdges, { projectPattern?, recipientPattern? })
    → { nodes, edges }  // マッチした事業/支出先と、それに連結する府省庁・budget・recipient のみ残す
  ```

- `filterTopN` 本体は**変更しない**（既に18引数・944行のため、これ以上の引数追加を避ける）。page.tsx の呼び出し直前（現状 page.tsx:1325 付近）に scope ステップを合成する
- UI: フィルタパネルに検索入力を追加（事業/支出先の対象切替 or 両方）。無効な正規表現はプレーン部分一致にフォールバック
- 既存の `scaleBudgetToVisible` が部分表示時の予算スケーリングを担うため、スコープ後の整合はその仕組みに乗る

### マッチ動作の検証済み事実

- `/マイナンバー/` → project-spending 13件（SG-2検索APIと同一件数）
- `/エヌ・ティ・ティ/` → recipient 17件（表記ゆれが別ノードのため複数。名寄せはSG-3の正規化キーと意味を揃える）

### 留意点

- グラフJSONの支出先は12,521ノードで、全支出先26,823件より少ない（グラフ生成時に名寄せ・集約済み）。**「その他の支出先」に集約された支出先は検索に掛からない**可能性があるため、ヒット0件時に「集計データ（/api/search/recipients）には存在する可能性」を案内する文言を検討（SG-3完了後）
- 正規表現はユーザー入力のため、ReDoS対策としてパターン長の上限と `try/catch` でのコンパイルを行う
- SG-1〜5（AIフレンドリー化）とは独立して実装可能。人間ユーザー向けの検索体験という意味で、本計画のUX側の副産物に当たる

## 実装順序と依存関係

```
SG-1 (入口) ──────────────────────────┐
SG-2 (検索API + 共通規約) ─→ SG-4 (リンク埋め込み) ─→ llms.txt 更新(SG-1の改訂)
SG-3 (逆引きインデックス + API) ─→ SG-4 ─→ SG-5 (プロフィール + 企業中心Sankey)
SG-6 (検索結果のSankey表示) … 独立。任意のタイミングで着手可（クライアント完結）
```

- SG-1 と SG-2 は独立して着手可能。SG-4 は SG-2/3 の links ヘルパーとAPIの存在に依存。SG-5 は SG-3 に依存
- PR分割は SG 単位を基本とし、各SG完了時に `/quality-check` + 既存ページのスモーク（`/subcontracts`, `/quality` の表示確認）を行う
- いずれも既存UIの変更は SG-4 の `/subcontracts` リンク追加と SG-5 のみ。レスポンシブ対応等のUX課題とは独立

## レイヤー設計ルールとの整合チェック

| ルール | 本計画での扱い |
|---|---|
| `scripts/` はCSV処理のみ・UI/APIロジック禁止 | SG-3 の生成スクリプトはデータ変換のみ。入力が成果物JSONである点は「パイプライン後段」として data-pipeline-guide に位置づけを記載 |
| `app/lib/` はPure・HTTP/React禁止 | 検索・Sankey生成はPure関数。loader（fs+キャッシュ）はHTTPハンドラではなくデータアクセスのため可（既存 `app/lib/subcontracts` と同格）。`NextResponse` 等は使用しない |
| `app/api/` はハンドラのみ、ロジックは `app/lib/` へ | quality-scores route 内の読込ロジックを app/lib に抽出してから検索APIを実装 |
| `client/components/` は直接APIコール禁止 | RecipientSankey は props 受領のみ。fetch は page.tsx |
| Multi-block は filter().reduce() で合算 | インデックスは appearance を全件保持し、集計は常に reduce。known bug を踏まない |
