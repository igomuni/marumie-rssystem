## 0. 使える内容データの棚卸し（すでに持っているもの）

内容ベース調査の原料はすでにデータ内にある。未活用なのは検索・集計の手段:

| データ | 場所 | 内容 |
|--------|------|------|
| 事業の概要・目的・課題 | project-details（purpose / overview / currentIssues） | 事業単位のテキスト |
| **支出先の使途** | quality-recipients の `role`（事業を行う上での役割）・`cc`（契約概要） | **支出行単位の使途テキスト**（74,639行/年） |
| 委託チェーン | `chain`・ブロック名 | 資金経路の文脈 |
| 事業区分 | project-details `category` | 粗い分類（既存） |

## 1. 内容調査スタック（3段構え）

### 1-1. 全文検索の拡張（決定論・実装小・最初にやる）

- `/api/search/projects?scope=details` — 概要・目的・課題を検索対象に（PR #244 キュー2）
- **新設 `/api/search/spending?q=`（使途検索）** — `role`/`cc`/ブロック名を対象に**支出行を横断検索**。応答: マッチ行（pid・事業名・支出先・金額・cc抜粋）+ 金額合計 + 事業別集計。「広報にいくら使われている?」「システム改修を受注しているのは誰?」という**使途起点の問い**が初めて解ける
- 前提パイプライン: `scripts/generate-spending-search-index.ts` — quality-recipients JSON から検索用のコンパクトな行インデックスを生成（新規CSV処理は不要、既存成果物からの導出）

### 1-2. AI事前タグ付け（バッチ・トークンは生成時のみ・本命）

支出行に**支出種別タグ**（システム開発/運用・広報/イベント・調査研究・給付/交付・工事・委託運営…）、事業に**政策分野タグ**（子育て・エネルギー・防衛・DX…）を LLM バッチで一括付与する。

- 実行パターンは `score-project-quality-ai.py` で実証済み（5,003事業×2年・モデル比較・コスト見積もりの実績あり）。**推論コストは生成時の一回だけ**で、以後の検索・集計・図はタグで決定論的に回る
- `SankeyQuery.filter` に `tags` を追加 → 「広報費だけのサンキー図」「給付型事業だけの図」が webView 込みで実現。曖昧な自然言語（「広告っぽい支出」）とデータの間の**意味の橋**はここで架ける
- embedding 意味検索は**保留**——タグで曖昧クエリの大半が解けるはず。タグの不足が観測されてから再検討（自己改善ループの観測対象）

### 1-3. プラットフォーム（どこで動かすか・段階論）

| 段階 | 形態 | 特徴 |
|------|------|------|
| 現在 | ローカル Claude Code / VSCode | 実験場。プレイブック準拠 |
| 公開第1段 | 外部エージェント（llms.txt + 公開API） | 追加開発が最小。利用者は Claude/ChatGPT 等に URL を渡すだけ |
| 第2段 | **リモートMCPサーバー（Vercel上）** | Claude Desktop 等からツールとして接続。API群の薄いラッパーで、query API・プレイブックが整った今は現実的（20260611 検討の再浮上） |
| 第3段 | サイト内蔵アシスタント | 一般人の曖昧指示に最も近いUXだが、LLM APIコスト・キー管理・ガードレールが伴う。第1〜2段の利用実績を見て判断 |

タグ付け・highlights（§3）は事前計算のため、どの段階でも共通基盤になる。

## 2. ユーザーシナリオ — 曖昧な指示 → 具体化 → 深掘り

### UX原則（3つ）

1. **解釈の宣言**: 曖昧語をどう解釈したかを必ず明示し、修正可能にする（appliedQuery エコーバックの思想をUXに昇格）。例:「『子育て』を、事業名・概要に『子育て|こども|保育』を含む事業（N件・◯兆円）と解釈しました。保育所整備に絞ることもできます」
2. **最初の答えは数字1つ+図1枚**: 総額と webView。詳細は出さない
3. **深掘りメニューの提示**: 答えの直後に「次の質問」を2〜3個提示する。サイトの核「良い質問の生成」のUI化であり、各選択肢は既存APIの1コールに対応させる

### シナリオ表（曖昧指示の代表5型）

| 曖昧な指示 | 具体化（解釈） | 使う機能 | 深掘りメニュー例 |
|-----------|--------------|---------|----------------|
| 「子育て支援って何に使われてるの?」 | 概要検索 `子育て\|こども\|保育` → N事業・総額 | scope=details + query summary + タグ内訳 | 最大の事業は? /「その他」比率は? / 前年からの増減は? |
| 「マイナンバーって結局いくら?」 | 事業名+概要検索 → 総額と年度推移 | query + compareYears | 何に使われた?(使途内訳) / ベンダーは誰? |
| 「広告・広報に税金いくら?」 | **使途検索** `広報\|広告\|PR` or タグ=広報 | /api/search/spending | 受注上位の企業は?(法人番号逆引き) / 省庁別は? |
| 「無駄遣いっぽいのを教えて」 | **提案システム(§3)へ**: 異常度上位を「無駄とは断定できないが説明の薄い/急増した事業」とフレーミングして提示 | /api/highlights | 各事業の深掘りレポートへ |
| 「うちの県に関係する事業は?」 | 支出先名で自治体検索 → 県・市町村への交付一覧 | search/recipients + recipients/[key] | どの省庁から? / 何の事業で? |

### ウォークスルー例（シナリオ1）

```text
利用者:「子育て支援って何に使われてるの？」
① 解釈宣言: 概要検索で 214事業・3.2兆円 と解釈提示（数値は例）
② 最初の答え: 総額 + webView 1枚 +「7割が給付・交付型（タグ内訳）」
③ 深掘りメニュー: (a) 最大の5事業 (b) 資金の行き先が見えない部分（「その他」・執行団体）
   (c) 去年からの変化
④ 利用者が (b) を選択 → 「その他」比率上位の給付事業リスト + 品質スコア
   → 個別事業の詳細ダイアログ / 調査レポート生成へ
```

「無駄遣い」型への注意: データ正確性最優先の原則から、**サイトは「無駄」を判定しない**。提示できるのは「説明の薄さ」「急増」「集中」という観測可能な異常度であり、フレーミングを必ずこの語彙に揃える（プレイブックに追記する）。

## 3. 提案システム — ログとデータから「興味深い事業」を提示する

### 構造: 2系統のシグナルの掛け算

**系統1: データ側の異常度指標（決定論・LLM不要・先行実装）**

過去レポートが人力で見つけた「発見の型」を指標に蒸留し、全事業を機械スキャンする:

| 指標 | 由来（発見の型） |
|------|----------------|
| 前年比増減率・新規/消滅 | KSK +221億（e-Tax レポート） |
| 「その他」比率 × 金額 | 交付金の72%「その他」（実験①）・こども家庭庁67% |
| 集中度 topShare1/3 × 金額 | NTTデータ単独85%（e-Tax） |
| 品質スコア低 × 予算大 | 実験①の pid=1900 |
| 執行/予算乖離 | 防衛 7517（予算の5.7倍執行） |
| 再委託深度 | subcontracts 系レポート |

- 実装: `scripts/generate-highlights.ts`（ビルド時バッチ）→ `highlights-{year}.json` → **`/api/highlights`**（+ 将来トップページ「注目の事業」ウィジェット）
- **これが「無駄遣いっぽいのを教えて」への回答基盤**になる（§2 シナリオ4）

**系統2: 利用側の関心シグナル**

- 当面（ローカル/公開初期）: レポートコーパス（再現性ブロックの SankeyQuery）から頻出テーマ・省庁・支出先を集計
- 公開後: **URL=状態の設計のおかげで、アクセスログのクエリパラメータがそのまま関心テレメトリになる**（fnp/fm/fnr の集計 = 何が検索されたかの分布）。このサイトの構造的な利点
- プライバシー: 集計利用のみ・個人単位の追跡はしない。公開段階でポリシーを明記（公開判断のゲート項目に追加）

**提案の生成 = 異常度上位 ∩ 関心領域**。LLM の役割は「なぜ興味深いか」の1行説明の生成のみ（低トークン）。さらに highlights 上位をプレイブックの依頼文テンプレに流し込めば**レポートの自動種まき**ができ、コーパスが拡充されて自己改善ループ（20260705_1628）に還流する。
