## 目的

**利用者が「無駄遣いっぽいのを教えて」「面白い事業ない?」と聞いたとき、AIが答えられずに断るか、根拠なく答えてしまう状態を解消するため。** 過去レポートが人力で見つけた「発見の型」を指標化して全事業を機械スキャンし、観測可能な異常度として提示する。

## 語彙の規律（最重要・実装より先に固定）

サイトは「無駄」を判定しない。提示できるのは**観測可能なシグナル**のみ:

| 指標 | 観測している事実 | 由来（発見の型） |
|------|----------------|----------------|
| `spendingChange` | 前年からの支出の急増・急減、新規/消滅 | KSK +220億・電気ガス -64%（compareYears 実証済み） |
| `otherRatio` | 支出先「その他」への比率 × 金額規模（説明の薄さ） | 交付金72%「その他」（実験①） |
| `concentration` | 支出先上位1社シェア × 金額規模 | NTTデータ85%（e-Tax） |
| `lowScoreHighBudget` | 品質スコア（レビューシート記載の質）低 × 予算大 | 実験① pid=1900 |
| `execBudgetGap` | 執行額/予算額の乖離 | 防衛 7517（予算の5.7倍執行） |
| `subcontractDepth` | 再委託の深さ | subcontracts 系レポート |

チャット・API の応答文言もこの語彙に揃える（「注目」= シグナルが観測された、の意）。

## 設計判断1: 重み付き合成スコアを作らない（初版）

元検討の未決事項「指標の重みと閾値」は、**初版では重み付けを回避する設計**で解消する:

- `/api/highlights` は**指標ごとの上位リスト**（各10件）+ **複数指標に同時該当した事業リスト**（multiSignal）を返す
- 合成スコアを作ると「重みの正当性」という答えのない議論を背負う。指標別リストなら各数値がそのまま根拠になり、データ正確性最優先の原則と整合する
- 「特に注目」の代替は multiSignal（2指標以上に該当）。重みではなく共起で絞る

閾値も同様に「上位N位」ベースとし、絶対値の閾値（例: その他比率50%以上）は設けない（分布が年度で動くため）。

## 設計判断2: パイプライン新設せず in-process 計算（WP3-3 と同方針）

元検討は `scripts/generate-highlights.ts`（ビルド時バッチ）を想定していたが、必要な入力はすべてロード済みキャッシュにある:

- graph（その他比率・事業別集中度・支出合計）: `loadSankeyGraph`
- 年度間差分: `compareYearsSummary`（実装済み）
- 品質スコア・予算/執行・再委託深度: `loadQualityScores`（redelegationDepth も既に項目にある）

5,000事業 × 6指標の全スキャンは1回だけ実行して**年度別メモリキャッシュ**（loader 群と同じ流儀）。生成物・ビルド手順・Git 管理対象は増やさない。

## 構成

| Layer | ファイル | 役割 |
|-------|---------|------|
| Domain | `app/lib/highlights.ts`（新規・Pure） | 全事業スキャン: 指標計算・上位抽出・multiSignal 抽出 |
| Domain | `app/lib/api/highlights-loader.ts`（新規） | 入力ロード + 計算結果の年度別キャッシュ |
| API | `app/api/highlights/route.ts`（新規） | `GET /api/highlights?year=&metric=` |
| AI | `sankey-chat-agent.ts` | 新ツール `get_highlights`（metric 指定可・クランプ済み） |

## API 仕様（案）

`GET /api/highlights?year=2025`（`metric=` で単一指標に絞り込み可）

- `metrics.{指標名}`: 各上位10件。エントリは pid・事業名・府省庁・**指標の根拠数値**（例: otherRatio なら比率と金額、spendingChange なら両年度支出と増減率）・links（事業詳細・sankeyView）
- `multiSignal`: 2指標以上に該当した事業（該当指標名の配列付き、上位10件）
- `metadata.notes`: 語彙の規律を明記 —「本APIは異常・無駄の判定ではなく観測可能なシグナルの列挙である」「品質スコアはレビューシート記載の質の評価」「事業年度N=予算年度N-1実績」「その他比率は報告書に『その他』と記載された支出の比率」
- 指標の対象母集団: 金額の小さい事業でノイズが出る指標（otherRatio・concentration・subcontractDepth）は**支出額 10億円以上**の事業に限定（この下限だけが本設計唯一の絶対値パラメータ。未決事項1）
- spendingChange は year と year-1 の比較（2024 指定時は比較不能なので空 + notes で案内。実質 2025 専用）

## チャットツール `get_highlights`

- 引数: `{ metric?: 指標名 }`（省略時は multiSignal + 各指標の上位3件のダイジェスト）
- 「無駄遣いっぽいのを教えて」への応答規範をシステムプロンプトに追記: get_highlights で調べた上で、**「無駄とは判定できないが、説明の薄さ・急増・集中などのシグナルが観測された事業」**というフレーミングで提示する。断定表現の禁止を明記
- 提案チップの answerable リストに「注目シグナルのある事業は?」を追加

## レイヤー整合チェック

| ルール | 扱い |
|--------|------|
| `scripts/` CSV処理のみ | 触れない（in-process 計算の判断による） |
| `app/lib/` Pure | highlights.ts は入力を引数で受ける Pure 関数。loader は既存流儀 |
| 金額1円単位・「その他」vs「その他の支出先」 | otherRatio は支出先名「その他」（実データ）のみを対象とし、集約ノードは含めない（notes 明記） |
| 語彙の規律 | notes・ツール description・プロンプトの3箇所に固定 |

## 検証方針

1. tsc / lint
2. curl 実測: 各指標の上位に既知の発見が現れるか — spendingChange に電気ガス/KSK、concentration に e-Tax 系、execBudgetGap に防衛 7517（過去レポートの再現が正しさの基準）
3. 初回計算のレイテンシ（キャッシュ後は即時のはず。初回が秒級なら許容、それ以上なら報告）
4. チャット: 「無駄遣いっぽい事業を教えて」でフレーミング規律が守られた回答になるか

## 未決事項（ユーザー判断）

1. ノイズ抑制の支出額下限: **10億円**でよいか（otherRatio・concentration・subcontractDepth に適用。5億〜50億の間で調整可能）
2. multiSignal の該当条件: **2指標以上**でよいか（3指標以上だと該当がほぼ消える可能性。実装時に件数を報告して調整）
3. 将来のトップページ「注目の事業」ウィジェット（元検討 §3）は本設計のスコープ外とした — API とチャットのみでよいか
