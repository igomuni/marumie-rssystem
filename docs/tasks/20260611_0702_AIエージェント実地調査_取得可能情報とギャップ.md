# AIエージェント実地調査: ローカル起動で取得できた情報とギャップ

日時: 2026-06-11
前提ドキュメント: [20260611_0648_AIフレンドリー化_ユーザーストーリーとレートリミット検討.md](20260611_0648_AIフレンドリー化_ユーザーストーリーとレートリミット検討.md)
方法: `npm run dev` でローカル起動し、AIエージェント（Claude）が HTTP 経由で自由に探索。シナリオは「マイナンバー関連事業の支出と再委託先を調べる」

## 要約

- **データは想像以上に揃っている**。事業概要・契約概要・法人番号・委託チェーン（`組織→A→C` 形式）まで取得でき、「直接支出先→再委託先のつながり見える化」は**新規データ収集なしで実現可能**
- 一方、入口が全部塞がっている: `robots.txt`・`sitemap.xml`・`llms.txt` すべて404。検索APIなし。逆引き（支出先→事業）APIなし
- エージェントは「数MBのダンプを落として自前で走査する」ことで全タスクを完遂できたが、**外部のAIエージェントには現実的でない**（29MBダウンロード・コンテキスト制約）。ここがスモールゴールの的

## 1. 実験ログ: 今できたこと・できなかったこと

### シナリオ「マイナンバー関連事業の支出と再委託先を調べる」

| ステップ | 結果 | 方法 | 外部エージェントにとっての現実性 |
|---|---|---|---|
| ① サイトの自己紹介を得る | ✗ | robots.txt / sitemap.xml / llms.txt 全て404 | 入口なし。HTMLのmeta descriptionのみ |
| ② キーワードで事業を検索 | △ | 検索APIなし。`/api/quality-scores`（limit無視・全5,003件 3.9MB）を落として自前フィルタ → 「マイナンバー」13件ヒット | 3.9MBはツール応答としては不可。ローカルでない限り非現実的 |
| ③ 事業の詳細を読む | ◎ | `/api/project-details/706` → 目的・現状課題・事業概要のテキストが取れる。**AIの回答素材として極めて良質** | 可能（pidさえ分かれば） |
| ④ 再委託ツリーを見る | ◎ | `/api/subcontracts/712` → ブロック構造・**法人番号**・契約概要・金額。例: 地方公共団体情報システム機構(61.2億) → NTTコム(59.5億) → PwC(0.9億) | 可能（pidさえ分かれば） |
| ⑤ 委託経路の復元 | ◎ | `/api/quality-scores/recipients?pid=712` の `chain` フィールド（`組織→A→C`）+ 深度 `d` で完全復元可能 | 可能だがフィールドが短縮名（n,b,s,a2...）でAIが誤読しやすい |
| ⑥ 支出先からの逆引き | ✗(API) / ○(生データ) | APIなし。`subcontracts-2024.json`(29MB) を全走査 → **NTTコム(法人番号7010001064648): 直接受注94件451.0億円 + 再委託受注54件255.9億円** が府省庁横断で判明 | 29MBダウンロード+全走査。外部エージェントには不可能に近い |

### 副次的な発見

- `/api/sankey/mof-overview` は `metadata.notes` に注意書き（「予算書上の金額で重複含む」等）を同梱している。**全APIが見習うべきお手本**が既に1つある
- `/api/quality-scores` の応答には `pid`・事業名・府省庁・予算額・`hasRedelegation`・`redelegationDepth` が含まれ、**事実上の事業インデックス**。これの軽量版（検索付き）が作れれば②が解決する
- `recipients` API の短縮フィールド名はサイズ削減目的だが、コメント（route.ts内）を読まないと解読できない。AI向けには凡例をレスポンスに含めるか、ドキュメントへのURLを入れる必要がある
- 法人番号（corporateNumber）が支出先に付いている。**国税庁法人番号公表サイトや gBizINFO と突合できる横断キー**であり、AIフレンドリー化の最大の資産のひとつ

## 2. 何ができると「深掘り → 関連情報 → 次の情報」がつながるか

エージェント視点で今回詰まった箇所は、すべて「**次のリンクがない**」ことに帰着する。探索ループを成立させるのに必要なのは:

### (a) 入口: キーワード → pid のジャンプ（検索API）

```
GET /api/search/projects?q=マイナンバー → [{pid, name, ministry, budget, hasRedelegation, url}]
```
- 今回の実験では3.9MBダンプの自前フィルタで代替した。これを数KBの応答にするだけ
- 実装は quality-scores のメモリキャッシュ済みデータを `name.includes()` でフィルタするだけ。**新規データ生成不要、route 1本**

### (b) 横断: 支出先 → 事業群の逆引き（最重要ギャップ）

```
GET /api/recipients/7010001064648 → {name, 直接受注: [...], 再委託受注: [...], 府省庁別集計}
```
- 今回29MB全走査で得た「NTTコム 直接94件/再委託54件」をAPI1発にする
- 29MBを毎回走査するのは論外なので、**`scripts/` で法人番号→事業の逆引きインデックスを事前生成**（パイプラインの新成果物）。推定数MB、検索は法人番号 or 名前正規化キー
- これが**ユーザーの言う「直接支出先から再委託先へのつながり見える化」の土台そのもの**

### (c) 連鎖: 全応答に「次に呼べるURL」を埋め込む（HATEOAS的リンク）

- `project-details` の応答に `subcontractsUrl`, `qualityUrl`, `webUrl`(人間用ページ) を含める
- `subcontracts` の各 recipient に `recipientUrl`（逆引きAPIへのリンク）を含める
- これにより**エージェントは事前知識なしで「掘れる方向」を発見できる**。llms.txt が入口、リンクが経路
- 法人番号には外部リンク（gBizINFO等）も付与可能 → サイト外への関連情報展開も自動化される

### (d) 文脈: メタデータ・凡例の同梱

- mof-overview 方式の `notes` を全APIに展開（単位1円、年度、「その他」vs「その他の支出先」、データ留意事項URL）
- recipients API の短縮フィールドに凡例 or 通常名版を用意

(a)→(b)→(c) が揃うと、エージェントの探索は
**「キーワード → 事業リスト → 事業詳細 → 再委託ツリー → 気になる企業 → その企業の全受注 → 別の事業 → …」**
というループとして閉じる。今回は人力（ファイル直読み）でこのループを回したが、APIだけで回るようになるのがゴール。

## 3. 「直接支出先 → 再委託先のつながり見える化」の実現性

**結論: データは完全に揃っており、実現可能。** 必要なのは逆引きインデックスとUI/APIのみ。

### データの裏付け（今回確認済み）

- `subcontracts-2024.json` の各ブロックに `isDirect` / `originKind` / 法人番号 / 金額
- `project-quality-recipients` の `chain`（`組織→A→C`）+ 深度で**誰が誰に投げたかの経路が事業内で完全復元可能**
- 法人番号により表記ゆれを超えて同一企業を束ねられる（名寄せ済みに近い）

### 作れるもの（段階順）

1. **支出先プロフィールページ** `/recipients/[corporateNumber]`: 受注総額、直接/再委託の内訳、府省庁別、事業リスト。逆引きインデックスがあれば静的生成可能
2. **企業中心Sankey**: 中央=企業、左=発注元（府省庁/事業/元請）、右=さらに再委託した先。既存の `app/lib/` Sankey生成の新プリセットとして実装できる
3. **元請→下請ネットワーク図**: 「A社が元請の時、誰に投げているか」の集約。多重下請け構造・常連ペア（例: JLIS→NTTコム）の可視化。ここまで来ると報道・研究価値が高い

### 注意点

- 法人番号が空の支出先（個人・海外・「その他」集約）の扱いを最初に決める必要がある（名前正規化でのフォールバック or 対象外と明記）
- Multi-block問題（CLAUDE.md既知バグ）: 逆引き集計は必ず filter().reduce() 方式で合算
- 「再委託受注額」と「直接受注額」の合算は二重計上に注意（直接61億の中から59億が再委託されるケース。**足すと過大計上**になる。表示は分離が原則）

## 4. スモールゴール提案（この順で1つずつ）

| # | ゴール | 内容 | 規模 |
|---|---|---|---|
| SG-1 | 入口を作る | `llms.txt` + `robots.txt` + `sitemap.xml`。API仕様・統計サマリ・留意事項を記載 | 半日 |
| SG-2 | 検索を作る | `GET /api/search/projects?q=`（既存キャッシュのフィルタのみ）+ 全APIに `Cache-Control` と `notes` | 1日 |
| SG-3 | 逆引きを作る | `scripts/` に法人番号逆引きインデックス生成を追加 + `GET /api/recipients/[cn]` | 2-3日 |
| SG-4 | リンクで繋ぐ | 全API応答に関連URL埋め込み（HATEOAS化） | 1日 |
| SG-5 | 見える化 | `/recipients/[cn]` ページ（プロフィール+企業中心Sankey） | 数日 |

SG-1〜2 だけでも「外部エージェントが検索→詳細→再委託まで自走できる」状態になり、ユーザーストーリー US-1（記者）/US-3（市民・間接）が成立する。SG-3 以降が本プロジェクト固有の価値（再委託の横断可視化）。
