# AIチャットのストリーミング応答（WP2-2）設計 — 進行イベントのSSE配信

日時: 2026-07-10 06:33 (JST)
ステータス: 設計（実装前・ユーザーレビュー待ち)
関連: [AIチャット機能進化ロードマップ](20260709_0759_AIチャット機能進化ロードマップと実装計画.md) WP2-2 / [WP1ベースライン評価](20260709_0847_WP1深掘りツール実装とベースライン評価.md)（レイテンシ実測）

## 目的

**チャット利用者が、送信後14〜130秒（利用ログ実測）の間まったく反応のない画面を見続けて「壊れたのか」と不安になる状態を解消するため。**

現状は送信中スピナー + 固定文言「条件を組み立てています…」のみで、LLM 2〜9往復・ツール実行の進行が一切見えない。深掘りツール（WP1）と年度比較（WP3）で応答が長くなり、問題が拡大した。

## スコープ判断: 進行イベントのみ配信（トークンストリームはしない）

ロードマップ未決事項2の決定案。**v1 は「ツール実行の進行イベント」だけを SSE で流し、最終応答は従来どおり一括で返す**。

理由:

1. 体験目標（「進行が見える」）は進行イベントで達成できる。無反応時間の最大要因はツール実行を挟む多往復であり、往復の節目は数秒〜十数秒ごとに発生する
2. トークンストリームは OpenRouter の `stream:true` パース・プロバイダ差（hy3/Novita の癖）・Markdown 部分描画・`submit_result`（ツール呼び出しで確定するため本文が流れない）との整合など、コストが体験改善に見合わない
3. 進行イベント方式なら **LLM 呼び出し部（DI）は無変更**で、エージェントループに観測点を足すだけで済む

## 全体フロー

```text
ユーザー送信
   │ page.tsx: POST /api/ai/sankey-chat（body に stream: true を追加）
   ▼
route.ts: Content-Type: text/event-stream で ReadableStream を返す
   │ runSankeyChatAgent(..., onProgress) を起動し、
   │ onProgress コールバック → SSE イベントに変換して逐次 enqueue
   ▼
event: progress  {"kind":"llm_round","round":2}
event: progress  {"kind":"tool","tool":"run_sankey_query","matched":12}
event: progress  {"kind":"retry","waitMs":10000}
event: result    {（従来の SankeyChatResponse と同一形）}   ← 終端
   ▼
page.tsx: SSE をパースし progress → 進行ラベル state 更新、result → 従来の応答処理
AiChatPanel: sending 中の固定文言を進行ラベル（プロップ）に差し替え
```

## イベント設計

### 型（types/sankey-ai-chat.ts に追加）

| イベント | ペイロード | 発生元 |
|---------|-----------|--------|
| `progress` kind=`llm_round` | `{ round: number }` | エージェントループ（LLM 呼び出し直前） |
| `progress` kind=`tool` | `{ tool: string, ...結果要約 }` — run_sankey_query は `matched`（事業件数）、search 系は `hits`、深掘り系はツール名のみ | ループ（ツール実行直後） |
| `progress` kind=`retry` | `{ waitMs: number }` | **route**（リトライは route の callLlm が所有） |
| `result` | 従来の `SankeyChatResponse` と同一 | ループ完了時 |
| `error` | `{ error: string }`（従来の 502/500 文言と同一） | route |

- イベントは**構造化データのみ**を運び、日本語ラベルへの変換は UI 層（AiChatPanel 内の対応表）で行う。`app/lib/ai/` に UI 文言を置かない
- SSE のコメント行 `: ping` を **15秒間隔**で送出（LLM 1往復は最長60秒沈黙するため、プロキシ・ブラウザのアイドル切断を防ぐ。UI はコメント行を無視）

### エージェントループへの観測点（app/lib/ai/sankey-chat-agent.ts）

`runSankeyChatAgent` に省略可能な第4引数 `onProgress?: (ev: SankeyChatProgressEvent) => void` を追加する。

- 純粋なコールバック注入であり、fetch・NextResponse・React は持ち込まない（**DI 構造の維持**。LlmCaller と同じ思想）
- 未指定なら従来と完全同一の挙動（非ストリーミング経路・評価ハーネス・既存テストは無変更で動く）
- 発火点は2箇所のみ: 各ラウンドの `callLlm` 直前（`llm_round`）、各ツール実行直後（`tool` + 結果要約）

## API 仕様（route.ts）

- 要求: 既存 POST に `stream: true`（body の省略可能フィールド）を追加。**未指定なら現行の JSON 応答のまま**（後方互換。評価ハーネス・curl 検証はそのまま使える）
- 応答（stream 時）: `Content-Type: text/event-stream` / `Cache-Control: no-store`。`event:` 行 + `data:`（JSON 1行）の標準 SSE 形式。`result` または `error` が終端イベント
- 入力検証エラー（400系）は SSE を開始せず従来どおり JSON で返す（検証はストリーム開始前に完了している）
- リトライ時は route が `progress` kind=`retry` を送出（待機がユーザーに「フリーズ」と見えるのを防ぐ）
- 利用ログ（usage-log）は従来どおり完了時に1行（変更なし）
- ガード（isAiChatEnabled・404）・タイムアウト・入力クランプはすべて現行のまま

## クライアント設計

### page.tsx（送信処理の拡張）

- `fetch` の応答 Content-Type で分岐: `text/event-stream` なら `response.body` のリーダーで SSE をパース（EventSource は POST 不可のため fetch ストリーム読みで実装）、JSON ならフォールバックとして従来処理
- 新 state: `aiChatProgress: SankeyChatProgressEvent | null`（最新イベントのみ保持。履歴は持たない）。`result`/`error` 受信で null に戻し従来の応答処理へ
- SSE パースが途中で切断された場合は従来のエラー表示（「通信に失敗しました」）に落とす

### AiChatPanel（表示のみ）

- プロップに `progress?: SankeyChatProgressEvent | null` を追加
- sending 中の固定文言を対応表で差し替え:

| イベント | 表示例 |
|---------|--------|
| llm_round (round=1) | 「要求を解釈しています…」 |
| llm_round (round≥2) | 「結果を確認しています…（N回目）」 |
| tool: run_sankey_query | 「クエリを実行しました — 12事業がマッチ」 |
| tool: search_projects / search_recipients | 「語彙を検索しています…」 |
| tool: get_* / compare_years | 「詳細データを取得しています…」 |
| retry | 「混雑のため待機して再試行します…」 |

- fetch はせず、状態機械（idle → sending＋進行表示 → done/error）の表示のみを担う（レイヤー規約維持）

## レイヤー整合チェック

| ルール | 本設計での扱い |
|--------|---------------|
| `app/lib/` は HTTP/React 禁止 | onProgress は純コールバック。SSE 化・ping・retry イベントはすべて route 側 |
| `app/api/` はハンドラのみ | route は「コールバック→SSE 変換 + ストリーム管理」のみ追加。ループ制御は持たない |
| `client/components/` は fetch 禁止 | AiChatPanel はプロップ受け取りと文言対応表のみ |
| Pages は状態管理・API 呼び出し | SSE 読み・progress state は page.tsx |
| 既存互換 | `stream` 未指定の全経路（ハーネス・外部 curl）は挙動不変 |

## テスト・検証方針

1. 非ストリーミング回帰: 評価ハーネス全シナリオが従来どおり動く（`stream` 未指定）
2. `curl -N -X POST ... -d '{"messages":[...], "stream": true}'` で progress → result のイベント列を目視確認（深掘り系プロンプトで tool イベントが複数出ること・retry 発生時のイベントを確認）
3. ブラウザ: 送信直後から進行ラベルが更新されること、途中切断（dev サーバー停止）でエラー表示に落ちること

## 未決事項（ユーザー判断）

1. ハーネスにストリーミング検証モード（`--stream`: イベント列を記録）を追加するか（推奨: 追加。progress イベントの回帰確認が自動化できる）
2. 進行ラベルの粒度: ツール結果の件数（「12事業がマッチ」）まで出すか、種別のみ（「クエリを実行しています」）に抑えるか（推奨: 件数まで。適用前の期待値が伝わる）
3. `stream: true` を将来既定にするか（v1 はオプトイン。UI からの呼び出しのみ stream を付け、既定切替は運用実績を見て判断）
