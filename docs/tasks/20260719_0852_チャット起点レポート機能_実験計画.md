# チャット起点レポート機能 — 実装前のユーザー実験計画

作成日: 2026-07-19（JST）
位置づけ: 「チャットからレポート生成への導線」の実装可否・設計を決めるための、
**現行チャットのままユーザーが実施できる実験**の観点リスト。結果のチャットログを
docs/.ignore/ に保存してもらい、解析して設計（レポートボタンの仕様）に落とす。
関連: 20260705_1544（自然言語レポート生成の現在地）、docs/.ignore の人手レポート群
（防衛・こども家庭庁等 = 品質の比較基準）、20260719_0845 チャットログ（調査セッション実例）。

## 0. 先に知っておくべき構造的制約（実験の予想を立てる根拠）

| # | 制約 | レポート化への含意 |
|---|---|---|
| C1 | 会話履歴は user/assistant の**テキストのみ**をAPIに送る（ツールの生データは前ターンから消える） | 「ここまでを要約して」は **assistant 発話に残った数値だけ**が素材。発話に書かれなかった詳細は再取得しない限り使えず、幻覚リスク箇所 |
| C2 | 履歴上限 20メッセージ・合計8000字 | 長い調査セッションでは序盤の発見が黙って切り捨てられる |
| C3 | 1リクエスト = LLM 8往復・ツール10回まで | ワンショットの「レポートを書いて」で掘れる深さの上限 |
| C4 | 応答は1メッセージ（数千字想定・ストリームなし） | 長文レポートは途切れ・タイムアウトの可能性（先日の「続きが消えています」） |

## 1. 実験メニュー（各15分程度・チャットログを保存）

### E1: ワンショット・レポート要求（C3の限界測定）

> 「蓄電池の製造サプライチェーン強靱化支援事業について、資金の流れとデータ品質の
> 観点でレポートを書いてください」

観察: ツール上限内で何を取捨するか / 構成の質（docs/.ignore の防衛レポートと比較）/
数値に出典（どのツール由来か）が付くか / 語彙規律（断定回避・年度注記）の維持。

### E2: 対話積み上げ → 最後に要約（C1の実証・本命）

数ターン深掘り（詳細→品質→再委託→年度比較など）した後に:

> 「ここまでの調査をレポートとしてまとめてください」

観察: **会話に書かれた数値だけで書けているか、書かれていない数値を捏造しないか**（C1）/
再取得（ツール再実行）を自発的にやるか / まとめの構成力。

### E3: 再現性ブロックの要求（レポートコーパス規約への適合性）

E2 の続きで:

> 「この結論を第三者が再現できるように、適用したフィルタ条件（JSON）と、
> 各数値がどのツール・条件から得られたかを付記してください」

観察: SankeyQuery を正しく吐けるか / 数値↔出典の対応付けの精度（ここが弱いと
レポート機能はサーバ側での自動付記が必須になる = 設計の分岐点）。

### E4: 履歴上限の断崖（C2の体感）

20発話を超える長い調査の終盤で、序盤にしか出ていない発見に言及を求める:

> 「最初のほうで見つけた◯◯も含めてまとめて」

観察: 切り捨ての気づきにくさ / 「その情報は履歴にない」と正直に言うか。

### E5: 年度比較レポート（compare_years 主体の型）

> 「マイナンバー関連の事業について、2024→2025 の変化をレポートにしてください」

観察: 増減の根拠数値 / 「事業年度N=予算年度N-1実績」注記 / 新規・消滅の扱い。

## 2. 実験結果から決めたい設計分岐

1. **「レポート化」ボタンの実体**: (a) 会話履歴をそのまま整形要求する1ターン（C1 の制約内）
   か、(b) 会話から調査対象を抽出し**ツール再実行つきの専用エージェント**（別プロンプト・
   上限緩和）を回すか。E2/E3 で (a) の品質が不足なら (b)
2. **出典・再現性の担保**: エージェント任せ（E3 の精度次第）か、実行したツール呼び出しを
   クライアント側で記録してレポートに機械添付するか（BYOK ではツール履歴はブラウザに
   全部あるため機械添付が可能 — こちらが堅い可能性が高い）
3. **成果物の置き場**: 発見メモ（実装済）の拡張としてレポートを IndexedDB 保存 +
   Markdown コピー、が最小。共有・公開（市民版レポートコーパス）は次段
4. 上限（C2/C3/C4）の緩和が必要か: レポート用の専用モード（maxトークン・往復数の拡大、
   BYOK なら使用者コストなので緩めやすい）

## 3. 実験の作法

- BYOK（自分のキー・実務水準のモデル）で実施推奨（free モデルの途切れ・不安定を排除）
- 各実験のチャットログを `docs/.ignore/2026MMDD_HHMM_実験E*.md` に保存 → 解析依頼
- 良かった応答・破綻した応答の両方が設計材料（20260719_0845 と同様に評価コーパス化）
