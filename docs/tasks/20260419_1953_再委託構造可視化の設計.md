# 再委託構造可視化の設計（改訂版）

## 目的

予算データを調査するユーザーが「この事業の予算はどこまで再委託されているか」を把握できないという課題を解消するため。事業ごとのブロック間フロー（DAG）を専用ページで自前SVG描画し、資金フローの透明性を高める。

---

## データ制約の前提（既確認）

金額ベースで再委託チェーンの **90%はブロック間フローまでしか正確に表現できない**（ブロック内に支出先が複数存在するため、どの支出先がどの支出先に再委託したかはCSVに存在しない）。

本実装は **ブロックをノード** とするDAGを描画することで、この制約を回避する。支出先はブロックに内包する形で表示し、支出先間の矢印は引かない。

---

## 使用CSVと列定義

### 5-2: `5-2_RS_{YEAR}_支出先_支出ブロックのつながり.csv`

ブロック間フロー（再委託構造の骨格）。

| 列インデックス | 列名 | 用途 |
|---|---|---|
| 2 | 予算事業ID | 事業の識別キー |
| 3 | 事業名 | 表示用 |
| 6 | 府省庁 | 事業の帰属省庁 |
| 13 | 支出元の支出先ブロック | ソースブロック番号（空=担当組織） |
| 14 | 支出元の支出先ブロック名 | ソースブロック名（空=担当組織名） |
| 15 | 担当組織からの支出 | `TRUE` = 国から直接支出のフロー |
| 16 | 支出先の支出先ブロック | ターゲットブロック番号 |
| 17 | 支出先の支出先ブロック名 | ターゲットブロック名 |
| 18 | 資金の流れの補足情報 | エッジのラベル（任意）|

**「担当組織からの支出=TRUE」の行**がルートフロー（国→ブロック）。ブロック番号が空の場合は担当組織（府省庁）がソース。

### 5-1: `5-1_RS_{YEAR}_支出先_支出情報.csv`

ブロック内の支出先一覧（ブロックノードの内容を埋める）。

| 列インデックス | 列名 | 用途 |
|---|---|---|
| 2 | 予算事業ID | 事業の識別キー |
| 13 | 支出先ブロック番号 | ブロックの識別キー |
| 14 | 支出先ブロック名 | ブロック表示名 |
| 15 | 支出先の数 | ブロック内支出先数（0の行はブロックサマリ行） |
| 17 | ブロックの合計支出額 | ブロックノードの高さ基準 |
| 18 | 支出先名 | 支出先の表示名 |
| 19 | 法人番号 | 支出先の識別 |
| 23 | 支出先の合計支出額 | 支出先の個別金額 |
| 24 | 契約概要 | 支出先ごとの契約内容の説明（詳細パネルに表示） |

**`支出先名` が空の行** はブロックのサマリ行（ブロック名・合計額のみ持つ）。支出先行は `支出先名` が非空の行。

### 5-3: `5-3_RS_{YEAR}_支出先_費目・使途.csv`

支出先ごとの費目・使途内訳（ブロックノードをクリックしたときの詳細情報）。

| 列インデックス | 列名 | 用途 |
|---|---|---|
| 2 | 予算事業ID | 事業の識別キー |
| 13 | 支出先ブロック番号 | ブロックの識別キー（5-1 col13 と結合） |
| 14 | 支出先名 | 支出先の識別（5-1 col18 と結合） |
| 15 | 法人番号 | 支出先の識別（同上） |
| 16 | 契約概要 | 5-1 col24 と同内容（5-1 に存在しない場合の補完用） |
| 17 | 費目 | 費目分類（例: 雑役務費、旅費） |
| 18 | 使途 | 使途の説明（例: 研修実施請負業務） |
| 19 | 金額 | 費目×使途ごとの金額（1円単位） |

5-3 は `(予算事業ID, 支出先ブロック番号, 支出先名, 法人番号)` をキーに 5-1 の支出先と結合する。1支出先につき複数行（費目ごとに1行）。

### 2-1: `2-1_RS_{YEAR}_予算・執行_サマリ.csv`

事業の予算額・執行額（ページタイトル部分の数値）。

| 列インデックス | 列名 | 用途 |
|---|---|---|
| 2 | 予算事業ID | 事業の識別キー |
| 14 | 当初予算(合計) | 表示用予算額 |
| 19 | 執行額(合計) | 表示用執行額 |

## データ生成パイプライン

### 新規スクリプト: `scripts/generate-subcontracts.ts`

**入力**: `5-1`・`5-2`・`5-3`・`2-1` の4 CSV（同年度）

**処理**:
1. 5-2 を読み込み、事業単位でブロック間エッジを収集
2. 5-1 を読み込み、事業×ブロック単位で支出先一覧・合計額・契約概要を収集
3. 5-3 を読み込み、`(事業ID, ブロック番号, 支出先名, 法人番号)` をキーに費目・使途の内訳を収集し、5-1 の支出先に付加
4. 2-1 から事業の予算額・執行額を収集
5. 事業ごとに `SubcontractGraph` 型のオブジェクトを生成し、`projectId` をキーとして出力

**出力**: `public/data/subcontracts-{YEAR}.json.gz`

```
{
  "{projectId}": {
    projectId: number,
    projectName: string,
    ministry: string,
    budget: number,        // 当初予算
    execution: number,     // 執行額
    blocks: BlockNode[],
    flows: BlockEdge[]
  },
  ...
}
```

```
interface BlockNode {
  blockId: string          // ブロック番号: "A", "B", ..., "AA", ...
  blockName: string        // ブロック名
  totalAmount: number      // ブロックの合計支出額（5-1 col17）
  isDirect: boolean        // 担当組織から直接支出されるブロックか
  recipients: {
    name: string                // 支出先名（5-1 col18）
    corporateNumber: string     // 法人番号（5-1 col19）
    amount: number              // 支出先の合計支出額（5-1 col23）
    contractSummary?: string    // 契約概要（5-1 col24）
    expenses?: {                // 費目・使途内訳（5-3）
      category: string          // 費目（5-3 col17）
      purpose: string           // 使途（5-3 col18）
      amount: number            // 金額（5-3 col19）
    }[]
  }[]
}

interface BlockEdge {
  sourceBlock: string | null   // null = 担当組織（ルート）
  targetBlock: string
  note?: string                // 資金の流れの補足情報
}
```

### `compress-data` への追記

`scripts/compress-data.sh` で `subcontracts-{YEAR}.json` を `.gz` に圧縮するエントリを追加。
`scripts/decompress-data.sh` で `.gz` → `.json` 展開エントリを追加。

## 一覧ページ `/subcontracts/`

### 目的

再委託構造を持つ事業の全体像を把握し、気になる事業の詳細ページへ素早く到達できる入口。

### 表示内容（1行 = 1事業）

| 列 | 内容 | ソース |
|---|---|---|
| 事業名 | 事業名（PID付き） | 5-2 col3 |
| 府省庁 | 所管府省庁 | 5-2 col6 |
| 予算額 | 当初予算 | 2-1 col14 |
| 執行額 | 執行額 | 2-1 col19 |
| 再委託階層数 | DAGの最大深さ（直接支出が深さ1） | 5-2から計算 |
| 直接ブロック数 | 担当組織から直接支出されるブロック数 | 5-2 col15=TRUE の件数 |
| 総ブロック数 | 事業内の全ブロック数 | 5-1のユニークブロック数 |
| 総支出先数 | 事業内の全支出先数 | 5-1の支出先行数 |

### フィルタ

- **テキスト検索**: PID・事業名・府省庁・ブロック名・支出先名に対するインクリメンタル検索（前方一致または部分一致）
- **年度切替**: `2024` / `2025` のセレクタ

### ソート

以下の列でソート可能（昇順/降順トグル）：

- 予算額
- 執行額
- 再委託階層数
- 総ブロック数
- 総支出先数

デフォルトは **PID昇順**。

### データ設計への影響

`subcontracts-{YEAR}.json` に一覧表示用のサマリフィールドを追加する：

```
interface SubcontractGraph {
  ...既存フィールド...
  // 一覧ページ用サマリ（generate-subcontracts.ts で事前計算）
  maxDepth: number          // DAGの最大深さ
  directBlockCount: number  // 直接支出ブロック数
  totalBlockCount: number   // 総ブロック数
  totalRecipientCount: number // 総支出先数
}
```

一覧ページは `subcontracts-{YEAR}.json` の全エントリを読み込み、クライアントサイドでフィルタ・ソートする（データ量が小さいため、APIページネーション不要）。

## SVGレイアウト（自前実装）

### 座標系の設計

**X軸**：ブロックの階層深さ（0 = 担当組織、1 = 直接ブロック、2 = 再委託ブロック、...）
**Y軸**：同一深さ内でのブロックの縦位置（合計金額の降順で配置）

sankey-svg と同様に **固定ノード幅・高さ金額比例** のレイアウトを採用する。

### レイアウト計算 (`app/lib/subcontract-layout.ts`)

1. **深さ計算（BFS）**: `flows` のエッジを辿り、各ブロックの深さを計算。ルート（`sourceBlock === null`）の直接の子が深さ1。

2. **Fan-In 対応**: 同一ブロックが複数の親から参照される場合（Fan-In）、ブロックノードは**最も深いパスの深さ**を採用する（Sugiyama法のレイヤー割り当てに準ずる）。

3. **Y位置計算**: 各深さ内でブロックを `totalAmount` 降順に並べ、上から順に配置。ブロック間に固定パディングを入れる。

4. **エッジパス**: ブロック間のエッジは sankey-svg の `ribbonPath()` と同様に、ソース右端中央 → ターゲット左端中央 への三次ベジェ曲線で描画。

### ノード種別と表示

| ノード種別 | 表示 |
|---|---|
| 担当組織（ルート） | 府省庁名。固定高さ（金額比例しない）|
| 直接ブロック（isDirect=true） | ブロック名＋金額。青系の色 |
| 再委託ブロック（isDirect=false） | ブロック名＋金額。オレンジ系の色 |

**ブロックノード内の支出先リスト**:
- 金額上位5件の支出先名・金額をノード内部に小さく表示
- 6件以上のとき「他X社」を表示

**ブロックノードのクリック時（詳細パネル）**:
- 支出先ごとに契約概要（5-1 col24）を表示
- 費目・使途内訳（5-3）を折り畳みで表示（`費目: 使途 / 金額` の行形式）

**曖昧さの表示**:
- ブロック内支出先が2件以上 かつ そのブロックが他ブロックへのフロー元になっている場合、ブロックのボーダーを点線で表示し「委託元不特定」バッジを付ける

## Fan-Out / Fan-In の実例（PID=27: デジタル庁）

```
担当組織（デジタル庁）
  ├──→ A: アクセンチュア（4.2億）
  ├──→ B: アクセンチュア（2.4億）
  ├──→ C: 株式会社クラウドネイティブ（0.1億）
  ├──→ D: 株式会社クラウドネイティブ（0.0億）
  └──→ E: 株式会社SHIFT（7.0億）
             ├──→ F: furuCRM（0.6億）
             ├──→ G: 株式会社さうなし（0.1億）
             ├──→ H: 株式会社A-STAR（0.1億）
             ├──→ I: クラスメソッド（0.1億）
             ├──→ J: ROSCA（0.1億）
             └──→ K: 株式会社ニコシス（0.0億）
```

Fan-In実例（PID=141）: ブロックA・B・C が同じブロックDへ流入する構造。DAGとして描画し、Dは1つのノードとして表示。
