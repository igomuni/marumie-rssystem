## 1. 目的（Why）

`/quality` を見る利用者が、**2024年度でも2025年度と同じ世代・同じ基準のスコア**を見られるようにするため。現在の2024データは全件 `aiSource=heuristic`（実AI未評価）かつ有効性が旧 0-3 スケールのため、UI で「レベル: 3/10 → 100点」のような矛盾表示が出る。これを2025と同じ実LLM・0-10スケールへ揃える。

加えて、ユーザーの依頼は次の3点:

1. 2024再生成の **実行に必要なもの** を整理する
2. **どのような処理が行われるか** を把握する
3. **最初はAPIを使わず、チャットのプロンプトで同様の処理ができるか** を確認する

## 3. どのような処理が行われるか

`scripts/score-project-quality-ai.py` の `main()` の流れ。

```text
1. 入力ロード: recipients / scores（機械signal）/ project-details
2. キャッシュロード（data/cache/ai-quality-cache-<model>.json、--force で無視）
3. 全支出先行を走査 → ユニークタプル抽出 ＋ ゲート即決
     - opaque名（辞書 or o フラグ）→ identify=0 で即確定（AI呼ばない）。purpose は機械判定
     - それ以外 → AI判定キューへ
4. AI判定（OpenRouter or APIキー無し時 heuristic）
   4a. 軸A/B（支出先単位）: JUDGE_SYSTEM プロンプトで identifiability(0-3)・purpose(0-3)・reason
   4b. 軸E（事業単位）: EFFECT_SYSTEM プロンプトで effective(0-10)・reason（effect_key で重複排除）
5. プロジェクト集計・機械計算・総合再計算（事業ごと）
6. scores JSON を新スキーマで上書き（--limit 時はテストモードで上書きしない）
```

### 3-1. AI判定の中身（プロンプト要旨）

- **軸A/B JUDGE_SYSTEM**: 「行政事業レビューの品質監査官」として、納税者が『誰にいくら何のために払ったか』を追跡・検証できるかを評価。
  - `identifiability 0-3`: 0=特定不能（その他/未定/個人等）/1=曖昧（辞書未登録・文脈薄）/2=ほぼ特定可（出先機関・自治体・改組大学・海外団体、CNなくても文脈で明確）/3=完全特定可（正式法人名＋法人番号 or 公的機関正式名称）
  - `purpose 0-3`: 0=記載なし/1=定型・抽象（『業務委託』『一式』）/2=概ね具体/3=具体的で検証可能
  - 個人・外国法人・国の機関は法人番号を持たないことがあるため、**CN欠落のみを理由に不当に下げない**。
- **軸E EFFECT_SYSTEM**: 「行政事業レビューの評価官」として、事業の目的・現状課題・概要から**国民生活への寄与が明確・妥当に説明されているか**を 0-10 で採点。**実測成果ではなく成果設計（意図）の明確さ**。中間値も積極利用、reason に60字程度で根拠。

### 3-2. 機械計算とレベル→点数

- `identify/purpose 0-3` → 点数 `{0:0, 1:40, 2:70, 3:100}`。事業内は **金額(a2)加重平均**（金額欠落時は件数加重）。
- **軸C 収支整合性**: `gapRatio ≤ 0.10` なら100点、超過分は線形減衰。
- **軸D 構造整合性**: 旧 `axis4` から**再委託深度減点を除外**し、金額不整合＋孤立ブロックのみ。**総合には不算入（参考表示）**。
- **軸E**: `effective(0-10) × 10` で 0-100。
- **総合**: 重み A28／B22／C15／E35（Noneは除外し残り重みで再配分）。`aiSource` を付与。

### 3-3. 出力スキーマ（事業ごとに付与/更新される主フィールド）

`axisIdentify` / `axisPurpose` / `axisBudget` / `axisStructure` / `axisEffective` / `identifyLevelAvg` / `purposeLevelAvg` / `effectiveLevel` / `effectiveReason` / `aiSource` / `totalScore`。

### 3-3-bis. `heuristic` フォールバックの具体的な中身

`aiSource="heuristic"`（および実AI時の `heuristic(fill)` 穴埋め）は、**外部APIもAIも一切呼ばず、ローカルのPython関数で決定的に採点する**経路。意味理解はせず、**文字数の閾値・辞書/正規表現マッチ・キーワードのカウント**だけで近似する。`OPENROUTER_API_KEY` 未設定時は全件これで採点され、**現在の2024データもこの経路で生成されている**。

**軸A/B `heuristic_judge()`**（`scripts/score-project-quality-ai.py`）

- identifiability(0-3): opaque名→**0** ／ `valid`＋CNあり→**3** ／ `gov`・`supp`→**2** ／ `valid`(CN欠落)→**2** ／ `invalid`→**1** ／ `unknown`→契約概要≧12字 or 役割≧8字なら**2**、未満は**1**
- purpose(0-3): cc/role の長い方を `text` とし、空 or 定型句正規表現（`業務委託|委託業務|請負|物品購入|役務|その他|一式|—|-|不明`）一致→**0** ／ 8字未満→**1** ／ 20字未満→**2** ／ 20字以上→**3**

**軸E `heuristic_effect()`**

- 目的+概要が20字未満→**0点で即終了**。それ以外はベース3点に、本文≧60字で+2、便益キーワード（`国民|生活|安全|削減|向上|改善|支援`…）の種類が2以上で+2・4以上でさらに+2、数値+単位（`◯%`・`◯件`・`◯億`等）を含めば+1。最終 2〜10にクランプ。

**実AI実行時も部分的に動く箇所**

- APIバッチが4回リトライしても失敗した分は `heuristic(fill)` で穴埋め（ただし**キャッシュに保存されない**ため、再実行で実AIに上書きされる）。
- opaque のゲート即決は identify=0 を確定させ、purpose のみ heuristic を流用。

→ 「heuristic 結果」は高度な処理ではなく**決定的な近似採点**であり、実AI品質には `OPENROUTER_API_KEY` 経由の実行が必要。`heuristic_judge` の identify は 0/40/70/100点（レベル0-3）にしか割り振らず、AIのような細かい弁別を持たない点が2024問題（旧0-3スケールとの組み合わせで「3/10→100点」）の根。

### 3-4. レイヤー整合性（CLAUDE.md）

本処理は `scripts/`（データパイプライン）に閉じる。UI/APIロジックは含まず、API層（`app/api/quality-scores/*`）は生成済みJSONを読むだけ。`app/lib/` 不介在。レイヤー設計ルールと矛盾しない。

## 4. 「APIを使わず、チャットのプロンプトで同様の処理ができるか」の確認

### 4-1. 結論

**できる（サンプル規模では実証済み）。** 軸A/B/E の判定主体は LLM であり、チャットのアシスタント（Claude）自身が同じ LLM。スクリプトの `JUDGE_SYSTEM`／`EFFECT_SYSTEM` プロンプトとレベル定義をそのままチャットに与えれば、同じ形式（`identify`/`purpose` 0-3、`effective` 0-10 ＋ reason）の判定を生成できる。

ただし **全件（軸A/B 87,653 ＋ 軸E 5,642）をチャットで処理するのは非現実的**。チャットは少数サンプルでの **方式検証・プロンプト調整・出力品質の目視確認** に向き、全件の確定生成はスクリプト＋APIキー経由が前提。

### 4-2. 実証（2024実データのサンプル判定）

下記は本書作成時に、スクリプトと同一ルーブリックをチャット上で実データに適用した結果。スクリプトのゲーティング/機械判定とも整合する。

**軸A/B（pid=1「内閣人事局経費」の支出先より）**

| 支出先 | 入力（status / CN / 契約概要） | identify | purpose | 判定根拠 |
|--------|-------------------------------|:--:|:--:|----------|
| 三井不動産ビルマネジメント株式会社 | valid / あり / 研修会場等の借上げ | 3 | 3 | 正式法人名＋法人番号、契約概要が具体的 |
| 麹町税務署 | gov / なし / 講師謝金等の源泉徴収 | 2 | 2 | 国の出先機関で実体明確、CN無しは減点せず／用途は概ね具体 |
| 個人A | invalid / なし / 講師への謝金及び旅費 | 1 | 2 | 仮名で個人特定不可だが講師個人と文脈で判別 |
| その他 | invalid・opaque / なし / 講師への謝金及び旅費 | 0 | 2 | 不透明名で特定不能（スクリプトではゲート即決 identify=0） |

**軸E（事業テキスト）**

| 事業 | effective(0-10) | 判定根拠（要旨） |
|------|:--:|----------------|
| pid=1 内閣人事局経費 | 7 | 対象（全府省職員・新任幹部等）・課題（受験者減/離職増）・活動（各種研修）が具体的。国民便益はやや間接 |
| pid=4 情報システムの整備 | 7 | 対象・課題（システム移管・統括）・活動（プロジェクト監理→予算反映）が具体的。国民便益（デジタルサービス向上）は間接的 |

→ いずれもスクリプトのレベル定義どおりに判定でき、**チャットによる処理は方式として成立**する。

### 4-3. チャット方式の限界（全件確定には不適）

- **件数**: 軸A/B 87,653 ＋ 軸E 5,642。チャットの1ターンで安全に処理できる件数を大きく超える。
- **再現性・冪等性**: スクリプトは `temperature=0`・JSON強制・寛容パーサ・永続キャッシュ・金額加重集計を機械的に保証。チャット手作業ではこの一貫性・集計が担保しにくい。
- **集計**: 事業スコアは支出先の金額加重平均。判定後の機械集計はスクリプトの役割。

### 4-4. 推奨の進め方

1. **チャットでプロンプト/出力品質を検証**（本書 §4-2 のような少数サンプル）。必要ならルーブリックやモデルを調整。
2. 妥当性を確認後、**スクリプト＋`OPENROUTER_API_KEY` で全件確定生成**。キャッシュで中断耐性あり。
3. `.gz` 化して Git 反映 → `/quality` で2024の新軸表示・分布を確認。

## 5. 留意点

- `OPENROUTER_API_KEY` 未設定で実行すると現状と同じ heuristic 結果になる（2024問題は解消しない）。本番は実LLM必須。
- モデルは2025生成と揃えると年度間の基準が一致（2025は `google/gemini-3.5-flash`）。キャッシュ名はモデル末尾セグメントで正規化されるため、同一実モデルならプロバイダ跨ぎで再利用可。
- `data/cache/` は `/data/` 配下で gitignore 済（追加対応不要）。
- 軸D（構造）は総合不算入の参考表示。再生成後も総合は A/B/C/E のみで再配分される。
- 軸Eは**実測成果でなく成果設計（意図）の明確さ**。UI注記「※実測成果ではない」と整合させる。
