# Claude Code 設定の最適化

> 作成日: 2026-05-29 06:34 (JST)
> 目的: このプロジェクトの Claude Code 設定を整理・最適化する。初心者向けに用語解説つきで記録し、あとから復習できるようにする。

---

## 0. まず用語を整理（初心者向け）

Claude Code の挙動は、いくつかの仕組みの組み合わせで決まります。混同しやすいので最初に整理します。

| 用語 | これは何か | どこに置く |
|------|-----------|-----------|
| **CLAUDE.md** | プロジェクトのルール・前提を書いた指示書。毎回読み込まれる | リポジトリ直下 / `.claude/` |
| **settings.json** | 権限・hook・モデル等の設定ファイル（**チームで共有**＝Git管理する想定） | `.claude/settings.json` |
| **settings.local.json** | 自分のマシン専用の設定（**Git管理しない**＝`.gitignore` 済み） | `.claude/settings.local.json` |
| **permissions（権限）** | 「このコマンドは確認なしで実行してよい」という許可リスト | settings の `permissions.allow` |
| **hook（フック）** | 「編集のあとに自動で○○する」など、決まったタイミングで自動実行される処理。**Claude ではなくツール本体が実行する**ので、メモリやCLAUDE.mdでは代用できない | settings の `hooks` |
| **skill（スキル）** | 特定作業の手順をまとめたもの。`/名前` で呼び出す | 標準搭載 or `.claude/commands/` |
| **plugin（プラグイン）** | skill / agent / hook / command を束ねた配布パッケージ | `/plugin` で導入 |

## 1. 質問と回答

**質問:** このプロジェクトの Claude Code 設定を最適化するような「公式プラグイン」はある？

**回答:** **専用プラグインは存在しない。**
公式マーケットプレイス（`/plugin` で見られる）のプラグインは「ワークフロー用の機能パック」であって、設定を最適化するツールではありません。
代わりに「設定の最適化」は **Claude Code 標準搭載のスキル** が担当します。プラグインを入れるより、こちらを使うのが正解です。

- `/fewer-permission-prompts` … 権限プロンプトを減らすため、許可リストを整理して提案してくれる標準スキル
- `update-config` スキル … hook や権限などの設定変更を行う標準スキル

## 2. 現状の調査結果

調査コマンド: `.claude/` 配下と各 settings ファイルを確認。

```
.claude/
├── CLAUDE.md                  # ワークスペースルール（Git管理）
├── commands/                  # 9個のスラッシュコマンド（Git管理）
│   ├── data-update.md  pipeline.md  plan.md  pr.md
│   ├── quality-check.md  resolve-reviews.md  review.md
│   └── sankey.md  sync.md
└── settings.local.json        # ★Git管理されていない（gitignore済み）
```

わかったこと:

1. **共有 `settings.json` が無い**
   権限設定は `settings.local.json`（＝このマシン専用・Git管理外）にしか存在しない。
   → 他の人や CI では権限が引き継がれず、毎回プロンプトが出る状態。

2. **`settings.local.json` の許可リストが肥大化＆汚れている**
   - 50件以上の許可ルールがある。
   - そのうち数件は、`gh issue create` の**本文（バグ報告・機能要望の長文）がまるごと許可ルールとして登録**されている。これは一度きりの使い捨てで、二度と同じ文字列にマッチしない「ゴミ」。

3. **hook が未設定**
   CLAUDE.md には「変更後に `npm run lint` / `npx tsc --noEmit` を実行」というルールがあるが、これは手動頼み。hook 化すれば自動実行できる。

## 3. 最適化プラン

優先度順。

### ステップ A: 許可リストの整理（`/fewer-permission-prompts`）
- 過去セッションを走査し、安全な read-only 系コマンドを整理した許可リストを提案してもらう。
- 使い捨ての `gh issue create` フルbodyルールを削除。

### ステップ B: 共有 `settings.json` の作成（Git管理）
- プロジェクト共通で安全な権限だけを `.claude/settings.json` に切り出す。
- 個人マシン固有のものは `settings.local.json` に残す。
- → チーム・CI でもプロンプトが減る。

### ステップ C: hook の追加（任意・`update-config`）
- 編集後に lint / 型チェックを自動実行する hook を検討。
- ※自動実行は便利だが重くなる場合もあるので、入れるかは要相談。

## 4. 会話の再開（CLI を閉じても履歴は消えない）

Claude Code を閉じたあとでも、前の会話を再開できます。

| 方法 | コマンド / 操作 | 用途 |
|------|----------------|------|
| **直前のセッションを続ける** | `claude --continue`（`claude -c`） | 一番手軽。さっきの続きをすぐ再開 |
| **過去の一覧から選ぶ** | `claude --resume`（`claude -r`） | 複数の履歴から選んで再開 |
| **セッション中に選ぶ** | `/resume` | 起動中に別の会話へ切り替え |

仕組みと注意点:

1. **履歴はローカルに JSON で保存される。**
   このプロジェクトの場合 `~/.claude/projects/-Users-igomuni-MyGitHub-marumie-rssystem/` 配下。
2. **プロジェクト（作業ディレクトリ）ごとに履歴が紐づく。**
   再開は同じディレクトリで実行するのが基本。別の場所で起動すると、そのプロジェクトの履歴は出てこない。
3. **長くなった会話は自動で要約（圧縮）される。**
   再開時は「要約＋残りの文脈」が引き継がれるので、続きから作業できる。

> 要するに「閉じても消えない、`claude -c` か `claude -r` で戻れる」。

## 5. モデルの切り替えと `/effort`（コストと品質の調整）

「**誰に頼むか（モデル）**」と「**どれだけ考えてもらうか（effort）**」は独立した2軸。組み合わせて調整する。

### 5-1. モデルの切り替え

| 方法 | やり方 | 説明 |
|------|--------|------|
| セッション中に切替 | `/model` | 選択画面が開く。Opus / Sonnet / Haiku / Default から選ぶ。その場限り |
| 直接指定 | `/model claude-opus-4-8` など | ピッカーを出さず即切替 |
| 起動時に指定 | `claude --model claude-opus-4-8` | 最初から特定モデルで開始 |

現在のモデルID:
- **Opus 4.8** `claude-opus-4-8` … 最上位
- **Sonnet 4.6** `claude-sonnet-4-6` … 速度・コストのバランス
- **Haiku 4.5** `claude-haiku-4-5-20251001` … 軽量・最速

**`/fast`（Fast モード）**: Opus を**出力高速化したまま**使う機能。小さいモデルに格下げするわけではない。Opus 4.8 / 4.7 / 4.6 で利用可。

### 5-2. `/effort`（推論の深さ＝考える量）

回答前に**どれだけ考えるか**を制御。深いほど品質↑だが、遅く・トークン消費↑。

| レベル | 向く作業 |
|--------|---------|
| **low** | リネーム・grep・簡単な質問。速くて安い |
| **medium** | 通常のコーディング。多くの場合のデフォルト |
| **high** | 複雑なタスク向けの深い推論 |
| **max（xhigh）** | トークン制約なしの最大推論。そのセッション限り |
| **auto** | デフォルトに戻す |

使い方: `/effort` を押すたびに循環、または `/effort high` のように直接指定。

### 5-3. 使い分けの目安

| 状況 | おすすめ |
|------|---------|
| 軽い修正・調査 | Sonnet/Haiku + `/effort low` |
| 通常の実装 | Opus + `/effort medium` |
| 設計判断・難しいバグ | Opus + `/effort high`〜`max` |
| 品質そのまま速くしたい | `/fast` |

## 6. 今の設定・コストの確認方法

「今どのモデル・どれだけ使ったか」を確認するコマンド。

| コマンド | 何がわかるか |
|---------|------------|
| **`/usage`**（別名 `/cost` `/stats`） | セッションのコスト・プラン使用量・利用状況。APIユーザーはトークン消費と推定金額、Pro/Maxプランは利用枠に対する進捗 |
| **`/status`** | 現在のモデル・作業ディレクトリ・アカウント情報など現在の状態 |
| **`/context`** | コンテキストウィンドウの使用内訳（system prompt / tools / memory / 会話履歴）を色分け表示。重い場合は最適化の提案も |
| **`/config`** | 設定の確認・変更 |
| 画面下のステータスライン | 現在のモデルと、コンテキスト使用率（％）が常時表示される |

ポイント:
- **コストを見たい → `/usage`**（このプランがどう課金されているかが出る）
- **コンテキストが重くなってきた → `/context`** で内訳を確認し、必要なら `/compact` や新規セッションを検討
- **今のモデルを確認 → `/status` か `/model`**、画面下のステータスラインでも常時見られる
- CLI外で日別・プロジェクト別に集計したい場合は、サードパーティ製の `ccusage`（ローカルのJSONLを解析）もある

## 8. `settings.local.json` のクリーンアップ

> ⚠️ `settings.local.json` は**個人マシン専用・Git管理外**。ここを削ると、その許可は失効し再度プロンプトが出る。データパイプライン作業に必要なものは残す前提。

| 種別 | 該当エントリ例 | 対応 |
|------|---------------|------|
| **ゴミ（使い捨て・断片）** | 巨大 `gh issue create` 本文、`/tmp` 一時ファイル参照、`for`/`do`/`done` 断片、`Bash(# …)` コメントブロブ、壊れたルール | ✅ **削除済み（18件）**。実コマンドにマッチしない or 広いワイルドカードでカバー済みのため機能影響ゼロ |
| **共有へ昇格済み** | `npx tsc:*` `npm run lint:*` | ✅ 共有 `settings.json` に安全な形（`--noEmit` / `lint` 固定）で追加済み。local 側は重複だが害はない |
| **任意コード実行の広いワイルドカード** | `python3:*` `node:*` `npx tsx:*` `npx ts-node:*` `node -e:*` | ⏸ **温存**。本来は危険だが、このプロジェクトのパイプライン作業で多用するため。狭いスクリプト単位に絞る案は要相談 |
| **変更系（正常）** | `git add/commit/push:*` `gh pr create:*` 等 | ⏸ 温存。個人の利便で許可している分 |

→ 低リスクなゴミ掃除は完了（163→145件）。残るは「任意コード実行ワイルドカードを狭めるか」の判断だが、パイプライン作業の利便とのトレードオフなので現状維持でも可。
