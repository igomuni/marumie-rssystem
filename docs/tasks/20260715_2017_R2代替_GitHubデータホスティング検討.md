# R2 代替: GitHub でのデータホスティング検討

作成日: 2026-07-15（JST）
目的: Cloudflare R2 がクレジットカード登録必須だったため、代替として **GitHub をデータ配信基盤に使えるか** を検討する。
関連: `docs/tasks/20260613_0405_静的データR2offloadと帯域試算.md`（前提サイズ・帯域試算はこちら）

---

## 0. 結論（先出し）

**可能。推奨は「jsDelivr CDN + 既存のコミット済み `.gz` + クライアント側解凍（DecompressionStream）」。**

- リポジトリは公開（未認証 raw アクセスで 200 を実機確認済み）なので前提を満たす。
- 配信対象の `.gz` は**すでに全部 Git にコミット済み**（最大 4.5MB、jsDelivr の 20MB/ファイル制限内）。パイプライン変更・アップロード作業が一切不要。
- URL にコミット SHA を埋めれば **immutable 長期キャッシュ**になり、6/13 doc で懸念した「年度のみのファイル名でキャッシュが割れない」問題も同時に解決する。
- カード登録・アカウント作成・トークン発行・バケット運用がすべて不要（R2 案のインフラ手作業がゼロになる）。

## 1. GitHub 配信の選択肢 4 つ

| 案 | 配信元 | ファイル上限 | 帯域 | CORS | キャッシュ | 評価 |
|---|---|---|---|---|---|---|
| A | **jsDelivr** (`cdn.jsdelivr.net/gh/...`) | **20MB/ファイル** | OSS 向け無制限 | `*` | バージョン固定で immutable | ◎ 推奨 |
| B | **GitHub Releases** アセット | 2GB/ファイル | 公式上限なし | `*` | リダイレクト先の署名URL、長期キャッシュ不向き | ○ bulk 配布向け |
| C | **raw.githubusercontent.com** | 100MB（Git 上限） | 非公式・高負荷で throttle | `*`（実機確認済） | `max-age=300` 固定 | △ 試作・フォールバック |
| D | **GitHub Pages**（データ専用リポジトリ） | 100MB/ファイル、サイト 1GB | **100GB/月（ソフト）** | 設定不可（同一オリジン不可なので実質 `*` 頼み不可） | `max-age=600` 固定・カスタムヘッダ不可 | △ Vercel と同じ 100GB 制限で妙味薄 |

### 案A: jsDelivr（推奨）

- 公開 GitHub リポジトリのファイルをそのまま CDN 配信する無料サービス。東京 PoP あり。OSS プロジェクトのデータ配信は本来用途で ToS 上も適合。
- URL 形式: `https://cdn.jsdelivr.net/gh/igomuni/marumie-rssystem@<ref>/public/data/<file>`
  - `<ref>` = コミット SHA / タグ / ブランチ。**SHA・タグ固定なら永続キャッシュ**（jsDelivr 側もエッジで permanent 扱い）。`@main` は 12h キャッシュで更新遅延があるため本番では使わない。
- 制約: **20MB/ファイル**（超過は 403）。raw JSON の subcontracts 30MB は不可だが、**`.gz`（≤4.5MB）を配って client 解凍すれば全ファイル余裕**。
- リスク: サードパーティ無料サービス（障害・サ終の可能性）。→ 後述のフォールバック（同一オリジン `/data`）で吸収。ファイルは `public/` に残るので Vercel 配信は常に生きている。

### 案B: GitHub Releases

- ページロード用というより、**bulk `.gz` 直リンク DL の受け皿**として有効（6/13 doc §6-2 の帯域ベクタを Vercel から切り離せる）。
- リリースにアセット添付（2GB/ファイルまで）。帯域上限の公式規定なし。CORS も許可される。
- 難点: `releases/download/...` → 署名付き URL へのリダイレクトが挟まり、エッジ長期キャッシュが効きにくい。初回表示のデータ取得には案A より不利。

### 案C: raw.githubusercontent.com

- 実機確認結果（本セッション）:

```text
HTTP/2 200
access-control-allow-origin: *
content-type: application/octet-stream
cache-control: max-age=300
```

- CORS は通るが、GitHub は本番 CDN 用途を非推奨としており、無認証アクセスは高負荷時に throttle されうる。`max-age=300` 固定で長期キャッシュ不可。**プロトタイプ検証と非常用**の位置づけ。

### 案D: GitHub Pages

- ソフトリミットが **100GB/月**で Vercel Hobby と同じスケール。しかもカスタムヘッダ（`Cache-Control`・`Content-Encoding`）が一切設定できず、データ専用リポジトリの追加運用も発生する。**わざわざ選ぶ理由がない**。

## 2. 技術上の唯一の論点: `Content-Encoding: gzip` が付けられない

R2 案では「事前 gzip 済みバイトを `Content-Encoding: gzip` で置き、ブラウザに透過解凍させる」設計だった（R4 対策）。GitHub 系はどの案もレスポンスヘッダを制御できないため、`.gz` は `application/octet-stream` のバイナリとして届く。

→ **クライアント側で解凍する**。全モダンブラウザ（Chrome 80+ / Safari 16.4+ / Firefox 113+）にネイティブ実装済みの `DecompressionStream` を使えば追加ライブラリ不要:

```ts
async function fetchGzJson<T>(url: string): Promise<T> {
  const res = await fetch(url);
  if (!res.ok || !res.body) throw new Error(`fetch failed: ${res.status}`);
  const stream = res.body.pipeThrough(new DecompressionStream('gzip'));
  return new Response(stream).json();
}
```

- wire サイズは gzip 実測値（sankey 1.43MB / subcontracts 4.45MB / quality-scores 0.63〜1.0MB）。Vercel の brotli（1割弱小さい）にわずかに劣るが誤差レベル。
- 解凍コストはメインスレッドをブロックしないストリーム処理で、現行の「Vercel が br 解凍させる」のとユーザー体感は同等。

## 3. 実装スケッチ（R2 案のコード側プランをほぼ流用）

1. `app/lib/data-url.ts` を新設:

```ts
export function dataUrl(file: string): string {
  const base = process.env.NEXT_PUBLIC_DATA_BASE_URL; // 未設定ならローカル/フォールバック
  return base ? `${base}/${file}.gz` : `/data/${file}`;
}
```

2. 本番の base URL は**デプロイコミットの SHA で自動固定**できる:

```ts
// next.config.ts などで組み立て
const sha = process.env.VERCEL_GIT_COMMIT_SHA;
// → https://cdn.jsdelivr.net/gh/igomuni/marumie-rssystem@<sha>/public/data
```

   - データとコードが常に同一コミットで一致し、デプロイごとに URL が変わるので **immutable キャッシュと更新反映が両立**する。環境変数の手動更新も不要。
3. fetch 箇所 4 ファイル（`app/sankey-svg/page.tsx:1337`、`app/subcontracts/page.tsx:217`、`app/subcontracts/[projectId]/page.tsx:1210`、`app/mof-budget-overview/transfer-detail/page.tsx:18`）を `dataUrl()` + `fetchGzJson()` に差し替え。base URL 側は `.gz` + 解凍、フォールバック時は素の `/data/*.json` を従来どおり取得。
4. **フォールバック**: jsDelivr への fetch が失敗したら同一オリジン `/data/*.json` に自動リトライ（ファイルは `public/` に残り続けるため Vercel 配信は常時利用可能）。R6（可用性二重化）を R2 案より簡単に満たせる。
5. `app/layout.tsx` に `<link rel="preconnect" href="https://cdn.jsdelivr.net">` を追加（R1 対策）。

## 4. R2 案との比較まとめ

| 観点 | R2（独自ドメイン+Cache Rules） | GitHub + jsDelivr（案A） |
|---|---|---|
| カード登録 | **必須**（今回のブロッカー） | 不要 |
| インフラ手作業 | バケット・ドメイン・CORS・トークン | **ゼロ**（既存リポジトリのまま） |
| アップロード工程 | パイプライン末尾に追加必要 | 不要（コミットが配置を兼ねる） |
| 帯域上限 | egress 無料で事実上なし | OSS 向け無制限（明文の帯域課金なし） |
| キャッシュ制御 | 自由（immutable 可） | SHA 固定 URL で実質 immutable |
| 圧縮配信 | `Content-Encoding: gzip` で透過解凍 | クライアント解凍（DecompressionStream） |
| 可用性 | Cloudflare 依存 | jsDelivr 依存だが**同一オリジンフォールバック内蔵** |
| 撤退容易性 | 高 | 高（`NEXT_PUBLIC_DATA_BASE_URL` を消すだけ） |

## 5. 留意点・未確認事項

- **jsDelivr への疎通は本作業環境から未確認**（サンドボックスのネットワークポリシーで `cdn.jsdelivr.net` への CONNECT が 403。jsDelivr 側の拒否ではない）。着手前にブラウザで `https://cdn.jsdelivr.net/gh/igomuni/marumie-rssystem@main/public/data/sankey-svg-2024-graph.json.gz` が取得できることだけ確認するとよい。
- リポジトリを **private 化すると案A/C は即死**する（公開が前提条件）。
- 6/13 doc の結論（「Brotli が効いており当面 R2 offload 不要、Usage を観測してから動く」）は今も有効。本件は「動くと決めたときの実装先が R2 → GitHub に変わる」という話であり、**着手タイミングは引き続き Vercel の帯域 Usage 次第**でよい。準備コストが R2 案より大幅に小さいので、先行実装して環境変数で寝かせておく選択も安価。

## 6. 次のアクション（実装する場合）

1. ブラウザから jsDelivr URL の疎通確認（5 分）
2. `dataUrl()` + `fetchGzJson()` + フォールバックの実装（本 doc §3、fetch 4 箇所）
3. `preconnect` 追加
4. Vercel 環境変数（または `VERCEL_GIT_COMMIT_SHA` 組み込み）を設定してデプロイ → 実機で wire サイズ・レイテンシ確認
