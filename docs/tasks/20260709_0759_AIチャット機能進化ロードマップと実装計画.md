### v1 の現在地（コードで確認済み）

- エージェント: `app/lib/ai/sankey-chat-agent.ts`（ツール4種: `run_sankey_query` / `search_projects` / `search_recipients` / `submit_result`、LLM往復6回・ツール8回上限、LLM は DI）
- API: `app/api/ai/sankey-chat/route.ts`（ガード + OpenRouter 接続）、UI: `client/components/SankeySvg/AiChatPanel.tsx`
- できること: 自然言語 → フィルタ条件 → 結果カード → URL 反映。できないこと: **データの中身に答える**（金額の内訳・品質・再委託・年度比較・使途）

## 1. ロードマップ全体像

進化の軸は「**フィルタ設定アシスタント → データに答えるアシスタント → 問いを提案するアシスタント → 公開**」の4段。各 Phase は独立に価値が出る単位で区切る。

```text
Phase 1  深掘りツール拡張        「この事業の品質スコアは?」「NTTデータへの支出は?」に答える
Phase 2  応答UXの進化            解釈宣言・深掘りメニュー・ストリーミング
Phase 3  内容ベース検索の取り込み  年度比較・概要検索・使途検索・集中度（API拡張と連動）
Phase 4  提案システム連携         「無駄遣いっぽいの教えて」→ highlights ベースの観測可能な異常度で回答
Phase 5  公開（Vercel搭載）       レートリミット・コスト監視・利用分析。別途ユーザー判断
```

依存関係: Phase 1・2 は v1 の上に独立に積める。Phase 3 は API/パイプライン側の先行実装が前提（チャットへの露出自体は薄い）。Phase 4 は highlights バッチが前提。Phase 5 はすべての利用実績を見てからの判断ゲート。

## 2. ワークパッケージ別実装計画

実装担当の割り当て方針:

- **Sonnet 向け（プレスクリプティブ計画）**: 既存パターンの複製で完結する作業。ファイル・手順・受け入れ基準まで固定し、設計判断を残さない
- **Opus 向け（ゴール定義計画）**: 新規パイプライン・スキーマ設計・ストリーミング等、設計裁量が必要な作業。ゴール・制約・非目標を定義し、設計は docs/tasks に起こしてから実装する

共通ルール（全 WP に適用）:

- レイヤー規約厳守: ツール実行は `app/lib/` の既存関数へ委譲、`app/lib/ai/` に fetch/React を持ち込まない
- ツール追加の定型: `TOOLS` 配列に JSON Schema 定義 → ループの `switch` に実行分岐 → システムプロンプトに使い方を追記 → `MAX_TOOL_CALLS` の妥当性を再確認
- 金額は1円単位。multi-block spending は `filter().reduce()` で合算（CLAUDE.md 既知バグ）
- 完了時 `/quality-check`（lint + tsc）。curl での Step 1 相当の応答品質確認（設計書 §段階分割の流儀）を踏襲
- PR は draft・ユーザー許可後のみ

### WP1-1: 深掘りツール4種の追加【Sonnet 向け】

「絞り込んだ後にそのまま質問できる」を実現する最小拡張。結果カード（フィルタ適用）とテキスト回答（深掘り）の2モード共存になる。

| 新ツール | 委譲先（すべて実装済みローダ/lib） | 返す内容 |
|---------|--------------------------------|---------|
| `get_project_detail` | `loadQualityScores(year)` から pid 照合 + project-details ローダ | 目的・概要・現状課題・予算/執行・府省庁 |
| `get_quality_scores` | `loadQualityScores(year)`（pids 指定・上限10件） | 6軸スコア + 総合。スコアの意味（説明の質の評価であり事業の善悪ではない）を応答規範としてプロンプトに明記 |
| `get_recipient_detail` | `loadRecipientIndex(year)` の法人番号キー照合 | 直接/再委託受領額・関与事業数・上位事業 |
| `get_subcontract_chain` | subcontracts 系ローダ（`app/lib/api/` 既存） | 指定事業の再委託構造の要約（深度・上位チェーン） |

手順:

1. `app/lib/ai/sankey-chat-agent.ts`: `TOOLS` に4定義追加、`switch` に実行関数追加（`executeSearchProjects` と同じ形の薄い関数を4つ）。返却 JSON は件数・文字数をクランプ（1ツール応答 4,000 字目安）
2. システムプロンプトの「手順」に深掘りモードを追記: フィルタ要求なら従来通り `submit_result`、**データへの質問ならツールで調べてテキストで回答（`submit_result` は呼ばない）**。数値は必ずツール応答から転記（捏造禁止）
3. 上限調整: `MAX_TOOL_CALLS` 8 → 10、`MAX_LLM_ROUNDS` 6 → 8（深掘りは往復が1〜2回増える）
4. `AiChatPanel.tsx` は変更不要（テキスト回答は既存の message 表示に乗る）

受け入れ基準: curl で「マイナンバー関連に絞って」→ 結果カード、続けて「その中で品質スコアが低いのは?」→ スコア付きテキスト回答、の2往復が成立する。

### WP1-2: 応答品質の評価ハーネス【Sonnet 向け】

設計書 Step 1 の curl 評価を再現可能にする。`scratchpad` ではなく `scripts/dev/`（または `docs/tasks/_assets/`）に、代表プロンプト10件（実験①②の要求文 + 深掘り2往復シナリオ）を流して message/result/toolCalls を記録する小スクリプトを置く。以後の全 WP の回帰確認に使う。モデル差し替え評価（§4）もこれに乗せる。

### WP2-1: 解釈宣言と深掘りメニュー【Sonnet 向け】

内容ベース検討 §2 の UX原則を v1 に実装する。

1. `types/sankey-ai-chat.ts`: `SankeyChatResult` に `interpretation?: string`（曖昧語をどう解釈したか）と応答に `suggestions?: string[]`（次の質問 2〜3個）を追加
2. `submit_result` ツールの引数に `interpretation` / `suggestions` を追加し、システムプロンプトに規範を追記: 曖昧語（「子育て」「大型」等）を含む要求では解釈を必ず宣言する。suggestions は**追加ツールなしで答えられる問いのみ**提案する（現有ツールで answerable な範囲に限定）
3. `AiChatPanel.tsx`: 結果カードに解釈行を表示、suggestions をチップ（タップで入力欄に転記して送信）として描画。fetch はせずコールバック経由（レイヤー規約）

受け入れ基準: 「子育て支援って何に使われてるの?」→ 解釈宣言 + 件数 + 図ボタン + 深掘りチップ2〜3個が1応答で出る。

### WP2-2: ストリーミング応答【Opus 向け】

v1 で明示的にスコープ外とした項目。ゴール: 送信後の無反応時間（現状 LLM 数往復分 = 10〜30秒）を「進行が見える」体験にする。

- 制約: `app/lib/ai/` の DI 構造を保つ（ストリーム処理をどの層に置くかが主要な設計判断）。全文トークンストリームは必須ではなく、**ツール実行の進行イベント**（「クエリ実行中… 12件マッチ」）だけでも体験目標は達成できる — SSE でイベント種別を設計すること
- 非目標: 会話のサーバー側セッション化（ステートレス維持）
- 先に docs/tasks に設計（イベントスキーマ・route の SSE 化・AiChatPanel の受信状態機械）を起こし、ユーザーレビュー後に実装

### WP3-1: 年度間比較 + 集中度指標のツール露出【Sonnet 向け・API拡張完了が前提】

前提: 20260705_1544 §3-1（`compareYears`）と §3-4（`topShare1/3` の summary 同梱）の `app/lib/sankey-query.ts` 側実装。**2026-07-09 時点で未実装をコード確認済み**（`compareYears` / `topShare` / `scope=details` いずれも grep ヒットなし）。先に API 側 WP（Opus 向け・20260705_1544 の §3-1/3-4/3-2 が仕様の出発点）を起票する必要がある。

チャット側の作業は薄い: `run_sankey_query` の返却 summary に topShare が自動で乗る（プロンプトに読み方を1行追記）+ `compare_years` ツールを1つ追加（同一 SankeyQuery の2年度差分を返す既存関数へ委譲）。システムプロンプトに「予算年度 = 事業年度 − 1」の年度ズレ注記を追加。

受け入れ基準: 「マイナンバー関連は去年から増えた?」→ 差分（増減・新規/消滅）を含むテキスト回答。

### WP3-2: 概要全文検索のツール反映【Sonnet 向け・scope=details 実装が前提】

`search_projects` ツールに `scope` 引数（`name` | `details`）を追加し、既存 `project-search.ts` の scope=details 実装へ委譲。プロンプトの手順2に「事業名で0件なら scope=details で概要を検索する」を追記。計上のねじれ（例: 国税システム群がデジタル庁計上）に届くようになる。

### WP3-3: 使途検索ツール `search_spending`【Opus 向け】

ゴール: 「広報にいくら使われている?」「システム改修を受注しているのは誰?」という**使途起点の問い**に答える。

- 前提パイプライン: `scripts/generate-spending-search-index.ts`（quality-recipients の `role`/`cc`/ブロック名から行インデックス生成、74,639行/年）。新規CSV処理は不要、既存成果物からの導出 — `/pipeline` スキル準拠
- 設計判断が必要な点: インデックスの粒度と圧縮（.gz 運用に乗せるか）、ローダのメモリキャッシュ戦略、チャットツールの返却形（マッチ行 + 金額合計 + 事業別集計のクランプ）
- 内容ベース検討 §1-1 が仕様の出発点。API `/api/search/spending` とチャットツールは同じ lib 関数を共有すること（別実装禁止）

### WP3-4: AIタグ付け → `filter.tags`【Opus 向け・実行判断は別途】

内容ベース検討 §1-2。支出種別タグ・政策分野タグのバッチ付与 → `SankeyQuery.filter.tags` 追加 → チャットで「広報費だけの図」が成立する本命。コスト見積もり（quality-AI 実績から算出）とタグ体系案を先に docs/tasks に出し、ユーザーの実行判断を待つ。embedding 検索は引き続き保留。

### WP4-1: highlights 連携 —「興味深い事業」の提案【Opus 向け】

ゴール: 「無駄遣いっぽいのを教えて」型に、**観測可能な異常度**（前年比急増・「その他」比率・集中度・品質スコア低×予算大・執行/予算乖離・再委託深度）で答える。

- 前提: `scripts/generate-highlights.ts` + `highlights-{year}.json` + `/api/highlights`（内容ベース検討 §3 系統1。LLM 不要で着手可能）

## 4. チャット実行モデル（LLM）への含意 — hy3 レポートから

- **既定 `google/gemini-3.5-flash` を維持**。品質スコアでの判定品質実績に加え、チャットはツール呼び出しの正確さ（JSON Schema 準拠・金額の桁変換）が要で、中央寄せ傾向の無料モデル（hy3 等）に差し替える動機がない
- 無料モデルの用途は**疎通・プロンプト検証まで**（品質比較レポートの結論をチャットにも適用）
- コスト圧縮が必要になった場合の候補は `deepseek-v3.2`（20260620_0950 のコスパ評価）。ただし差し替え前に WP1-2 ハーネスで**ツール呼び出し互換性**（プロバイダによる `response_format`/tool calling の対応差は hy3 レポート §1 で実証済みの罠）を必ず疎通確認する
- `SANKEY_AI_CHAT_MODEL` 環境変数で差し替え可能な構造は既にあるため、実装変更は不要
