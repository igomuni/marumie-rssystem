# sankey-svg レスポンシブ対応設計（Issue #195）

- 対象: [/sankey-svg](app/sankey-svg/page.tsx)（メインページ）
- 関連 Issue: [#195](https://github.com/igomuni/marumie-rssystem/issues/195)
- 関連調査: [docs/tasks/20260506_0802_sankey-svg_フォントサイズ調整_独立調査.md](20260506_0802_sankey-svg_フォントサイズ調整_独立調査.md)
- 検証ツール: [tests/e2e/responsive-investigation.spec.ts](../../tests/e2e/responsive-investigation.spec.ts)（本タスクで追加）
- 検証アセット: [docs/tasks/_assets/20260528_responsive/](_assets/20260528_responsive/)

## 目的

- 「タブレット・スマホで /sankey-svg を開いた利用者」が「上部UIと列見出し・本体ラベルが重なって読めない／操作できない状態」を解消するため。
- 「PCで閲覧する利用者」が「閲覧環境（ノートPC・TVモニター・スクリーン投影）に応じた情報密度で表示できるようになるため」。

Issue #195 の「やりたいこと」のうち、本設計は **狭幅レイアウト再設計** と **表示モードプリセットの基盤** を扱う。タッチ操作のヒットエリア最適化は構造変更とセットで進める。

## 現状の確認

### Playwright による幅別キャプチャ（2026-05-28）

| 幅 | 観察された問題 |
|----|----------------|
| 1440x900 | 問題なし。検索・年度・フォント/TopN/設定の3クラスタが1段で収まる。 |
| 1280x800 | ほぼ問題なし。 |
| 1024x768 (タブレット横) | 上部ツールバー残り余白が乏しい。列見出しは表示できるがフォント縮小が始まる。 |
| 768x1024 (タブレット縦) | 上部ツールバーが破綻直前。検索ボックス幅が縮み、フォント・TopN・設定アイコン群がぎりぎり収まる。列見出しは生きているがクリックターゲットが小さい。 |
| 390x844 (iPhone) | **検索ボックスと右側のアイコン群が同一行上でほぼ接触**。列見出し（省庁／事業／…）が極小フォントで上部に並ぶ。サイドパネル前提のレイアウトのため詳細閲覧導線が不明瞭。 |
| 360x800 (Android狭幅) | 390pxと同じ問題がさらに悪化。 |

> 横スクロールは発生していない（`documentSize.w === viewport.w`）。問題は「同一行に詰め込まれた要素のオーバーラップ／極小化」と「サイドパネル前提のUI導線がモバイルで機能しない」点に集約される。

### 現状の構造上の制約

[app/sankey-svg/page.tsx](app/sankey-svg/page.tsx) のうち、レスポンシブに関わる主な定数・分岐：

- `SEARCH_BOX_RESERVE`（[app/sankey-svg/page.tsx:729](app/sankey-svg/page.tsx#L729)）: `svgWidth < 1100 ? 92 : 56`。**唯一の幅分岐**で、上部ツールバーが2段に折り返すケースを想定して描画領域の上余白を確保している。
- `SIDE_PANEL_WIDTH_DEFAULT = 310`（[app/sankey-svg/page.tsx:90](app/sankey-svg/page.tsx#L90)）: 固定値。狭幅でも310pxを占有する前提。
- `COLUMN_LABEL_FONT_PX_DEFAULT = 12`（[app/sankey-svg/page.tsx:71](app/sankey-svg/page.tsx#L71)）: `scaleFont()` で `fontScale` をかける。タッチ向けの下限保証はない。
- 上部ツールバーは `position: absolute` の小要素を `top: 12px` 付近に並べる構成（[app/sankey-svg/page.tsx:2290](app/sankey-svg/page.tsx#L2290) 近辺）。**フレックス/グリッドの折り返しではない**ため、幅が縮むと衝突する。
- レイアウト幅・高さ判定は `ResizeObserver` 経由で `svgWidth/svgHeight` を取るのみ。「現在の表示モード」概念は存在しない。

## 設計方針

### 1. 表示モードという第1級概念を導入する

幅・閲覧目的に応じた **DisplayMode** を `app/sankey-svg/page.tsx` の状態として持つ。

- 値の例: `compact-mobile` / `compact-tablet` / `standard-desktop` / `dense-laptop` / `presentation-tv` / `projection`
- 決定ロジック:
  1. URLクエリ・ローカルストレージで明示指定があればそれを採用（プリセット切替UIから設定）
  2. なければ `containerWidth`（既に取得済み）と `matchMedia('(pointer: coarse)')` から自動判定
- モード値はトークン定数オブジェクトに紐付ける（後述「2. レイアウトトークン化」）。

これにより Issue #195 「将来的に表示目的別のプリセットを持てる設計」の基盤を作る。実装は「自動判定 + 手動上書き」のシンプル構成に留め、プリセットの拡充は後続タスクへ。

### 2. レイアウトトークンを集約する

現在 page.tsx 各所に散らばっている「幅・余白・フォントサイズ・サイドパネル幅・列ラベル余白」の数値を、モードごとのトークンオブジェクトにまとめる（types に型定義、`app/lib/` または `app/sankey-svg/_layout.ts` に値を配置）。

例（値は実装時に確定）:

```
{
  topToolbarLayout: 'single-row' | 'two-row' | 'sheet',
  searchBoxReservePx: number,
  sidePanelMode: 'docked' | 'bottom-sheet' | 'fullscreen',
  sidePanelWidthPx: number,         // docked 時のみ意味を持つ
  columnHeaderMode: 'with-amount' | 'name-only' | 'hidden',
  columnLabelFontPx: number,
  controlIconMinHitPx: number,      // タッチ向け 44px 基準
  fontScaleClampMin: number,
}
```

このトークンを単一の source-of-truth として page.tsx と子コンポーネントに渡す。`SEARCH_BOX_RESERVE` の `svgWidth < 1100` 直書き分岐は撤去する。

### 3. 上部ツールバーを「クラスタの段組」に作り直す

現状の3クラスタ（左: 検索、中: 年度、右: フォント/TopN/設定）を、モードに応じて以下に切り替える。

| モード | 上部レイアウト |
|--------|---------------|
| `standard-desktop` / `dense-laptop` | 1段。現状踏襲。 |
| `compact-tablet` (≦1024px相当) | 2段。1段目: 検索 + 年度。2段目: フォント・TopN・表示設定アイコン群。`searchBoxReservePx` を 2段分に拡張。 |
| `compact-mobile` (≦480px相当) | コンパクトツールバー: 検索のみ常時表示。年度はピル型ボタンで右肩に、フォント/TopN/表示設定は **「表示設定シート」** に集約（ハンバーガー1ボタン）。 |

実装は CSS flex の折り返しに頼らず、トークン値を見て要素配置を切り替える（要素自体は子コンポーネントとして抽出可能だが、本タスクのスコープは page.tsx 内のJSX分割で十分）。

### 4. サイドパネルをモード別に切り替える

- `docked`（デスクトップ）: 現状の固定左パネル（既存のリサイズ機能維持）。
- `bottom-sheet`（タブレット縦・大型スマホ）: 画面下に高さ可変のシート。半開き / 全開 / 閉の3段スナップ。背景の Sankey はパン操作可能なまま。
- `fullscreen`（小型スマホ）: ノード選択時はサイドパネルを全画面オーバーレイにし、戻るボタンで一覧へ。

ロジックとしての `selectedNodeId` / パネル内容は共通。**プレゼンテーション層だけ差し替える**。既存の `sidePanelWidth` / `sidePanelResizeRef` 等は `docked` モード時のみ動作するよう条件分岐する。

### 5. 列見出しの省略モード

- `with-amount`（既定）: 列名 + 集計金額。
- `name-only`（タブレット縦以下）: 列名のみ。金額はノードのツールチップ・サイドパネル側で確認。
- `hidden`（最狭幅）: 列見出し非表示。代わりに横スクロール時のアクセントとして列区切り線のみ残す。

`COLUMN_LABEL_FONT_PX` を `Math.max(columnLabelFontPx, mode.fontScaleClampMin)` でクランプし、極小化を防ぐ。

### 6. タッチ操作のヒットエリア

- 上部ツールバー・ミニマップ・フォントコントロール等のボタンは `compact-tablet` / `compact-mobile` で **最小タップサイズ 44px** を確保する（`controlIconMinHitPx` トークン）。
- 既存のホバー前提UI（ツールチップなど）は、タッチ環境ではタップで開閉できるよう同等の状態管理を追加。設計対象は「ヒットエリアの基準化」までとし、各ボタンの個別調整は実装フェーズで網羅する。
- パン/ズームは pointer events ベースで既存実装が動く想定。タッチ・ピンチズーム未対応箇所が検証で見つかれば本タスク範囲に含めて修正する。

## レイヤー設計との整合

CLAUDE.md のレイヤー設計ルールに照らし、本設計は以下を守る：

- **`scripts/` には何も追加しない**（データパイプライン無関係）。
- **`app/lib/` の Sankey ロジックは変更しない**。`computeLayout` / `filterTopN` の入出力は不変。レスポンシブ判定はあくまでビュー層の責務。
- **モード判定・トークン定義** は `app/sankey-svg/` ローカル（例: `app/sankey-svg/_layout.ts`）に置く。複数ページで再利用する必要が生じた段階で `app/lib/` または `client/components/` へ昇格を検討する。
- **新規追加のサブコンポーネント**（例: BottomSheet, CompactToolbar）は `client/components/SankeySvg/` 配下に置き、直接の `fetch` / API 呼び出しは行わない。propsで状態を受け取る純粋UI部品とする。

## 実装方針: 並行ルートで開発 → 置換

[app/sankey-svg/page.tsx](app/sankey-svg/page.tsx) は 4,500行超で状態管理が密結合しており、本設計の変更（トークン化・ツールバー段組み・サイドパネル方式切替）を直接適用すると回帰リスクが高い。また既存 e2e（[tests/e2e/sankey-svg.spec.ts](../../tests/e2e/sankey-svg.spec.ts)）はデスクトップ前提のため、過渡期に落ちやすい。

そのため、以下の手順で進める：

1. **新ルート `/sankey-svg-next`** を作成し、`app/sankey-svg-next/page.tsx` として実装する。
2. データ層（`app/lib/`）は共通利用し、ビュー層のみ複製。
3. `/sankey-svg-next` で Issue #195 の完了条件を満たした段階で、旧 `/sankey-svg` ルートを `sankey-svg-next` の内容で置換し、`/sankey-svg-next` ルートは削除する（CLAUDE.md の「backwards-compat shim を作らない」方針に従い、互換リダイレクト等は残さない）。
4. 既存 e2e は新ルート向けにコピーして移行、最終的に旧スペックを廃止。

トレードオフ: 短期的に UI コード重複が発生するが、本番ページの安定性と引き換え。

- モックは **静止モックアップ**であり、React/Next.js 実装にそのまま貼り込まない。レイアウト・寸法・状態遷移の参照資料として扱う（[CSS変数・余白・角丸の値はそのまま採用してよい](_assets/20260528_responsive/Mobile%20Sankey%20Layout.html)）。
- モック内のラベル配置補正（事業ラベルと支出先ラベルの衝突回避、フォーカス時の非選択ラベル非表示）は **実装に必須**。既存 page.tsx のラベル配置ロジックでは衝突が再発しうるため、`compact-mobile` モード時の専用ブランチが必要。
- フロー描画方式は既存 page.tsx も同様の帯状パスを使っているため互換性問題なし（[app/sankey-svg/page.tsx:2280-2290](app/sankey-svg/page.tsx#L2280) 近辺の link path 生成参照）。

- TopN スライダーが上部ツールバー2段目に常時露出する点は、デスクトップ版の `showTopNSliders` 折りたたみ挙動（[app/sankey-svg/page.tsx:252](app/sankey-svg/page.tsx#L252)）とは異なる。`compact-tablet` モード時は常時表示で固定する。
