# 次タスク検討: UX改善 vs AIフレンドリー性向上（MCP対応など）

日時: 2026-06-11
ステータス: 検討（実装前の方針判断）

## 結論（先に要約）

**次の一手としては「AIフレンドリー性向上」の方が筋が良い。ただし小さく始める**（読み取り専用MCPサーバー + `llms.txt` 程度のスコープ）。

理由は3点:

1. **実装コストが構造的に安い** — レイヤー設計のおかげで、MCP対応は既存資産の薄いラッパーで済む
2. **UXの大物（レスポンシブ対応）は直近で意図的に棚上げした判断がある** — 今すぐ蒸し返すのは判断の一貫性に欠ける。残りのUX改善は漸進的な磨き込みで、緊急性が低い
3. **差別化と「データ基盤」への昇格** — 政府予算データをAIエージェントから引ける窓口は現状ほぼ存在せず、本サイトの価値を「可視化UI」から「データソース」に拡張できる

## 前提整理

### プロジェクトの現状

- 公開ページ: `/sankey-svg`（メイン）、`/subcontracts`、`/mof-budget-overview`、`/quality`
- 直近のコミット（#223〜#227）はすべてスマホ幅のUX修正 → **UXの目先の不具合は一巡している**
- レスポンシブ対応（Issue #195）: 並行ルート `/sankey-svg-next` は 2026-06-01 に削除済み。「本番は非レスポンシブのまま、再開時は履歴から復元」という**明示的な棚上げ判断**がある

### AIフレンドリー観点での既存資産

| 資産 | 状態 | MCP対応への寄与 |
|------|------|----------------|
| `app/lib/`（ドメインロジック） | Pure関数、HTTP/React非依存 | **そのままMCPツールの実装に転用可能** |
| `/api/sankey` ほか4エンドポイント | `docs/api-guide.md` で仕様文書化済み | ツールスキーマがほぼ書き起こすだけで定義できる |
| 構造化JSON仕様書 | `docs/構造化JSON仕様書.md` | データモデルの説明をAIに渡せる |
| データ | 1円単位で正規化済み、5,003事業・26,823支出先 | クエリ可能な粒度がすでにある |

つまり「AIフレンドリー化」はゼロからの構築ではなく、**既にある層を一段公開するだけ**。レイヤー設計ルール（CLAUDE.md）が利いている。

## 両案の比較

### A案: UX改善

**候補となる残タスク**
- レスポンシブ対応の再開（Issue #195、履歴から復元）
- spending-block層の視認性課題（別アプローチ検討中）
- フォントサイズ記憶機能（2026-06-01 検討済み）

**強み**
- エンドユーザー（一般の閲覧者）に直接届く
- サイトの本質的価値は「人間が予算を理解できること」であり、UXはその根幹

**弱み**
- 大物のレスポンシブは10日前に棚上げしたばかり。再開するなら「なぜ今か」の根拠が必要
- 残りは磨き込み系で、1件あたりの効果が小さい
- 直近5コミットがすべてUX修正 → 投資が偏っており、限界効用が下がっている局面

### B案: AIフレンドリー性向上

**候補となる施策（コスト昇順）**

1. `llms.txt` / `llms-full.txt` の設置 — サイト構造・API仕様・データの留意事項（「その他」vs「その他の支出先」等）をAI向けに記述。**半日以下**
2. APIレスポンスへのメタ情報強化（単位・出典・注意書きの埋め込み）— **小**
3. 読み取り専用MCPサーバー — `app/lib/` を呼ぶツール群（事業検索、支出先検索、Sankeyデータ取得、品質スコア取得）。**中（数日）**
4. MCPレジストリへの公開・README整備 — **小**

**強み**
- 上記の通り実装コストが構造的に安い（資産の再利用率が高い）
- 競合がほぼいない領域。「行政事業レビューのデータをAIで引ける」は研究者・記者・政策ウォッチャーに刺さる
- 自分自身の分析ワークフロー（Claude等での予算分析）が即座に楽になる＝ドッグフーディングで品質検証できる
- UX改善と排他ではない。むしろAI経由で「どのビューが求められているか」の発見があれば、UX改善の優先順位付けに還元できる

**弱み・リスク**
- **配布の問題**: 一般閲覧者はMCPサーバーをインストールしない。受益者は当面、技術リテラシーの高い少数層
- Vercelホスティングとの兼ね合い（リモートMCPにするか、npmパッケージのローカルMCPにするか）の設計判断が必要
- 「その他」問題のような**データの罠をAIが誤読するリスク** — ツールの説明文・注意書きの設計が品質の肝になる（データ正確性が最優先という本プロジェクトの方針に直結）

## 判断の軸

| 軸 | UX改善 | AIフレンドリー |
|----|--------|---------------|
| 実装コスト | 大物は重い／小物は効果小 | 既存資産の再利用で安い |
| 受益者の広さ | 広い（一般閲覧者） | 狭い（当面は技術層） |
| 1施策あたりのインパクト | 漸進的 | 新カテゴリの開拓 |
| 直近の判断との整合 | レスポンシブ棚上げと矛盾 | 矛盾なし |
| データ正確性方針との関係 | 中立 | 注意書き設計が必須（リスクでもあり強化機会でもある） |

## 推奨アプローチ

**B案を小さく始める。順序:**

1. **`llms.txt` + API仕様のAI向け整備**（半日）— 最安で効果検証できる。MCPを作らなくても、Claude等のWebフェッチ経由で既にAIフレンドリーになる
2. **読み取り専用MCPサーバーの設計ドキュメント**（`/plan` スキルで作成）— リモート型 vs ローカル型、ツール一覧、データの罠への注意書き設計を先に固める
3. **MCP実装**（数日）— `app/lib/` のラッパーとして実装

UX側は「spending-block視認性」「フォントサイズ記憶」のような**小粒タスクを合間に挟む**運用とし、レスポンシブ再開はAI対応が一巡してから改めて判断する。

## 保留・未決事項

- MCPのホスティング形態（Vercel上のリモートMCP / npm配布のローカルMCP）
- 対象ユーザーの解像度（研究者向け？ 自分用？ 一般公開？）→ ツール設計の粒度に影響
- `rs-vis` 側（兄弟プロジェクト）との関係 — データ提供の窓口をどちらに置くか
