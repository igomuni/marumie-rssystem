# AIフレンドリー化: ユーザーストーリーとレートリミット検討

日時: 2026-06-11
前提ドキュメント: [20260611_0642_次タスク検討_UX改善vsAIフレンドリー.md](20260611_0642_次タスク検討_UX改善vsAIフレンドリー.md)
ステータス: 検討（設計前のスコープ定義）

## 要約

- ユーザーストーリーは5本。共通するのは「**生JSONは大きすぎてAIに渡せない**（13〜29MB）→ サーバー側で絞り込んだ小さな応答を返す窓口が必要」という点
- 提供物は3層: **① llms.txt（案内）→ ② クエリAPI（絞り込み済みJSON）→ ③ MCPサーバー（②のラッパー）**。②が本体で、③は薄い皮
- レートリミットは「コストの高い動的処理を守る」発想で設計する。幸いデータが年度単位の静的データなので、**CDNキャッシュで吸収できる部分が大半**。現状は `project-details` に加え `search/projects`・`search/recipients`・`recipients/[key]`・`subcontracts`・`sankey/mof-overview` が `API_CACHE_CONTROL` 設定済み。リミット導入の前に、既存キャッシュ面の検証・文書化を優先する（未設定ルートが残っていないか確認）

## 1. ユーザーストーリー

### US-1: 記者・ライター「特定テーマの支出を調べて記事の裏取りをしたい」

> 「『マイナンバー』関連の事業にいくら使われていて、最大の支出先はどこか、ChatGPT/Claudeに聞いて一次データ付きで確認したい」

- **行動**: AIチャットに質問 → AIがWeb検索 or API経由で本サイトのデータに到達 → 出典URL付きの回答を得る
- **必要なもの**:
  - キーワードで事業・支出先を検索できるAPI（現状は存在しない。生JSONをDLする以外の手段がない）
  - 応答に**出典・年度・単位（1円）・データの留意事項**が埋め込まれていること（AIの誤読防止）
  - 各事業・支出先に**人間が確認できる固定URL**（例: `/subcontracts/[projectId]`）が応答に含まれること → 裏取り可能
- **成功条件**: AIの回答が「151.12兆円中、〇〇事業が△△円、出典: このURL」の形になる

### US-2: 政策研究者・アナリスト「自分のAI環境から横断分析したい」

> 「Claude Code / Claude Desktop にMCPを繋いで、府省庁横断の支出先重複や再委託構造を対話的に分析したい」

- **行動**: MCPサーバーを設定 → 「文科省と経産省の両方から受注している支出先トップ20を出して」のような対話分析
- **必要なもの**:
  - MCPサーバー（ツール例: `search_projects`, `search_recipients`, `get_project_detail`, `get_subcontract_tree`, `get_ministry_summary`, `get_quality_score`）
  - ページネーションと件数上限（AIのコンテキストに収まる応答サイズ設計。1応答 ≦ 数十KB目安）
  - ツール説明文に「その他」vs「その他の支出先」等の罠を明記
- **成功条件**: 生JSONを手元にDLせず、対話だけで横断集計が完結する

### US-3: 一般市民（AI経由の間接利用）「税金の使い道を会話で知りたい」

> 「『子育て支援に国はいくら使ってる？』と聞いたら、AIがこのサイトのデータで答えてくれる」

- **行動**: 本人はサイトもAPIも意識しない。AI製品側のWeb検索・クロールが本サイトを情報源として拾う
- **必要なもの**:
  - `llms.txt` / `llms-full.txt`（サイト構造・API仕様・統計サマリ・留意事項）
  - 各ページのメタデータ整備（タイトル・description に統計値を含める、構造化データ）
  - クローラーが拾える静的なサマリページ（例: 府省庁別年間サマリのHTML/JSON）
- **成功条件**: AI製品の回答の出典に本サイトが現れる。**これは配布問題（一般人はMCPを入れない）への回答でもある**

### US-4: シビックテック開発者「このデータの上にアプリを作りたい」

> 「予算データを使った別の可視化・通知Bot・データセット統合をしたい。生JSON 29MBを毎回DLしたくない」

- **行動**: 公開APIを叩いてアプリを構築。場合によっては高頻度アクセス
- **必要なもの**:
  - APIドキュメントの公開（`docs/api-guide.md` をサイト上に公開 or OpenAPIスキーマ）
  - 利用規約・出典表記ルール・レートリミットの明示
  - 大量取得者向けには「生JSON（.gz）の一括DL」へ誘導（API連打よりCDN配信が双方に安い）
- **成功条件**: APIを「使っていいのか」「どこまで叩いていいのか」が明文化されている
- **備考**: **このストーリーがレートリミット要件の主因**。US-1〜3は単発・低頻度

### US-5: 運営者自身「自分の分析・検証ワークフローを楽にしたい」

> 「データ更新後の検証や、プロンプトログに残しているような探索的分析を、Claude Codeから直接やりたい」

- **行動**: ローカルMCP（リポジトリ内のJSONを直接読む）で分析・回帰チェック
- **必要なもの**: ローカル動作するMCP（API経由不要、レートリミット無関係）
- **成功条件**: 「この更新で総額・事業数が変わってないか」をAIに一言で検証させられる
- **備考**: **最初のドッグフーディング対象**。ここで磨いたツール定義をリモートMCPに昇格させる順序が低リスク

## 2. 提供物マップ（ストーリー × 提供物）

| 提供物 | US-1 記者 | US-2 研究者 | US-3 市民 | US-4 開発者 | US-5 自分 | コスト |
|---|---|---|---|---|---|---|
| llms.txt + メタデータ整備 | ◎ | ○ | ◎ | ○ | — | 半日 |
| 検索・絞り込みAPI（新規） | ◎ | ◎(裏側) | ○(裏側) | ◎ | — | 数日 |
| MCPサーバー（ローカル） | — | ○ | — | — | ◎ | 1-2日 |
| MCPサーバー（リモート） | ○ | ◎ | — | ○ | ○ | +数日 |
| APIドキュメント公開 + 利用規約 | ○ | ○ | — | ◎ | — | 1日 |
| 生JSON一括DLの正規ルート化 | — | ○ | — | ◎ | — | 半日 |

**読み取れること**: 検索・絞り込みAPIがほぼ全ストーリーの土台。MCPは「②の皮」なので、**API設計を先に固めるのが正しい順序**。実装順の推奨は llms.txt → ローカルMCP（US-5でツール設計を検証）→ 検索API → リモートMCP公開。

### 検索APIの最小セット案

```text
GET /api/search/projects?q=マイナンバー&year=2024&limit=20
GET /api/search/recipients?q=○○株式会社&year=2024&limit=20
GET /api/ministries/summary?year=2024
GET /api/projects/[projectId]            # 既存 project-details の公開版
GET /api/projects/[projectId]/subcontracts  # 既存 subcontracts の公開版
```

共通仕様:
- 応答は常に `metadata`（年度・単位・生成日時・出典URL・留意事項へのリンク）+ `data` + `pagination`
- `limit` 上限は 100。応答サイズ上限の目安 100KB（AIコンテキスト対応）
- 全文検索は不要。部分一致 + 正規化（全角半角・法人格表記ゆれ）で開始

## 3. レートリミットの考慮

### 3.1 現状のコスト構造（実装確認済み）

| 経路 | 実装 | 1リクエストのコスト |
|---|---|---|
| `public/data/*.json` 静的配信 | CDN(Vercel Edge) | 安い（ただし29MBファイルは帯域課金に注意） |
| `/api/project-details/[id]` | `readFileSync`(13MB) + 抽出。**唯一 `Cache-Control` あり**（s-maxage=86400） | キャッシュヒット時は安い |
| `/api/subcontracts/[id]` | `readFileSync`(29MB) 毎回 | **高い**（キャッシュなし） |
| `/api/quality-scores` 系 | `readFileSync`(4〜28MB) 毎回 | **高い**（キャッシュなし） |
| `/api/sankey/mof-overview` | `readFileSync`(4KB) | 安い |

問題の本質: サーバーレス関数が**毎回巨大JSONを同期読み込み**しており、AI経由のアクセス増はそのまま関数実行時間・メモリ課金に跳ねる。リクエスト「数」より1リクエストの「重さ」が先に問題になる。

### 3.2 設計方針: リミットの前にキャッシュ（守る対象を減らす）

データは年度単位の静的スナップショット（更新はデプロイ時のみ）。つまり**ほぼ全応答が無期限キャッシュ可能**。

1. **全APIルートに `Cache-Control: public, s-maxage=86400, stale-while-revalidate=604800` を付与**（デプロイでCDNキャッシュは自然に切り替わる）
   → 同一クエリの2回目以降はCDNで完結し、関数が動かない。AIエージェントは同じ質問を何度もするため、ヒット率は高いと期待できる
2. 新設する検索APIも**クエリ正規化**（パラメータ順・大文字小文字・全半角）してキャッシュキーの分裂を防ぐ
3. 大量取得は `.gz` 一括DL（静的配信）へ誘導し、API連打のインセンティブ自体を消す

これだけで US-1/2/3/5 の負荷は実質吸収できる見込み。レートリミットが本当に必要なのは「キャッシュミスを量産するアクセス」= US-4 の機械的クロールと、悪意あるパラメータ総当たり。

### 3.3 レートリミット本体の設計案

**段階導入とする。最初から重装備にしない。**

**Phase 1（API公開と同時・必須）**
- Vercel の WAF / Firewall ルールで IPベースの粗い上限（例: 60 req/分/IP）。アプリコード変更不要
- 429 応答に `Retry-After` ヘッダを必ず付ける（**AIエージェントはRetry-Afterを尊重する実装が多く、明示するだけで行儀が良くなる**）
- `limit` パラメータ上限・応答サイズ上限をAPI側で強制（1リクエストの重さに天井）

**Phase 2（利用が観測されたら）**
- 匿名: 緩い共有リミット / APIキー登録（無料）: 個別リミット、の2層化
  - キーはRedis等を増やさず、Vercel KV または Upstash（無料枠）+ `@upstash/ratelimit` のスライディングウィンドウで実装するのが定石
- MCPリモート版はこのAPIキーをそのまま使う（MCP側に独自認証を作らない）

**Phase 3（必要になったら考える）**
- 商用利用・大量利用の問い合わせ窓口、データセット配布（Hugging Face等）への切り出し

**やらないこと**: 課金、複雑なクォータ管理、ユーザーダッシュボード。個人運営の無償公開サイトの維持コストに見合わない。

### 3.4 リミット値の初期案（根拠つき）

| 対象 | 値 | 根拠 |
|---|---|---|
| 匿名 IP/分 | 60 | AI対話1ターンでのツール連続呼び出しは多くて10回程度。対話的利用を阻害しない |
| 匿名 IP/日 | 2,000 | 全事業5,003件の半分以上を1日で舐めるのは対話利用ではなくクロール → 一括DLへ誘導 |
| `limit` パラメータ上限 | 100 | 応答100KB目安から逆算 |
| 429時 Retry-After | 60秒 | エージェントのリトライ間隔として現実的 |

※ 値は仮置き。Phase 1 で Vercel のアクセスログ（関数実行時間・キャッシュヒット率）を見て調整する前提。

### 3.5 レートリミット以外の保護

- **帯域**: 29MBの生JSONが `public/` に裸で置かれている。AIクローラーが定期取得し始めると帯域課金が先に痛む可能性 → `.json` は robots.txt でクロール拒否し、`.gz` への明示リンクだけ案内する
- **robots.txt / llms.txt の整合**: クローラー向け（拒否対象）とAIエージェント向け（API案内）を矛盾なく書き分ける
- **悪用面**: データは全て公開情報のためデータ漏洩リスクはなし。守るのはインフラコストのみ、という整理で良い

- APIキー導入の閾値（Phase 2 へ進む判断基準: 例「キャッシュミス由来の関数実行が ○万回/月 を超えたら」）
- 検索の表記ゆれ正規化の仕様（法人格・全半角・旧字体）— `/pipeline` の正規化資産が流用できるか要確認
- リモートMCPのホスティング（Vercel上の `/api/mcp` ルートで足りるか、別デプロイか）
- 利用規約・出典表記の文面（元データのRSシステム側の利用条件との整合確認が必要）
