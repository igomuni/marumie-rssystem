# 支出先なし表示設計

## 目的

データ閲覧者が「支出先のない事業」をサンキー図で視覚的に確認でき、支出先列との接続が途切れていないことを認識できるようにするため。

---

## 現状の問題

直接支出先がない事業（`budgetAmount > 0 && spendingAmount === 0`）の `project-spending` ノードは、JSONデータにアウトゴーイングエッジが存在しない。

フィルタ処理の結果として：

- `effectivelyHiddenIds` は `n.value > 0` の条件でのみ生成されるため、0円事業は除外されない
- しかし `project-spending → r-no-spending` のエッジが存在しないため、支出先列との接続が描画されない
- 事業(支出)ノードがサンキー図に表示されても、支出先列に向かうリボンが一切なく、視覚的に孤立している

また、`支出先なし` ノード自体が選択可能なノードとして存在していないため、「どの事業が支出なしか」をパネルで一覧できない。

---

## レイアウトエンジンの現状確認

`computeLayout`（`app/lib/sankey-svg-filter.ts`）を調査した結果、以下が確認された：

- **0値ノード**: `Math.max(1, node.value * ky)` → 最低 1px の高さが保証される
- **0値エッジ**: `MIN_LINK_W = 1` として処理される（`link.value === 0` 分岐が既実装）

すなわち、フィルタ後の出力に 0値エッジ・0値ノードが含まれさえすれば、**レイアウトエンジン側の変更は不要**。

---

## 変更スコープ

### 1. `scripts/generate-sankey-svg-data.ts`

`recipientMap` に `__no-spending__` キーでエントリを追加する。

- プロジェクトループ（6c）内で `budgetAmount > 0 && spendingAmount === 0` を検出した際、`recipientMap.get('__no-spending__')` にそのPIDを追加
- 6d ループで `__no-spending__` キーのエントリを処理する際、連番IDではなく固定ID `r-no-spending` を使用
- 生成ノード: `{ id: 'r-no-spending', name: '支出先なし', type: 'recipient', value: 0, skipLinkOverride: true }`
  - `skipLinkOverride: true` により、レイアウトエンジンがリンク合計でノード高さを上書きするのを防ぐ
- 生成エッジ: 各該当PIDに対して `{ source: 'project-spending-${pid}', target: 'r-no-spending', value: 0 }`

**既存の `recipientMap` → 6d ループのパイプラインをそのまま活用するため、特殊ステップは不要。**

### 2. `app/lib/sankey-svg-filter.ts`

`windowRecipients` / `tailRecipients` の分割後に、`r-no-spending` を tail から window の末尾へ昇格させる。

```
tailRecipients に r-no-spending が含まれる場合:
  → tailRecipients から除去
  → windowRecipients の末尾に追加
```

これにより、以下が成立する：

- `windowRecipientIds` に `r-no-spending` が含まれる
- 6d ループの `project-spending → window recipients` エッジ処理（`allEdges` のスキャン）で 0値エッジが filtered edges に含まれる
- `r-no-spending` ノードが layout に渡される

**`__agg-recipient` への影響**: `tailValue` の計算は tail 昇格後の `tailRecipients` を使うため、0値の `r-no-spending` が tail から抜けても合計値に変化なし（0円加算がなくなるだけ）。

### 3. `app/sankey-svg/page.tsx`

**変更なし。** 既存の `panelSections` ロジックがそのまま機能する。

- `r-no-spending` 選択時 (`ntype === 'recipient' && !selectedNode.aggregated`):
  - `graphData.edges` から `e.target === 'r-no-spending'` のエッジを集約 → 264件（2024年）の project-spending ノードを事業タブに表示
  - 支出先タブ: 自身 (`支出先なし: 0円`) を表示
- 支出なし事業の選択時:
  - `graphData.edges` から `e.source === spendingNode.id` のエッジを取得 → `r-no-spending` が支出先タブに `支出先なし: 0円` として表示される

---

## 表示上の振る舞い

| 状況 | 表示 |
|------|------|
| デフォルト全体表示 | `r-no-spending` は支出先列の最下部に 1px の高さで常時表示 |
| 支出なし事業を選択 | 事業(支出)ノードから `r-no-spending` に 1px リボンが描画される |
| `支出先なし` を選択 | 264件の関連事業がパネル事業タブに一覧表示される |
| 府省庁・全体フォーカス | 対象事業の支出なし分が `支出先なし` ノードへのリボンとして接続される |

---

## 影響範囲

| 指標 | 変更内容 |
|------|----------|
| JSONサイズ | ノード +1、エッジ +264（2024年） / +292（2025年） |
| `totalRecipientCount` メタデータ | 1増加（`r-no-spending` をカウントから除外する場合は生成スクリプトで調整） |
| レイアウトエンジン | 変更なし |
| page.tsx | 変更なし |
| filter.ts の window/tail 境界 | `r-no-spending` が tail から常時 window に移動するため、実質的な `totalRecipientCount` のカウントが 1 変動 |

---

## 未解決・要確認

1. **`totalRecipientCount` の扱い**: `r-no-spending` は実際の支出先ではない。メタデータの `recipientCount` や UI の「支出先N件」表示に含めるか否かを確認する。
2. **府省庁・全体ビューでの表示**: 全府省庁分の 264 件が全て `r-no-spending` へのリボンを描画すると、`r-no-spending` ノードに多数の 1px リボンが集中する。視覚的に許容可能か確認が必要。
