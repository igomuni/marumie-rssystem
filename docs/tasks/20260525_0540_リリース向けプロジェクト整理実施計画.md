# リリース向けプロジェクト整理 実施計画

作成日: 2026-05-25 05:40 JST
ステータス: 計画案（レビュー待ち）

## 目的

**プロジェクトオーナーが、公開対象を最小限の4ページに絞り込んだ状態でリリース準備を進められるようにするため。**

現状は実験的ページ（/sankey, /sankey2, /sankey-lab, /entities など）が混在しており、トップページから多数の入口が露出している。公開価値が定まった4ページ（**/sankey-svg、/subcontracts、/mof-budget-overview、/quality**）に絞り、それ以外のページ・APIをソースから除去することで、保守対象を縮小し、リリース後の説明・サポート負荷を下げる。

## スコープ

### 公開対象（残すページ）

| ルート | 役割 |
|---|---|
| `/`（新トップ） | `/sankey-svg` へのリダイレクト |
| `/sankey-svg` | 直接支出サンキー図（SVG版）。事実上のメインコンテンツ |
| `/subcontracts` | 再委託構造ブラウザ |
| `/mof-budget-overview` | MOF予算全体ビュー |
| `/quality` | 事業別データ品質スコア |

`/mof-budget-overview` と `/quality` はトップから動線を張らず、URL直打ちでのアクセスを想定。

### 削除対象（ページ）

`app/` 配下から以下を削除：

- `app/sankey/`
- `app/sankey2/`
- `app/sankey-lab/`
- `app/sankey-lab-d3/`
- `app/budget-drilldown/`
- `app/entities/`
- `app/entities-v2/`
- `app/entity-labels-csv/`
- `app/globe/`
- `app/map/`
- `app/spending-bottomup/`

### 削除対象（API）

公開4ページが使用するのは以下のみ。

- `/api/sankey/mof-overview`
- `/api/project-details/[…]`
- `/api/quality-scores`
- `/api/quality-scores/recipients`
- `/api/subcontracts/[…]`

これ以外（`/api/sankey/route.ts`、`/api/entities`、`/api/entities-v2`、`/api/entity-labels-csv`、`/api/map/*` 等）は削除。`app/api/sankey/route.ts` が `mof-overview` サブルートに影響しないよう、削除時に `app/api/sankey/` ディレクトリ自体は残し `route.ts` のみ消す点に注意。

### 削除対象（client/components, app/lib, components）

公開4ページから到達不能になるものを削除。代表的には以下：

- `client/components/Sankey2/*`
- `client/components/SankeyBudgetDrilldown.tsx`
- `client/components/SankeySpendingBottomup.tsx`
- `client/components/BudgetBoardView.tsx`
- `client/components/GlobeView.tsx`
- `app/lib/sankey-generator.ts`（旧 `/sankey` 用）
- `app/lib/mof-sankey-generator.ts`（`/mof-budget-overview` が使うのは `mof-transfer-sankey-generator.ts` のみ。実依存を再確認した上で判断）

判定基準: **削除後に `npx tsc --noEmit` と `npm run build` が成功すること**を最終確認とする。

### 削除対象（scripts/）

「公開ページが使うものだけ残す」方針。

- **残す対象**: 以下JSONを生成するパイプラインスクリプトおよびその依存ユーティリティ
  - `public/data/sankey-svg-*.json`（/sankey-svg）
  - `public/data/subcontracts-*.json`（/subcontracts）
  - `public/data/mof-budget-overview-2023.json`（/mof-budget-overview）
  - `public/data/project-quality-scores-*.json`、`project-quality-recipients-*.json`（/quality）
- **削除対象**: 上記に寄与しない `analyze-*.ts`/`*.py`、旧 `/sankey` 用 generator スクリプト、 entity-labels-csv 関連、map/globe 関連スクリプト。
- 残すべきものを実装フェーズで grep（generate スクリプトの import グラフを辿る）で確定する。判定に迷うものは `scripts/_archive/` への移動ではなく、Gitタグでの保全に委ねて削除する。

### types/

公開4ページが参照する型のみ残す。

- 残す: `sankey-svg.ts`, `subcontract.ts`, `mof-budget-overview.ts`, `mof-transfer.ts`, `project-details.ts`, `sankey.ts`（`SankeyNode` を `/sankey-svg` が参照しているため）、`structured.ts`（パイプライン基底型）, `rs-system.ts`（同）, `preset.ts`/`view-state.ts` は依存確認後に判断。

### public/data

不要となるデータ（`entity-labels-csv.json`、`entity-normalization.json` 等）は削除候補。ただし `entity-labels.json`・`houjin-lookup.json` 等は他ページの生成パイプラインで参照されている可能性があるため、grep で確認してから判断。

### ドキュメント

- `docs/tasks/` 配下は履歴として全保持。
- `CLAUDE.md`・`README.md` の Quick Reference・Skills 表・トップページ紹介文を、公開4ページに合わせて更新。
- `.claude/skills/sankey/` などスキルファイルに `/sankey`・`/sankey2` 前提の記述があれば更新。

## アーカイブ戦略

実装作業前に**現状のmainを完全保全**する。

1. `main` の最新コミットに対し、注釈付きタグ `archive/pre-release-20260525` を作成しリモートへ push。
2. 同一コミットからブランチ `archive/pre-release-20260525` を作成し、リモートへ push。
3. 以降の整理作業は `chore/release-cleanup` 等のフィーチャーブランチで行い、PRはユーザー指示後に作成する（[CLAUDE.md のエージェント行動ルール](../../CLAUDE.md) 準拠）。

復元は `git checkout archive/pre-release-20260525` で常に可能とする。

## トップページ仕様

- `app/page.tsx` を、Next.js App Router の `redirect('/sankey-svg')` を呼び出す Server Component に置き換える。
- 既存リンクカード群（`/sankey`・`/sankey2`・`/entities` 等）の表示は廃止。
- `/sankey-svg` のURL自体は維持（外部リンク・ブックマーク互換のため）。

## 実装フェーズ（順序）

順序は依存関係を逆向きに辿る。

1. **アーカイブ確定** — タグ・ブランチ作成・push。
2. **作業ブランチ作成** — `chore/release-cleanup`。
3. **ページ削除** — 「削除対象（ページ）」のディレクトリを `rm -rf`。
4. **API削除** — 公開4ページの使用APIのみ残し、他を削除。
5. **コンポーネント・lib削除** — 到達不能になるファイルを `npx tsc --noEmit` の未参照警告と grep で特定し削除。
6. **トップページ差し替え** — `app/page.tsx` を redirect 版に置換。
7. **types/ 整理** — 未参照型ファイル削除。
8. **scripts/ 整理** — 公開ページのデータ生成パイプラインに連なるものだけ残し、他を削除。`scripts/` 内 import グラフは grep で辿る。
9. **public/data 整理** — 未参照JSON/gz を削除。
10. **ドキュメント更新** — `app/page.tsx` の旧リンク説明・README・CLAUDE.md の Skills 表を整合。
11. **品質チェック** — `npm run lint` / `npx tsc --noEmit` / `npm run build` / `npm run dev` で公開4ページの動作確認。
12. **コミット** — 削除を意味単位で分割（pages → api → libs → scripts → docs）。
13. **PR作成** — **ユーザー指示後**に実施。

## レイヤー設計ルール整合性チェック

[CLAUDE.md](../../CLAUDE.md) のレイヤールールとの矛盾は無い：

- `scripts/` 整理はCSV処理スクリプトの範囲内で完結。UI/APIロジックは混入しない。
- `app/lib/` の削除は generator 系の整理のみで、HTTP/React 依存を新たに導入しない。
- `client/components/` 側の変更はファイル削除のみで、新たな直接APIコールを導入しない。
- 新 `app/page.tsx` は `redirect()` のみで状態管理・ロジックを持たない（Pages レイヤーの責務に適合）。

## リスク・留意点

- **隠れた相互参照**: 削除予定ファイルから残存ファイルへの import がある場合は自明だが、逆（残存→削除予定）の依存が見落とされると build 失敗する。各ステップ後に `npx tsc --noEmit` を必ず通すこと。
- **`/sankey-svg` から `/sankey`・`/sankey2` への内部リンク**: 公開ページに残置されていないか grep で確認し、あれば削除または `/sankey-svg` 内ナビに置換。
- **prebuild の `.gz` 展開**: `public/data/*.gz` を削除する場合、対応する `prebuild` の挙動（存在しないファイルへの `gunzip` がエラーにならないか）を確認。
- **Vercel デプロイ**: 削除コミットを main にマージした時点で本番が切り替わるため、PR マージは公開タイミングと一致させる。
