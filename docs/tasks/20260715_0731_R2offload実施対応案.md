> **方針変更（同日）**: R2 は有効化に支払い情報登録が必須のため見送り。**8節の GitHub Pages 案を採用**する。1〜7節は R2 案の記録として残す。コード側の骨格（`dataUrl()` + env 切替 + バージョン付き URL）は両案共通。

## 1. 対象と現状

ブラウザが直接 fetch する静的データ（= Vercel 帯域を消費するもの）を R2 へ移す。

| fetch 箇所 | ファイル |
|---|---|
| `app/sankey-svg/page.tsx:1337` | `sankey-svg-{year}-graph.json` |
| `app/subcontracts/page.tsx:217` | `subcontracts-{year}.json` |
| `app/subcontracts/[projectId]/page.tsx:1210` | `project-quality-scores-{year}.json` |
| `app/mof-budget-overview/transfer-detail/page.tsx:18` | `mof-budget-overview-2023.json` |

サーバ側 API だけが読む大物（recipient-index 等）は対象外（帯域に乗らない）。

## 2. 方針（設計ドキュメントの結論を踏襲）

「独自ドメイン + Cache Rules + 事前 gzip + `Content-Encoding: gzip`」をセットで実施。
さらに今回、**URL にバージョンセグメントを入れて `immutable` 長期キャッシュ化**する（前回ドキュメント 6-1 の宿題の解決）。

```text
https://data.<独自ドメイン>/v20260715/sankey-svg-2024-graph.json
                            ^^^^^^^^^ データ更新ごとに新プレフィックス
```

- バージョンは `NEXT_PUBLIC_DATA_BASE_URL` に含める（例 `https://data.example.com/v20260715`）。
- データ更新時: 新プレフィックスへアップロード → env var 更新 → 再デプロイ。旧バージョンは残るのでロールバックも即時。
- これにより `Cache-Control: public, max-age=31536000, immutable` が安全に使える（キャッシュ汚染リスクなし）。

## 3. コード側の変更（実装タスク）

### 3-1. `dataUrl()` ヘルパ新設 — `app/lib/data-url.ts`

```ts
export function dataUrl(path: string): string {
  const base = process.env.NEXT_PUBLIC_DATA_BASE_URL;
  return base ? `${base}/${path}` : `/data/${path}`;
}
```

- **env 未設定なら従来どおり同一オリジン `/data`** → ローカル `npm run dev`・プレビュー環境は無変更で動く。
- Pure 関数なので vitest 対象（`app/lib` のテスト方針に合流）。

### 3-2. fetch 4箇所を `dataUrl()` に差し替え

1節の4ファイル。`fetch(dataUrl(`subcontracts-${year}.json`), …)` の形。

### 3-3. `app/layout.tsx` に preconnect（R1 対策）

```tsx
{process.env.NEXT_PUBLIC_DATA_BASE_URL && (
  <link rel="preconnect" href={new URL(process.env.NEXT_PUBLIC_DATA_BASE_URL).origin} crossOrigin="anonymous" />
)}
```

### 3-4. アップロードスクリプト — `scripts/upload-data-to-r2.sh`

wrangler（`npx wrangler r2 object put`）または `aws s3 cp`（S3互換 API）で、対象7ファイルを:

- **gzip 済みバイトを put**（raw を置かない）
- メタデータ: `Content-Encoding: gzip` / `Content-Type: application/json; charset=utf-8` / `Cache-Control: public, max-age=31536000, immutable`
- put 先: `v$(TZ=Asia/Tokyo date +%Y%m%d)/<ファイル名>.json`（拡張子は `.json` のまま、実体は gzip）

`/data-update` スキルのフローの末尾に「R2 アップロード + env var 更新」の手順を追記する。

## 4. インフラ側（手作業・前提条件）

| # | 作業 | 備考 |
|---|---|---|
| 1 | Cloudflare アカウント + **独自ドメインの Cloudflare ゾーン登録** | ★最大の前提。未保有なら取得が先 |
| 2 | R2 バケット作成（ロケーションヒント APAC） | 無料枠: 10GB ストレージ / Class B 1,000万/月 |
| 3 | バケットにカスタムドメイン接続（`data.<ドメイン>`） | **r2.dev は使わない**（レート制限・CDNキャッシュなし） |
| 4 | Cache Rules: `data.<ドメイン>/*` を Cache Everything + Edge TTL 長め | オリジンの `Cache-Control` を尊重させる設定でも可 |
| 5 | バケット CORS: `Access-Control-Allow-Origin` に本番オリジンを設定 | fetch はカスタムヘッダ無しなので preflight は発生しない |
| 6 | アップロード用 API トークン発行（R2 書き込み権限のみ） | ローカル `.env` / wrangler login で保持。Git に入れない |

## 5. ロールアウト手順

1. コード側（3-1〜3-4）を実装・マージ。**env 未設定のため挙動は現状と完全同一**（安全にデプロイ可能）。
2. インフラ構築（4節）→ スクリプトで初回アップロード。
3. `curl -sI -H 'Accept-Encoding: gzip' https://data.<ドメイン>/v.../subcontracts-2024.json` で検証:
   - `content-encoding: gzip` / `content-type: application/json` / `cf-cache-status: HIT`（2回目）
4. Vercel の環境変数に `NEXT_PUBLIC_DATA_BASE_URL` を設定 → 再デプロイで切替。
5. 問題があれば env var を削除して再デプロイ（即ロールバック、Vercel 直配信に戻る）。

## 6. 残論点

- **bulk `.gz` の直DL**（前回 6-2）: `public/data/*.json.gz` は本対応のスコープ外。急がないが、R2 に寄せるなら同じバケットに置いて `public/` から外すのが自然（`prebuild` 展開フローとの整合の検討が必要なので別タスク）。
- `app/robots.ts` の Disallow はパス変更後も `/data/*.json` のままでよい（R2 側ドメインはクロール対象外だが、必要なら R2 側にも robots を置ける）。
- **費用**: 本規模（合計 ~11MB・GET 数百万/月以下）なら R2 無料枠内。egress 課金なし。

## 7. 見積り

- コード側: 小規模（ヘルパ + 4箇所差し替え + preconnect + スクリプト）。半日以内。
- インフラ側: ドメインが既にあれば 1〜2時間。**ドメイン未保有ならその調達が律速**。

## 8. 採用案: GitHub Pages 版（支払い情報不要）

### 8-1. R2 案との差分

| 観点 | R2 | GitHub Pages |
|---|---|---|
| 費用・登録 | 無料枠内だが**カード登録必須** | 完全無料・登録不要 |
| 帯域 | egress 無制限無料 | **100GB/月ソフトリミット**（Vercel Hobby と同水準） |
| 効果 | 帯域上限の消滅 | データ帯域を Vercel から**分離**（枠が実質2倍、データ急増時もアプリ本体は無事） |
| CORS | バケット設定が必要 | `Access-Control-Allow-Origin: *` が自動付与、設定不要 |
| 独自ドメイン | 必須（r2.dev 不可） | 不要（`github.io` で可） |
| アップロード | wrangler / S3 API | **git push だけ** |
| CDN | Cloudflare（東京PoP） | Fastly（東京PoP） |

前回試算（現実的なトラフィックで 100GB/月は十分）を踏まえ、本規模では Pages で実用上問題なし。超過してもソフトリミット（通知ベース、即停止ではない）。

### 8-2. 配信形式の選択 — **`.json.gz` + クライアント解凍を本命**とする

Pages に置くファイル形式は2通りある:

- **方式A（raw `.json` を置く）**: Pages/Fastly の自動圧縮に期待する。
  - 懸念1: 13〜30MB 級への自動圧縮が効くかは**閾値依存で未確認**（効かないと wire 30MB）。
  - 懸念2: raw 一式 ≈ **97MB/バージョン**がデータリポジトリの git 履歴に積み上がる。
- **方式B（gzip 済み `.json.gz` を置き、クライアントで `DecompressionStream` 解凍）**: **採用**。
  - wire サイズが**常に gzip 実測値で確定**（subcontracts 4.4MB / sankey 1.3MB）。Pages の圧縮挙動に依存しない。
  - リポジトリ増分は **≈ 11MB/バージョン**で肥大しない。
  - `DecompressionStream('gzip')` は全モダンブラウザ対応（Chrome 80+ / Firefox 113+ / Safari 16.4+）。
  - Pages は `.gz` を `application/gzip` のまま返す（`Content-Encoding` なし・二重圧縮なし）ので挙動が安定。

### 8-3. データリポジトリ構成

リポジトリ名は公開 URL に露出するため「まるみえ」を避け、例: **`rs-data`**（public 必須 — free プランの Pages は public リポジトリのみ。データ自体が公開情報なので問題なし）。

```text
rs-data/
├── README.md            # データの出所・ライセンス・更新履歴
├── .nojekyll            # Jekyll 処理をスキップ（ビルド高速化・必須ではないが推奨）
└── v20260715/
    ├── sankey-svg-2024-graph.json.gz
    ├── sankey-svg-2025-graph.json.gz
    ├── subcontracts-2024.json.gz
    ├── subcontracts-2025.json.gz
    ├── project-quality-scores-2024.json.gz
    ├── project-quality-scores-2025.json.gz
    └── mof-budget-overview-2023.json.gz
```

- Pages 設定: Settings → Pages → **Deploy from a branch**（`main` / root）。push だけで数十秒後に配信。
- 配信 URL: `https://<user>.github.io/rs-data/v20260715/subcontracts-2024.json.gz`
- 旧バージョンは直近 2〜3 個だけ残して削除（作業ツリーから消しても git 履歴には残るが、~11MB/版なら当面問題なし。肥大が気になった時点で orphan ブランチで作り直す）。

### 8-4. コード側の変更（3節からの差分）

1. `dataUrl()`（3-1）はそのまま。**外部 base 設定時はファイル名に `.gz` を付与**する:

   ```ts
   export function dataUrl(path: string): string {
     const base = process.env.NEXT_PUBLIC_DATA_BASE_URL;
     return base ? `${base}/${path}.gz` : `/data/${path}`;
   }
   ```

2. fetch → JSON の共通ヘルパ `fetchDataJson()` を `app/lib/data-url.ts` に追加し、4箇所の fetch を差し替え:

   ```ts
   export async function fetchDataJson<T>(path: string, init?: RequestInit): Promise<T> {
     const url = dataUrl(path);
     const res = await fetch(url, init);
     if (!res.ok) throw new Error(`fetch failed: ${res.status} ${url}`);
     if (!url.endsWith('.gz')) return res.json();
     const stream = res.body!.pipeThrough(new DecompressionStream('gzip'));
     return new Response(stream).json();
   }
   ```

   - ローカル（env 未設定）は従来どおり `/data/*.json` を素の JSON で取得 → 挙動不変。
   - `AbortController` は `init.signal` 経由でそのまま使える（subcontracts 系の既存実装を維持）。

3. preconnect（3-3）は origin を `https://<user>.github.io` に。実装は同じ（env から `new URL().origin`）。

4. 公開スクリプト `scripts/publish-data.sh`（3-4 の置き換え）:
   - `public/data/*.json.gz` のうち配布対象 7 ファイルを、隣接クローンの `../rs-data/v$(TZ=Asia/Tokyo date +%Y%m%d)/` へコピー
   - `git add && git commit && git push`（データリポジトリ側）
   - 最後に「Vercel の `NEXT_PUBLIC_DATA_BASE_URL` を新バージョンに更新して再デプロイ」を促すメッセージを表示
   - `/data-update` スキルの末尾フローに本手順を追記

### 8-5. ロールアウト手順（5節の置き換え）

1. `rs-data` リポジトリ作成 → Pages 有効化 → 初回データ push。
2. 実機検証:

   ```bash
   curl -sI https://<user>.github.io/rs-data/v20260715/subcontracts-2024.json.gz
   # 期待: HTTP 200 / content-type: application/gzip / access-control-allow-origin: * / x-served-by: cache-...（Fastly）
   ```

3. 本体側のコード変更（8-4）をマージ。env 未設定なので挙動は現状と同一。
4. ローカルで `NEXT_PUBLIC_DATA_BASE_URL=https://<user>.github.io/rs-data/v20260715 npm run dev` にして 4 ページの表示を確認（クロスオリジン + gz 解凍の実地テスト）。
5. Vercel に同 env を設定 → 再デプロイで切替。
6. ロールバックは env 削除 → 再デプロイ（Vercel 直配信に戻る）。

### 8-6. 残論点（6節の更新）

- Pages 側の `Cache-Control` は GitHub 固定（`max-age=600`）で変更不可。バージョン付き URL なので実害は小さい（更新は URL 切替で伝搬し、10分キャッシュはむしろ再訪を安価にする）。`immutable` 長期キャッシュにしたくなったら、その時が R2/独自ドメイン再検討のタイミング。
- 帯域がソフトリミット 100GB/月 に近づいた場合も R2 再検討（カード登録の判断はその時点で）。
- bulk `.gz` 直DL・robots の論点は R2 案と同じ（スコープ外・急がない）。
