# 支出先 `status=unknown` の経緯 — 辞書の鮮度問題

**日時**: 2026-06-24 07:46 (JST)
**発端**: 法人番号 `3310001006466`「株式会社西日本流体技研」（pid=4572 の支出先）が houjin.db に実在するのに、品質データ上は `status=unknown` だった理由の調査。
**結論**: `status` は**古い辞書への名前完全一致のみ**で決まり、houjin.db も法人番号も参照しない。辞書が2024以前ベースで古く、**2025支出先名の約45%が `unknown`** 化している（実在企業多数を含む）。

---

## 1. `status` の決まり方（houjin.db も法人番号も使わない）

`scripts/score-project-quality.py` で、支出先「名」を `recipient_dictionary.csv`（26,192エントリ）に**完全一致**で検索し、status を付与する。

```text
L325  if recipient_name in dict_map:        # 厳密辞書に「名前」が載っているか
          ... valid / invalid / gov / supp
L342  else:
L343      row_status = 'unknown'            # 辞書に名前が無ければ unknown
```

- 判定キーは **支出先名（文字列完全一致）のみ**。
- **法人番号での照合なし**、**houjin.db への live 照合なし**。
- 辞書 `recipient_dictionary.csv` には `corporate_number`/`name_in_db`/`cn_in_db` 等の列があり、**辞書ビルド時には houjin.db と照合**しているが、それは「辞書に載っている名前」についての付加情報であって、**辞書に無い名前は救済されない**。

---

## 2. 西日本流体技研が `unknown` だった直接原因

- **2025新出**: 2024データに0件、2025に3件（すべて `unknown`）。
- 辞書に**名前でも法人番号でも0件**、表記ゆれ候補（流体技研/西日本流体）も0件。
- → 辞書生成時の支出先名リストに含まれず、辞書エントリに存在しないため `unknown`。
- houjin.db には実在（法人番号 `3310001006466`／長崎県佐世保市）だが、status パイプラインは houjin.db を引かないため救済されない。

---

## 3. 氷山の一角 — 2025の約45%が `unknown`

| 指標 | 件数 |
|------|------|
| 辞書エントリ | 26,192 |
| 2025 支出先ユニーク名 | 26,331 |
| **うち辞書に無い（→`unknown`化）** | **11,804（約45%）** |

`unknown` には**明らかな実在企業が大量に**含まれる（例: NTT西日本株式会社、クラウドエース株式会社、三菱電機ビルソリューションズ株式会社茨城支店、シミックソリューションズ株式会社 …）。houjin.db には載っているのに status では `unknown`。

→ **根本原因は「辞書の鮮度」**。`recipient_dictionary.csv` は2024以前のデータで構築されたまま2025向けに再生成されておらず、2025新出名・カバレッジ外の名前がすべて `unknown` に落ちる。

---

## 4. 重要: `unknown` は「実在しない」を意味しない

`status=unknown` ＝「（古い）辞書に名前が無い」だけ。特定不能や架空を意味しない。西日本流体技研は実在企業で houjin.db に法人番号付きで存在する。**旧 status は支出先の実在性ではなく「辞書カバレッジ」を測っている**（これは品質スコア再設計の動機そのもの。設計: `20260616_1749_AI支出先データ品質スコアリング再設計.md`）。

---

## 5. AI再設計による補正（実害は限定的）

新しい `/quality`（AI評価）は、status に依存せず **名前＋法人番号(有無)＋契約概要から identifiability を直接判定**する。実例:

- 西日本流体技研（status=unknown）→ **AI が identify=3** と判定（pid=4572 検証 `20260623_2010`）。axisIdentify=100。

→ **旧 status は辞書次第で不正確（45%unknown）だが、AIスコアは name+CN から特定可能性を判定するため、この欠陥を実質的に克服している。** ただし per-recipient の `s`（status）フィールド自体は依然として古い辞書由来の値であり、UI やゲーティング（opaque判定とは別）で status を直接表示・利用する箇所がある場合は誤解を招きうる。

---

## 6. 再生成時の留意点 / 改善案

- **辞書 `recipient_dictionary.csv` を最新の支出先名（2025含む）で再生成**すれば `unknown` 率は大きく下がる。生成スクリプトはリポジトリ内に見当たらず（事前ビルド成果物）、houjin.db との照合で `name_in_db`/`cn_in_db`/`valid` を再計算する仕組みと推測される。
- もしくは **status 判定に法人番号→houjin.db の live 照合を追加**すれば、辞書未収録でも CN があれば実在判定で救済できる（西日本流体技研のようなケース）。
- 当面は **AIスコア（axisIdentify 等）が最終指標**なので致命的ではないが、status を表示・フィルタに使う箇所では「unknown=辞書未収録であって実在性ではない」点に注意。

---

## 7. houjin.db カバレッジ検証 — 「DBの再DLは前提として不要」（2026-06-24）

「辞書を直す前に元DB(houjin.db)を再DLすべきか」を実データで検証した。

| 項目 | 値 |
|------|------|
| houjin.db 鮮度 / 件数 | **2026-01-30 更新 / 5,741,920 件**（十分新しい） |
| 2025 支出先で 法人番号13桁あり | 157,729 行 |
| **そのうち現 houjin.db に法人番号で存在** | **約 85%**（サンプル2,000で1,719件） |
| unknown企業の名前ヒット（サンプル6社） | 5社ヒット（西日本流体技研/クラウドエース/伊良コーラ/桑和貿易/旭日産業）、AVITAのみ名称表記差で0 |

### 結論

- **現在の houjin.db だけで、法人番号付き支出先の約85%が引ける**。unknown化していた実在企業の大半は**既に現DBに収録済み**。
- → **問題は DB の鮮度・内容ではなく「status パイプラインが houjin.db を法人番号で引いていないこと」**（名前完全一致の古い辞書のみ参照）。**再DLしても unknown はほとんど減らない**。
- 残り約15%（DBに無い分）は主に **2026-01-30以降の新規登録**（再DLで一部回収）／**個人・外国法人・番号書式エラー**（構造的で再DL不可）。→ **再DLの効果は限定的**。
- 生CSV（法人番号付き `data/year_2025/5-1_*.csv`）と現 houjin.db が**ともにローカルにある**ため、**再DLなしで「CN→houjin.db 照合」を即実装/辞書再生成できる**。

### アクション順（更新）

1. **houjin.db 再DLは保留**（現DBで85%カバー、まず仕組みを直す）。
2. **status 判定に「法人番号→houjin.db 照合」を追加**（`score-project-quality.py` の `else: unknown` 前に1段）、または辞書を **CN照合ベースで再生成**。名前一致（AVITA の表記差で漏れた例）より **CN照合の方が堅牢**。
3. 最大カバレッジが要る段階で houjin.db を最新化（NTAは定期更新。上積みは限定的）。
