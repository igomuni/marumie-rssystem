# サブコントラクトエッジ 代表選択ロジックの排除

## 問題

現在の`generate-sankey2-data.ts`のサブコントラクトエッジ生成で、ブロック名が支出先ノードに一致しない場合に**ブロック内の最大金額支出先を代表として選ぶ**ロジックが存在する。これはCSVにない関係を捏造する誤った処理である。

## 具体例: PID=5874（地域計画策定推進緊急対策事業）

5-2 CSVのブロック接続:

```text
担当組織(農林水産省) → A(地方農政局等)
A(地方農政局等) → B(都道府県)
B(都道府県) → C(市町村)
C(市町村) → D(農業委員会)
```

ブロックCの支出先（5-1 CSV、その他除外・金額順）:

| 支出先 | 金額 |
|--------|------|
| 中野市 | 0.11億 |
| 鶴岡市 | 0.11億 |
| 奥州市 | 0.10億 |
| ... | ... |

ブロックDの支出先: 朝倉市農業委員会（0.02億）等

**現在の生成結果**: `中野市 → 朝倉市農業委員会 0.02億 pids=[5874]`

CSVにこの関係はない。ブロック名「市町村」が支出先名に一致しないため、ブロックC内の最大金額支出先「中野市」が代表として選ばれた結果の**捏造エッジ**。

## 同種の問題: PID=3440

```text
B(都府県) → D(市町村営事業)
```

ブロックBの代表として「熊本県」（最大金額58.47億）が選ばれ、`熊本県 → 朝倉市 12.88億` が生成される。CSVにこの関係はない。

## 排除すべきロジック

`generate-sankey2-data.ts` のサブコントラクトエッジ生成（5iセクション）:

```typescript
// 現在のコード（誤り）
const sourceRecipients = (blockSpending.get(`${pid}:${conn.source}`) ?? [])
  .filter(r => r.name !== 'その他');
if (sourceRecipients.length === 0) { skippedNoSource++; continue; }

// 代表 = 最大金額の支出先 ← この選択が捏造の元
const representative = sourceRecipients.reduce((best, r) => r.amount > best.amount ? r : best);
```

このロジックは「ブロック名 ≠ 支出先名」の場合に発動し、**ブロック内の無関係な支出先を委託元として偽装する**。

## 正しい挙動

ブロック名が支出先ノードとして存在しない場合、サブコントラクトエッジは**生成しない**。

CSVが提供するのはブロック間の接続情報（B→D）のみであり、ブロック内の個別支出先間の対応は不明。不明な関係をエッジとして表現すべきではない。

## 影響

ブロック名が支出先名に一致しない接続は2,933件中約2,180件（ソース支出先が複数のケース）。これらのサブコントラクトエッジが消失する。

ただし:
- パネルの`subcontractChains.blockChain`には影響なし（ブロック名で表示）
- ブロック構造の情報はパネルから引き続き参照可能
- 68.4%のブロック接続はソース支出先が1件で、代表選択問題が発生しない

## 付随する問題: 間接支出先のレイアウト消失

サブコントラクトエッジの排除に連動して、以下の問題がある:

`compute-sankey2-layout.ts`の府省庁帰属計算は`project-spending → recipient`の直接エッジのみで決定される。間接のみの支出先（13,391ノード）は帰属が決まらず、layoutNodeMapから除外される。

現在はサブコントラクトエッジ経由の帰属伝搬で対応しているが、捏造エッジを排除するとこの伝搬経路も失われ、間接支出先がレイアウトから消失する。

帰属計算にサブコントラクトエッジだけでなく、`subcontractChains`のブロック接続情報（projectId付き）を使う等の別アプローチが必要。
