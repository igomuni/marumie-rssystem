# コンテキスト効率化整理計画

作成日: 2026-04-30

## 背景

会話ごとにロードされるコンテキスト量を削減し、Claude の作業精度と応答速度を改善する。

## 現状のコンテキスト構成

毎会話ロードされるもの：

| ファイル | 行数 | 備考 |
|---|---|---|
| `CLAUDE.md` | 115行 | プロジェクト設定・構成・ガイド参照 |
| `MEMORY.md` | 46行 | 自動メモリ（過去会話から蓄積） |
| system-reminder | 可変 | スキル一覧・deferred tools など |

## 問題点

### 問題1: MEMORY.md に不要なセクションがある（高優先）

| セクション | 問題 | 推奨対処 |
|---|---|---|
| **Key Architecture Notes** | `CLAUDE.md` の Quick Reference・Main Entry Points と完全に重複 | 削除 |
| **Bug Fixes Applied** | 実装済み・`CLAUDE.md` の Known Bugs にもルールあり | 削除（ルールは CLAUDE.md に維持） |
| **Data Patterns** | `accountFilter` の実装状態など、コードで確認できる | 削除 |

削減見込み：約 20 行（46行 → 26行程度）

### 問題2: CLAUDE.md に `/sankey-svg` の記載がない（中優先）

最近の開発の主戦場になっている `/sankey-svg` が Main Entry Points テーブルに存在しない。
作業のたびに Claude がファイル探索を行う無駄が生じている。

追加すべきファイル：

| File | Purpose |
|------|---------|
| `app/sankey-svg/page.tsx` | SVG Sankey UI・Canvas描画・ドリルダウン・サイドパネル |
| `app/lib/sankey-svg-filter.ts` | フィルタロジック（府省庁・事業・支出先・金額） |
| `app/lib/sankey-svg-constants.ts` | レイアウト定数・ノード幅・間隔 |

### 問題3: docs/tasks/ の171ファイルが整理されていない（低優先）

直接ロードはされないが「設計ガイド」と「作業ログ」が混在。古い設計ドキュメントを Claude が誤参照するリスクがある。

## アクションプラン

### Tier 1: 即実施（コンテキスト削減）

1. **MEMORY.md 整理**
   - `Key Architecture Notes` セクション削除
   - `Bug Fixes Applied` セクション削除
   - `Data Patterns` セクション削除
   - 残すもの：Conventions / Feedback / Project Status

2. **CLAUDE.md の Main Entry Points に `/sankey-svg` を追加**
   - `### /sankey-svg（SVG ベース動的 Sankey 図）` セクションとして追加済みの場合は確認のみ
   - ※ 実際には `CLAUDE.md` の `## Main Entry Points` に `/sankey-svg` セクションがある（確認済み）
   - → **対応不要**（既に記載あり）

   補足: CLAUDE.md を再確認したところ `/sankey-svg` セクションは既存。MEMORY.md 整理が主要タスク。

### Tier 2: 任意（整理）

3. **docs/tasks/ のアーカイブ**
   - 2026年3月以前の task ドキュメント（約100ファイル）を `docs/tasks/archive/` へ移動
   - 直接のコンテキスト削減効果はないが、プロジェクト状況の見通しが改善される

## 実施後の期待効果

- MEMORY.md が約 43% 削減（46行 → 26行）
- 毎会話の冗長な重複情報がなくなり、Claude が混乱するリスクが低下
- Known Bugs のルール（`projects.find → filter+reduce`）は CLAUDE.md に残るため、実用上の知識損失なし
