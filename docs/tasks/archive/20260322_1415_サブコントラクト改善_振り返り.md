# サブコントラクト改善 振り返り

## 目的

今回のタスク（サブコントラクトエッジの事業帰属修正）で、うまくいったアプローチと失敗したアプローチを整理し、パイプライン再設計の判断材料とする。

## うまくいったこと

### 1. subcontractエッジへのprojectIds付与

**内容**: サブコントラクトエッジに `projectIds: number[]` を追加し、どの事業由来のフローかを追跡可能にした。

**効果**:
- 18,270件のサブコントラクトエッジのうち92%（16,858件）が単一事業由来 → 帰属の特定が容易
- パネルの事業表示、BFSの探索範囲制御の両方に使える汎用的な情報

**成功要因**: エッジ生成時に情報を付与する設計は、下流のUI・BFSロジックをシンプルに保てた。

### 2. SubcontractGroupListの事業帰属修正

**内容**: パネルの委託グループ表示が、グラフ探索（228エッジを総当たり）で事業を推測していたのを、`edge.projectIds`から直接特定に変更。

**効果**: 朝倉市←北海道の委託エッジが正しくPID=3440（農業施設災害復旧等事業）に帰属するようになった。

**成功要因**: 根本原因の特定後は修正がシンプルだった。

### 3. ゼロ支出directBlockの表示

**内容**: 直接支出ブロックに支出データがない場合でも、パネルにブロック情報を表示するようにした。

**効果**: PID=18（政府共通ネットワーク）の委託構造が完全に見えるようになった。

### 4. データソース整理ドキュメント

**内容**: 使用CSV列の棚卸しと`outflows`の正体を明文化。

**効果**: 再委託構造が「二重構築」されている問題を可視化できた。

## 失敗したこと

### 1. 根本原因の誤認: BFS vs パネル

**経緯**:
1. 朝倉市に「高校生の地域留学」（PID=113）が表示される問題を調査
2. BFSが原因と判断し、BFSにprojectIds制約伝搬を実装
3. ユーザーから「まだUIでは繋がったままです」と指摘
4. 実際の原因はパネルの`SubcontractGroupList`がグラフ探索で事業を推測していたこと

**失敗要因**:
- パネル表示のコードを十分に読まずにBFSに着手した
- 「UIで繋がって見える」＝「BFSで繋がっている」と早合点した
- 実際にはBFS修正も必要だったが、**順序が逆**だった

**教訓**: 表示の問題は、データ→パイプライン→UI表示の順で原因を追うべき。UI側のロジック（パネル）を先に確認する。

### 2. subcontractTargetsアプローチの追加と撤回

**経緯**:
1. 朝倉市農業委員会のエッジ（PID=5874）が`indirectOnlyPairs`で除去されて孤立
2. 「サブコントラクトエッジのターゲットなら直接エッジを残す」ロジックを追加
3. PID=18（政府共通ネットワーク）の直接支出がUI上で51億→28億のはずが膨張
4. 原因: 深い再委託チェーン（レベル4以降）の支出先7件が直接エッジとして復活
5. 全面撤回し、indirect-onlyエッジは無条件削除に戻す

**失敗要因**:
- 個別の症状（朝倉市農業委員会の孤立）を場当たり的に修正しようとした
- `indirectOnlyPairs`の除去ルール自体を再検討せず、例外条件を追加する方向に走った
- 深い再委託チェーンの存在（4段目以降）を考慮していなかった

**教訓**: 場当たり的な例外追加は副作用を生む。除去ルールそのものを見直すべきだった。

### 3. compute-sankey2-layout.tsのprojectIdsパススルー漏れ

**経緯**: generate-sankey2-data.tsでprojectIdsを付与したが、compute-sankey2-layout.tsのGraphEdge/LayoutEdge型にフィールドを追加せず、レイアウトデータでは`projectIds: undefined`になっていた。

**失敗要因**: パイプライン全体（生成→レイアウト→UI）を通して確認しなかった。

**教訓**: 型にフィールドを追加したら、パイプライン上の全ステージで伝搬を確認する。

## 複雑化の根本要因

### A. structured.jsonへの二重依存

現在のパイプライン:

```
CSV (5-1, 5-2)
  → generate-structured-json.ts → structured.json (outflows生成)
  → generate-sankey2-data.ts    → sankey2-graph.json
       ↑ structured.jsonを読み込み
       ↑ 同時に5-2 CSVも直接読み込み
```

**問題点**:
- `outflows`（structured.json経由）と5-2 CSV直接読み込みの**二つの経路**で同じ再委託情報を処理
- `outflows`は名前マッチの脆弱性あり（ブロック名と支出先名の不一致）
- `isDirectFromGov`も`structured.json`経由で取得 → 5-2 CSVの`支出元の支出先ブロック`が空かどうかで判定可能なのに

### B. 間接エッジの処理が後付け

1. 5-1 CSVの全支出行からエッジを生成（直接・間接区別なし）
2. structured.jsonの`isDirectFromGov`で間接のみのペアを特定
3. 後からエッジを削除・金額修正

この「全部作ってから間引く」アプローチが:
- `indirectOnlyPairs` → 孤立ノード問題を引き起こした
- `subcontractTargets`のような場当たり的修正の温床になった

### C. ブロック番号ベースの構造と支出先名ベースのエッジが分離

- **ブロック接続**（5-2 CSV）: `A→B→C→D` のような構造的な接続情報
- **サブコントラクトエッジ**: `recipient-X → recipient-Y` の支出先名ベース
- 両者の紐づけは `outflows.recipients[].name` による名前マッチ → 脆弱

## 改善提案

### 方針: 5-2 CSVのブロック接続を第一優先にする

```
CSV (5-1, 5-2)
  → generate-sankey2-data.ts → sankey2-graph.json
       5-2 CSV: ブロック接続からサブコントラクトエッジを直接生成
       5-1 CSV: 各ブロックの支出先・金額を取得
       structured.json: 不要（outflows依存を排除）
```

### 具体的な改善点

#### 1. isDirectFromGovの判定を5-2 CSV由来にする

**現在**: structured.jsonの`isDirectFromGov`フラグ
**改善**: 5-2 CSVで`支出元の支出先ブロック`が空の行 → そのブロックは「直接支出」

これにより:
- structured.jsonへの依存を1つ排除
- 「全部作ってから間引く」ではなく「最初から直接分だけエッジを作る」

#### 2. サブコントラクトエッジをブロック接続から直接生成

**現在**: outflows.recipients[] → 支出先名マッチ → recipient→recipientエッジ
**改善**: 5-2 CSVのブロック接続（A→B）→ 5-1 CSVのブロックAの支出先 → ブロックBの支出先 → エッジ生成

これにより:
- outflows依存を排除
- 名前マッチの脆弱性を解消（ブロック番号ベースで確実に紐づく）
- projectIdsは5-2 CSVの`予算事業ID`から直接取得可能

#### 3. パネル用データとグラフ用データの統一

**現在**: `subcontractChains.blockChain`（パネル用）と`edges[edgeType='subcontract']`（グラフ用）が別構築
**改善**: 同一のブロック接続処理から両方を生成

#### 4. structured.json依存の残存確認

`isIndirect`と`chainPaths`の付与にもstructured.jsonを使用中。これらも5-2 CSVベースで生成可能か検討が必要。

## まとめ

| 区分 | 項目 | 状態 |
|------|------|------|
| 成功 | projectIds付与 | 完了・効果的 |
| 成功 | パネル事業帰属修正 | 完了・根本解決 |
| 成功 | BFS制約伝搬 | 完了・必要な修正 |
| 失敗 | 根本原因の誤認（BFS先行） | 時間ロス |
| 失敗 | subcontractTargets追加→撤回 | 副作用で撤回 |
| 失敗 | レイアウトパススルー漏れ | 後から気づいて修正 |
| 根本 | structured.json二重依存 | 未解決 |
| 根本 | 間接エッジの後引き方式 | 未解決 |
| 根本 | ブロック番号と支出先名の分離 | 未解決 |

今回の修正はprojectIdsという「情報の追加」で対処したが、パイプラインの構造的な問題（二重依存・後引き・名前マッチ）は残っている。5-2 CSVのブロック接続を唯一のソースとしてパイプラインを再構築すれば、これらの問題を根本的に解消できる。
