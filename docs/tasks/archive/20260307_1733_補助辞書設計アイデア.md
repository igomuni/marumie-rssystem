# 補助辞書設計アイデア

## 目的

厳密辞書（`data/result/recipient_dictionary.csv`）は法人番号DBとの名称照合を基準としており、これは変更しない。
一方、同辞書で `valid=False` となっている8,052種・47,867行のうち、特定のパターンで「実態として有効な支出先」と判別可能なものが多数存在する。

補助辞書を設けて、**厳密辞書invalidの中から辞書完全一致で有効名称を拾い出す**仕組みを設計する。

## 現状の invalid 全量把握

| カテゴリ | 種類数 | 行数 | 補助辞書化方針 |
|---------|------|------|------------|
| 補助辞書に収録済みだが厳密辞書でinvalid | 775種 | 8,680行 | **A. 既存辞書から機械的に生成** |
| 政府地方機関（法人DB名称は親省庁名） | 288種 | 7,354行 | **A. 機械的に生成** |
| 個人系（個人A/B/C..., 個人(イ)...） | 246種 | 7,727行 | **B. パターン辞書化（要設計判断）** |
| 不透明集約（その他、未定等） | 5種 | 7,223行 | 補助辞書化しない（正当invalid） |
| 民間企業（社名変更・旧社名） | 1,279種 | 5,564行 | **C. 旧社名辞書（要設計判断）** |
| 職員X/委員X/出張者X系 | 100種 | 2,276行 | **B. パターン辞書化（要設計判断）** |

| 大学（改組・改名） | 83種 | 389行 | **A. 機械的に生成** |
| 残り | 多数 | 約8,000行 | 個別確認 |

## カテゴリA: 機械生成 — 最も確実・最優先（推定16,000行カバー）

### A-1: 既存補助辞書との照合

`field_office_names.csv`, `ministry_names.csv`, `municipality_names.csv` 等の辞書カテゴリに収録済みだが、
厳密辞書では `valid=False` になっているもの（775種, 8,680行）。

具体例（行数上位）:
| 支出先名 | 厳密辞書invalid理由 | 収録辞書 |
|---------|------------------|---------|
| 九州農政局 | CN=農林水産省で名称不一致 | field_office_names |
| 大阪労働局 | CN=厚生労働省で名称不一致 | field_office_names |
| 関東農政局 | CN=農林水産省で名称不一致 | field_office_names |
| 東京労働局 | CN=厚生労働省で名称不一致 | field_office_names |
| 関東地方整備局 | CN=国土交通省で名称不一致 | field_office_names |

**生成方法**: `field_office_names.csv` 等から name 列を読み込み、厳密辞書で `valid=False` のものを補助辞書に追加。

### A-2: 政府地方機関（補助辞書未収録）

厳密辞書で `cn_in_db=True` かつ `db_name_by_cn` が省庁名（農林水産省, 厚生労働省等）で、名称不一致のもの（288種, 7,354行）。

代表的なパターン:
- **農政局**: 九州農政局, 関東農政局, 東北農政局, 北陸農政局, 東海農政局, 中国四国農政局, 近畿農政局
- **労働局**: 都道府県労働局 (東京, 大阪, 神奈川, 埼玉, 千葉, 愛知, 兵庫, 福岡, 北海道 等)
- **税務署**: 麹町税務署, 名古屋税務署 等
- **法務局**: 東京法務局, 名古屋法務局, 大阪法務局, 福岡法務局 等
- **その他出先**: 沖縄総合事務局

**生成方法**: 厳密辞書から `cn_in_db=True` かつ `db_name_by_cn` が省庁名のエントリを抽出してリスト化。
ただし `field_office_names.csv` に未収録のものは手動確認を推奨。

## カテゴリB: パターン辞書化 — 設計判断が必要（推定10,000行）

### B-1: 個人の匿名化（246種, 7,727行）

**個人の実在する支出先を個人情報保護のために匿名化したもの。**

| パターン | 件数 | 例 | 文脈 |
|---------|------|-----|------|
| 個人A/B/C...（アルファベット） | 推定2,600行 | 個人A, 個人B | 各種補助金の個人受給者 |
| 個人(イ)/(ロ)...(カタカナ) | 推定300行 | 個人(イ), 個人(ヌ) | 道路用地取得の土地所有者 |
| 個人1/2/3...（数字） | 推定150行 | 個人1, 個人2 | 各種 |
| 個人（括弧なし単体） | 144行 | 個人 | 不透明（誰かわからない）|

**設計上の論点**: `個人A`, `個人B` はデータとして「実在する個人への支払い」を示す。名称品質は「実在を確認できないが、実在するものとして記録されている」状態。
- **valid として拾う場合**: 補助辞書に `個人A`〜`個人Z` を収録（26エントリ）
- **invalid のままにする場合**: 誰への支払いか特定できないため invalid を維持

※ `個人`（単体）は不透明辞書（opaque_recipient_keywords）で level=2 として既に扱い済みのため対象外。

### B-2: 職員・委員・出張者の匿名化（100種, 2,276行）

内部的な個人への支払い（旅費, 謝金等）で個人名を匿名化したもの。

| パターン | 種類数 | 行数 | 例 |
|---------|------|------|-----|
| 職員X | 38種 | 1,300行 | 職員A, 職員B...職員Z等 |
| 委員X | 23種 | 540行 | 委員A, 委員B...委員I等 |
| 出張者X | 27種 | 287行 | 出張者A, 出張者B, 出張者 |
| 有識者X | 12種 | 149行 | 有識者A, 有識者B |

**設計上の論点**: 職員A/B/C...は「実在する職員への旅費・手当支払い」。カテゴリとして valid か否かは設計方針次第。
- **valid として拾う場合**: 補助辞書に exact match で `職員A`〜`職員Z` 等を収録
- 各種変形（`職員(A)`, `職員１`等）も考慮すると exact match では漏れが生じる

## カテゴリC: 旧社名辞書 — 判断がもっとも難しい（1,279種, 5,564行）

法人番号が1件特定されているが、DB上の正式名称と支出先名が異なるケース。
主因は**社名変更**（合併・改称）で、支出当時は正しい名称だった。

| 支出先名（旧名称） | 現DB名 | 行数 |
|--------------|-------|------|
| エヌ・ティ・ティ・コミュニケーションズ株式会社 | ＮＴＴドコモビジネス株式会社 | 448 |
| 東日本電信電話株式会社 | ＮＴＴ東日本株式会社 | 206 |
| 株式会社エヌ・ティ・ティ・データ経営研究所 | 株式会社ＮＴＴデータ経営研究所 | 198 |
| 富士ソフト株式会社 | Ｆマネジメント株式会社 | 104 |
| 国立大学法人東京工業大学 | 国立大学法人東京科学大学 | 160 |
| 西日本電信電話株式会社 | ＮＴＴ西日本株式会社 | 80 |

**設計上の論点**:
- 支出当時の名称で記録 → データの記録として正しい → **valid とする考え方**
- 現在は存在しない法人名 → 名称品質として invalid という厳密辞書の判断は合理的
- 補助辞書に入れた場合、「過去の社名が使われている」という情報が失われる

**折衷案**: `category=old_name` として補助辞書に収録し、スコアへの影響はあくまでオプション扱いにする。

## カテゴリD: JV・共同体 — 正規表現対応（270種, 675行）

建設JV（特定建設工事共同企業体）やコンサルの設計共同体。プロジェクトごとに名称が変わるため exact match では辞書化困難。

| パターン | 例 |
|---------|-----|
| `企業名・企業名特定建設工事共同企業体` | 前田・鴻池・田中特定建設工事共同企業体 |
| `企業名・企業名設計共同体` | 道路新産業開発機構・長大設計共同体 |
| `企業名・企業名共同企業体` | 三菱・鹿島共同企業体 |

**exact matchの限界**: exact match で収録しようとすると270エントリが必要になり、毎年の支出データ更新のたびに新規JV名が発生する。

**対応案**:
- `opaque_recipient_keywords.csv` と同様の構造で **pattern辞書**（match_type=regex）を新設
- パターン例: `^.+・.+(共同企業体|設計共同体|建設共同体)$`
- これは exact match に特化する方針とは異なるため、別ファイルとして管理

## 補助辞書ファイル設計案

```
public/data/dictionaries/supplementary_valid_names.csv
```

| フィールド | 説明 |
|---------|------|
| `name` | 支出先名（完全一致） |
| `category` | 分類（government_branch / old_name / university_rename / anonymous_individual / anonymous_internal） |
| `note` | 補足（例: 農林水産省の地方農政局, NTT→ドコモビジネスに改称 等） |

### スコアリングへの反映

スコアリングスクリプト（`score-project-quality.py`）での適用イメージ:

1. 厳密辞書ロード（既存）
2. **補助辞書ロード（新設）** → supplementary_map: name → category
3. 支出先名判定:
   - 厳密辞書 `valid=True` → valid_count++
   - 厳密辞書 `valid=False` かつ補助辞書にあり → **supp_valid_count++**
   - 厳密辞書 `valid=False` かつ補助辞書にもなし → invalid_count++

4. 軸1スコア計算:
   - 現行: `axis1 = valid / (valid + invalid) * 100`
   - 補助考慮: `axis1 = (valid + supp_valid) / (valid + supp_valid + invalid) * 100`

JSON出力にも `suppValidCount` を追加すれば、どのくらいが補助辞書で救済されたか可視化できる。

## 対応優先順位まとめ

| 優先 | カテゴリ | 対象行数 | 作業量 |
|-----|---------|---------|------|
| 1 | A-1: 既存辞書収録済み → 機械生成 | 8,680行 | 低（スクリプト1本） |
| 2 | A-2: 政府地方機関（農政局・労働局・税務署等） | 7,354行 | 低（厳密辞書から抽出） |
| 3 | A-3: 大学改組 | 389行 | 低（小規模） |
| 4 | B-1: 個人(X)匿名化（設計判断後） | 7,727行 | 中（方針決定が必要） |
| 5 | B-2: 職員/委員/出張者(X) | 2,276行 | 中 |
| 6 | C: 旧社名（設計判断後） | 5,564行 | 中〜高 |
| 7 | D: JV・共同体（正規表現） | 675行 | 高 |

カテゴリA（最優先3つ）だけで約**16,400行**をカバーできる見込み。
全体の invalid 47,867行の約**34%**が機械的に解決可能。
