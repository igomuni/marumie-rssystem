# Claude Code 向けツール整備計画

作成日: 2026-04-30

Claude Code の作業効率と再発明防止を目的に、CLIツール・Pythonスクリプト・Skills の3種類を整備する計画。

---

## 1. Python スクリプト（`scripts/` 配下に新規作成）

### `scripts/check-project-state.py`（優先度: 高）

会話開始時に Claude が1コマンドで現状把握できるサマリーを出力する。

```
$ python scripts/check-project-state.py
Branch: chore/xxx (3 commits ahead of main)
Data:   rs2024-structured.json.gz ✓
        sankey2-graph.json.gz     ✓
        sankey2-layout.json       ✗ (要: npm run compute-sankey2-layout)
Lint:   0 errors, 4 warnings (pre-existing)
Types:  OK
```

**解決する問題**: 毎回「どのファイルがあるか」「lintは通るか」を個別に確認している手間を1コマンドに集約する。

出力項目:
- 現在のブランチ・main との差分コミット数
- データファイル（.gz / .json）の存在状況と次に必要なコマンド
- lint・TypeScript チェック結果

---

### `scripts/task-archive.py`（優先度: 高）

古い task ドキュメントを一括でアーカイブする。

```
$ python scripts/task-archive.py --before 2026-03-01
Moved 98 files → docs/tasks/archive/

$ python scripts/task-archive.py --dry-run --before 2026-03-01
[dry-run] Would move 98 files → docs/tasks/archive/
```

**解決する問題**: `docs/tasks/` に 171ファイルが混在し、Claude が古い設計ドキュメントを誤参照するリスクがある。定期実行で抑制する。

オプション:
- `--before YYYY-MM-DD`: 指定日より前のファイルを対象
- `--dry-run`: 実際には移動せず対象一覧のみ表示

---

### `scripts/data-status.py`（優先度: 中）

データパイプラインのどのステップが完了しているか確認する。

```
$ python scripts/data-status.py
[✓] data/year_2024/          (normalized CSV)
[✓] public/data/rs2024-structured.json.gz
[✓] public/data/sankey2-graph.json.gz
[✗] public/data/sankey2-layout.json   ← 要: npm run compute-sankey2-layout
[✗] public/data/sankey2-layout.json.gz ← 要: npm run compress-data
```

**解決する問題**: `data-update` スキル実行前に Claude が必要なステップを判断するため、毎回個別のファイル確認を行っている。`check-project-state.py` のデータ部分を詳細化したもの。

※ `check-project-state.py` を先に作る場合、データ確認ロジックをそこに含めれば本スクリプトは不要になる可能性あり。

## 2. Skills（`.claude/commands/` 配下に新規作成）

### `/quality-check`（優先度: 高）

lint と TypeScript チェックを1コマンドで実行する。

```
/quality-check
```

**解決する問題**: `pr` スキル実行前だけでなく、実装中にも何度も走らせるが、毎回 Claude が2コマンドを組み立て直している。

実行内容:
```bash
npm run lint && npx tsc --noEmit
```

---

### `/investigate`（優先度: 高）

`scripts/` 配下の分析スクリプト群を用途別に案内・実行する。

```
/investigate label-coverage     → analyze-label-coverage.py
/investigate invalid            → analyze-invalid-recipients.py
/investigate project-quality    → score-project-quality.py
/investigate mof                → analyze-mof-*.ts 系
/investigate houjin             → investigate-houjin-quality.py
/investigate unlabeled          → investigate-unlabeled-with-houjin.py
```

**解決する問題**: `scripts/` に `analyze-*.py`・`investigate-*.py`・`score-*.py` が 30本以上あるが、Claude が存在を知らず毎回ゼロから調査コードを書こうとするリスクが高い。スキル内で用途→スクリプトのマッピングを明示することで再発明を防ぐ。

スキル内に含める情報:
- 用途別スクリプト一覧と実行コマンド
- 各スクリプトの出力形式と読み方
- 新規調査が必要な場合の既存スクリプト流用方針

---

## 3. 既存スキルとの関係

| 新規 | 既存スキルとの棲み分け |
|---|---|
| `/quality-check` | `/pr` は PR 作成時のみ。開発中の頻繁な品質確認は `/quality-check` |
| `/investigate` | `/sankey` はコード実装。データ調査・分析は `/investigate` |
| `check-project-state.py` | `/data-update` は更新実行。状態確認は別コマンド |

---

## 実施優先順

| 優先 | アイテム | 理由 |
|---|---|---|
| 1位 | `/investigate` スキル | 30本超のスクリプトの再発明防止が最大効果 |
| 2位 | `scripts/check-project-state.py` | 会話開始コストの削減 |
| 3位 | `/quality-check` スキル | 実装が2行・頻出パターン |
| 4位 | `scripts/task-archive.py` | ドキュメント肥大化の定期抑制 |
| 5位 | `scripts/data-status.py` | `check-project-state.py` と重複する可能性あり |
