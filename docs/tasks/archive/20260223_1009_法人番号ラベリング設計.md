## 1. 目的

Step 1（完全一致辞書）・Step 2（格パターン）でカバーできない名称のうち、法人番号が付与されているものを `法人種別` コードで追加分類し、金額カバレッジを向上させる。特に `399 その他設立登記法人`（全国健康保険協会等 13.92兆円）が主要ターゲット。

### 2-4. houjin.db の実態（`build-houjin-sqlite.py` のバグ）

> **重要**: `build-houjin-sqlite.py` には列定義のバグがある。`houjin.db` の `type_code` 列は法人種別ではなく**更新区分**（最終更新の事由）が格納されている。

NTA ZIP（全件ファイル）の実際の列構造（30列）：

| インデックス | 内容 | build スクリプトの認識 | 実際の値例 |
|------------|-----|---------------------|----------|
| `[2]` | **更新区分** | 「法人種別」と誤認 → `type_code` に格納 | `01`（新規）, `12`（住所変更）, `71`（更正） |
| `[8]` | **NTA 3桁法人種別コード** | 「郵便番号」と誤認 → 未収録 | `101`, `201`, `301`, `399`... |
| `[15]` | 郵便番号（7桁） | `[8]` と誤認 → 未収録 | `0850824` |

更新区分コードの分布（`houjin.db` 全 574万件）：

| type_code | 更新区分の意味 | 件数 |
|-----------|-------------|-----:|
| `01` | 新規 | 3,928,525 |
| `12` | 国内所在地の変更 | 922,666 |
| `21` | 清算の結了等 | 743,825 |
| `11` | 商号又は名称の変更 | 114,308 |
| `71` | 提供情報更正 | 29,280 |

**結論**: `houjin.db` は法人種別の参照には使えない。法人種別 3桁コードの権威ソースは NTA ZIP のカラム `[8]`。

## 3. データ品質調査結果（`scripts/investigate-houjin-quality.py` 実行結果）

### 3-1. 法人番号フォーマット異常 — 2件（軽微）

| 支出先名 | 法人番号 | 問題 |
|---------|---------|------|
| 新技術活用による... | `0` | 支出先名欄に契約概要が誤入力されたと推定 |
| みずほリサーチ&テクノロジーズ株式会社 | `901000102685`（12桁） | 先頭 `9` が余分（正規は13桁） |

影響は軽微。対処はデータパイプライン側で検討。

### 3-2. 同一支出先名で複数の法人番号 — 305件

「大阪府」「埼玉県」「北海道」等の都道府県・政令市名に複数の法人番号が混在する。

代表例: 「埼玉県」に4種類の法人番号（本庁・教育委員会・警察・特殊機関が各自の番号で記入）。

> **評価**: 単純な入力ミスではなく、**RS CSV の運用上の構造的問題**（「法人 = 部局単位」と記録している省庁・都道府県が存在）。Step 3 のラベリングには直接影響しないが、名称から法人番号を一意に特定する補完処理の限界となる。

### 3-3. 同一法人番号で複数の支出先名 — 660件 / 23.52兆円

同一の法人番号に複数の支出先名が記録されているケース。**ラベリング精度に直接影響する最重要問題**。

| 法人番号 | houjin.db 公式名 | RS CSV 支出先名の数 | 主な支出先名例 |
|---------|---------------|:---------:|----------|
| `2000012100001` | 国土交通省 | **62個** | 中国地方整備局、北海道開発局、東京航空局... |
| `8000020130001` | 東京都 | **8個** | 東京消防庁、警視庁、東京都水道局... |
| `7000020010006` | 北海道 | **8個** | 北海道警察、北海道企業局... |

> **解釈**: 省庁・都道府県は**1法人番号を全部局で共有**し、支出先名は部局名で記録している。NTA ZIP カラム `[8]` の法人種別は「法人」単位（国土交通省全体 → 101）だが、RS CSV では部局名ごとに `法人種別` が別コードに設定されることがある。

### 3-4. 法人種別コードの誤分類 — 28件 / 7.92兆円

RS CSV の `法人種別` と NTA ZIP カラム `[8]` の不整合。主な誤分類パターン：

| パターン | RS法人種別 | NTA正解 | 件数 | 代表例 |
|--------|-----------|--------|----:|-------|
| 地方公共団体を誤分類 | `399` or `499` | `201` | 7件 | 東京消防庁・警視庁(499)、北海道警察各部(399)、千葉県警察(399) |
| 国の機関を誤分類 | `301` or `399` | `101` | 4件 | 国土交通省系部局(301)、国立医薬品食品衛生研究所(301)、林野庁(399) |
| 会社形態の旧コード残存 | `305` or `302` | `301` | 10件 | 合同会社→株式会社転換後も旧コードのまま等 |

> **最大の問題**: 国土交通省 `[2000012100001]` に RS法人種別=`301`（株式会社）が設定されており、NTA 正解は `101`（国の機関）。3.57兆円規模の誤分類。

### 3-5. 法人番号なし補完候補 — 118件（高精度） / 0.05兆円

法人番号が空の支出先名 6,392件（90.34兆円）のうち、houjin.db への名称完全一致で補完可能なもの：

- **単一ヒット（高精度）**: 118件 / 約500億円
- **複数ヒット（同名法人が複数）**: 37件（補完困難）
- 上位例: 日本司法支援センター（343億円）、出入国在留管理庁（108億円）

> 法人番号なしの90兆円大半は「その他」系集合名称や行政機関部局名であり、名称完全一致では補完できない。**補完の効果は限定的（0.05兆円）のため、優先度は低い。**

## 4. Step 3 設計

### 4-1. ラベリング対象と処理順序

Step 3 を以下の2段階で構成する：

**Step 3a（法人種別コードによるラベリング）**
Step 1・Step 2 でカバーされていない支出先名のうち、`法人種別` コードが存在するもの。法人種別コードは RS CSV の値を基本とし、NTA ZIP カラム `[8]` で誤分類補正を適用した後の値を使う。

| 法人種別コード | 未カバー件数 | 金額 | 追加ラベル |
|-------------|----------:|-----:|---------|
| `399` その他設立登記法人 | 490件 | **13.92兆円** | 設立登記法人（特殊法人等） |
| `201` 地方公共団体 | 199件 | 0.04兆円 | 地方公共団体 |
| `499` その他 | 188件 | 0.04兆円 | その他法人 |
| `101` 国の機関 | 47件 | 0.02兆円 | 国の機関 |
| `301` 株式会社 | 46件 | 0.01兆円 | 民間企業（株式会社） |
| `401` 外国会社等 | 29件 | 0.01兆円 | 外国法人 |
| 計 | **≈ 1,000件** | **≈ 14.03兆円** | |

**Step 3b（法人番号なし補完）**

houjin.db 名称完全一致で単一ヒットの 118件に法人番号・法人種別コードを付与し、Step 3a のラベリングに接続する。金額効果は 0.05兆円と小さいため、実装は後回しでよい。

### 4-2. 法人種別コード → ラベルのマッピング

Step 1・Step 2 の分類体系に揃えて、以下のラベルを付与する：

| コード | L1（大分類） | L2（中分類） | 備考 |
|--------|-----------|-----------|------|
| `101` | 国の機関 | 国の機関 | |
| `201` | 地方公共団体 | 地方公共団体 | |
| `301` | 民間企業 | 株式会社 | |
| `302` | 民間企業 | 有限会社 | |
| `303` | 民間企業 | 合名会社 | |
| `304` | 民間企業 | 合資会社 | |
| `305` | 民間企業 | 合同会社 | |
| `399` | 設立登記法人（その他） | 特殊法人等 | 全国健康保険協会・日本年金機構等、名称から格を特定できない法人 |
| `401` | 外国法人 | 外国会社等 | |
| `499` | その他法人 | その他法人 | |

### 4-3. 法人種別コードの確定ルール

同一支出先名に複数の法人種別コードが存在する場合（§3-2の305件等）は、以下の順で解決する：

1. **NTA ZIP カラム `[8]` の値が取得できる場合**: その値を正とする（§3-4の誤分類28件も同時に修正される）
2. **NTA ZIP 未ヒットの場合**: RS CSV 法人種別コードの**優先順位**で解決
   優先順位: `201` > `101` > `399` > `301`〜`305` > `401` > `499`

## 6. 実装方針

### 6-1. スクリプト構成

| スクリプト | 役割 |
|---------|------|
| `scripts/investigate-houjin-quality.py` | データ品質調査（実装済み） |
| `scripts/analyze-label-coverage.py` | Step 3a セクションを追加（次のタスク） |

### 6-2. analyze-label-coverage.py への追加処理フロー

```
1. RS CSV から各ユニーク名称の 法人番号・法人種別コードを集計

2. NTA ZIP を読み込み、法人番号 → NTA 3桁コード（カラム[8]）の辞書を構築
   ※ RS CSV に出現する有効法人番号のみ対象（全件走査は負荷が高いため必要分のみ）

3. 法人種別コード確定
   for each ユニーク名称:
     cn_set = RS CSV で記録された法人番号セット
     if cn_set ∩ nta_zip_hits ≠ ∅:
       determined_code = NTA ZIP カラム[8] の値（cn_set 内で最多数決 or 優先順位）
     else:
       determined_code = RS CSV type_set 内で優先順位上位のコード

4. Step 3a ラベリング
   for each 未カバー名称（Step 1・Step 2 ともヒットしなかったもの）:
     if determined_code ≠ '':
       label = KTYPE_MAP[determined_code]

5. Step 3a カバレッジ集計・レポート出力

6. （Step 3b: オプション）houjin.db 名称照合による補完候補を表示
```

### 6-3. NTA ZIP 読み込みの実装上の注意

- ZIP 全件読み込み（574万件）は実行時間が長い可能性があるため、RS CSV に出現する有効法人番号セットを先に構築し、ヒット時のみ辞書に追加する
- NTA ZIP が存在しない場合は RS CSV の `法人種別` 列をそのまま使う（フォールバック）
