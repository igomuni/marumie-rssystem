# /sankey-svg ノード高さ設計考察

## 現状の高さ基準（実装ベース）

| ノード | 現在の高さ基準値 | 備考 |
|--------|----------------|------|
| 総計 | ウィンドウ内支出先の合計 | 全支出でなくウィンドウ表示分のみ |
| 府省庁（Top N） | 各省庁のウィンドウ内支出合計 | |
| 府省庁（その他集約） | 非TopN省庁のウィンドウ内支出合計 | |
| 事業(予算)（Top N） | **元の予算額**（CSV予算値） | `skipLinkOverride:true` で保護。支出先が非表示の場合は `layoutCap` で支出額に制限 |
| 事業(予算)（その他集約） | ウィンドウ内支出合計（`otherProjectWindowTotal`） | ← 予算データでなく支出データになっている |
| 事業(支出)（Top N） | **ウィンドウ内支出額のみ**（`projectWindowValue`） | tail支出先へのエッジは高さに含まない |
| 事業(支出)（その他集約） | ウィンドウ内支出合計 + `layoutCap` で上限あり | |
| 支出先（ウィンドウ内） | **全エッジの合計**（ウィンドウ外からの流入も含む） | オフセット前の省庁・事業からの支出も加算 |
| 支出先（tail集約） | tail支出先の合計 + 非TopN事業のtail流 | `layoutCap` で上限あり |

---

## あるべき姿の前提

- **予算系ノード**（総計・府省庁・事業(予算)）：**予算データ**を高さの基準にする
- **支出系ノード**（事業(支出)・支出先）：**支出データ**を高さの基準にする

---

## 課題とトレードオフ

### 1. ウィンドウ表示との整合性

現在、支出先はオフセットスライダーで「ウィンドウ」表示される。これにより：

- ウィンドウ内支出先の合計 < 全支出 （ウィンドウ外の支出先が存在する場合）
- 「予算系は予算額・支出系は支出額」を厳密に守ると、ウィンドウ移動のたびに予算ノードと支出ノードの高さ比が変わらないが、支出先ノードとの釣り合いが崩れる

**選択肢 A: ウィンドウ内合計を基準にする**（現在に近い）
- メリット：画面上のリンク幅と高さが一致しやすい
- デメリット：予算ノードの「全体」感が失われる

**選択肢 B: 全体合計を基準にする**
- メリット：予算・支出の絶対量が画面に表示される
- デメリット：ウィンドウ外にある支出先のリンクが画面外に消えるため、高さとリンクの整合が取れない

**選択肢 C: 列ごとに異なる基準を使う（現実的な折衷案）**
- 予算系（総計・府省庁・事業(予算)）：全体合計 → 固定スケール
- 支出系（事業(支出)・支出先）：ウィンドウ内合計 → 動的スケール
- 課題：列間でスケールが異なるため、リンクの太さと両端ノードの高さが不一致になる

### 2. 集約ノードの高さキャップ問題

現在、集約ノード（`__agg-*`）はレイアウトエンジンの「リンク和でノード高さを上書き」機構により意図しない膨張が起きるため、`layoutCap` で上限を設けている。

**根本原因：**
- `__agg-project-spending` は中間ノード（受信：省庁→集約予算、送信：集約支出→支出先）
- srcSum = ウィンドウ内 + tailフロー全合計 → ノード高さが過大になる
- `__agg-recipient` も同様にtail+ウィンドウ外のフローを受けるため過大になりうる

**あるべき姿（案）：**

| ノード | 推奨高さ基準 | キャップの要否 |
|--------|-----------|-------------|
| 事業(予算)（TopN） | 元の予算額（全体） | ウィンドウ内支出額が上限（支出先非表示時） |
| 事業(予算)（集約） | 集約事業の予算額合計 | 同上 |
| 事業(支出)（TopN） | ウィンドウ内支出額 | キャップ不要（`skipLinkOverride`で制御） |
| 事業(支出)（集約） | ウィンドウ内支出額合計 | キャップ不要（直接設定すれば） |
| 支出先（TopN） | ウィンドウ内受取額 | キャップ不要 |
| 支出先（集約） | tailの受取額合計 | キャップ不要（直接設定すれば） |

### 3. 集約ノードの「直接設定」アプローチ

現在の実装：
1. ノードに `value` を設定
2. レイアウトエンジンが `node.value = max(srcSum, tgtSum)` で上書き（`skipLinkOverride:false` の場合）
3. `layoutCap` で事後的に制限

**改善案：集約ノードにも `skipLinkOverride:true` を使う**
- 集約ノードの `value` を意図した値（ウィンドウ内合計など）に設定してから `skipLinkOverride:true` を付与
- `layoutCap` の複雑なキャップ計算が不要になる
- ただしリンク幅とノード高さの整合は別途調整が必要

## 「ウィンドウ」の概念

支出先をランキング（金額順）した時に、**表示する範囲を切り取ったもの**。

```text
全支出先 26,823件を金額順に並べると:

1位  社会保険診療報酬支払基金   10兆円
2位  日本年金機構             5兆円
...
100位 ○○株式会社
101位 △△法人      ← ここからtail（ウィンドウ外）
...
26,823位
```

`topRecipient=100, recipientOffset=0` なら **1〜100位がウィンドウ内**、101位以降がtail。

オフセットスライダーで `recipientOffset=50` にすると **51〜150位がウィンドウ内**、1〜50位はプレウィンドウ（非表示）、151位以降がtailになる。

## 列合計の差分（事業(支出) vs 支出先）

**事業(支出)合計 < 支出先合計** となる原因:

- 事業(支出)ノードの値は `projectWindowValue`（ウィンドウ内支出先への流れのみ）
- 支出先ノードの値は `allRecipientAmounts`（全エッジの合計）

**差分の正体:**
```
支出先合計 - 事業(支出)合計
= TopN事業のtail支出先（topRecipient位以降）への支出合計
```

例：医療保険給付費国庫負担金等（10兆円事業）の場合、
上位100位以内の支出先への金額は `projectWindowValue` に含まれるが、
101位以降（△△市への10億円など）はリボンとして `__agg-recipient` に流れ、
事業(支出)ノードの合計からは脱落する。

オフセットスライダーを動かすとウィンドウ範囲が変わるため、この差分も変化する。

## 列ラベルの金額表示について

列ラベルの金額は `rawValue ?? value` を使用（キャップ前の元の金額）。
ただしこれは各列の「集計定義」が異なるため、列間の比較には注意が必要：

| 列 | 集計定義 |
|----|---------|
| 総計 | ウィンドウ内支出先合計 |
| 府省庁 | 各省庁のウィンドウ内支出合計 |
| 事業(予算) | **元の予算額合計**（ウィンドウ非依存） |
| 事業(支出) | ウィンドウ内支出先への支出合計のみ（tail除外） |
| 支出先 | 全エッジ合計（ウィンドウ外からの流入含む） |

## 未解決の問題

- **支出先ノードの高さが全エッジ合計になっている**：ウィンドウ外の省庁・事業からの流入も加算されるため、ウィンドウ内の支出リンクの合計とノード高さが一致しない。意図通りかどうか要確認。
- **事業(支出)列合計がtailを除外**：上記の通り、TopN事業のtail支出先への支出が列合計に含まれない。
