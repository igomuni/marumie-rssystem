# d3-sankey 列固定レイアウト実験計画

## 目的

開発者が d3-sankey の `sankeyJustify` による列シフト問題（下流リンクのないノードが最右列に移動する）を解消し、Global View のスライダー操作時に安定した4列構造を維持できるようにするため。

## 問題の根本原因

d3-sankey の `align.js` にある `justify` 関数:

```js
export function justify(node, n) {
  return node.sourceLinks.length ? node.depth : n - 1;
}
```

`sourceLinks` が空（= 下流リンクなし）のノードは **常に最右列 `n - 1`** に配置される。
Global View の4列構造（ministry-budget → project-budget → project-spending → recipient）において、特定レベルで recipient ノードが少ない/0個になると、project-spending ノードが最右列に押し出される。

### d3-sankey のレイアウト処理フロー

1. **`computeNodeDepths`**: BFS で各ノードの `depth`（= 最短パスの深さ）を計算
2. **`computeNodeHeights`**: 逆方向BFS で `height`（= 最長後方パスの深さ）を計算
3. **`computeNodeLayers`**: `align(node, maxDepth+1)` で列インデックスを決定 → `x0`/`x1` を計算
4. **`computeNodeBreadths`**: 列内のy座標を反復緩和法で計算（6回イテレーション）

問題は **ステップ3** で `align = justify` が末端ノードを最右列に寄せること。

## 実験計画

小規模テストデータを使った実験UIを作り、以下の4つのアプローチを比較する。

### 実験用テストデータ

Global View の4列構造を模した小規模データ（3パターン）:

| パターン | 内容 | 検証ポイント |
|---------|------|------------|
| A: 完全形 | 3省庁→3事業→3支出先 | 基準（正常に4列になるか） |
| B: 支出先少 | 3省庁→3事業→1支出先 | 支出先が少ない時の列崩れ |
| C: 支出先なし | 3省庁→3事業→0支出先 | 最悪ケース（recipientゼロ） |

各ノードタイプは実データと同じ id prefix を使用: `ministry-budget-*`, `project-budget-*`, `project-spending-*`, `recipient-*`

### 実験1: `sankeyLeft` への変更

`sankeyLeft` は `node.depth` をそのまま列インデックスに使うため、下流リンクの有無に関係なくBFSで決まった位置を維持する。

- **期待**: 列シフトは発生しない
- **懸念**: `sankeyJustify` は列間を均等分散するが `sankeyLeft` は左寄せになる。列数が動的に変わるとx座標の間隔が詰まる可能性

### 実験2: `sankeyCenter` への変更

`sankeyCenter` は以下のロジック:

```js
export function center(node) {
  return node.targetLinks.length ? node.depth
      : node.sourceLinks.length ? min(node.sourceLinks, targetDepth) - 1
      : 0;
}
```

- 入力リンクあり（project-budget, project-spending, recipient）→ `node.depth`（BFS深さ通り）
- 入力リンクなし＋出力リンクあり（ministry-budget）→ 出力先の最小depth - 1
- 両方なし → 0

- **期待**: `justify` と違い「sourceLinksが空 → 最右列」にならないため、project-spending の列シフトが防止される
- **懸念**: ministry-budget（起点ノード）の列計算が `min(sourceLinks, targetDepth) - 1` になるため、depth=0 が保証されない可能性。実データで列位置が正しいか要検証

### 実験3: カスタム align 関数（列をノードtypeで固定）

ノードの `type` (id prefix) に基づいて固定列を割り当てるカスタム関数を `nodeAlign` に渡す。

```ts
function customAlign(node: { id: string; depth: number; sourceLinks: unknown[] }, n: number): number {
  const id = node.id as string;
  if (id.startsWith('ministry-budget')) return 0;
  if (id.startsWith('project-budget')) return 1;
  if (id.startsWith('project-spending')) return 2;
  if (id.startsWith('recipient') || id.startsWith('subcontract')) return 3;
  return node.sourceLinks.length ? node.depth : n - 1; // fallback
}
```

- **期待**: 列が完全に固定される。d3-sankey の y座標計算（`computeNodeBreadths`）はそのまま活用可能
- **懸念**: `computeNodeLayers` が返す `columns` 配列のインデックスに空の列が生まれる場合、x座標の間隔が不均一になりうる（d3-sankey は `(x1 - x0 - dx) / (maxDepth)` で等間隔配分するため、実際の列数とmaxDepthがずれると隙間が生じる）

### 実験4: ダミーノード挿入

各列に不可視の「アンカーノード」を配置し、d3-sankey に列構造を強制する。

- 各列に `value: 1`（極小値）のダミーノードを追加
- ダミーノード間を最小値のリンクで接続（全4列を貫通するチェーン）
- レイアウト後にダミーノード・リンクを除外して出力

- **期待**: `sankeyJustify` でも列構造が維持される（ダミーノードが各列に存在するため）
- **懸念**: ダミーノードが y座標の計算に影響し、実ノードの配置が微妙にずれる可能性。ダミーリンクの `value` 調整が難しい

### 実験5: レイアウト後のx座標上書き

d3-sankey の計算をそのまま実行し、レイアウト後に各ノードの `x0`/`x1` をノードタイプに基づいて強制上書き。

```ts
// d3-sankey 実行後
for (const node of layoutNodes) {
  const col = getColumnForType(node.type); // 0,1,2,3
  node.x0 = col * columnSpacing;
  node.x1 = node.x0 + NODE_WIDTH;
}
// リンクパスを再生成
generator.update(graph); // y座標のみ再計算
```

- **期待**: y座標は d3-sankey の最適化を活用しつつ、x座標だけ固定
- **懸念**: x座標を変えた後に `update()` しても y 方向のリンク接続座標（`link.y0`, `link.y1`）は既に計算済みなので整合性は保たれるが、リンクのSVGパス（水平ベジェ曲線）は x 変更を反映しない。`sankeyLinkHorizontal` で再生成が必要

## 実験UI

`app/sankey-lab/page.tsx` に実験用ページを作成する。

### 構成

- 左サイドバー: テストデータパターン選択（A/B/C）+ アプローチ選択（実験1〜5）
- メイン領域: d3-sankey レイアウト結果のSVG描画
- 右サイドバー: 各ノードの `depth`, `height`, `layer`, `x0`, `y0` をテーブル表示（デバッグ用）

### 評価基準

| 基準 | 説明 |
|------|------|
| 列固定 | 4列構造が全パターンで維持されるか |
| y座標品質 | ノード・リンクの縦位置が自然か（重なり・隙間） |
| 実装複雑度 | 本番コードへの変更量・影響範囲 |
| 汎用性 | 「その他」集約ノード等の特殊ノードにも対応できるか |
