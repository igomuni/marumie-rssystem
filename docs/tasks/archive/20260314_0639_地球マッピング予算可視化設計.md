# 地球マッピング予算可視化設計

## 目的

予算の利用者（国民・研究者・行政関係者）が、**149.72兆円の支出全体を直感的に把握しにくい状態**を解消するため。

地球の表面積（約510兆m²）を器として、実質支出額を面積にマッピングした球体UIを構築する。Google Mapsのようなズーム・パン操作で、マクロ（全体俯瞰）からミクロ（個別事業）までシームレスに探索できるようにする。

### 組織階層（BudgetTree）

`rs2024-structured.json` の `budgetTree` は最大7階層:

```
府省庁（37省庁）
 └─ 局・庁（BureauNode）
    └─ 部（DepartmentNode）
       └─ 課（DivisionNode）
          └─ 室（OfficeNode）
             └─ 班（GroupNode）
                └─ 係（SectionNode）
```

各ノードが `projectIds[]` と `totalBudget` を保持。中間階層は疎（全省庁が7階層持つわけではない）。`BudgetRecord` の `hierarchyPath: string[]` で非空階層だけの配列も利用可能。

### 球体レンダリング

**採用技術: Three.js（直接使用）**

選定理由:
- WebGLによる球体の3D描画・テクスチャマッピングが自然
- `three/examples/jsm/controls/OrbitControls` でGoogle Maps的なパン・ズーム・チルトが標準装備
- LOD（Level of Detail）制御でズームレベルに応じた詳細度変更が実現可能
- Turbopackとの互換性が確保される（React Three FiberはTurbopack環境で `react-reconciler` の互換性問題あり）

代替案と不採用理由:
- **@react-three/fiber + @react-three/drei**: Turbopackとの互換性問題（`ReactCurrentOwner` エラー）により不採用
- **Globe.gl / react-globe.gl**: 地理データ前提のAPI設計。カスタムテッセレーションには不向き
- **CesiumJS**: 実地理データ向け。抽象的な面積マッピングにはオーバースペック
- **D3 geo orthographic**: 2D投影のみ。球体の裏面が描画できない。ズーム体験がMapと乖離する
- **deck.gl**: GIS向け。球体面への自由なポリゴン配置には制約がある

### 球面テッセレーション（面積分割）

**採用方式: 階層的Voronoi球面分割**

4,617事業を球面上に面積比例で配置する問題。以下の手順で解決する:

#### Step 1: 階層的な重心配置

`budgetTree` の階層構造に従い、球面を再帰的に分割:

1. **第1階層（府省庁）**: 球面全体を37省庁の実質支出比で分割
   - 各省庁の重心を球面上に均等配置（Fibonacci格子など）
   - 重心をもとにPower Voronoiで面積比例の領域を生成
2. **第2階層（局・庁）**: 省庁領域内をさらに局・庁の支出比で分割
3. **以下再帰**: 部→課→室→班→係→個別事業まで

階層が深くなるほど領域が小さくなり、ズームしないと見えない — Google Mapsの市区町村→番地と同じ構造。

#### Step 2: 球面Voronoi

Voronoi on sphere の実装:
- `d3-geo-voronoi` パッケージが球面ドロネー三角形分割→球面Voronoiを提供
- 各生成点に「重み」を付けることでPower Voronoiに近似し、面積を支出比例に近づける
- 反復的にフィードバック補正（現在面積と目標面積の差分で重心を移動）

#### Step 3: LOD（Level of Detail）

ズームレベルに応じて描画する階層深度を変える:

| カメラ距離 | 描画される階層 | 表示される単位 |
|-----------|-------------|-------------|
| 遠い（全球俯瞰） | 第1階層のみ | 府省庁（37領域） |
| 中距離 | 第1-2階層 | 局・庁（~150領域） |
| 近い | 第1-3階層 | 部・課レベル |
| 非常に近い | 全階層 | 個別事業（4,617領域） |

→ カメラ距離に応じて、表示する領域の粒度を動的に切り替える。遠い時は府省庁の1ポリゴンだけ、近づくにつれてサブ領域に分裂するアニメーション。

### 階層的マージン

事業間のマージン（余白）は階層レベルに応じて幅を変える。Google Mapsで国境→県境→市区町村境が順に見えてくるのと同じ構造。

#### マージン幅の階層対応

| 境界の種類 | マージン幅（球面角度） | 視覚メタファー | ズームレベル |
|-----------|---------------------|-------------|------------|
| 府省庁間 | 広い（~1°） | 海峡・大洋 | 全球俯瞰で視認 |
| 局・庁間 | 中程度（~0.3°） | 河川・山脈 | 中距離で出現 |
| 部・課間 | 細い（~0.1°） | 道路・街路 | 近距離で出現 |
| 個別事業間 | 極細（~0.03°） | 垣根・区画線 | 最大ズームで出現 |

#### マージンの面積配分

球面の**約30%をマージン（「海」）**に割り当てる:

- **事業面積**: 球面の70%（≈ 357兆m²）に149.72兆円をマッピング
- **マージン**: 球面の30%（≈ 153兆m²）を階層境界として使用
- **補正後の換算係数**: 1円 ≈ 2.384 m²

この70:30の配分は地球の実際の陸海比率（29:71）を反転させた形。「地球のほとんどが予算で埋まっている」というインパクトを維持しつつ、階層構造を視覚的に分離する。

#### マージンの表現

- **遠景**: 府省庁間の「海」だけが見える。大陸のように省庁領域が分かれている
- **中距離**: 府省庁の「海」は消え（領域が拡大して画面を埋める）、代わりに局・庁間の「川」が出現
- **近景**: 局・庁の「川」も消え、部・課間の「道路」が見え、個別事業の「区画」が判別可能に

マージンはLOD切替と連動して描画する:
- 遠景でレンダリングされる府省庁ポリゴンは、子要素のマージンを含んだ合算形状
- ズームイン時に親ポリゴンが子ポリゴン群に分裂し、マージンが「開いていく」アニメーション

### 配色

球面の「地形」として:

| 属性 | 配色ルール |
|------|----------|
| 府省庁 | 省庁ごとに固有色相（HSLのH値を37分割） |
| 執行率 | 彩度・明度で表現（高執行率=鮮やか、低=くすむ） |

### 情報パネル

選択された領域の詳細を表示:
- 階層パス（パンくず）: 府省庁 › 局 › 課 › 事業名
- 実質支出額・予算額・執行率
- 面積換算値（「山手線○個分」など直感的な表現）
- 品質スコア（該当時）
- 子領域の一覧（ズーム前のプレビュー）

### API設計

`GET /api/map/globe`

レスポンス: 全事業の階層・座標・面積情報

```typescript
interface GlobeMinistry {
  name: string;
  totalSpending: number;
  areaFraction: number; // 全球に対する面積比率（マージン除外後）

interface GlobeBureau {
  name: string;
  totalSpending: number;
  areaFraction: number;
  centroid: [number, number];
  children: GlobeProject[];
}

interface GlobeProject {
  projectId: number;
  name: string;
  spending: number;
  areaFraction: number;
  centroid: [number, number];
  hierarchyPath: string[];
  qualityScore: number | null;
  accountCategory: string;
  executionRate: number;

```

注: 中間階層（部・課・室・班・係）は疎なため、APIでは3階層（府省庁→局・庁→事業）に畳む。表示時にhierarchyPathで詳細を補完する。位置の近接性は局・庁単位で確保される。

### Fibonacci格子による初期配置

府省庁の重心を球面上に均等分布させるためのFibonacci格子:

球面上にN点を準均等に配置する標準的手法。各点iの座標:
- `theta = arccos(1 - 2*(i+0.5)/N)` （極角）
- `phi = π * (1+√5) * i` （方位角、黄金角）

これを初期シードとし、面積比例のPower Voronoiで最終位置を調整する。

### パフォーマンス考慮

- **初期ロード**: Voronoi計算はAPI側（Node.js）で事前計算し、座標情報だけを配信。クライアントはポリゴン描画のみ
- **LOD**: 全4,617事業のポリゴンを一度に描画しない。カメラ距離に応じてマージされたポリゴンを表示
- **テクスチャベーク**: 省庁レベルの配色をテクスチャとしてベークし、個別ポリゴン描画を回避（遠景時）
- **InstancedMesh**: 同一形状のポリゴンが多い場合にインスタンス化で描画コスト削減

## スコープ外メモ: 地殻レイヤー構想

本設計のスコープは**地表レイヤー（球面への事業マッピング）**のみ。以下は将来的な拡張の着想メモ。

球体の内部に支出先ごとの球体（細胞）を内包する構造:

- **地表**: 事業（面積 = 実質支出額）
- **地殻**: 支出先（体積 = 受領金額）の細胞が球体内部に存在
- **毛細血管**: 地表の事業から地殻の支出先細胞へ資金が流れるトレース

地表レイヤー（2D面積）と地殻レイヤー（3D体積）の世界観。事業から支出先への資金フローが毛細血管のように球体内部を走る。
