## 目的

支出先ラベリングシステムの現ステップ（Step 1: 辞書一致 / Step 2: 格パターン）でラベルが付与されなかった支出先のうち、**法人番号（`cn` フィールド）が存在するものを対象に、法人種別コード（`typeCodes`）をラベリング根拠として活用する** Step 3 を設計・実装する。

これにより、「支出先ラベリング確認UI」で「未ラベル」として表示される件数を減らし、支出先の分類状況を利用者が把握しやすくなる。

---

## 現状

| 指標 | 値 |
|------|----|
| ユニーク支出先名（総数） | 26,148件 |
| ラベル付与済み（Step 1 + Step 2） | ~22,095件（84.5%） |
| 未ラベル（source='none'） | ~4,053件 |
| 未ラベルのうち cn != '' のもの | 要調査（後述） |

> Step 1+2 の拡充（格パターン・辞書追加）で 84.5% を達成済み。
> 残余の未ラベルは「格を持たない法人」や「集合名称」等が中心と推定される。

### なぜ法人番号あり・未ラベルが残るのか

- **格不在型**: 名称に法人格ワードがない（例: 全国健康保険協会、日本年金機構 ← Step 1 の special_corporation_names.csv に収録済みのはずだが、未収録のものが残存）
- **辞書未収録型**: 法人番号は存在するが、辞書・格パターン・特殊法人リスト いずれにもマッチしない
- **typecodes 誤分類型**: RS CSV 上の typeCodes が NTA 正解と異なるケース（[法人番号ラベリング設計 §3-4](20260223_1009_法人番号ラベリング設計.md) 参照）

## 調査フェーズ

### 目的

Step 3 の実装前に「法人番号あり・未ラベル」の件数・金額・typeCode 分布を把握し、対応方針を確定する。

### 調査方法

**新規スクリプト `scripts/investigate-unlabeled-with-houjin.py`** を作成する。

- 入力: `public/data/entity-labels-csv.json`（`generate-entity-labels-csv.py` の出力）
- フィルタ条件:
  - `source == 'none'`（未ラベル）
  - `cn != ''`（法人番号が存在する）
- 集計内容:
  - typeCode 別の件数・金額
  - 上位 30 件リスト（金額降順、名称・typeCodes・金額を表示）
  - 比較: cn == '' の未ラベルの件数・金額（参考値）

### 期待される出力フォーマット

```
=== ラベルなし法人番号あり 深堀り調査 ===
source='none' かつ cn!='' の支出先: XXX件 / X.XX兆円

[typeCode 別内訳]
  399 その他設立登記法人: XXX件 / X.XX兆円
  201 地方公共団体     :  XX件 / 0.XX兆円
  101 国の機関         :  XX件 / 0.XX兆円
  301 株式会社         :  XX件 / 0.XX兆円
  ...

[上位 30件]
  1. 全国健康保険協会（あれば）: X.XX兆円  typeCodes=[399]
  2. ...

[参考] cn='' の未ラベル: XXX件 / X.XX兆円
```

### 調査結果に基づく方針分岐

| typeCode 件数 | 金額 | 追加対応 |
|-------------|------|---------|
| 399 が多い（> 100件）| ≫ 1兆円 | Step 3a を最優先実装 |
| 201 / 101 が多い | 相応の金額 | 辞書追加（ministry_supplement 等）が有効か検討 |
| 件数・金額ともに小さい | < 0.1兆円 | Step 3a で十分、NTA ZIP 補正は不要 |

## 実装フェーズ: Step 3 の設計

### 概要

`generate-entity-labels-csv.py` に **Step 3（typeCode によるフォールバックラベリング）** を追加する。Step 1・Step 2 でヒットしなかった支出先名に対して、収集済みの `typeCodes` からラベルを付与する。

### 処理フロー

```
[Step 1: 辞書一致]
[Step 2: 格パターン]
↓（未ラベル残余）
[Step 3: typeCode によるフォールバック]
  条件: labels.get(name) is None かつ name_data[name]['cn'] != ''
  処理:
    1. typeCodes から優先順位に従って代表コードを 1 つ選択
    2. コード → L1 / L2 のマッピングでラベルを付与
    3. source = 'typecode' でマーキング
```

### typeCode → L1 / L2 マッピング

| typeCode | L1 | L2 |
|----------|----|----|
| `101` | 国の機関 | 国の機関 |
| `201` | 地方公共団体 | 地方公共団体 |
| `301` | 民間企業 | 株式会社 |
| `302` | 民間企業 | 有限会社 |
| `303` | 民間企業 | 合名会社 |
| `304` | 民間企業 | 合資会社 |
| `305` | 民間企業 | 合同会社 |
| `399` | 設立登記法人（その他） | 特殊法人等 |
| `401` | 外国法人・国際機関 | 外国会社等 |
| `499` | その他法人 | その他法人 |

> `399` に「設立登記法人（その他）」という新規 L1 を導入する。既存カテゴリ「特殊法人・特別の法人」は Step 1（special_corporation_names.csv）のみが対象であり、typeCode=399 でヒットした未知法人とは性質が異なるため、区別する。

### 複数 typeCode の解決ルール

同一支出先名に複数の typeCode が記録されている場合（例: `['399', '201']`）は以下の優先順位で解決する:

```
201 > 101 > 399 > 301 > 302 > 303 > 304 > 305 > 401 > 499
```

> 根拠: より上位の行政体分類（地方公共団体・国の機関）を優先することで、誤分類（国土交通省部局が 301 と記録されるケース等）に対して保守的な方向の判断を取る。

### cn 要件の扱い

- **cn != '' を必須条件とする**: Step 3 は cn が存在する支出先にのみ適用する
- 理由: typeCodes は cn なしの行にも記録されているが、cn がない場合は法人の実体確認ができず誤分類リスクが高い
- cn == '' の未ラベル支出先は「集合名称・匿名化名称・内部経費科目名」が大半であり、typeCode ラベリングは不適切

### NTA ZIP による補正（オプション・後回し）

typeCode の誤分類（[法人番号ラベリング設計 §3-4](20260223_1009_法人番号ラベリング設計.md)）は RS CSV の `法人種別` 列の精度問題から発生する。補正には NTA ZIP（全件ファイル）の読み込みが必要で実装コストが高い。

**この設計では NTA ZIP 補正は実装スコープ外とし、RS CSV の typeCodes をそのまま使う。**
- 影響範囲: 誤分類が確認されている 28件 / 7.92兆円（ただし大半は Step 1/2 で正しくラベル済みのはず）
- 必要になった場合は別タスクで対応
