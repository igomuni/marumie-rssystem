# sankey2 データソース整理

## 目的

sankey2のデータ生成で使っているCSV列と中間データ（`outflows`）の関係を明確にし、再委託構造の構築ロジックの問題点を把握するため。

## `outflows`とは何か

`outflows`は**structured.json内の中間データ**であり、CSVから直接読むものではない。

### 生成元

`scripts/generate-structured-json.ts` の `buildBlockFlowMap()` が5-2 CSVを処理して生成:

1. 5-2 CSVの各行で `支出元の支出先ブロック` が非空 → 委託関係あり
2. **支出元ブロック**に紐づく支出先（5-1 CSV）を特定
3. **支出先ブロック**に紐づく支出先（5-1 CSV）を `recipients` として紐づけ
4. 結果: `支出先A.outflows = [{projectId, sourceBlockName, targetBlockName, amount, recipients}]`

### 問題点

1. **間接的なデータ**: 5-2 CSVのブロック接続 → structured.jsonのoutflows → sankey2のsubcontractエッジ、と2段階の変換を経る
2. **名前マッチの脆弱性**: outflowsのrecipients名で支出先ノードをマッチング。5-1 CSVと5-2 CSVのブロック名が一致しないケースがある
3. **深い再委託チェーンでの欠落**: outflowsは「支出先Aから支出先Bへの流れ」を追跡するが、チェーンの末端（4段目以降）ではoutflowsが生成されないケースがある

## 現在の再委託構造の二重構築

sankey2では再委託構造を**2つの独立した方法**で構築している:

| データ | ソース | 用途 |
|--------|--------|------|
| `subcontractエッジ` | structured.jsonの`outflows` | グラフのrecipient→recipientエッジ |
| `subcontractChains.blockChain` | 5-2 CSV直接 | パネルの「資金の流れ」表示 |

**問題**: 同じCSV（5-2）由来なのに、一方はstructured.json経由、もう一方は直接読み込み。不整合の原因。

## あるべき姿

5-2 CSVの`支出元の支出先ブロック`を**第一優先**で再委託構造を構築する:

1. 5-2 CSVのブロック接続（A→B→C→D）を読み取る
2. 各ブロックの支出先を5-1 CSVから取得
3. ブロック接続から`isDirectFromGov`を判定（`支出元の支出先ブロック`が空＝直接）
4. subcontractエッジもブロック接続から直接生成（outflows不要）

これにより:
- structured.jsonの`outflows`への依存を排除
- 再委託構造の一貫した構築
- 5-2 CSVのブロック接続が唯一の信頼できるソース
