# サブコントラクトエッジへのprojectIds付与

## 目的

/sankey2 のユーザーが、支出先間の委託エッジ（subcontract）がどの事業に由来するかを正しく把握できるようにするため。

## 現状の課題

### サブコントラクトエッジがprojectIdを持たない

`generate-sankey2-data.ts` でサブコントラクトエッジを生成する際、outflowsのrecipients情報を `source→target` のキーで集約し、projectIdを捨てている。

```typescript
// 現在のコード: projectIdを失う
const edgeKey = `${sourceNode.id}→${targetNode.id}`;
subcontractAmounts.set(edgeKey, (subcontractAmounts.get(edgeKey) || 0) + r.amount);
```

### 結果: パネルが無関係な事業名を表示する

例: 支出先「朝倉市」を選択した場合

- サブコントラクトエッジ: `recipient-北海道` → `recipient-朝倉市`（12.88億、PID=3440 農業施設災害復旧等事業由来）
- パネルの`SubcontractGroupList`が委託元（北海道）の事業帰属を特定する際、**北海道への接続228事業をグラフ探索で推測**していた
- 結果、最初にヒットした「高校生の地域留学の推進のための高校魅力化支援事業（PID=113）」が誤って表示される
- 実際の資金源であるPID=3440（農業施設災害復旧等事業）は表示されない

### 集約の実態

| 分類 | 件数 | 割合 |
|------|------|------|
| 1事業のみ | 16,858 | 92% |
| 複数事業にまたがる | 1,450 | 8% |

92%のエッジは単一事業由来であり、projectIds付与による帰属特定は十分に有効。

## 改善内容

### A. エッジデータへのprojectIds追加

**対象**: `scripts/generate-sankey2-data.ts`

サブコントラクトエッジ生成時に、edgeKeyごとにprojectIdのSetも管理し、最終エッジに`projectIds: number[]`を付与する。

### B. 型定義の更新

**対象**: `client/components/Sankey2/types.ts`

`LayoutEdge.projectIds?: number[]` を追加。

### C. パネルの事業帰属ロジック修正

**対象**: `client/components/Sankey2/Sankey2View.tsx` の `SubcontractGroupList`

委託元/委託先の事業グループ化で、グラフ探索による推測ではなく `edge.projectIds` から `project-spending-{pid}` を直接特定する。projectIdsがない場合のみ従来のフォールバックロジックを使用。

### D. BFSの改善

**対象**: `client/components/Sankey2/Sankey2View.tsx` の `bfsHighlight`

サブコントラクトエッジ通過時にprojectIds制約を伝搬し、上流のproject-spending探索と下流のサブコントラクト展開の両方でフィルタリング。無関係な事業への拡散を防止。
