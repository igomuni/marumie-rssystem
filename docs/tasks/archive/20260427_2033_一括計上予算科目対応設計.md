# 一括計上予算科目の事業に予算額が反映されない問題 — 対応設計

> Issue: [#135](https://github.com/igomuni/marumie-rssystem/issues/135)

## 目的

データを閲覧するユーザーが「予算ゼロ・支出あり」の事業を正確な情報で確認できるようにするため。
現状、224件（～6,559億円）の事業がサンキー図に予算との接続なしで孤立して表示されており、データの欠落に見えてしまう。

---

## 原因の特定

### 2-1 CSV の実態

「一括計上予算科目」の事業（例: PID=11 第一期政府共通プラットフォーム）は、2-1 CSV に 2023年度行が**存在する**が、`計(歳出予算現額合計)` が 0 になっている。

| カラム | 2021年度 | 2022年度 | 2023年度 |
|--------|----------|----------|----------|
| 計(歳出予算現額合計) | 13,916,000,000円 | 0円 | **0円** |
| 執行額(合計) | 13,521,000,000円 | 15,821,000,000円 | **12,896,926,000円** |

`TARGET_BUDGET_YEAR = YEAR - 1 = 2023` でフィルタしたとき:
- `totalBudget = 0` → `budgetAmount = 0`
- `executedAmount = 12,896,926,000円` — しかしこの値は現在使われていない

### コード上の問題箇所（generate-sankey-svg-data.ts）

```ts
L267: const budgetAmount = budget?.totalBudget || 0;
  // budget.executedAmount は budgetMap に格納済みだが参照されていない

L293: if (budgetAmount > 0)  // ministry→project-budget エッジが生成されない
L302: const flowValue = Math.min(budgetAmount, spendingAmount);
      // = Math.min(0, X) = 0 → project-budget→project-spending エッジも生成されない
```

### なぜ予算が 0 なのか

デジタル庁が「情報システム関係予算を段階的に一括計上する」（PID:4 事業概要より）仕組みのため、個別システム事業（例: PID:20079 国税総合管理システム）は 2-1 CSV に予算行が存在するが `計(歳出予算現額合計)` が 0 になる。予算はデジタル庁の一括計上事業 ID（PID:4 など）側でまとめて保有され、個別事業 ID の `計(歳出予算現額合計)` は 0 のまま、実際の**執行額は個別事業 ID に帰属**して `執行額(合計)` に記録される。

### CSV 間の明示的リンクの有無

PID:4（一括計上側）と個別事業 PID との親子関係を示すフィールドは、**全 CSV ファイルを調査した結果、存在しない**。

| CSV | 調査フィールド | 結果 |
|-----|--------------|------|
| 1-5（関連事業） | `関連事業の事業ID` | 両方とも空欄 |
| 2-2（予算種別） | PID:20079 の行 | 空エントリのみ（項・目なし） |
| 6-1（備考） | 備考フィールド | 空欄 |

唯一の間接的な手がかりは**事業名に含まれる予算科目名**のみ:

- PID:4: `情報システムの整備`**`(情報通信技術調達等適正・効率化推進費)`**
- PID:20079: `国税総合管理(KSK)システム`**`(情報通信技術調達等適正・効率化推進費)`**

同じ科目名を事業名に持つ事業は **210件** あり、これらすべてが一括計上側（PID:4 等）の予算を使う個別事業と推測されるが、機械的なマッピングは現行 CSV データでは不可能。

**結論**: 一括計上側（PID:4）の予算をどの個別事業に按分したかを CSV から復元する手段はない。個別事業の `執行額(合計)` を用いた表示対処が現実的な唯一の手段となる。

---

## 対応方針

### 方針: 2-1 CSV の `執行額(合計)` をフォールバック予算額として使用

`totalBudget = 0` かつ `executedAmount > 0` のとき、`budgetAmount = executedAmount` として扱う。
この値は 2-1 CSV（予算・執行サマリ）内の公式実績値であるため、データ根拠のある代替表示として妥当。

| 比較軸 | 採用案（執行額フォールバック） | 不採用案（5-1 CSV 支出額をそのまま） |
|--------|-------------------------------|--------------------------------------|
| データ根拠 | 2-1 CSV 官式値 | 5-1 CSV（直接支出のみ・フィルタ後） |
| 精度 | 間接含む全実績 | 直接支出のみ |
| 実装箇所 | generate-sankey-svg-data.ts のみ | 同上（大差なし） |
| 意味的正確性 | 「一括計上 → 執行額を代替表示」として説明可能 | 「予算ゼロだが支出額を予算として表示」は不自然 |

### 視覚的差別化

一括計上科目の事業は「予算額ではなく実績ベースで表示している」ことをユーザーに伝えるため、専用フラグ `isConsolidatedBudget` を `RawNode` に付与する。UI 側でこのフラグを使ってツールチップ等に注記を表示する。

## 変更スコープ

### 1. `types/sankey-svg.ts`

`RawNode` に以下フィールドを追加:

```typescript
/** True when budgetAmount is derived from executedAmount (一括計上科目) */
isConsolidatedBudget?: boolean;
```

### 2. `scripts/generate-sankey-svg-data.ts`

**変更箇所 A: L267 — budgetAmount の決定ロジック**

```typescript
// Before
const budgetAmount = budget?.totalBudget || 0;

// After
const isConsolidatedBudget = !!(budget && budget.totalBudget === 0 && budget.executedAmount > 0);
const budgetAmount = budget?.totalBudget || budget?.executedAmount || 0;
```

**変更箇所 B: L272-280 — project-budget ノード生成に isConsolidatedBudget を付与**

`project-budget` ノードの push に `...(isConsolidatedBudget ? { isConsolidatedBudget: true } : {})` を追加。

**変更箇所 C: L293 / L302-307 — エッジ生成条件**

`budgetAmount` が正の値になれば既存ロジックのまま動作するため、追加変更不要。

### 3. `app/sankey-svg/page.tsx`

ツールチップ（ポップアップ）の表示箇所で `node.isConsolidatedBudget === true` のとき:

- ラベル: `「予算（一括計上）`  
- 注記: `※ 予算額は計上されておらず、執行実績額を代替表示しています`

またはシンプルに、既存の `rawValue` メカニズムを流用:

```text
rawValue = 0  // 本来の予算額
value = executedAmount  // 表示に使う値
isConsolidatedBudget = true
```

→ ツールチップ: `予算: 一括計上（実績: 129億円）`

## 影響範囲

| 指標 | 現状 | 修正後 |
|------|------|--------|
| 予算ゼロ事業数 | 224件 | 0件（executedAmount>0 の全件に対応） |
| 対応支出合計 | 孤立（6,559億円） | サンキー図に接続 |
| totalBudget への影響 | 変更なし（totalBudget=0 の事業は集計から除外済み） | 変更なし |
| 府省庁ノードへの影響 | 変更なし（ministry-level は totalBudget ベース） | 変更なし（別途検討） |

**注意**: 府省庁ノードの `value`（= 府省庁の予算合計）は `budgetMap[pid].totalBudget` の合算で算出されており、一括計上科目の executedAmount は含まれない。府省庁→事業(予算)エッジは追加されるが、府省庁ノード自体の総計との整合性が崩れる可能性がある。**この点は実装前に要確認。**

---

## 未解決事項・要検討

1. **府省庁ノード値との整合性**: 一括計上科目の executedAmount を project-budget に足すと、`ministry → project-budget` のエッジ合計が ministry ノードの value を超える場合がある。府省庁ノードも同様に executedAmount を加算する必要があるかを確認する。

2. **`'total'` ノード（予算総計）への影響**: 現状 `totalBudget` の合算。一括計上科目の executedAmount を加算すべきか。

3. **実績額 > 直接支出額のケース**: `Math.min(budgetAmount, spendingAmount)` は `budgetAmount = executedAmount` では `executedAmount` に設定されるが、5-1 CSV ベースの `spendingAmount` と乖離する場合がある。ただし `Math.min` 自体は維持するため、フロー値は必ず正しく bounded される。
