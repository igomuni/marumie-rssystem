# 再委託構造ビュー改善設計

設計日: 2026-05-09
対象年度: 2024年度 / 2025年度
スコープ: データ層 + ドメイン層 + UI層フル対応
主たる利用者: EBPM分析者

## 目的

EBPM分析者が「事業単位で予算・支出・再委託構造を正確に読み取りたい」状態になるため、以下を解消する。

- 5-2 の構造的に「府省庁ルートだけでは説明できない別起点ブロック」（財投借入・回収金・自己収入・利水者等・移替・参考的制度フロー）が、現行の `subcontracts-{YEAR}.json` と `/subcontracts/{projectId}` ビューでは「直接支出でも再委託でもない」中間状態として正しく表現されていない。
- ノードクリックでサイドパネル全体の主語が「ブロック詳細」に切り替わるため、「事業全体の構造を読みたい」のか「特定ブロック内の支出先を読みたい」のかが UI 上で混在している。
- 5-3 の費目・使途や、5-2 の `国自らが支出する間接経費` がデータレイヤーで取り込まれていない（または捨てられている）。

これにより、JICA有償資金協力（PID 1406）、防災科研運営費交付金（PID 1695）、水資源開発事業（PID 4046）、石油備蓄事業利子補給金（PID 19873）といった事業の「実際の資金構造」がEBPM観点で誤って読み取られるリスクがある。

## 設計方針

### 主語固定とイベント分離

- ページの主語は常に「事業」（PID）。
- ブロックノードクリックは「サイドパネル主語の切り替え」ではなく、「資金フロー図／資金の流れリスト／ブロック一覧」上のハイライト・スクロールに留める。
- ブロック詳細（支出先・契約・費目）は二段目の操作（タブ切替もしくは詳細パネル展開）で初めて開く。

### 「ブロック接続」と「ブロック内訳」の分離

- 5-2 由来の `flows`（ブロック間接続）と、5-1 由来の `blocks`（ブロック内訳）は別レイヤーとして扱う。
- 既存型 `SubcontractGraph` はすでに `blocks` と `flows` を分離済み。今回はそれぞれの粒度に「種別」「補足」「フラグ」を追加して情報量を上げる。

### フロー種別の明示

5-2 行から CSV だけで判定できる種別を `BlockEdge` に付与し、UI 側ではそれを見て描画方法を切り替える。種別は次の通り：

| 種別 (`origin`) | 判定条件 | UIでの扱い |
|---|---|---|
| `direct` | `担当組織からの支出=TRUE` かつ `支出元の支出先ブロック` 空 | 通常ルート（実線・赤系） |
| `transfer` | `補足情報` に `移替` を含む直接ルート | 通常ルートに「移替」バッジを付与 |
| `separate-origin` | 構造条件（後述）を満たす別起点ブロックからのフロー | 別起点フロー（破線・グレー系） |
| `subcontract` | `担当組織からの支出=FALSE` のブロック間フロー | 既存の再委託フロー（実線・橙系） |
| `reference` | 補足に `参考`、もしくは下流ブロックの `5-1金額=0` で `融資`/`政府保証借入`/`利子補給` 等の補足 | 細い破線で参考フローとして表示 |
| `indirect-cost` | `targetBlock` が空で `国自らが支出する間接経費=TRUE` | フロー図には載せず、別セクション |

`separate-origin` の判定は [tasks/20260509_0853_EBPM事業単位_複数支出元検知ロジック.md](20260509_0853_EBPM事業単位_複数支出元検知ロジック.md) の構造条件をデータパイプライン側に取り込む。データレイヤーで一度だけ計算する点に注意。

### ブロック種別の明示

`BlockNode` には `originKind` を持たせる：

| 種別 (`originKind`) | 意味 |
|---|---|
| `direct` | 担当組織からの直接支出ブロック |
| `subcontract` | 上流が `direct` または `subcontract` のブロック（通常の再委託） |
| `separate-origin-broad` | 広めの別起点（支出元として登場するが、支出先として登場せず、直接支出ブロックでもない）。上流ルートが欠落している可能性も含む |
| `separate-origin-strong` | 強い別起点（広めの別起点 かつ `sourceFeedsMerge=true`）。本命候補として強調表示 |

`isTerminal` は下流が無いリーフブロックを示す独立フラグ。`originKind` を上書きせず別フィールドとする。

なお、`analyze-multiple-project-funding-sources.ts` で使われていたキーワード分類（`借入・財投・保証`／`回収・償還` 等の `originCategory`）は、構造分析を主軸とする方針のため Phase 1 のスコープから外す。`originKind` で「広めの別起点」と「強い別起点」を区別すれば EBPM 観点では十分と判断した（後続フェーズで必要になれば追加検討）。

### 金額ゼロブロックの取り扱い

`blocks[*].totalAmount === 0` かつ `recipients.length === 0` のブロックは、フロー図上で太さ・面積を持たせない（最小サイズの参考的ノード）。事業レベルで全ブロックがそうなる場合（PID 19873 など）は、ヘッダーに「制度フロー」バッジを表示。

### 間接経費の独立扱い

5-2 で `国自らが支出する間接経費=TRUE` の行は `flows` に入れず、`SubcontractGraph.indirectCosts: IndirectCost[]` として別配列で持つ。UI では資金の流れリストとは別の独立セクションで列挙。

## データ層の設計変更

### 型定義の拡張 — [types/subcontract.ts](../../types/subcontract.ts)

```text
BlockNode: {
  ...既存,
  + originKind: 'direct' | 'subcontract' | 'separate-origin-broad' | 'separate-origin-strong'
  + isTerminal: boolean       // 下流が無いリーフブロック
  + recipientCount: number    // recipients.length をデータ生成時に固定
  + hasExpenses: boolean      // 5-3 の費目・使途あり
}

BlockEdge: {
  ...既存,
  + origin: 'direct' | 'transfer' | 'separate-origin' | 'subcontract' | 'reference'
  + isReference: boolean   // 既存の note と独立した参考標記フラグ
  + amount?: number        // 5-2 由来で金額が判明している場合のみ
}

IndirectCost: {
  blockHint: string        // 旧 "支出元の支出先ブロック名"（参考表示用）
  category: string         // "国自らが支出する間接経費の項目"
  amount: number
  note?: string
}

SubcontractGraph: {
  ...既存,
  + indirectCosts: IndirectCost[]
  + hasSeparateOrigin: boolean         // 別起点ブロックが1つ以上あるか
  + hasReferenceFlow: boolean          // 参考的制度フローを含むか
  + isInstitutionalFlowOnly: boolean   // 全ブロックが金額ゼロ
  + separateOriginAmount: number       // 別起点ブロック合計金額（フィルタ用）
}
```

`projectName`、`ministry`、`budget`、`execution` はそのまま。

### パイプライン拡張 — [scripts/generate-subcontracts.ts](../../scripts/generate-subcontracts.ts)

既存処理に以下を追加する。新しいスクリプトは作らず、`generate-subcontracts.ts` を拡張する方針（CSV読み込みコストを抑えるため）。

1. **5-2 解析時**:
   - `targetBlock` が空かつ `国自らが支出する間接経費=TRUE` の行は `indirectCosts` 配列へ振り分け、`flows` には入れない。
   - 各 `flow` に `origin` を付与。`subcontract` は既定値、`direct` は既存の `directBlocks` 判定を流用、`transfer` は補足情報に `移替` が含まれる `direct` 行。
   - 補足情報に `参考` を含む行は `isReference=true`。
2. **別起点判定（事業単位）**:
   - [analyze-multiple-project-funding-sources.ts](../../scripts/analyze-multiple-project-funding-sources.ts) の構造条件（`targetBlocks`／`directTargetBlocks`／`sourceBlocks`）を流用し、`separateOriginBlocks: Set<blockId>` を作る。
   - 別起点判定で `sourceFeedsMerge` も併せて判定し、より強い別起点候補と区別可能にする（`originCategory` の補助情報として付与）。
   - 該当する `flows[*]` の `origin` を `separate-origin` に更新。
3. **5-1 集計後の Block 種別決定**:
   - `originKind` を BFS の起点（`direct` ブロック）から伝播し、`subcontract` を割り当て。
   - `separateOriginBlocks` に該当するブロックは `separate-origin` で上書き。
   - 下流が空なら `terminal` フラグ的に判定（`originKind` 上書きはせず、別フラグでもよい）。
4. **`originCategory` 分類**:
   - ブロック名・役割・補足情報の文字列マッチで分類タグを決定。`analyze-multiple-project-funding-sources.ts` のキーワード辞書を共通モジュールへ切り出すか、本スクリプト内で同じ規則を直書き。
5. **5-3 取り込み（既存）**:
   - 現状は `recipient.expenses` に格納している。`SubcontractGraph` 単位で `hasExpenses` フラグも付与しておくとリスト UI で軽く絞りやすい。
6. **参考的制度フロー判定**:
   - 事業内の全ブロックが `totalAmount === 0` かつ `recipients.length === 0` の場合 `isInstitutionalFlowOnly=true`。
   - 個別フローでは、`origin === 'subcontract'` かつ下流ブロックが `totalAmount === 0` で補足に `融資`／`政府保証借入`／`利子補給` を含むものを `origin='reference'` に格上げ。

### 出力ファイル

- 既存の `public/data/subcontracts-2024.json`、`public/data/subcontracts-2025.json` をそのまま上書き（型拡張のため Web 側もビルド時に再型チェック）。
- `.gz` 管理ルールに従い、`prebuild` フックで再展開可能であること。

## ドメイン層の設計変更 — [app/lib/subcontract-layout.ts](../../app/lib/subcontract-layout.ts)

`subcontract-layout.ts` は React/HTTP に触れない純粋関数レイヤー。今回もこの制約を維持する。

### depth 算出の更新

- `direct` 起点だけでなく、`separate-origin` ブロックも depth=1 に並べる（ただし「別レーン」として水平方向にオフセット）。
- `direct` レーンと `separate-origin` レーンの間に視覚的なギャップ（`SEPARATE_ORIGIN_LANE_GAP`）を確保。
- `separate-origin` ブロックから下流ブロックへ伸びる `flows[*].origin === 'separate-origin'` のエッジを別パスとして識別できるよう、`LayoutEdge` に `origin` を引き継ぐ。

### LayoutEdge と LayoutBlock の拡張

```text
LayoutEdge: {
  ...既存,
  + origin: BlockEdge['origin']
  + isReference: boolean
}

LayoutBlock: {
  ...既存,
  + originKind: BlockNode['originKind']
  + isTerminal: boolean
  + isZeroAmount: boolean
}
```

レイアウト計算で `originKind === 'separate-origin'` のブロックは、別レーン配置・幅縮小（`NODE_W * 0.85` 程度）を行う。`isZeroAmount` のブロックは最小高さ（既存 `NODE_MIN_H` を 0.7 倍）で配置。

### サイドパネルのタブ構成

ノードクリック・タブ切替どちらでもパネル主語は「事業」のまま。タブで開く対象が変わる。

| タブ | 内容 |
|---|---|
| `流れ` | 5-2 由来の資金フロー一覧。`from / to / origin種別 / 補足 / 金額` の表。`separate-origin` は別グループとして折りたたみ表示。`reference` は淡色。 |
| `ブロック` | 5-1 由来のブロック一覧。**`totalAmount` 降順のフラットリスト**で `blockId / blockName / originKindバッジ / 役割 / 合計支出額 / 支出先数 / 費目使途有無` を表示。フィルタチップ: `すべて` / `直接のみ` / `再委託のみ` / `別起点のみ`。検索: ブロック名・役割。 |

- ノードクリック → `selectedBlock` を更新 → `支出先タブ` を自動アクティブ化 + 該当行をスクロール表示。

- ブロックタブの行クリック → `selectedBlock` 更新 + フロー図上で対応ノードをハイライト＆視野中央に寄せる。

  - `direct`
  - `transfer`
  - `subcontract`
  - `separate-origin`
  - `reference`
- `BlockNode.originKind` を追加する。
  - `direct`
  - `subcontract`
  - `separate-origin-broad`（広めの別起点）
  - `separate-origin-strong`（強い別起点。`sourceFeedsMerge=true`）
- `BlockNode.isTerminal`、`BlockNode.recipientCount`、`BlockNode.hasExpenses` を追加する。

| 判定 | 条件 | 用途 |

| 広めの別起点 | 支出元として登場するが、支出先として登場せず、担当組織からの直接支出ブロックでもない | 注意喚起・調査対象 |

## 最初の実装スコープ

一度に全UIを作り替えると差分が大きい。最初の実装は次に絞る。

1. Phase 1 の型・データ生成拡張
2. Phase 2 の `LayoutEdge.origin` 伝搬
3. 詳細画面の既存サイドパネルに、まず `資金の流れ` と `ブロック一覧` の2セクションを追加
4. 既存の支出先詳細表示は残し、ノードクリック時の挙動だけ「選択・ハイライト」に寄せる

この最小スコープで、JICA PID 1406 の誤読、つまり `財政融資資金借入金` と `回収金等` が通常支出先に見える問題をまず潰す。その後、タブ化・一覧フィルタ・間接経費表示を広げる。

## 確認事項と決定事項

### 決定済み（2026-05-09 ユーザー確認）

| 論点 | 決定 | 設計への反映 |
|---|---|---|
| 間接経費の表示位置 | **独立タブとして実装** | サイドパネル `間接経費` タブで `indirectCosts` を表示。`hasIndirectCost` がない事業ではタブ無効化。 |
| ブロックタブの並び順 | **`totalAmount` 降順のフラットリスト** | `originKind` 別グループは作らない。代わりに各行に `originKind` バッジを並べて視認性を確保。フィルタチップで絞り込みは可能。 |
| `originCategory` 分類のキーワード辞書 | **必須ではない（構造分析を主軸）** | Phase 1 では `originKind`（構造判定）のみを必須とし、`originCategory`（キーワード分類）は実装スコープから外す。型定義からも省く（後続フェーズで必要になれば追加）。 |
| タブ状態のURL同期 | **スコープ外** | URLは `?year=` のみ維持。タブ状態は React state で管理。 |

### 保留（具体例調査後に判断）

- **参考標記フローの扱い**: 補足に `参考` を含む行、または金額0で `融資`／`政府保証借入`／`利子補給` を含む行をどこまで分離するか。  
  Phase 1 では `BlockEdge.isReference` フラグだけ立てて保持し、UI 側では当面 `reference` を `subcontract` と同じ描画にする。具体例（5-2 で `参考` を含む行のサンプル抽出、PID 19873 のような金額ゼロケースの実画面）を調査ログとして集めてから、Phase 3 で描画方法を確定する。

### 設計への反映差分

上記決定を踏まえて以下の調整を行う。

- **`BlockNode.originCategory` を型・パイプラインから削除**: Phase 1 の必須項目から外す。`originKind` のみで起点種別を表現する。
- **Phase 1 の必須出力フィールドから `separateOriginAmount` を保持・`hasMerge`/`maxMergeWidth` 等の合流統計は維持**: 構造分析が主軸のため、合流幅と別起点件数の数値は引き続きデータ層で算出する。
- **ブロックタブの仕様確定**: `totalAmount` 降順のフラットリスト。各行に `originKind` バッジ（`直接` / `再委託` / `別起点(強)` / `別起点(広)`）を表示。フィルタチップ（`すべて` / `直接のみ` / `再委託のみ` / `別起点のみ`）で絞り込み可能。
- **参考フロー描画は Phase 3 で再検討**: Phase 1 では `isReference` をデータに持たせるだけ、Phase 2 では `LayoutEdge.isReference` を引き継ぐだけ、Phase 3 で具体例を見ながら線種・色を決定する。
