# 事業別 支出先データ品質評価 設計

作成日: 2026-03-07

---

## 目的

**行政事業レビューシートの作成者（府省庁の担当者）** が、**支出先データの品質に課題がある事業を特定し、改善できる**ようにするため。

これまでの支出先評価（`recipient_dictionary.csv` による法人番号DB突合）で事業ごと・部課局ごとに品質のばらつきがあることが確認された。事業単位で支出先の品質をスコア化し、問題の全体像と改善優先度を定量的に把握する。

---

## 評価軸の設計

### 軸 1: 支出先名の品質（既存資産を活用）

`recipient_dictionary.csv` の valid/invalid 判定を事業単位で集計。

| 指標 | 内容 |
|------|------|
| valid_count | 事業内の valid 支出先名の件数 |
| invalid_count | 事業内の invalid 支出先名の件数 |
| valid_ratio | valid_count / (valid_count + invalid_count) |
| invalid_reasons | invalid の理由別内訳（入力ミス、集合表現、英字のみ、任意団体、etc.） |

### 軸 2: 法人番号の記入率

支出先 CSV の `法人番号` 列の記入状況を事業単位で集計。

| 指標 | 内容 |
|------|------|
| cn_filled_count | 法人番号が記入されている支出先数 |
| cn_empty_count | 法人番号が空の支出先数 |
| cn_fill_ratio | cn_filled_count / total |

### 軸 3: 予算と支出のバランス

予算サマリ（予算年度 2023）の執行額と、支出先 CSV の支出先合計支出額を事業単位で照合。

| 指標 | 内容 |
|------|------|
| exec_amount | 予算サマリの執行額 |
| spend_total | 支出先 CSV の合計支出額 |
| gap_ratio | \|exec_amount - spend_total\| / exec_amount |

> **注意**: spend_total がブロック間の二重計上で exec_amount のちょうど 2倍になるパターンが多い（再委託元+再委託先の両方で計上）。これ自体が品質指標になる。

### 軸 4: ブロック構造の妥当性

| 指標 | 内容 |
|------|------|
| block_count | 事業のブロック数 |
| has_redelegation | 再委託ブロックの有無 |
| redelegation_depth | 再委託の階層数（再委託/再々委託） |
| block_amount_consistency | ブロック合計支出額と支出先個別支出額の整合性 |

### 軸 5:「その他支出先」フラグの使用状況

| 指標 | 内容 |
|------|------|
| other_flag_true_count | `その他支出先 = TRUE` の行数 |
| other_flag_ratio | TRUE の行数 / 全行数 |

> 「その他支出先」が TRUE の行は支出先の詳細が省略されていることを意味し、透明性の低下を示す。

## 総合スコアの算出方針

各評価軸を **0〜100 の正規化スコア** に変換し、重み付き合計で総合スコアを算出する。

| 軸 | 重み（案） | 0点の条件 | 100点の条件 |
|----|-------:|----------|-----------|
| 支出先名の品質 | 40% | valid_ratio = 0% | valid_ratio = 100% |
| 法人番号の記入率 | 20% | cn_fill_ratio = 0% | cn_fill_ratio = 100% |
| 予算・支出バランス | 20% | gap_ratio > 100% | gap_ratio = 0% |
| ブロック構造 | 10% | 不整合あり | 整合 |
| その他支出先の抑制 | 10% | other_flag_ratio > 50% | other_flag_ratio = 0% |

> 重みは初期値。バッチ実行の結果を見ながら調整する。

## 実装方針

### 処理フロー

```
1. データロード
   - 2-1 予算サマリ（予算年度=2023の行を抽出） → 事業ID→{執行額}
   - 5-1 支出先情報 → 事業ID→{ブロック情報, 支出先リスト}
   - recipient_dictionary.csv → 支出先名→{valid/invalid, 理由}

2. 事業単位で5軸を計算

3. 総合スコアを算出

4. CSV出力（事業ID, 事業名, 府省庁, 局庁, 各軸スコア, 総合スコア）
```

### 出力ファイル

| ファイル | 内容 |
|---------|------|
| `data/result/project_quality_scores.csv` | 全5,664事業の品質スコア |

### バッチ実行の段階

| フェーズ | 対象 | 目的 |
|---------|------|------|
| Phase 1 | 10事業 | スクリプトの動作確認、出力形式の検証 |
| Phase 2 | 全事業 | 全件スコア計算、実行時間・メモリ計測 |

> Phase 1 は LLM を使わない Python スクリプトで完結する（辞書突合結果ベース）。LLM 評価は将来拡張として別途設計する。

## LLM 活用の検討（将来拡張）

現時点のスコア計算は辞書突合 + 数値照合で完結するため LLM は不要。
ただし、以下の評価は LLM の判断が必要になる可能性がある:

- 支出先名が「事業名」「プログラム名」として妥当かの判定
- ブロック名と支出先名の意味的整合性
- 契約概要と支出先の対応関係の妥当性

これらは Phase 2 の結果を確認した後に、別途設計する。

---

## 留意事項

- **予算年度**: 支出先データは予算年度 2023 の実績が主（事業年度 2024 のレビューシート）
- **二重計上**: ブロック間で同じ金額が再委託元・再委託先で二重に計上されるケースが多く、gap_ratio が 100% になることは必ずしもデータ品質の問題ではない。ブロック構造の特性として評価する
- **スクリプト配置**: `scripts/` にのみ配置。UI やAPI ロジックは含めない（CLAUDE.md レイヤー設計ルール準拠）
