# /sankey2 Phase 6〜8 設計

## 概要

Phase 5までに構築したフィルタ・検索・インタラクション機能をさらに強化する。
段階的にリリース可能な3フェーズに分割。

- **Phase 6**: ナビゲーション強化（ブラウザ履歴、ラベル情報量、ラベルフィルタ）
- **Phase 7**: 委託チェーン可視化（データ拡張、サイドパネル、BFS探索）
- **Phase 8**: パフォーマンス改善（計測、バンドリング、間引き、Canvas検討）

---

# Phase 6: ナビゲーション強化 ✅

## 6-1: ブラウザ戻る/進む対応 ✅

### 現状の課題
- URL同期に `router.replace()` を使用 → ブラウザ履歴に積まれない
- フィルタや選択を変更しても「戻る」で前の状態に戻れない

### 設計

**方針**: `router.replace` → `router.push` に変更し、状態変更を履歴に積む。
ただし、高頻度の変更（スライダー操作中、Pan/Zoom中）は `replace` のまま、
確定操作（フィルタ適用、ノード選択、検索実行）のみ `push` にする。

```text
操作                    → 履歴に積む？
─────────────────────────────────────
ノード選択/解除          → push
府省庁フィルタ変更       → push
検索実行（確定時）       → push
金額スライダー操作中     → replace（高頻度）
金額スライダー確定       → push（onMouseUp/onTouchEnd）
Zoom/Pan               → replace（高頻度、履歴不要）
```

**実装方針**:
- `pushRef` フラグで `push` / `replace` を制御
- スライダーの `onChange` → `replace`、`onMouseUp` → `pushRef = true` → 次の同期で `push`
- `popstate` イベントリスナーで URL → state 復元

### 注意点
- Next.js の `useSearchParams` は `popstate` で自動更新される → state復元のみ実装
- 初回ロード時の URL 復元ロジック（既存）はそのまま流用

## 6-2: ノードラベリング強化 ✅

### 現状の課題
- ラベルは `node.label` のみ（ホバー/選択時に金額追加）
- type情報やministry情報がラベルからは読み取れない
- 小さいノードのラベルが見切れる

### 設計

**Zoom依存の段階表示**:

```text
Zoom段階          表示内容
──────────────────────────────────────
screenArea < 400  ラベルなし（現行通り）
400 ≤ area < 2000 ノード名のみ（1行）
2000 ≤ area < 8000 ノード名 + 金額
area ≥ 8000       ノード名 + 金額 + 府省庁名/typeバッジ
```

**実装方針**:
- foreignObject内のdivに段階的にコンテンツを追加
- フォントサイズは現行ロジック維持（`min(h*0.25, w*0.18, 150)`）
- 金額・府省庁名は小さめのフォント（`fontSize * 0.6`）

## 6-3: ラベルフィルタ ✅

### 現状の課題
- Zoomを上げるとラベルが密集して読みにくい
- 特定typeのラベルだけ見たい場合がある

### 設計

**type別のラベル表示トグル**:
- フィルタパネル内に「ラベル表示」セクション追加
- type別チェックボックス（府省庁/事業(予算)/事業(支出)/支出先）
- チェックOFFのtypeはノード矩形は表示するがラベルを非表示

**ラベル密度制御**（オプション）:
- `LABEL_SCREEN_AREA` を動的に変更するスライダー
- 400（現行）〜 5000（ラベル少なめ）の範囲

# Phase 7: 委託チェーン可視化

## 7-1: データパイプライン拡張

### 現状のデータ構造

RS2024のStructuredJSONには委託チェーン情報が存在する:
- `sourceChainPath`: 委託経路（例: `"博報堂 → EYストラテジー"`）
- `isDirectFromGov`: 政府からの直接支出かどうか
- `outflows` / `inflows`: 再委託先/元のフロー情報

### 設計
- `LayoutNode` に `isIndirect?: boolean` と `chainPath?: string` フィールド追加
- `generate-sankey2-data.ts` でStructuredJSONの委託情報をグラフデータに付与

## 7-2: サイドパネル拡張

### 設計
- 支出先ノード選択時、委託チェーン情報をパネルに表示
- `isDirectFromGov: false` の場合「間接支出（委託経由）」バッジ表示
- `sourceChainPath` を経路として表示（例: `政府 → 博報堂 → EYストラテジー`）

## 7-3: 委託チェーンBFS探索

### 設計
- recipientクラスタ内に再委託エッジを追加（`generate-sankey2-data.ts` + `compute-sankey2-layout.ts`）
- BFS探索で再委託先→再々委託先まで深堀り可能に
- エッジタイプ（直接支出 / 再委託）で色分け

# Phase 8: パフォーマンス改善

## 8-1: パフォーマンス計測

### 設計
- React DevTools Profiler でコンポーネント別レンダリング時間を計測
- Performance API で Zoom/Pan 操作時のフレームレートを計測
- ボトルネック特定: DOM更新 vs useMemo再計算 vs レイアウト/ペイント

## 8-2: エッジバンドリング

### 現状の課題
- 74,390本のエッジが個別Bezierで描画 → 低Zoom時にノイズ化
- 同一府省庁→同一クラスタ方向のエッジが重なり合う

### 設計

**クラスタ間バンドリング**:
- 同一 `(sourceMinistry, targetCluster)` ペアのエッジを1本の太い帯に束ねる
- 帯の幅 = 束ねた金額合計の対数比例
- Zoom閾値以上で個別エッジに分解（LOD的挙動）

**実装方針**:
- `compute-sankey2-layout.ts` でバンドル情報を事前計算
- `LayoutEdge` に `bundleId?: string` フィールド追加
- 描画時: 低Zoom → bundleIdごとに1本描画、高Zoom → 個別描画

## 8-3: エッジ間引き

### 設計
- 低Zoom時にvalue下位のエッジをカリング
- エッジの面積ベースLOD: `edge.width * edgeLength * zoom² < threshold`
- ノードLODと同じ思想で段階的に表示

## 8-4: visibleNodes/Edgesの差分更新

### 設計
- 現在の `useMemo` はtransform/filter変更のたびにフル再計算
- 前回結果との差分のみ更新する仕組みを検討
- `useRef` で前回のvisible setを保持、新旧比較で追加/削除のみ処理

## 8-5: Canvas描画への移行検討

### 検討事項

| 方式 | メリット | デメリット |
|------|---------|-----------|
| SVG (現行) | DOM統合、イベント処理が容易、React管理 | 要素数に比例するDOM操作コスト |
| Canvas 2D | 描画高速、要素数無制限 | ヒットテスト自前実装、テキスト描画制限 |
| WebGL (pixi.js等) | 最高性能 | 導入コスト大、テキスト描画が困難 |

### 方針
8-1〜8-4の最適化後に60FPS未達の場合のみCanvas 2D移行を検討。
移行する場合:
1. エッジ層のみCanvas化（ノード矩形はSVG維持）
2. ラベルのCanvas化（foreignObjectの代替）
3. 全面Canvas化（最終手段）
