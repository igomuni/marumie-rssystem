# ズーム限界と金額規模の関係

## 概要

/sankey2のズーム倍率の実効限界は、**総予算額と最小ノード金額の比率**で決まる。151兆円の総予算に対して30円のノードを表示するには約130,000%のズームが必要であり、これは偶然ではなくtreemapの面積配分から導かれる数学的制約である。

## 理論式

ノードが画面上で表示されるための条件:

```
screenArea = node_area × k² ≥ MIN_SCREEN_AREA
```

ここで:
- `node_area` = treemapが割り当てた面積（レイアウト座標系）
- `k` = ズーム倍率
- `MIN_SCREEN_AREA` = LOD閾値（現在: 4 px²）

treemapにおけるノード面積は金額に比例:

```
node_area ≈ CLUSTER_AREA × (amount / totalAmount) × GAP_FACTOR
```

ここで:
- `CLUSTER_AREA` = 4000 × 4000 = 16,000,000 px²
- `totalAmount` = 151,123,034,375,145 円（≈ 151.1兆円）
- `GAP_FACTOR` ≈ 0.64（比例gap 20%による面積損失: (1-0.2)² = 0.64）

これを解くと、あるノードを表示するために必要なズーム倍率:

```
k = sqrt(MIN_SCREEN_AREA × totalAmount / (CLUSTER_AREA × amount × GAP_FACTOR))
```

## 金額別の必要ズーム倍率

| 金額 | レイアウト面積 (px²) | 必要k | 必要ズーム% |
|------|---------------------|-------|-----------|
| 30円 | 2.34×10⁻⁶ | 1,308 | 130,780% |
| 100円 | 6.72×10⁻⁶ | 771 | 77,146% |
| 385円 | 2.77×10⁻⁵ | 380 | 38,033% |
| 1,000円 | 6.93×10⁻⁵ | 240 | 24,030% |
| 1万円 | 6.70×10⁻⁴ | 77 | 7,726% |
| 10万円 | 6.62×10⁻³ | 25 | 2,458% |
| 100万円 | 6.63×10⁻² | 8 | 777% |
| 1000万円 | 6.61×10⁻¹ | 2 | 246% |
| 1億円 | 6.63 | 1 | 78% |

## 制約の本質

```
k_max = sqrt(MIN_SCREEN_AREA / min_node_area)
      ≈ sqrt(MIN_SCREEN_AREA × totalAmount / (CLUSTER_AREA × minAmount × GAP_FACTOR))
      ≈ sqrt(4 × 1.51×10¹⁴ / (1.6×10⁷ × 30 × 0.64))
      ≈ 1,308
```

つまり**ズーム限界 ≈ sqrt(総予算 / 最小金額) のオーダー**である。

151兆 ÷ 30 ≈ 5×10¹² → sqrt ≈ 2.2×10⁶ だが、CLUSTER_AREA(1.6×10⁷)とMIN_SCREEN_AREA(4)で正規化されて k ≈ 1,308 となる。

## 現在の設定

| パラメータ | 値 | 備考 |
|-----------|-----|------|
| MAX_ZOOM | 10,000 | k=10,000まで（1,000,000%） |
| MIN_SCREEN_AREA | 4 | ≈ 2×2px相当 |
| CLUSTER_SIZE | 4000×4000 | 16M px² |
| GAP_RATIO | 0.2 | ノード寸法の20%以下 |

MAX_ZOOM=10,000 > 必要k=1,308 のため、現在の設定で全ノード（30円まで）の表示が可能。

## 変更時の影響

- **CLUSTER_SIZEを拡大**: ノード面積が増え、必要kが下がる（低ズームで小額ノード表示可能）。ただしエッジパス座標・ファイルサイズに影響
- **MIN_SCREEN_AREAを下げる**: 必要kが下がるが、ノードが小さくなりすぎて視認性低下
- **GAP_RATIOを下げる**: ノード面積の損失が減り必要kが下がるが、ノード間の視覚的分離が悪化
