# Globe Phase 4 — 領域割り当てアルゴリズム検討

## 目的

Globe可視化のユーザーが、府省庁ごとの領域を**直感的な形状**で認識できるようにするため。
現在の緯度帯割り当ては面積比例を正確に実現しているが、全領域が水平ストライプになるため、
「地図上の国のような」認知しやすい形状にできるかを検討する。

## 現状（Phase 3）

### アーキテクチャ

```
API (route.ts)                    Client (GlobeView.tsx)
┌──────────────────┐             ┌──────────────────────────┐
│ rs2024-structured │             │ generateIcosphere(6)     │
│ → 33省庁 + その他 │──fetch──→  │ → 81,920面               │
│ → areaFraction    │             │ assignFaces(ico, data)   │
│ → seed [lon,lat]  │             │ → 緯度帯で北→南割り当て  │
└──────────────────┘             └──────────────────────────┘
```

### 割り当てアルゴリズム: 緯度帯方式

1. 全81,920面の重心Y座標を計算
2. Y降順（北極→南極）にソート
3. 支出額降順の府省庁に、`areaFraction × faceCount` 個ずつ連続割り当て
4. 最後の府省庁が残り全面を吸収

### 長所

- **面積比例が正確**: 各府省庁の面数 = `round(areaFraction × 81,920)` で誤差±1面
- **実装がシンプル**: ソート＋カーソル走査のみ、O(N log N)
- **境界が明確**: 隣接する府省庁は常に水平境界で接する
- **視認性**: 北極=最大省庁（厚労省）という直感的な配置

### 限界

- **全領域が水平ストライプ**: 地図的な「領土」感がない
- **境界のジャギー**: icosphereの三角形配置が正二十面体由来のため、同一緯度でもY座標が微妙にばらつき、境界線がギザギザになる
- **隣接関係が固定**: 支出額2位と3位は常に隣接（緯度で決まるため）

## 代替アルゴリズムの検討

### 1. 球面Voronoi分割

各府省庁に1つの種点（seed）を置き、全面を最寄りの種点に割り当てる方式。

**メリット**:
- 領域が「丸みを帯びた多角形」になり、地図的な見た目になる
- 種点の位置で隣接関係を自由に制御できる

**根本的な問題**:
- Voronoiは**等面積に収束する**性質がある（Lloyd緩和）
- 厚労省65.7% vs 公正取引委員会0.001% のような**極端な面積比**を実現不可能
- 過去の試行でmulti-seed（面積比に応じて種点を増やす）も試したが「見た目的にかなり歪」と判断済み

**結論**: 面積比例との相性が悪く、採用困難

### 2. HEALPix（Hierarchical Equal Area isoLatitude Pixelization）

天文学（宇宙マイクロ波背景放射の解析）向けに開発された球面分割方式。

**特徴**:

| 項目 | 内容 |
|------|------|
| ピクセル形状 | 曲線四角形（三角形ではない） |
| 面積均等性 | **厳密に均等**（icosphereは近似的） |
| 階層構造 | 12基底ピクセル × N² = 12N² |
| 緯度帯順序 | **ring ordering**で自然に緯度帯ソート済み |
| 近傍関係 | 各ピクセルの8近傍が定式化されている |

**参考にできる要素**:

1. **ring ordering**: ピクセル番号がそのまま緯度帯順になる。現在のicosphereではソート（O(N log N)）が必要だが、HEALPixライクな番号体系を導入すればソート不要になる
2. **厳密な等面積**: icosphereの三角形は正二十面体の頂点付近で面積が歪むが、HEALPixなら全ピクセルが厳密に同面積 → 面積比例の精度向上
3. **近傍定義**: 境界滑らか化アルゴリズムの入力として使える

**採用する場合の課題**:

- Three.jsの`BufferGeometry`で曲線四角形を描画するには、各ピクセルを三角形に分割する必要がある（追加の変換レイヤー）
- JS実装の選択肢が少ない（Python `healpy` が主流）
- 現在のicosphereからの移行コストが大きい

**結論**: 要素として参考になるが、フル採用は移行コストに対してメリットが小さい

### 3. 緯度帯 + 境界滑らか化（現方式の改善）

現在の緯度帯割り当てを維持しつつ、**境界のジャギーを軽減**する方式。

**手法: 境界面ローカルswap**

1. 緯度帯で割り当て（現在のロジック）
2. 境界面（隣接する面が異なる府省庁に属する面）を検出
3. 各境界面について、swap後に境界長が短くなるなら隣接府省庁にswap
4. 面数の増減が許容範囲（±1%）内に収まるよう制約

**メリット**:
- 現在の実装に追加するだけで済む（破壊的変更なし）
- 面積比例は維持される
- 境界線が滑らかになり、視認性向上

**必要な追加データ**: 面の隣接関係（adjacency graph）。icosphereでは辺を共有する面が隣接。

### 4. 細分割レベル上げ（レベル7）

81,920面 → 327,680面にすることで境界のジャギーを4倍細かくする。

**メリット**:
- 変更箇所が `ICO_SUBDIVISIONS = 7` の1行のみ
- 境界が自然に細かくなる
- 下位省庁（34位まで）にも面が割り当たる

**デメリット**:
- 頂点数が4倍 → 頂点カラー配列も4倍 → メモリ・描画負荷増
- `toNonIndexed()` 後の頂点数: 81,920×3 = 245,760 → 327,680×3 = 983,040
- モバイルでの性能懸念

## データの性質と制約

| 指標 | 値 |
|------|-----|
| 府省庁数（集約後） | 33 + その他 = 34 |
| 最大面積比率 | 65.7%（厚生労働省） |
| 上位3省庁の合計 | 79.2% |
| 上位9省庁の合計 | 96.9% |
| 最小省庁（集約前） | 0.00002%（公害等調整委員会） |
| 面積比の最大/最小比 | 約3,000,000:1 |

この極端な偏りが、Voronoiや等面積分割系のアルゴリズムが機能しない根本原因。
**緯度帯方式はこの偏りと最も相性が良い。**

## 推奨アプローチ

| 優先度 | 手法 | 効果 | リスク |
|--------|------|------|--------|
| A | 境界滑らか化（ローカルswap） | 境界のジャギー軽減 | 隣接グラフ構築の追加コスト |
| B | 細分割レベル7 | 解像度4倍向上 | メモリ・GPU負荷 |
| C | HEALPix ring ordering参考 | ソート不要化 | 効果が小さい（ソートは一瞬） |

**A（境界滑らか化）が最も費用対効果が高い。** 緯度帯の面積比例を維持しながら、境界線だけを改善できる。Bは性能テスト次第で併用可能。
