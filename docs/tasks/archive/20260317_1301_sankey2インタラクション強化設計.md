# /sankey2 Phase 4: インタラクション強化 設計

## 背景

Phase 3で基本的なSVG描画・Zoom/Pan・ホバーハイライトが完成。
現状のインタラクションは「ホバーで直接接続ノード・エッジをハイライト」のみ。

**課題**:
- ホバーを外すとハイライトが消え、フロー追跡を維持できない
- ノードの詳細情報（金額内訳・接続先リスト）を確認する手段がない
- 全体俯瞰時に現在のビューポート位置が分からない
- 37,258ノードでホバーごとに全エッジ走査 → パフォーマンス改善余地あり

## 4-1: ノードクリック → 選択状態 + サイドパネル詳細表示

### 概要

ノードをクリックすると「選択状態」になり、右側にサイドパネルが表示される。
選択中は接続ノード・エッジが常時ハイライトされ、ホバーと独立して動作する。

### 状態設計

```text
hoveredNodeId:  string | null   ← 既存（マウスオーバー）
selectedNodeId: string | null   ← 新規（クリック）

ハイライト判定:
  activeNodeId = selectedNodeId ?? hoveredNodeId
  connectedSet = activeNodeId の直接接続ノード集合
```

- クリック → `selectedNodeId` をセット
- 同じノードを再クリック or パネル閉じるボタン or Escキー → `selectedNodeId = null`
- 背景クリック → `selectedNodeId = null`

### サイドパネル仕様

```text
┌──────────────────────────────────┬──────────────┐
│                                  │ ✕ 閉じる     │
│         SVG描画領域              │              │
│                                  │  ノード名     │
│                                  │  金額         │
│                                  │  type         │
│                                  │  府省庁       │
│                                  │              │
│                                  │ ─ 接続先 ─── │
│                                  │  ↑ 流入元     │
│                                  │  node1: 金額  │
│                                  │  node2: 金額  │
│                                  │  ↓ 流出先     │
│                                  │  node3: 金額  │
│                                  │  node4: 金額  │
│                                  │              │
└──────────────────────────────────┴──────────────┘
```

**パネル内容**:
| セクション | 内容 |
|-----------|------|
| ヘッダー | ノード名、金額（formatAmount）、typeバッジ（色付き） |
| 属性 | 府省庁名（ministry）、事業ID（projectId） |
| 流入元 | source → このノードのエッジ一覧（金額降順、上位20件 + 残件数） |
| 流出先 | このノード → targetのエッジ一覧（金額降順、上位20件 + 残件数） |
| 接続先クリック | パネル内のノード名クリックで選択切替 + ズーム移動 |

**実装**:
- `Sankey2View.tsx` に `selectedNodeId` stateを追加
- パネルは SVG外の HTMLオーバーレイ（`absolute right-0`、幅320px）
- パネル表示中は SVG領域の幅を `calc(100% - 320px)` に縮小
- 接続エッジの検索は事前構築のインデックス（Map）を使用

### エッジインデックス（事前構築）

```typescript
// data変更時に一度だけ構築
const edgeIndex = useMemo(() => {
  if (!data) return { bySource: new Map(), byTarget: new Map() };
  const bySource = new Map<string, LayoutEdge[]>();
  const byTarget = new Map<string, LayoutEdge[]>();
  for (const edge of data.edges) {
    if (!bySource.has(edge.source)) bySource.set(edge.source, []);
    bySource.get(edge.source)!.push(edge);
    if (!byTarget.has(edge.target)) byTarget.set(edge.target, []);
    byTarget.get(edge.target)!.push(edge);
  }
  return { bySource, byTarget };
}, [data]);
```

これにより `connectedSet` 計算も O(edges) → O(接続数) に高速化。

## 4-2: BFS探索によるマルチホップハイライト

### 概要

現在は直接接続（1ホップ）のみハイライト。BFS探索で上流・下流を最大N段階まで辿り、フロー全体を可視化する。

### 仕様

- **デフォルト**: 1ホップ（現行動作と同じ）
- **Shiftキー + ホバー/クリック**: 上流・下流それぞれ最大3ホップまで展開
- **距離に応じたグラデーション**: ホップ数が増えるほど opacity を下げる

```text
距離0（選択ノード）: opacity 1.0
距離1（直接接続）:   opacity 0.8
距離2:              opacity 0.5
距離3:              opacity 0.3
```

### アルゴリズム

```typescript
function bfsHighlight(
  startId: string,
  edgeIndex: EdgeIndex,
  nodeMap: Map<string, LayoutNode>,
  maxDepth: number,
): Map<string, number> {  // nodeId → distance
  const distances = new Map<string, number>([[startId, 0]]);
  const queue: [string, number][] = [[startId, 0]];

  while (queue.length > 0) {
    const [nodeId, depth] = queue.shift()!;
    if (depth >= maxDepth) continue;

    // 下流探索（source → target方向）
    for (const edge of edgeIndex.bySource.get(nodeId) ?? []) {
      if (!distances.has(edge.target)) {
        distances.set(edge.target, depth + 1);
        queue.push([edge.target, depth + 1]);
      }
    }
    // 上流探索（target → source方向）
    for (const edge of edgeIndex.byTarget.get(nodeId) ?? []) {
      if (!distances.has(edge.source)) {
        distances.set(edge.source, depth + 1);
        queue.push([edge.source, depth + 1]);
      }
    }
  }
  return distances;
}
```

### 注意点

- BFSは `selectedNodeId` または `hoveredNodeId` をトリガーに `useMemo` で計算
- 37,258ノード・74,390エッジだが、3ホップ以内の到達範囲は通常数百〜数千ノード → 十分高速
- Shiftキーの状態は `useEffect` でキーイベントをリッスン

## 4-3: Minimap

### 概要

画面左下に小さな全体俯瞰マップを表示。現在のビューポート位置を矩形で示す。

### 仕様

```text
┌─────────────────────────────────────────┐
│                                         │
│            SVG描画領域                   │
│                                         │
│                                         │
│  ┌─────────┐                            │
│  │ Minimap │                            │
│  │ ┌──┐    │                            │
│  │ │VP│    │                            │
│  │ └──┘    │                            │
│  └─────────┘                            │
└─────────────────────────────────────────┘
```

- **位置**: 左下（HUDの下）
- **サイズ**: 幅200px × 高さ自動（アスペクト比維持）
- **内容**: 5クラスタの矩形（簡略化、個別ノードは描画しない）
- **ビューポート矩形**: 現在表示中の領域を半透明の青枠で表示
- **クリック/ドラッグ**: Minimapをクリック・ドラッグでビューポートを移動

### 実装方針

- `<canvas>` 要素で描画（SVGネストはパフォーマンス懸念）
- クラスタ5面の境界矩形 + ビューポート矩形のみ → 描画コスト極小
- `transform` state変更時に `requestAnimationFrame` で再描画

## 4-4: hover callbackメモ化

### 概要

PR #97 の Nitpick指摘。NodeRectの `onHoverStart`/`onHoverEnd` がレンダーごとに新規関数を生成し、37k NodeRectが全て再レンダーされる問題。

### 実装

```typescript
// NodeRect を React.memo でラップ
const NodeRect = React.memo(function NodeRect({ ... }: NodeRectProps) {
  // 既存実装
});

// 親コンポーネントでstableな callback map を構築
const hoverHandlers = useMemo(() => {
  if (!data) return new Map();
  const map = new Map<string, { start: () => void; end: () => void }>();
  for (const node of data.nodes) {
    map.set(node.id, {
      start: () => setHoveredNodeId(node.id),
      end: () => setHoveredNodeId(null),
    });
  }
  return map;
}, [data]);
```

- `React.memo` により props が変わらないNodeRectは再レンダーをスキップ
- `hoverHandlers` は `data` 変更時のみ再構築（37,258エントリ、メモリ ~3MB）
- `EdgeLine` も同様に `React.memo` 化

## 実装順序

| 順序 | タスク | 理由 |

| 1 | 4-4 メモ化 | 他タスクの基盤。先に最適化しておくことで以降のタスクでパフォーマンス問題を回避 |

| 4 | 4-3 Minimap | 独立機能。最後に追加 |
