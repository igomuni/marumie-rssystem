# 厳密辞書リファクタリング設計

## 目的

データ管理者が厳密辞書の品質を継続的に向上させやすくするため:
- 不要・過広なフラグを整理し、辞書の意味を明確にする
- 都道府県prefix付き市区町村名を厳密辞書内でvalidにし、補助辞書に依存しない形にする
- 辞書ファイルをGit管理対象の `public/data/dictionaries/` へ移動する

---

## 現状の厳密辞書フラグ分析

### ファイル構造

`data/result/recipient_dictionary.csv`（26,192件、Git管理外）

```
name, normalized_name, corporate_number, cn_count,
name_in_db, cn_in_db, name_cn_match, cn_name_contained, db_name_by_cn,
valid
```

### valid 判定ロジック（現状）

```python
valid = name_in_db AND best_cn AND name_cn_match
```

> **注意**: スクリプト冒頭コメントに `valid = name_in_db OR name_cn_match` と記載されているが、実装は AND 条件。ドキュメントバグ。

### フラグ組み合わせ実態（全26,192件）

| name_in_db | cn_in_db | name_cn_match | cn_name_contained | valid | 件数 | 解釈 |
|:---:|:---:|:---:|:---:|:---:|---:|---|
| True | True | True | False | **True** | 18,140 | 完全一致（正常） |
| False | False | False | False | False | 6,166 | CN無し・DB未登録（JV・略称等） |
| False | True | False | False | False | 957 | CN誤紐付き・部分名称 |
| False | True | False | True | False | 693 | DB名が支出先名に包含 |
| True | False | False | False | False | 236 | DB名称一致・CN未記入 |
| True | True | False | False | False | 100 | DB登録あり・CN紐付き名称不一致 |
| True | True | False | True | False | 4 | エッジケース |

### cn_name_contained の問題点

現在の定義: `cn_in_db AND NOT name_cn_match AND db_name_norm IN norm_key`

693件の内訳:
- **都道府県prefix付き市区町村** 93件: `北海道剣淵町` → DB:`剣淵町`（正当な別名表記）
- **支店・営業所付き社名** ~350件: `いであ株式会社東北支店` → DB:`いであ株式会社`
- **複数法人連名** ~100件: `(株)NTTドコモ,株式会社筑水キャニコム` → DB:`株式会社ＮＴＴドコモ`
- **その他** ~150件: 名称省略・競合CN等

「広すぎる定義」であり、93件の市区町村 と 600件の非対象を区別できていない。
valid 判定に使えない理由がここにある。

## フラグ見直し案

### 削除 or 改名候補: `cn_name_contained`

現状の `cn_name_contained` は「DB名が支出先名のどこかに含まれる」という広義の定義であり、
支店名・連名・略称など有象無象が混入している。この用途のまま残すと誤解を招く。

**案A: `cn_name_contained` を削除し、`pref_municipality` に置き換える**

新フラグ `pref_municipality` の定義:
```python
pref_municipality = (
    cn_in_db
    AND NOT name_cn_match
    AND db_name.endswith(('市', '区', '町', '村'))
    AND normalized_name.endswith(normalize(db_name))
    AND normalized_name != normalize(db_name)
)
```

このフラグは「市区町村の正式名称（DB名）に都道府県名・郡名を前置した表記」のみを捕捉する。
誤検知例の排除:
- `さいたま市立病院` → normalized_name が `さいたま市` で終わらない → 除外 ✓
- `愛媛県教育委員会` → DB名 `愛媛県` は市区町村サフィックスなし → 除外 ✓
- `北海道剣淵町` → DB名 `剣淵町`（町）, normalized_name が `剣淵町` で終わる → 対象 ✓

**案B: 両フラグを残し、pref_municipality を追加**

後方互換性のため `cn_name_contained` を残しつつ `pref_municipality` を追加。
列が増えるが移行期のデバッグに有利。

→ **推奨: 案A**（辞書を再生成するリファクタリングを行うなら不要な列は削除すべき）

### `name_in_db` の扱い

現状: valid 判定に使用（`name_in_db AND best_cn AND name_cn_match`）

`name_cn_match=True` な全18,140件で `name_in_db=True` も成立している（実データ確認済み）。
論理的には:

- `name_cn_match=True` → CNから引いたDB名 == normalized_name
- `name_in_db=True` → normalized_name がDB名称として存在する

この2条件は実質的に同値に近いが、`name_in_db` は「同じ名前で別CNが存在するか」の
クロスチェックとして機能している（同名異法人の誤検知防止）。

**結論: `name_in_db` は残す**。ただし valid 定義のコメントを正確に書き直す。

## 都道府県prefix市区町村のvalid化

### 対象: 93件

`recipient_dictionary.csv` 内で `pref_municipality=True` となる93件を `valid=True` にする。

### 新しい valid 判定ロジック

```python
# 従来条件（変更なし）
strict_match = name_in_db and bool(best_cn) and name_cn_match

# 新条件: 都道府県prefix付き市区町村
pref_municipality = (
    cn_in_db
    and not name_cn_match
    and db_name.endswith(('市', '区', '町', '村'))
    and norm_key.endswith(normalize(db_name))
    and norm_key != normalize(db_name)
)

valid = strict_match or pref_municipality
```

### 補助辞書との関係

現在 `supplementary_valid_names.csv` の `government_branch` カテゴリ（524件）に含まれる
都道府県prefix付き市区町村93件は、厳密辞書の valid 化によって**冗長になる**。

ただし `score-project-quality.py` のロジックは:
1. 厳密辞書でvalid → `valid_count++`
2. 厳密辞書でinvalidかつ補助辞書に存在 → `supp_valid_count++`

という二段構えのため、厳密辞書でvalid化された後は補助辞書の該当エントリは参照されない。
**即座に削除は必須ではないが、整合性のため同時に削除することを推奨**。

## 保存場所変更

### 現状と問題

| 項目 | 現状 | 問題 |
|------|------|------|
| パス | `data/result/recipient_dictionary.csv` | `.gitignore` により Git管理外 |
| 生成 | `scripts/build-recipient-dictionary.py` | 要: `data/houjin.db`（~1GB） |
| サイズ | ~2.4MB | Git管理可能 |

厳密辞書は **動的生成ではなく固定辞書** として機能しており、スコアリングの主要インプット。
Git管理外では再生成環境のないマシンでスコアリングスクリプトが動かない。

### 移動先

`public/data/dictionaries/recipient_dictionary.csv`

`.gitignore` に既に `!/public/data/dictionaries/*.csv` の許可ルールがあるため、
追加設定不要。

### 影響ファイル

| ファイル | 変更内容 |
|---------|---------|
| `scripts/build-recipient-dictionary.py` | `OUTPUT_CSV` パスを変更 |
| `scripts/score-project-quality.py` | `DICT_CSV` パスを変更 |
| `docs/quality-scoring-guide.md` | パス記述を更新 |

## 実装スコープまとめ

| タスク | 対象ファイル | 内容 |
|--------|------------|------|
| フラグ変更 | `scripts/build-recipient-dictionary.py` | `cn_name_contained` → `pref_municipality`（定義絞り込み）、valid 条件追加、コメント修正 |
| 再生成 | `data/result/recipient_dictionary.csv` → `public/data/dictionaries/recipient_dictionary.csv` | 新フラグ・新valid・新パスで出力 |
| スコアリング修正 | `scripts/score-project-quality.py` | DICT_CSV パス更新 |
| 補助辞書整理 | `public/data/dictionaries/supplementary_valid_names.csv` | 都道府県prefix市区町村93件を削除 |
| ドキュメント更新 | `docs/quality-scoring-guide.md` | 新しい軸1フロー・パス反映 |

### スコープ外

- `valid=True` 条件のさらなる緩和（支店名・JV 等のcn_name_contained 600件への対応）
- `cn_count > 1`（同名異法人）の扱い変更
- 補助辞書の government_branch（地方整備局等 431件）のさらなる見直し

## 期待されるスコア影響

厳密辞書の valid+93件により、軸1算出の分子が増加:
- 現在: `(valid_count + supp_valid_count) / total × 100`
- リファクタリング後: 93件が `valid_count` に移動（`supp_valid_count` から差し引き）

**スコア自体は変わらない**（valid+supp_valid の合計は同じ）。

ただし:
- UI の「+補助X件」表示が減る（実態に即した表示に改善）
- スコアリングの軸1が補助辞書なしでも正確になる
