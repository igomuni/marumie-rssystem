# 自前Sankeyデータパイプライン再設計

## 目的

開発者が `/sankey-lab` の自前レイアウトエンジン用のクリーンなデータを生成できるようにするため。
現行 `/sankey` のデータパイプライン（`sankey-generator.ts`）はnivo/d3-sankey前提の設計で、自前エンジンに渡すと以下の問題が発生する。

## 現行パイプラインの問題点

### 1. ノードtype の混乱

| ノード | 現行type | 問題 |
|--------|---------|------|
| `total-budget`（予算総計） | `ministry-budget` | 省庁と同じtype → 同じ列に配置される |
| `ministry-budget-other`（TopN以外省庁） | `ministry-budget` | 通常省庁と同じtype → 区別不能 |
| `project-budget-other-global` | `project-budget` | 同上 |
| `recipient-other-named`（「その他」） | `recipient` | 集約ノードと通常ノードが同一type |
| `recipient-other-aggregated`（TopN以外） | `recipient` | 同上 |

自前エンジンでは**typeで列を決定する**ため、typeが正確でないと列配置が崩れる。

### 2. 「その他」と「TopN以外」の混在

現行パイプラインは2種類の集約ノードを生成:

| ノードID | 名前 | 意味 |
|----------|------|------|
| `recipient-other-named` | その他 | 支出先名が「その他」の**実データ**（グローバル合計~26兆円、TopN表示時は絞り込まれ約16.7兆円） |
| `recipient-other-aggregated` | 支出先(TopN以外) | TopNに入らなかった支出先の**集約ノード**（TopN設定により変動） |

両方とも `type: 'recipient'` で、id/name の文字列パターンでしか区別できない。UI側で `isOtherNode()` のようなハックが必要になる。

### 3. 直接支出と間接支出の混在

現行パイプラインは直接支出先と間接支出先（再委託先）を区別せずに `recipient` ノードとして出力している。しかし、再委託チェーンにおける「どの支出先からどの支出先への再委託か」は金額ベースで90%が特定不能（ブロック内に複数支出先がいるため）。不正確なフローを表示するより、直接支出先に限定すべき。

### 4. ランタイム生成の複雑さ

現行 `sankey-generator.ts` は:
- 2,700行超のモノリシックファイル
- `selectData`（データ選択）と `buildSankeyData`（ノード・リンク構築）が密結合
- Global View / Ministry View / Project View の分岐が入り組む
- TopN計算、ドリルダウン、集約を全てランタイムで実行
- `generateSankeyData` の引数が12個

## /sankey2 パイプラインの良い点

`generate-sankey2-data.ts` は対照的にクリーンな設計:

| 特徴 | /sankey2 | /sankey（現行） |
|------|----------|----------------|
| ノードtype | `total`, `ministry`, `project-budget`, `project-spending`, `recipient` の5種 | 6種 + 同一typeに異なる意味のノードが混在 |
| 予算総計 | `type: 'total'` | `type: 'ministry-budget'`（省庁と同じ） |
| 生成方式 | ビルド時スクリプト（CSV→JSON） | ランタイムAPI（JSON→計算→レスポンス） |
| TopN集約 | なし（全データ出力、UIで制御） | パイプライン内でTopN計算 + 集約ノード生成 |
| ファイルサイズ | ~350行 | ~2,700行 |

## 再設計方針

### 支出先の範囲: 直接支出先のみ

サンキー図用データには**直接支出先のみ**を含める。再委託先（間接支出先）は含めない。

| 区分 | 金額 | 割合 | サンキー図での扱い |
|------|------|------|-------------------|
| 直接支出 | 129.73兆円 | 83.7% | `recipient` ノードとして出力 |
| 間接支出（再委託等） | 25.31兆円 | 16.3% | 含めない |

**理由:**
- 再委託チェーンは金額ベースで90%が「どの支出先からの再委託か」特定不能（[調査結果](20260326_0852_再委託チェーンのデータ化制約と設計方針.md)）
- 不正確なフローを描画するより、正確な直接支出フローに集中すべき
- 再委託データは別構造（`subcontract-chains.json`）で別途データ化する

### 直接支出先の判定方法

5-2 CSV（`5-2_RS_2024_支出先_支出ブロックのつながり.csv`）の `担当組織からの支出=TRUE` で判定する。

処理フロー:
1. 5-2 CSVを読み込み、`担当組織からの支出=TRUE` の `(予算事業ID, ブロック番号)` ペアを直接支出ブロックセットとして構築
2. 5-1 CSVを読み込み、各行の `(予算事業ID, 支出先ブロック番号)` が直接支出ブロックセットに含まれるかで分類
3. マッチした行の支出先 → `recipient` ノードとしてエッジを生成

ブロック番号自体に直接/間接の意味はない（Block A〜Zのいずれにも直接支出が存在する）。

### ノードtype の整理

新しいtype体系（/sankey2 に準拠 + 集約ノード用typeを追加）:

| type | 列 | 対象ノード |
|------|-----|-----------|
| `total` | 0 | 予算総計（1ノード） |
| `ministry` | 1 | 府省庁 |
| `project-budget` | 2 | 事業（予算側） |
| `project-spending` | 3 | 事業（支出側） |
| `recipient` | 4 | 支出先（直接支出先のみ） |

集約ノード（「TopN以外」）は**同じtypeを使い、`aggregated: true` フラグで区別**する:

```ts
interface SankeyNode {
  id: string;
  name: string;
  type: 'total' | 'ministry' | 'project-budget' | 'project-spending' | 'recipient';
  value: number;
  aggregated?: boolean;  // TopN以外の集約ノード
  originalId?: number;   // 元データID（集約ノードにはなし）
}
```

### 「その他」の扱い

支出先名「その他」は**実データであり集約ノードではない**。通常の `recipient` ノードとして扱う:

| ノード | type | aggregated | 並び順 |
|--------|------|-----------|--------|
| 年金受給者（24.6兆） | `recipient` | なし | TopN内（value降順） |
| その他（16.7兆） | `recipient` | なし | TopN内（value降順で自然に位置決定） |
| 支出先(Top10以外) | `recipient` | `true` | 常に最下部 |

### 生成方式: ビルド時事前計算

/sankey2 と同様に、**ビルド時にCSVから直接生成**する:

```
5-1 CSV + 5-2 CSV → スクリプト → sankey3-graph.json → クライアント
```

ランタイムAPIを経由しない。理由:
- 自前レイアウトエンジンはクライアント側でO(N)で計算可能
- TopN選択もクライアント側でスライス可能（全データがJSONに含まれる）
- `/sankey` の `sankey-generator.ts` への依存を断ち切れる

### 出力JSON構造

```ts
interface SankeyGraphData {
  metadata: {
    generatedAt: string;
    totalBudget: number;
    totalSpending: number;
    directSpending: number;      // 直接支出合計（129.73兆円）
    indirectSpending: number;    // 間接支出合計（25.31兆円）— データには含まない
    ministryCount: number;
    projectCount: number;
    recipientCount: number;      // 直接支出先のユニーク数
  };
  nodes: SankeyNode[];
  edges: SankeyEdge[];
}

interface SankeyEdge {
  source: string;  // ノードID
  target: string;  // ノードID
  value: number;   // 金額（1円単位）
}
```

### スクリプト構成

| ファイル | 役割 | 入力 |
|---------|------|------|
| `scripts/generate-sankey3-data.ts` | サンキー図用データ生成 | 5-1 CSV + 5-2 CSV |

`/sankey2` の `generate-sankey2-data.ts` をベースに、以下を変更:
- 5-2 CSVを読み込み、直接支出ブロックセットを構築
- 5-1 CSVの支出先を直接支出ブロックのみにフィルタ
- ノードtype を新体系に変更（`total`, `ministry` 等）
- 支出先名「その他」は通常 `recipient` ノードとして出力

### クライアント側の責務

スクリプトは**直接支出先の全データを出力**し、TopN選択・集約はクライアントが担当:

| 処理 | 担当 |
|------|------|
| CSV→ノード・エッジ生成（直接支出先のみ） | スクリプト（ビルド時） |
| TopN省庁の選択 | クライアント（スライダー） |
| TopN事業の選択 | クライアント（スライダー） |
| TopN支出先の選択 | クライアント（スライダー） |
| 集約ノード（「TopN以外」）の生成 | クライアント（選択結果から算出） |
| レイアウト計算（x/y座標） | クライアント（自前エンジン） |
| リボンパス生成 | クライアント |

### 検討事項: データサイズ

/sankey2 の `sankey2-graph.json` は ~25MB（gzip ~3MB）。全ノード・全エッジを含むためこのサイズになる。

新パイプラインは直接支出先のみのため、/sankey2 より小さくなる可能性がある（間接支出の84,866行分のエッジが不要）。圧縮後3MB以下であれば初回ロード時のフェッチは許容範囲。
