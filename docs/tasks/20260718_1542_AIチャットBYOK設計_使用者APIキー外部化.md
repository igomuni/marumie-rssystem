# AIチャット BYOK 設計 — 使用者自身の API キーで動かす

作成日: 2026-07-18（JST）
背景: Vercel 本番での AI チャット有効化を確認済み。運営側キーのままでは LLM コストと
乱用リスクを運営が全額負うため、**使用者が自分の API キーを登録して使う**（BYOK:
Bring Your Own Key）方式を検討する。

PdM 議事メモ（起点）:

```text
☆あなたのapiキーを使って、という感じにするといいのでは？
ブラウザのIndexedDBに保存 - ブラウザとopenrouterで直接やり取り
```

## 1. 現行アーキテクチャの確認

```text
ブラウザ ── POST /api/ai/sankey-chat（SSE進行イベント）
              └─ runSankeyChatAgent（app/lib/ai/sankey-chat-agent.ts・960行）
                   ├─ LlmCaller（DI済）── fetch → OpenRouter（キーは環境変数）
                   └─ ツール実行 ── サーバローダ直呼び（fs 読み・7ローダ）
```

- `LlmCaller` は既に DI（コールバック注入）になっており、LLM 呼び出し部の差し替えは容易。
- 一方**ツール実行はサーバローダ直結**（switch 文内で `loadSubcontracts` 等を直接呼ぶ）。
  ここがクライアント移植の主要な結合点。
- ツールが使うデータの公開 API 対応状況:

| ツール | 対応する公開 API | Vercel 公開状態 |
|---|---|---|
| search_projects / search_recipients / search_spending | `/api/search/*` | **公開済み** |
| get_project_detail | `/api/project-details/[id]` | 公開済み |
| （再委託系） | `/api/subcontracts/[id]` | 公開済み |
| （品質系） | `/api/quality-scores` 系 | 公開済み |
| （注目シグナル） | `/api/highlights` | 公開済み |
| run_sankey_query | `/api/sankey/query` | **非公開**（`SANKEY_QUERY_API_ENABLED` ガード） |

→ クライアント実行の障害は実質 **run_sankey_query の公開判断だけ**。

## 2. アーキテクチャ選択肢

### 案A: サーバ経由 BYOK パススルー（最小変更）

ブラウザがキーをリクエストヘッダで送り、サーバは環境変数の代わりにそれを使う。
キーはサーバを**通過**するが保存はしない。

- 利点: 変更が最小（route のキー解決 + クライアントのキー管理UIのみ）。エージェント・ツールは無変更。
- 欠点: 「キーが運営サーバを通る」ため、使用者は運営がキーをログしないことを**信頼するしかない**。
  Vercel の関数ログ・エラートラッカーに漏れ込む事故リスクも構造的に残る。
  サーバの実行時間・コンピュートは引き続き運営負担。

### 案B: フルクライアント実行（PdM 案・推奨）

エージェントループをブラウザで実行し、LLM はブラウザ → OpenRouter 直接、
ツールは自サイトの公開 API を fetch する。

```text
ブラウザ ── エージェントループ（クライアント移植）
   ├─ fetch → OpenRouter chat/completions（キーは IndexedDB、CORS 対応済みの公式API）
   └─ fetch → 自サイト公開 API（/api/search/* ほか。同一オリジンなので CORS 不要）
```

- 利点:
  - **キーが運営サーバに一切触れない**。「キーはあなたのブラウザにだけ保存されます」と
    言い切れる（プライバシー・信頼の説明が最も単純）。
  - LLM コスト・レートは使用者自身の OpenRouter アカウント管理下。運営コストゼロ。
  - Vercel の関数実行時間制限から解放（ループはブラウザで回る）。SSE 進行イベント基盤も
    クライアント実行では**不要になる**（進行はローカルコールバックで直接 UI に流せる）。
- 欠点・コスト:
  - エージェントの分割リファクタが必要（3節）。
  - run_sankey_query の公開判断が必要（4節）。
  - サーバ側で会話を観測できなくなる → usage-log（未充足需要の観測）が効かなくなる（5節）。

### 結論

**案B を採用**。PdM 案の方向性どおりで、セキュリティ説明の単純さ・運営コストゼロ・
実行時間制約からの解放と、利点が構造的。案A は「Bの途中版」としても中途半端
（キー通過の信頼問題を抱えたまま UI 投資だけ先行する）なので、経由しない。

## 3. エージェントの分割リファクタ（案B の中核）

`sankey-chat-agent.ts`（960行）を3層に分割する:

| 層 | 内容 | 配置 |
|---|---|---|
| コア（環境非依存） | システムプロンプト・ツール定義（LlmToolDef）・ループ制御・進行イベント | `app/lib/ai/chat-core.ts`（fs/fetch とも import しない Pure 層） |
| ツール実行の抽象 | `ToolExecutor = (name, args, year) => Promise<unknown>` を DI | コアの引数 |
| サーバ実装 | 現行の switch 文 = ローダ直呼び（dev・自キー運用の既存経路を維持） | `app/lib/ai/tool-executor-server.ts` |
| クライアント実装 | 同じツール名を公開 API fetch にマップ | `client/lib/ai/tool-executor-client.ts` |

- `LlmCaller` は既存 DI をそのまま使う: サーバ版=環境変数キー、クライアント版=IndexedDB キーで
  OpenRouter を fetch（`Authorization: Bearer`。OpenRouter はブラウザ直接呼び出しを公式サポート）。
- **サーバモード（現行 API）は開発用の残置ではなく、製品としても選択可能な共存モード**:
  コアを共有する2実行モードとして恒久維持する（どちらを既定にするかは運用判断 — 3-1節）。
- ループ・プロンプト・上限制御（MAX_LLM_ROUNDS / MAX_TOOL_CALLS）はコアに残り両モード共通。

### 3-1. 実行モードの共存と選択

UI は2つの判定を重ねてモードを決める:

1. サーバモードの可否 = `/api/ai/sankey-chat` の疎通（現行と同じ。`SANKEY_AI_CHAT_ENABLED` 依存）
2. クライアントモードの可否 = IndexedDB に使用者キーが登録済みか

| 運用パターン | env 設定 | 想定シーン |
|---|---|---|
| BYOK のみ | サーバモード env なし | 本番の基本形。運営 LLM コストゼロ |
| 両方有効 | env あり + レートリミット必須 | 「お試しはサイト提供（回数制限つき）、本格利用は自分のキーで」の導線。運営キー側のレートリミット実装が前提条件 |
| サーバモードのみ | env あり | ローカル開発・Preview 検証 |

両方有効な場合の既定はクライアントモード（キー登録済みなら使用者キーを優先）とする。

注意: 両モードはツール実行の経路が異なる（ローダ直呼び vs 公開 API fetch）ため、応答の
同一性はコア共有だけでは保証されない。ToolExecutor の入出力型を共通化し、フェーズ1の
リファクタ時に「同じ質問 → 同じツール結果」の比較確認で担保する。

## 4. run_sankey_query の公開判断

選択肢:

1. **`/api/sankey/query` を公開する**（`SANKEY_QUERY_API_ENABLED` ガード撤去 + レートリミット付与）
2. クライアント版ツールから run_sankey_query を外す（機能縮退）
3. クライアントに graph JSON を読ませてローカル実行（/sankey-svg ページは既に graph を
   fetch 済み — `executeQuery` は Pure 関数なのでブラウザでも動く）

**推奨は 3**: 追加の公開 API なしで済み、チャットを開く画面（/sankey-svg）は graph を
ロード済みなのでデータ二重取得もない。`executeQuery`（app/lib/sankey-query.ts・Pure）を
クライアント版 ToolExecutor から直接呼ぶ。1 の公開判断（レートリミット設計）は
チャットとは独立のテーマとして切り離せる。

## 5. キー管理 UI と保存

- **保存先: IndexedDB**（PdM 案どおり）。localStorage でも脅威モデルは同じ（XSS で読める）が、
  IndexedDB は構造化保存・将来の複数プロバイダ拡張に向く。
- 脅威モデルの正直な整理: ブラウザ保存キーは **XSS に対して防御不能**（HttpOnly にできない）。
  緩和は (a) サードパーティスクリプトを増やさない現状維持、(b) CSP 導入の検討、
  (c) 使用者への案内「**利用上限を設定したキーを使う**」（OpenRouter はキー別クレジット上限を
  設定可能 — 案内文に明記する）。
- UI 要件:
  - 設定パネル: キー入力（マスク表示・ペースト前提）・保存・削除・接続テスト
    （`GET /api/v1/models` 等の軽い呼び出しで検証）
  - モデル選択（使用者のコストなので選ばせる。既定は安価モデル）
  - 明示文言: 「キーはこのブラウザ（IndexedDB）にのみ保存され、当サイトのサーバには
    送信されません」/「上限設定済みキーの利用を推奨」
  - キー未登録時: チャット欄に登録導線を表示（現行の「available 判定で非表示」から変更）
- **キーを URL・usage-log・エラーレポートに決して載せない**。クライアント側の例外通知を
  将来入れる場合もヘッダをマスクする規約をコードコメントに明記。

## 6. usage-log（利用観測）への影響

クライアント実行になるとサーバは会話を観測できない。未充足需要の観測（Phase 0 の目的）を
継続するなら、**会話本文を送らない軽量テレメトリ**（イベント種別・ツール名・成否のみを
自サイト API に POST）を別途設ける必要がある。プライバシー文言（「会話はサーバに送信されません」）
と両立させるため、送る/送らないの線引きを明記した上で任意（オプトアウト可）とする。
→ 本設計のスコープ外とし、Phase 1 テレメトリ検討（20260707 の利用分析基盤）に接続する。

## 7. 実装フェーズ分解

1. **リファクタ**: chat-core 分割 + ToolExecutor DI（挙動不変。サーバ版で回帰確認。
   run_sankey_query のクライアント対応として executeQuery の Pure 呼びも整備）
2. **クライアント実行**: LlmCaller クライアント版（OpenRouter 直接）+ ToolExecutor
   クライアント版（公開 API fetch）+ ループのブラウザ実行と進行表示の接続
3. **キー管理**: IndexedDB ラッパ + 設定 UI + 接続テスト + 文言
4. **切替**: 公開 UI にモード判定（3-1節）を実装。サーバモードは共存モードとして恒久維持し、
   本番でどのパターンにするか（BYOK のみ / 両方有効）は運用判断。「両方有効」を選ぶ場合は
   運営キー側のレートリミット実装を前提条件とする

規模感: 1 が最大（960行の分割・回帰確認込みで 1日級）。2〜3 は各半日級。

## 9. リスク・残論点

- **OpenRouter 障害・仕様変更**が使用者体験に直結（運営プロキシがないため介入余地なし）。
  エラー文言でユーザー自身の OpenRouter ダッシュボード確認へ誘導する。
- モデルごとのツール呼び出し互換性（function calling 対応・引数 JSON の壊れ方）は
  モデル選択の自由化で幅が広がる。既定モデルでの動作保証 + 「その他は無保証」の線引き。
- 公開 API がチャット由来のアクセス増を受ける（特に search 系）。現状レートリミットなし —
  乱用検討（20260612）の宿題と合流する。
- 「あなたのAPIキーを使って」の文言・導線設計はサイト UI 一般名称ルール
  （AIアシスタント等）に従う。
