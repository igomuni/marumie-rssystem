# レポート駆動の自己改善ループ — 利用の観測から開発を導く仕組み

日時: 2026-07-05
ステータス: 考察メモ（[5つの利用モード](20260705_1613_サイトの使い方の全体像_5つの利用モード.md)の続編。プロダクトマネジメント視点でのループ設計）
発端: ユーザー発案「利用者の作成したレポートを分析し傾向を探ることで、機能のあるべき姿・リソース配分のヒントになる。このループを回せれば、サイト自身がAIのサポートで自発的に学習し可塑性を持つシステムにできるのでは」

## 1. このループは既に3周回っている（観測事実）

仮説ではない。レポート起点の機能改善が実績として3回発生している:

| 周回 | レポート（利用） | 発見された摩擦 | 生まれた改善 |
|------|----------------|--------------|------------|
| 1 | 実験①再エネ | 数事業のスコア取得に全件7MB | Issue #240 → `pids` バッチAPI（#242） |
| 2 | 実験②防衛×2 | プレイブック有無で品質差／API記録が判定不能 | プレイブック改訂（API記録必須化・逸脱許容） |
| 3 | e-Tax（Opus） | NTTデータ3法人の名寄せで詰まる・誤記載の存在 | **#243 法人番号名寄せ再設計**（最大級の機能開発に直結） |

特に3周目が象徴的で、「レポートを書こうとしたら名寄せで詰まった」という利用時の摩擦が、そのまま次の開発テーマになった。

## 2. 中核の考え方 — レポートはテレメトリである

レポートは単なるアウトプットではなく、「**利用者が何を問い、どこで手間取り、何が足りなかったか**」の構造化された観測記録として機能している。この解像度を上げる仕掛けは既に布石済み:

- プレイブックで「使用API（エンドポイントと概算呼び出し回数）の記録」を必須化（実験②の改訂）
- [現在地検討](20260705_1544_自然言語レポート生成の現在地と深掘り拡張の検討.md) 3-5 の**再現性ブロック**（レポート末尾に `SankeyQuery` JSON を残す）が入れば、レポート群は**機械可読なユーザーシナリオ・コーパス**になる

## 3. ループの定式化（足りないのは分析と triage の定例化だけ）

```text
利用: エージェントがレポート作成（プレイブック準拠・API記録つき）
  ↓ 蓄積
コーパス: docs/reports/（現在地検討 3-5 の分離案）
  ↓ N本ごと・または定期
分析: エージェントが横断分析
  - 頻出する問いの型（利用モードの分布）
  - 摩擦点（呼び出し回数の多い工程・自前計算していること・罠の踏み抜き）
  - 使われなかった機能・API
  ↓
提案: ロードマップ案の更新（機能候補 + 優先度 + 根拠となるレポート引用）
  ↓
判断: 人間が採否を決める ←【ここは自動化しない】
  ↓
実装 → プレイブック/api-guide 更新 → 次の利用へ
```

「人間が判断」を残すのが可塑性の設計上の要点（§4）。

## 4. 可塑性の勾配 — 自己改善が暴走しないための構造

システムの各層で「AIが変えてよい速度」が異なる。この勾配は既に事実上できており、意識的に維持する:

| 層 | 内容 | 変更の仕方 | 実績 |
|----|------|-----------|------|
| 柔らかい層 | プレイブック・api-guide | AIが実戦知見で直接追記（低コスト・可逆） | Opus が名寄せ・二重計上の罠を追記、Fable が実験結果で改訂 |
| 中間層 | API・summary の指標 | レポート起点で提案 → PR・レビューを通す | pids バッチ、compareYears（提案中） |
| 不変の骨格 | データ正確性最優先・1円単位・「その他」の区別・読み取り専用 | **学習で動かさない制約条件** | CLAUDE.md・プレイブック§1-2 |

骨格があるから可塑性が機能する。「AIサポートで自己改善するが、信頼性の核は変質しない」を保つ。

## 5. 現時点の限界 — コーパスの偏り

現在のレポートは実質2利用者（Fable / Opus = いずれも作り手側）によるもので、傾向分析は「**作り手の関心への過学習**」リスクを持つ。本当のシナリオ分布が見えるのは公開フェーズで外部の問いが入ってから。

- それまでのローカルループは「仕組みのプロトタイプ検証」と位置づける
- 裏返すと、**公開判断（ロードマップF）の価値項目に「シナリオ・コーパスの多様化」が加わった**

## 6. 次アクション（本PRをロードマップとする実装キュー）

実装はローカル Opus で実施予定。優先順は [現在地検討](20260705_1544_自然言語レポート生成の現在地と深掘り拡張の検討.md) §4 のとおり:

1. **年度間比較の一級化**（`/api/sankey/query` に compareYears モード。差分計算はAPI側）
2. **事業概要の全文検索**（`/api/search/projects?scope=details`）
3. **集中度指標**（summary.recipients に topShare1/topShare3。1と同時改修可）
4. **レポートライブラリ規約**（`docs/reports/` 分離 + 再現性ブロックのプレイブック追記）— 本ループのコーパス化に直結
5. **gBizINFO 接続実験**（APIキー取得後）

ループ運用そのものの初回実行（レポート横断分析 → ロードマップ更新）は、4 でコーパスの置き場が整い、レポートが2〜3本追加された時点が適期。
