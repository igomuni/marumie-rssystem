# ブロックノード層導入設計

## 目的

ユーザーが再委託チェーンの全体像を正確に把握できるようにするため。

現在、支出先ノードのみで再委託を表現しているが、CSVの再委託構造は「支出先ブロック」単位で定義されており、ブロックに複数支出先がある場合（全接続の27.3%）は原理的にエッジを生成できない。ブロック自体をノードとして導入することで、CSVの構造に忠実な再委託表現が可能になる。

背景調査: [20260322_1751_再委託チェーン表現の限界と次のステップ.md](20260322_1751_再委託チェーン表現の限界と次のステップ.md)

## 現状

```text
事業(支出) ──→ 支出先
               │
               └──→ 支出先（subcontractエッジ、ブロック名一致時のみ）
```

- ノードタイプ: `total`, `ministry`, `project-budget`, `project-spending`, `recipient`
- クラスタ: 5面（左→右に配置、各4000×4000px）
- 再委託: ブロック名が支出先名に一致する場合のみ `recipient→recipient` エッジを生成（カテゴリ名ブロックはスキップ）
- 間接支出先: 13,391ノードが府省庁帰属を持てず、レイアウトから除外

## 設計

### 新ノードタイプ: `spending-block`

事業(支出)と支出先の間に位置するブロックノード。5-2 CSVの支出先ブロックに対応。

- **ID形式**: `block-{pid}-{blockLetter}`（例: `block-3440-B`）
- **label**: ブロック名（5-2 CSVの支出先ブロック名。例: "都府県", "北海道"）
- **amount**: ブロック内支出先の合計金額（5-1 CSVのブロック合計支出額）
- **ministry**: 親事業の府省庁を継承
- **projectId**: 所属する事業ID

### エッジ構造の変更

現行の `project-spending → recipient` + `recipient → recipient (subcontract)` を以下に置換:

| エッジ種別 | source | target | 件数 | 生成元 |
|-----------|--------|--------|------|--------|
| `project-spending → block` | `project-spending-{pid}` | `block-{pid}-{letter}` | 13,088 | 5-2 CSV（担当組織からの支出=TRUE） |
| `block → block` | `block-{pid}-{src}` | `block-{pid}-{tgt}` | 6,909 | 5-2 CSV（非直接支出行） |
| `block → recipient` | `block-{pid}-{letter}` | `recipient-{name}` | 74,946 | 5-1 CSV（ブロック番号→支出先） |

現行の `project-spending → recipient`（直接支出エッジ）と `recipient → recipient`（subcontractエッジ）は廃止。

### クラスタ配置

```text
total | ministry | project-budget | project-spending | spending-block | recipient
 [0]     [1]          [2]              [3]                [4]            [5]
```

`CLUSTER_TYPES` に `spending-block` を追加（`project-spending` と `recipient` の間）。6面構成。

### spending-block クラスタのレイアウト: 事業トポロジー配置

spending-block クラスタは通常のtreemap配置ではなく、**親事業(project-spending)のY座標に揃えて配置**する。

```text
project-spending クラスタ        spending-block クラスタ
┌────────────────────┐          ┌────────────────────┐
│ ┌──────────┐       │          │ ┌─┬─┬─┐           │
│ │ 事業A    │       │  ───→    │ │A│B│C│  (事業Aのブロック) │
│ │          │       │          │ └─┴─┴─┘           │
│ └──────────┘       │          │                    │
│ ┌────┐             │          │ ┌─┬─┐             │
│ │事業B│             │  ───→    │ │A│B│  (事業Bのブロック)  │
│ └────┘             │          │ └─┴─┘             │
│ ...                │          │ ...                │
└────────────────────┘          └────────────────────┘
```

**配置アルゴリズム**:
1. project-spending クラスタの各ノードの `y` と `height` を取得
2. spending-block クラスタ内で、同じ `y` 〜 `y + height` の矩形領域にその事業のブロックノード群をtreemap配置
3. ブロックのないproject-spending ノードの領域は空白

**利点**:
- `project-spending → block` のエッジが短い水平線になり、視覚的に事業とブロックの親子関係が明確
- ブロック→ブロック（再委託）のエッジも事業内で完結するため交差が少ない
- 府省庁グルーピングが project-spending と自動的に揃う（同じY帯に配置されるため）

### 府省庁帰属の変更

現行: `project-spending → recipient` エッジから直接計算（間接支出先は帰属なし）

変更後: `block` ノードは親事業の府省庁を継承するため、`block → recipient` エッジ経由で全支出先に帰属が伝搬する。間接支出先13,391ノードの帰属問題が解消。

### 支出先1件ブロックの扱い

57%のブロックは支出先が1件のみ。ブロックノードを介してもパススルーになるが、構造の一貫性のために全ブロックをノード化する。レンダリング時にLOD（面積ベース表示閾値）で自然にフィルタされるため、表示上の問題は発生しない。

## 影響範囲

### scripts/generate-sankey2-data.ts

- `spending-block` タイプのノード生成を追加
- `project-spending → block` エッジの生成（5-2 CSVの直接支出行から）
- `block → block` エッジの生成（5-2 CSVの非直接支出行から）
- `block → recipient` エッジの生成（5-1 CSVのブロック番号→支出先から）
- 現行の `project-spending → recipient` 直接エッジを廃止
- 現行のサブコントラクトエッジ生成ロジック（5iセクション）を廃止
- `Sankey2Node.type` に `spending-block` を追加

### scripts/compute-sankey2-layout.ts

- `CLUSTER_TYPES` に `spending-block` を追加（5→6面）
- 府省庁帰属計算: `project-spending →` ではなく `block →` を見るか、blockノードのministry属性から直接取得
- treemap配置: spending-blockクラスタの府省庁グルーピング追加

### client/components/Sankey2/types.ts

- `LayoutNode.type` に `spending-block` を追加

### client/components/Sankey2/Sankey2View.tsx

- `TYPE_COLORS` に `spending-block` の色を追加
- `TYPE_LABELS` に `spending-block` の日本語ラベルを追加（例: "支出ブロック"）
- BFS探索: `spending-block` ノードを経由する探索パスの追加
- パネル表示: ブロックノードクリック時の情報表示

## 実装ロードマップ

### Phase 1: データパイプライン — ブロックノード・エッジ生成

**ファイル**: `scripts/generate-sankey2-data.ts`

**作業内容**:
1. `Sankey2Node.type` に `'spending-block'` を追加
2. 5-2 CSVの全ブロック（pid:blockLetter）に対して `spending-block` ノードを生成
   - ID: `block-{pid}-{letter}`、label: ブロック名、amount: ブロック合計支出額
   - ministry: 親事業の府省庁を継承、projectId: 事業ID
3. エッジ生成の置換:
   - 現行の `project-spending → recipient` 直接エッジを `project-spending → block` に変更
   - 現行のサブコントラクトエッジ（5iセクション）を `block → block` に変更
   - 新規: `block → recipient` エッジを5-1 CSVのブロック番号→支出先から生成
4. `subcontractChains` のflows生成ロジックを簡素化（ブロックノードが構造を持つためflowsは不要になる可能性あり）

**検証**: `npm run generate-sankey2` でエラーなく完了。graph.jsonにspending-blockノードが存在し、エッジが `project-spending → block → recipient` の経路を持つことを確認。

### Phase 2: レイアウト — 6面クラスタ・トポロジー配置

**ファイル**: `scripts/compute-sankey2-layout.ts`

**作業内容**:
1. `CLUSTER_TYPES` に `'spending-block'` を追加（index=4、recipientが5に移動）
2. 府省庁帰属計算の変更: `project-spending →` ではなく `block → recipient` エッジから計算（または blockノードのministry属性を直接利用）
3. spending-blockクラスタのトポロジー配置:
   - project-spendingクラスタの各ノード位置を取得
   - 同じ事業のブロックノード群を、親事業の `y` 〜 `y + height` の帯にtreemap配置
   - ブロックのない事業の帯は空白
4. 仮想空間サイズの更新（5面→6面分）

**検証**: `npm run compute-sankey2-layout` でエラーなく完了。layout.jsonにspending-blockノードが座標を持ち、親事業とY座標が揃っていることを確認。
