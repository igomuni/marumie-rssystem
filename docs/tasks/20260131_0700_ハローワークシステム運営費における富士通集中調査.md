## 4. 契約の特徴と課題

### 4.1 入札参加の困難性

大規模な一般競争契約において、入札参加者が1社のみ（富士通のみ）となるケースが確認されている。

#### 入札辞退理由（事業者ヒアリング結果）

1. **規模の問題**
   - 「案件の規模が大きく、人員を確保できないと考えた」
   - 「他案件との兼ね合いもあり、改修に必要な人員を確保できなかった」

2. **技術的難易度**
   - 「既存システムの改修であり規模が大きく、難易度が高く確実な履行ができないと判断した」

3. **既存システムへの依存**
   - 既存のハローワークシステムが富士通により構築されているため、改修には既存システムの詳細な知識が必要

### 4.2 随意契約の理由

機器の賃貸借（再リース）7件について、**「互換性」を理由とした随意契約**が行われている（69.10億円）。

これは既存システムとの互換性確保のため、他社製品への切り替えが困難であることを示している。

### 4.3 国庫債務負担行為

以下の契約が国庫債務負担行為として継続されている：
- システム運用・保守: 68.71億円
- 機器・回線賃貸借30件: 174.17億円

これらは複数年度にわたる契約であり、特定ベンダーへの長期的な依存関係を形成している。

## 5. ベンダーロックインの構造分析

### 5.1 ベンダーロックインの要因

1. **初期システム構築**: 富士通がハローワークシステムの初期構築を担当（2011年開始）
2. **技術的依存**: システムの詳細仕様や設計情報が富士通に集中
3. **人員確保の困難性**: 大規模システムの改修に必要な人員を他社が確保できない
4. **互換性の制約**: 既存機器・システムとの互換性により、富士通製品以外の選択肢が限定
5. **継続契約**: 国庫債務負担行為による複数年契約により、長期的な関係が固定化

### 5.2 競争性の欠如

| 項目 | 状況 |
|------|------|
| 主要契約の入札参加者数 | 1社（富士通のみ） |
| 総執行額に占める富士通の割合 | 80.7% |
| 随意契約の理由 | 互換性（事実上の随意契約） |
| 代替ベンダーの可能性 | 極めて限定的 |

## 6. 再委託先の状況

### 6.1 データの制約

**重要な発見**: 5-2 CSV（支出先_支出ブロックのつながり）を確認したところ、富士通株式会社からの再委託先データは公開データに**含まれていない**。

すべての支出ブロック（A〜I）が「担当組織からの支出=TRUE」となっており、以下の情報は取得できない：
- 富士通からどの企業に再委託されているか
- 再委託の金額
- 再々委託以降の資金の流れ

### 6.2 透明性の課題

641億円という大規模な支出のうち、どの程度が富士通内部で処理され、どの程度が外部企業に再委託されているかは、**現行の公開データでは不明**である。

## 7. システム更改の動向

### 7.1 クラウド化の方針

「デジタル社会の実現に向けた重点計画（2021年12月24日閣議決定）」に基づき、以下が求められている：
- クラウド・バイ・デフォルトの原則に基づくクラウド化
- 業務BPR
- 行政サービスのオンライン化

### 7.2 2023年度以降のシステム更改

最大の契約（259.17億円）は「ハローワークシステムの更改に係る全体アーキテクチャ設計・開発及び運用・保守等業務一式」であり、2023年度以降の新システムへの移行を含む。

ただし、この更改においても富士通が単独で受注しており、競争性の確保には至っていない。

## 9. 課題と考察

### 9.1 主要な課題

1. **過度なベンダー依存**
   - 80.7%という極めて高い集中度
   - 実質的な競争の欠如

2. **入札プロセスの形骸化**
   - 入札参加者が1社のみ
   - 総合評価方式でも競争が機能していない

3. **透明性の不足**
   - 再委託先の情報が公開されていない
   - 実際の業務遂行体制が不明

4. **長期的な依存関係の固定化**
   - 国庫債務負担行為による複数年契約
   - システム更改においても同一ベンダーが継続

5. **コスト最適化の困難性**
   - 競争原理が働かない環境
   - 価格妥当性の検証が困難

### 9.2 制度的な背景

現行の行政事業レビューシートでは、再委託以降の資金の流れについて十分な情報公開が行われていない。これにより、以下の問題が生じている：

- 実際の業務遂行主体が不明
- 中間マージンの有無や規模が不明
- 真の技術保有者が誰かが不明

## 10. 提言

### 10.1 競争性の確保

1. **システムの標準化・モジュール化**
   - ベンダー依存を減らすための技術的アーキテクチャ改善
   - オープンな標準規格の採用

2. **段階的な契約分割**
   - 大規模な一括契約を避け、機能単位で分割
   - 複数ベンダーによる競争環境の創出

3. **技術情報の開示**
   - システム仕様・設計情報の標準化と開示
   - 他社の参入障壁を低減

### 10.2 透明性の向上

1. **再委託情報の完全公開**
   - 再委託先の企業名、金額、業務内容の公開
   - 再々委託以降の情報も含む

2. **業務遂行体制の明確化**
   - 実際に作業を行う企業・人員の情報公開
   - 中間マージンの可視化

### 10.3 長期的な改善策

1. **クラウドサービスの活用**
   - 特定ベンダーへの依存を避けるマルチクラウド戦略
   - SaaSモデルの採用による柔軟性確保

2. **オープンソース化の検討**
   - 可能な部分のオープンソース化
   - コミュニティによる改善の促進

3. **段階的な移行計画**
   - 一度に全体を刷新するのではなく、段階的に競争環境を導入
   - リスクを抑えた移行戦略

## 11. まとめ

ハローワークシステム運営費における富士通株式会社への支出集中（80.7%）は、典型的な**ベンダーロックイン**の事例である。

### 主要な発見

1. 865億円の事業において、富士通への支出が641億円（執行比80.7%）と極めて高い
2. 主要契約の入札参加者が1社のみで、実質的な競争が機能していない
3. 再委託先の情報が公開されておらず、透明性に課題がある
4. 技術的依存、人員確保の困難性、互換性の制約により、他社への切り替えが極めて困難

### 今後の方向性

政府の「デジタル社会の実現に向けた重点計画」に基づくクラウド化は進められているものの、**システム更改においても同一ベンダーが受注している**現状では、根本的な競争環境の改善には至っていない。

ベンダーロックインの解消には、技術的アーキテクチャの見直し、契約方式の改善、情報公開の徹底など、**多面的かつ長期的な取り組み**が必要である。
