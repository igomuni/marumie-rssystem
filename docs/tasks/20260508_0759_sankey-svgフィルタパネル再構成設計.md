# /sankey-svg フィルタパネル再構成設計

## 背景・課題

省庁ノードクリックでフィルタ操作を連動できるようになった（[20260508_0610](20260508_0610_sankey-svg省庁ノードクリック_フィルタ連動設計.md)）一方、既存の検索/フィルタ UI には以下の課題が顕在化した：

1. **「金額フィルタ」という名前が実態と合わない**
   現状は会計区分・省庁・予算/支出金額を含む統合パネル。ここに事業/支出先のテキストフィルタも入る予定なので、もはや「金額」フィルタではない。

2. **検索モード/フィルタモードのトグルが煩雑**
   1 つの入力欄を `filterActive` で「検索」と「フィルタ」のどちらかに切り替える方式。「事業を検索しつつ支出先で絞り込む」のような複合操作が不可能。

3. **`filterTarget`（事業/支出先切替）が排他的**
   テキストフィルタを事業 OR 支出先のどちらか一方にしか使えない。両方同時に絞り込みたいニーズに対応できない。

## 仕様変更サマリ

| 項目 | 変更前 | 変更後 |
|------|--------|--------|
| パネル名 | 「金額フィルタ」 | **「フィルタ」** |
| モードトグル | `filterActive` で検索/フィルタ切替 | **廃止**（検索パネルとフィルタパネルが独立して常時利用可） |
| 事業/支出先切替 | `filterTarget` でいずれか1つ | **廃止**（事業名・支出先名 両方を独立フィールドに） |
| 正規表現トグル | 共通 1 個（`searchUseRegex`） | **事業名・支出先名 ごとに独立**（検索パネル用は別途維持） |
| URL パラメータ | `f`, `nft`, `nf` | **`f`/`nft`/`nf` を廃止し新パラメータに一本化** |

## 詳細設計

### 1. UI 構成

#### 検索パネル（変更最小）

- 左上の検索アイコン + 入力欄 + 結果ドロップダウン
- 用途: ノード探索・選択ジャンプ専用
- 状態: `searchQuery`, `debouncedQuery`, `searchUseRegex`, `searchPage`, `searchCursorIndex`
- フィルタとは完全に独立。検索結果はフィルタ除外を考慮して候補を絞る挙動は維持。

#### フィルタパネル（新構成）

ボタン名・パネル見出しを「フィルタ」に変更。展開すると以下を縦に並べる：

```
┌─ フィルタ ──────────────────────────────┐
│  会計区分  [一般 特会 両方 計上なし]      │
│  府省庁    [複数選択ドロップダウン]        │
│  事業名    [テキスト入力] [.* トグル]     │
│  支出先名  [テキスト入力] [.* トグル]     │
│  予算額    [Min] ～ [Max]                │
│  支出額    [Min] ～ [Max]                │
└────────────────────────────────────────┘
```

事業名・支出先名は独立して動作。両方入力すれば AND 条件で絞り込む。

### 2. State の変更

#### 削除

- `filterActive: boolean` — 廃止
- `filterTarget: 'project' | 'recipient'` — 廃止

#### 維持

- `filterMinistryNames: string[]`
- `filterMinBudgetText`, `filterMaxBudgetText`, `filterMinSpendingText`, `filterMaxSpendingText`
- `acGeneral`, `acSpecial`, `acBoth`, `acNone`
- `searchQuery`, `debouncedQuery`, `searchUseRegex`（検索パネル専用に名称ロール固定）
- `showAmountSliders` → `showFilterPanel` にリネーム（パネル全体の開閉なので名称も変える）

#### 新規追加

| State | 型 | デフォルト | URL | 用途 |
|-------|---|---------|-----|------|
| `filterProjectName` | `string` | `''` | `fnp` | 事業名フィルタ |
| `filterProjectNameRegex` | `boolean` | `false` | `fnpr` | 事業名フィルタの正規表現モード |
| `filterRecipientName` | `string` | `''` | `fnr` | 支出先名フィルタ |
| `filterRecipientNameRegex` | `boolean` | `false` | `fnrr` | 支出先名フィルタの正規表現モード |

### 3. URL パラメータ

#### 廃止

- `f` (filterActive)
- `nft` (filterTarget)
- `nf` (filterNameQuery; 検索クエリと兼用していたが、検索とフィルタの分離に伴い廃止)

#### 新規追加

- `q=<query>` 検索クエリ専用（旧 `nf` の代替）
- `fp=1` フィルタパネル開閉
- `fnp=<query>` 事業名フィルタ
- `fnpr=1` 事業名 正規表現 ON
- `fnr=<query>` 支出先名フィルタ
- `fnrr=1` 支出先名 正規表現 ON

> 旧 `nf` は検索クエリとフィルタの名前クエリを兼用していたため、検索/フィルタの分離に伴い **検索専用に `q` を新設** + **フィルタ専用に `fnp`/`fnr` を新設** の 2 段に分離する。

#### 維持

- `fm`（省庁、複数指定可）
- `fmb`/`fxb`/`fms`/`fxs`（予算/支出 Min/Max）
- `ac`（会計区分フラグ）

### 4. フィルタ計算ロジックへの影響

`filterExcludedIds` の `useMemo` 内のフラグ構築。

変更前：

```js
const hasName = filterActive && trimmedQuery.length >= 1;
const hasMinistry = filterMinistryNames.length > 0;
```

変更後：

```js
const hasProjectName = filterProjectName.trim().length >= 1;
const hasRecipientName = filterRecipientName.trim().length >= 1;
const hasMinistry = filterMinistryNames.length > 0;
```

Pass 1 でのマッチング判定：

```js
// project-budget ノード
const failProjectName = hasProjectName && !matchesProject(n.name);
// recipient ノード
const failRecipientName = hasRecipientName && !matchesRecipient(n.name);
```

`matchesProject` / `matchesRecipient` はそれぞれの正規表現フラグを参照する。

Pass 2 のカスケード除外条件にも `hasRecipientName` を追加する。

### 5. 検索パネルへの影響

検索パネルは `searchQuery`, `searchUseRegex` をフィルタとは完全に独立した state として保持し、入力欄のロールが切り替わらないようにする。

検索結果ドロップダウンの絞り込み（`searchResults` の `useMemo`）で `filterMinistryNames` と他フィルタを参照する点は維持（フィルタ適用後の候補から検索する仕様）。`filterTarget` 依存ロジックは削除。

### 6. 省庁ノードクリック連動への影響

[20260508_0610](20260508_0610_sankey-svg省庁ノードクリック_フィルタ連動設計.md) の機能は影響なし。`filterActive` 削除によりすでに「touch しない」設計なので、追加対応不要。

## マイグレーション戦略

旧 URL は `f=1&nft=p&nf=foo` 形式 → 新 URL は `fnp=foo` 形式。シンプルな置換だが、古いブックマークは動かなくなる。

ユーザー数・履歴の重要度から「破壊的変更で OK」と判断（外部からのリンクは限定的、内部で `fmc` 等を導入したばかりで `f` 系が安定的に使われていない）。

## 実装ステップ

1. State 削除・追加（`filterActive`, `filterTarget`, `searchPage` 周りは検索専用に整理）
2. `parseSearchParams` 更新（旧パラメータ削除・新パラメータ追加）
3. URL 同期 `useEffect` の更新
4. `filterExcludedIds` の `useMemo` を更新（事業名・支出先名を独立処理）
5. UI: ボタンラベル「金額フィルタ」→「フィルタ」、検索/フィルタトグル削除、事業/支出先 select 削除、フィルタパネル内に事業名・支出先名フィールドを追加
6. 検索パネルから `filterTarget` 依存箇所を整理
7. `npm run lint` + `npx tsc --noEmit` で検証

## 設計判断メモ

- **検索とフィルタを完全分離**: 兼用は柔軟性が低く、複合操作（検索しつつ別条件で絞る）に対応できない。独立化することで認知負荷も下がる。
- **事業名・支出先名の正規表現を独立トグル化**: 事業を完全一致、支出先を正規表現、のような混在ニーズに応えるため。共通トグルにすると一方を変えると他方も切り替わるので意外性が高い。
- **`q` を検索専用に新設**: `nf` を再利用せず別名にすることで、検索/フィルタの URL 表現も明確に分離。
- **省庁・会計・金額は維持**: 既存の non-text フィルタは挙動を変えない。テキスト系のみリファクタする最小スコープ。

## 検証ポイント

- 事業名・支出先名 単独 / 複合での絞り込みが効く
- それぞれの正規表現トグルが独立に動く（無効正規表現時の枠線赤も両方）
- 検索パネルがフィルタパネルと独立して動く（検索結果は引き続きフィルタを考慮する）
- 旧 URL（`f=1&nft=p&nf=foo`）を開くとパラメータが無視され、初期状態で起動（破壊的変更）
- 省庁ノードクリック連動 / focusRelated / autoFocusRelated は引き続き動作
