## 2. セキュリティ基準（次タスク向け・要対応候補）

### ✅ F-1: プロトタイプ汚染キーで 500 + 内部エラー文言の露出 → **本ブランチで修正済み**
- 再現（修正前）: `GET /api/recipients/__proto__` / `constructor` → **500** `{"error":"Cannot read properties of undefined (reading 'length')"}`
- 原因: `app/lib/api/recipient-index-loader.ts` の `index.recipients[key] ?? null`。`key="__proto__"` だとプロトタイプオブジェクト（truthy）が返り `?? null` で捕捉できず、後段 `entry.appearances.length` で例外。
- 修正: `getOwnEntry()`（`Object.prototype.hasOwnProperty.call` ガード）を導入し、`resolveRecipient` の2箇所の参照を置換。
- 確認: `__proto__` / `constructor` / `hasOwnProperty` / `name:__proto__` すべて **404**。

### ✅ F-2: 例外メッセージのクライアント露出 → **本ブランチで修正済み**
- 修正前: 全ルートの `catch` が `e.message` をそのまま返す（500）。ファイルパス・内部状態を含む文言が漏れる可能性。
- 修正: `api-notes.ts` に `serverErrorResponse(context, e)` を追加。詳細は `console.error` でサーバーログのみに記録し、応答は汎用 `{"error":"Internal server error"}`。6ルート（search×2 / recipients / quality-scores×2 / sankey/mof-overview）に適用。
- 残課題（任意）: `requestId` を返すと運用時の突合が容易。

### 🟠 F-3: 認証・レート制限なしの O(N) 全件スキャン検索（DoS 面）
- `searchRecipients` は毎リクエストで **25,811 エントリ × aliases** を走査し、各 alias に `normalizeRecipientName`（NFKC正規化＋複数 regex 置換）を実行。
- 実測: 1リクエスト ≈ 22〜37ms（ウォーム）。ユニーククエリ20件で 0.57s。
- CDN キャッシュ（`s-maxage=86400`）は**同一クエリにのみ有効**。攻撃者がクエリを毎回変える（cache-busting）と全件スキャンが素通りし CPU を消費。
- 対策案の候補:
  - レート制限（IP単位 / トークンバケット。Vercel Edge Middleware or 上流WAF）
  - クエリ最小長（例: 2文字以上）＋ 正規化結果をキャッシュ
  - 事前構築の転置インデックス / n-gram で O(N) 走査を回避
  - 検索結果にも `Cache-Control`（正規化クエリ単位で公開キャッシュ）

### 🟡 F-4: レスポンスサイズの上限
- `search` は limit 上限100、`/api/recipients/[key]` の appearances は上限200で**バウンド済み**（良好）。ただし `...entry` のスプレッドで `aliases` 等を全量返す。巨大エントリ（出現数の多い「国」等）で aliases が肥大しないか要確認。

### 🟡 F-5: robots / 公開範囲の方針
- `robots.txt` は `Disallow: /api/`（クロール抑制）だが**アクセス制御ではない**。`llms.txt` は API を積極公開。AI フレンドリー方針と DoS 対策のトレードオフを次タスクで明文化すべき。
