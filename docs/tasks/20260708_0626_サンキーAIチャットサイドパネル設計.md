# サンキーAIチャットサイドパネル設計 — 自然言語プロンプト → フィルター条件構築 → URL反映

日時: 2026-07-08
ステータス: 設計（実装前）
関連ドキュメント:

- [20260702_1454_AIフィルタークエリAPI設計検討.md](20260702_1454_AIフィルタークエリAPI設計検討.md)（`/api/sankey/query` の前史。「サイト内蔵チャット UI は当面扱わない」とした決定を本件で更新する）
- [docs/agent-playbook.md](../agent-playbook.md)（ローカルエージェントの探索ループ。本件はこのループの①〜④前半をサーバー内蔵化する）

## 目的

**サイト訪問者（記者・研究者・市民）が、Claude Code 等のローカルエージェント環境を持たなくても、「再エネ関連で予算100億円以上の事業だけ見たい」のような自然言語からブラウザだけで目的のサンキー図に到達できない状態を解消するため。**

`/api/sankey/query`（#231〜#242 で整備）により「AI がフィルタ条件を組み立て → summary で検証 → `links.webView` を人間に提示する」ループはローカルエージェントで実証済み（実験①②）。しかし実行主体がローカルの Claude Code 等に限られ、ブラウザだけのユーザーはフィルタパネルを手動操作するしかない。本件はこのループの実行主体をサーバー側（Next.js API + OpenRouter）に移し、`/sankey-svg` ページ内のチャット UI から使えるようにする。

### 本件のスコープ（ゴール）

**ユーザーのプロンプト → AI がフィルター条件（`SankeyQuery`）を構築 → クエリ実行・検証 → クライアントが URL へ変換してサンキー図に反映**、まで。

スコープ外（本件では実装しない）:

- 深掘り機能（品質スコア・事業詳細・再委託構造への言及、レポート生成）— ツール追加で拡張できる構造だけ確保する
- Vercel 本番公開（`/api/sankey/query` と同じくローカル実験ファースト。公開は別判断）
- 応答のストリーミング表示

| 資産 | 場所 | 本件での使い方 |

| 本番抑制ガード | `app/api/sankey/query/route.ts` の `isQueryApiEnabled` | 同型のガードを AI チャット API にも適用 |

### ギャップ（本件で新設するもの）

1. **LLM 呼び出し層がない**: 品質スコアの OpenRouter 利用はオフライン Python スクリプトであり、Next.js アプリ内から LLM を呼ぶ経路は存在しない
2. **エージェントループのサーバー内蔵版がない**: 「解釈 → クエリ → summary 検証 → 再調整」の反復は現状ローカルエージェント（人間が起動する Claude Code）が担っている
3. **チャット UI がない**: `/sankey-svg` の既存サイドパネル（ノード詳細）は**左側**固定。右側は未使用

## 設計方針

### 全体フロー

```text
ユーザー入力（自然言語）
   │  ページが POST /api/ai/sankey-chat（会話履歴 + 現在のビュー状態を同送）
   ▼
API Route（薄いハンドラ）
   │  OpenRouter へ chat.completions（tool calling）
   ▼
エージェントループ（app/lib/ai/、LLM呼び出しは注入）
   ├─ tool: run_sankey_query   … resolveSankeyQuery → buildFilterExcludedIds
   │                              → summarizeFilteredGraph（in-process）
   ├─ tool: search_projects / search_recipients … 語彙探索（0件時の再調整用）
   └─ 収束: AI が最終 SankeyQuery + 説明文を構造化出力
   ▼
応答 { message, result?: { query, summary } }
   │  クライアント: sankeyQueryToUrlParams(query) で URL 構築
   ▼
history.pushState + 既存の URL→状態復元ロジックで図を更新
（チャット履歴を保持したまま「リダイレクト」相当を実現）
```

核となる考え方は AIフィルタークエリAPI設計（20260702_1454）と同じ: **AI が試行錯誤する条件と人間が見る図は同一の `SankeyQuery` を共有し、別実装を作らない。** 本件で追加するのは「AI の実行主体をサーバーに置く」部分だけであり、フィルタロジック・URL 変換は一切複製しない。

### ツール実行は HTTP 自己呼び出しではなく in-process

エージェントループのツール（`run_sankey_query` 等）は `fetch('http://localhost/api/sankey/query')` のような自己 HTTP 呼び出しをせず、`app/lib/` の関数を直接呼ぶ。理由: (a) Vercel 環境で自ホスト解決が不安定、(b) `/api/sankey/query` は本番 404 ガードがありチャット API と有効化条件を独立に制御したい、(c) レイテンシ（1チャットでツールが複数回走る）。応答形式（summary の形・エラー文言）は `/api/sankey/query` と揃え、AI に与える仕様ドキュメントを共通化する。

### コンポーネント構成（レイヤー準拠）

| Layer | ファイル | 役割 |
|-------|---------|------|
| Types | `types/sankey-ai-chat.ts`（新規） | チャット API の要求/応答型（`ChatMessage`、`SankeyChatRequest`、`SankeyChatResponse`、`SankeyChatResult`） |
| Domain | `app/lib/ai/sankey-chat-agent.ts`（新規） | エージェントループ本体。システムプロンプト構築・ツール定義（JSON Schema）・ツール実行（`sankey-query.ts` / `search/` へ委譲）・反復上限・最終出力の検証。**LLM 呼び出しは関数として注入され（DI）、このファイル自体は HTTP コードを持たない** |
| API | `app/api/ai/sankey-chat/route.ts`（新規） | POST ハンドラ。有効化ガード → 入力検証 → OpenRouter 呼び出し関数（fetch）を組み立ててループに注入 → 応答整形。ロジックは持たない |
| UI Components | `client/components/SankeySvg/AiChatPanel.tsx`（新規） | 右側パネルの見た目のみ: メッセージリスト・入力欄・送信/ローディング状態・結果カード（「この条件で図を表示」ボタン）。**fetch はせず**、`onSend` / `onApplyResult` をコールバックで受ける |
| Pages | `app/sankey-svg/page.tsx` | チャット状態（メッセージ配列・送信中フラグ・パネル開閉）の保持、`/api/ai/sankey-chat` の呼び出し、結果適用（URL 変換 + 状態反映）。popstate ハンドラの本体を `applyParsedState(parsed)` として関数抽出し、popstate と AI 結果適用の両方から呼ぶ |

LLM 呼び出し（外部 HTTP）を API 層に置き、ループの制御構造とツール実行を Domain 層に置くことで、「`app/lib/` は HTTP 禁止」のルールを保ちながらループ本体を単体テスト可能にする（LLM をモック関数に差し替えられる）。

## API 仕様（案）

### POST /api/ai/sankey-chat

要求:

```json
{
  "messages": [
    { "role": "user", "content": "再エネ関連で予算100億円以上の事業だけ見たい" }
  ],
  "context": {
    "year": "2024",
    "currentQuery": { "（現在ページに適用中の SankeyQuery。未フィルタなら省略可）": "..." }
  }
}
```

- `messages`: 会話履歴の全量をクライアントが毎回送る（**サーバーはステートレス**。セッション永続化を持たない）
- `context.currentQuery`: ページの現在状態。「今の条件にさらに環境省だけ追加して」のような差分指示を成立させるため、システムプロンプトに埋め込む
- 入力上限: `messages` は直近 20 件・合計 8,000 字程度でクランプ（超過は 400）

応答（成功時）:

```json
{
  "message": "経産省・環境省の再エネ関連で予算100億円以上に絞りました。12事業・予算合計約1.2兆円が該当します。",
  "result": {
    "query": { "（resolveSankeyQuery 済みの SankeyQuery）": "..." },
    "summary": { "（summarizeFilteredGraph の結果: 件数・金額・上位）": "..." }
  },
  "usage": { "model": "google/gemini-3.5-flash", "toolCalls": 2 }
}
```

- `result` は省略可: 質問の意図が不明で聞き返す場合は `message` のみ返す（チャットとして自然な往復を許す）
- `result.query` はサーバー側で `resolveSankeyQuery` を通した正規化済みの値のみを返す（AI の生出力をそのまま返さない）。**URL への変換はクライアントが `sankeyQueryToUrlParams` で行う**（本件ゴールの通り）
- エラー応答: 400（入力不正）/ 404（無効化環境、下記ガード）/ 429 は設けない（ローカル実験フェーズ）/ 502（OpenRouter 障害。ユーザー向け文言は「AIが応答できませんでした」に丸め、詳細はサーバーログ）

### エージェントループ仕様

- OpenRouter の OpenAI 互換 chat.completions + tool calling を使用（非ストリーミング）
- ツール定義（v1）:

| ツール | 引数 | 返り値 | 実体 |
|--------|------|--------|------|
| `run_sankey_query` | `SankeyQuery`（filter 中心。view は通常既定値） | summary（件数・予算/支出合計・上位10件の名前と金額）。検証エラー時はエラー文言（AI が自己修正する） | `resolveSankeyQuery` → `buildFilterExcludedIds` → `summarizeFilteredGraph` |
| `search_projects` | `q`（文字列） | 上位ヒット事業名・pid・府省庁 | `app/lib/search/project-search.ts` |
| `search_recipients` | `q`（文字列） | 上位ヒット支出先名・法人番号 | `app/lib/search/recipient-search.ts` |

- 収束方法: 最終応答は `submit_result` ツール（`query` + ユーザー向け説明文を引数に取る）の呼び出しで確定させる。テキスト応答のみで終わった場合は `result` なしの聞き返しとして扱う。`submit_result` の `query` はサーバーで再度 `resolveSankeyQuery` + `run_sankey_query` 相当を実行して検証し、エラーならループに差し戻す
- 反復上限: ツール呼び出し **8 回**（agent-playbook の実測「探索1本 10〜15回」より、単一フィルタ構築は数回で収束する想定。上限到達時はその時点の最良候補を返すのではなく「条件を特定できなかった」旨の `message` のみ返す）
- タイムアウト: LLM 1呼び出しあたり 30 秒、リクエスト全体 60 秒
- システムプロンプトに埋め込む知識: `SankeyQuery` スキーマ、金額は1円単位（「100億円」→ `10000000000`）、府省庁名は完全一致リスト（グラフから取得した実在名を列挙）、「その他」vs「その他の支出先」等のデータの罠（agent-playbook §2 から抜粋）、`context.currentQuery`

### モデル・環境変数

品質スコア（`scripts/score-project-quality-ai.py`）の規約を踏襲する:

| 環境変数 | 既定値 | 意味 |
|----------|--------|------|
| `OPENROUTER_API_KEY` | なし（必須） | OpenRouter API キー。未設定時はチャット API が 404（機能自体を無効化） |
| `SANKEY_AI_CHAT_MODEL` | `google/gemini-3.5-flash` | 使用モデル。品質スコアのモデル検証（20260620_0950）で妥当性確認済みの既定を流用 |
| `SANKEY_AI_CHAT_ENABLED` | なし | Vercel 上で明示有効化する場合のみ `1`（下記ガード参照） |

キーはサーバー側でのみ参照し、クライアントへ一切露出しない（`NEXT_PUBLIC_` にしない）。

## UI 設計（右側サイドパネル）

- 配置: `position: fixed; right: 0`。既存のノード詳細サイドパネル（左側、zIndex 25）と同格の zIndex とし、両方同時に開ける。閉じた状態では右端に開閉タブ（左パネルの折りたたみタブと同じ操作感）
- 幅: 既存定数を流用（既定 400px / 最小 200px / 最大 800px）。v1 ではリサイズなし固定でも可（未決事項 3）
- 構成要素:
  1. メッセージリスト（user / assistant の吹き出し。assistant の `result` 付き応答には**結果カード**を付ける: マッチ件数・予算合計・「この条件で図を表示」ボタン）
  2. 入力欄 + 送信ボタン（送信中は入力をロックしスピナー表示）
  3. ヘッダ（タイトル・クリアボタン・閉じるボタン）
- 結果の適用（本件ゴールの「リダイレクト」）: 結果カードのボタン押下で
  1. `sankeyQueryToUrlParams(result.query)` で `URLSearchParams` を構築
  2. `history.pushState` で `/sankey-svg?...` へ URL 更新
  3. popstate ハンドラから抽出した `applyParsedState(parseSearchParams(...))` を呼び状態反映

  ページ遷移（フルリロード）にしない理由: チャット履歴とパネル開閉状態を保持したまま図だけ差し替えるため。pushState なのでブラウザバックで適用前の図に戻れる（チャット适用も既存のフィルタ操作と同じ履歴セマンティクスに乗る）
- 自動適用はしない: AI の `result` は必ずカード表示 → ユーザーがボタンで適用、の2段にする（AI の解釈違いで突然図が変わる体験を避ける + 適用前に件数を確認できる）
- モバイル（コンパクト幅）: パネルを全幅オーバーレイで表示（左パネルの既存レスポンシブ対応と同じ方針）。v1 はデスクトップ優先で、コンパクト幅では開閉タブのみ非表示にする選択も可（未決事項 3）
- 無効化環境（Vercel 本番等）でのフォールバック: チャット API が 404 の場合、パネル自体を出さない。判定はページ初回マウント時に軽量な疎通（後述の GET ハンドラ or 最初の送信失敗時に案内文言）で行う

## セキュリティ・コスト上の論点

実験フェーズ（ローカル）で入れる最小限:

- **本番抑制ガード**: `/api/sankey/query` の `isQueryApiEnabled` と同型（`VERCEL=1` なら素の 404、`SANKEY_AI_CHAT_ENABLED=1` で明示有効化、dev は `OPENROUTER_API_KEY` があれば有効）。LLM 呼び出しは**従量課金のため、クエリ API 以上に公開時の防御（レートリミット・認証）が必須**であり、ガードなしで main マージ → Vercel 公開されてはならない
- **入力クランプ**: メッセージ件数・文字数上限（前述）。ツール反復上限 8 回・全体タイムアウト 60 秒で 1 リクエストあたりのコスト上限を構造的に抑える
- **プロンプトインジェクション**: ツールは読み取り専用3種のみで書き込み系がなく、`result.query` はサーバー側で `resolveSankeyQuery` を通すため、注入が成立しても影響は「変なフィルタ条件の提案」に留まる。v1 はこの構造的な被害限定で足りるとする
- **正規表現**: `resolveSankeyQuery` 既存の防御（パターン長 128 字上限 + コンパイル検証）をそのまま通る

公開段階の検討事項（Vercel 搭載を判断した時点で着手。本件では実装しない）: レートリミット（IP 単位 + 日次総量）、コスト監視、乱用対策（CAPTCHA 等）、`Cache-Control`（チャットはキャッシュ不可のため `no-store` 明示）。

## 段階分割（実装順序）

| 段階 | 内容 | 単体での価値 |
|------|------|-------------|
| Step 1 | `types/sankey-ai-chat.ts` + `app/lib/ai/sankey-chat-agent.ts`（ツール定義・ループ・検証、LLM は DI）+ `app/api/ai/sankey-chat/route.ts`（ガード・OpenRouter 接続） | curl で「プロンプト → `result.query` + summary」が返る。UI なしで応答品質を評価できる |

- Step 1 の評価は agent-playbook の実験①で使った要求文（「再エネ大型事業」等）を流用し、ローカルエージェント実験と同等の条件が構築できるかを比較する
- Step 2 は既存挙動のスナップショット（ブラウザバック/フォワードでの状態復元）で回帰確認する

## レイヤー設計ルールとの整合チェック

| ルール | 本設計での扱い |
|--------|---------------|
| `scripts/` は CSV 処理のみ | 触れない（新規データ生成なし） |
| `app/lib/` は Pure・HTTP/React 禁止 | `sankey-chat-agent.ts` は LLM 呼び出しを関数注入（DI）で受け、fetch・`NextResponse`・React を一切含まない。ツール実行は既存 Pure 関数への委譲。グラフ読み込みは既存 `sankey-graph-loader.ts`（fs+キャッシュ、既存 loader 群と同格）を使う |
| `app/api/` はハンドラのみ | route.ts は「ガード → 入力検証 → OpenRouter fetch の組み立て → ループ呼び出し → 応答整形」のみ。ツール実行・プロンプト構築・収束判定は `app/lib/ai/` |
| `client/components/` は直接 API コール禁止 | `AiChatPanel.tsx` は表示専用。fetch は page.tsx が行いコールバックで接続 |
| Pages は状態管理・API 呼び出し・レイアウトのみ | チャット履歴・送信中フラグは page.tsx の state。フィルタ適用は既存の URL 同期機構に乗せる |
| 金額は1円単位 | `SankeyQuery` の既存規約のまま。「100億円」→ 1円単位の換算はシステムプロンプトで AI に指示し、`resolveSankeyQuery` の範囲検証で担保 |
| 「その他」vs「その他の支出先」 | システムプロンプトのデータの罠セクションに明記（AI の説明文が取り違えないように） |

## 決定事項（2026-07-08 ユーザーフィードバック）

1. **モデルの既定値**: `google/gemini-3.5-flash`（品質スコアと同じ）。`SANKEY_AI_CHAT_MODEL` で差し替え可能
2. **曖昧・解釈不能な要求**: 合理的な解釈が選べるなら既定解釈で即 `result` を返す（適用前に件数を確認できるため）。解釈できない場合は「解釈できなかった」旨と条件例をテキストで返信する（凝った聞き返しループは作らない）
