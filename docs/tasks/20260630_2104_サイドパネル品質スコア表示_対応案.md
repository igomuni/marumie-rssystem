## 1. 目的（Why）

サンキー図の利用者が、事業ノードを選んだとき**その事業の品質スコア（AI評価）をサイドパネル内でその場で確認できる**ようにするため。現状はスコアを見るには別ページ `/quality` を開く必要があり、サンキーの文脈から切れている。

---

## 2. 結合キー（確認済み）

- サンキーの事業ノード（`project-budget` / `project-spending`）は **`projectId`（整数）** を持つ。
- 品質スコアは **`pid`（文字列）** がキー。
- 同一ID空間・**1:1対応（2025: 5,794 = 5,794）**。
- ⚠️ **型差**: `projectId`(int) ↔ `pid`(string)。結合時は `String(projectId)` で揃える。
- 支出先レベルの結合は対象外（サンキーの recipient ノードは事業横断の共有ノードで `projectId` を持たない）。本機能は**事業レベル**に限定。

---

## 3. API設計（方式A: pid単体の軽量エンドポイント）

現状 `GET /api/quality-scores?year=` は**全5,794事業＋summary（約7MB）**を返すバルク専用で、pid単体取得が無い。サイドパネルのために 7MB を引くのは過大なため、**pid単体の軽量エンドポイントを新設**する。

### 3-1. エンドポイント

```text
GET /api/quality-scores/[pid]?year=2024|2025
```

- 既存の per-key パターン（`/api/project-details/[projectId]`、`/api/recipients/[key]`）に倣う。
- B層の規約を踏襲: `parseYear`（不正年度→400）、`serverErrorResponse`、`API_CACHE_CONTROL` ヘッダ。
- 見つからない pid は 404。

### 3-2. レスポンス（軽量プロジェクション）

全44フィールドではなく、**サイドパネル表示に必要な分だけ**返す:

```json
{
  "pid": "1",
  "name": "内閣人事局経費(研修事業)",
  "totalScore": 86.0,
  "axisIdentify": 100.0,
  "axisPurpose": 100.0,
  "axisBudget": 100.0,
  "axisStructure": 100.0,
  "axisEffective": 60.0,
  "effectiveLevel": 6,
  "effectiveReason": "…",
  "aiSource": "openrouter:google/gemini-3.5-flash"
}
```

- 旧 `axis1..axis5`・各種カウントは**返さない**（サイドパネルでは不要、ペイロード削減）。
- 必要に応じ `identifyLevelAvg` / `purposeLevelAvg` を追加可。

### 3-3. 実装メモ（コードは別PR）

- 既存 `app/lib/api/quality-scores-loader.ts`（年度別メモリキャッシュ付き全件ロード）を再利用し、**pid→item の索引を1本追加**して O(1) 取得にする。全件は初回ロード時のみ（既にバルクAPIと共有キャッシュ）。
- ⚠️ **loader の型が古い**: `quality-scores-loader.ts` の `QualityScoreItem` は #231 時点の旧定義（AI軸なし）。実JSONと `app/api/quality-scores/route.ts` 側の型には `axisIdentify` 等のAI軸がある。**pid単体APIを作る前に loader の型をAI軸込みへ更新**し、route.ts と型を一本化するのが望ましい（重複・乖離の解消）。

## 4. UI（サイドパネル表示案）

- 事業ノード選択時、サイドパネルの事業情報セクションに**「品質スコア」ブロック**を追加。
- 表示要素（最小）:
  - **総合スコア** `totalScore`（大きく・バッジ）
  - **4軸の内訳**: 特定可能性 / 使途 / 収支 / 有効性（軸D構造は参考扱い）
  - **有効性の根拠** `effectiveReason`（1行・「※実測成果でなく成果設計の明確さ」の注記）
  - `/quality` への詳細リンク（`projectLinks` 既存）
- 取得タイミング: **事業選択時に pid で fetch**（サイドパネル展開時のみ・遅延ロード）。サンキー初期ロードを重くしない。
- スコアが無い pid（`totalScore: null` 等）はその旨を表示。

---

## 5. 留意点

- **型変換**: `String(projectId) === pid`（§2）。
- **loader型の更新を先行**（§3-3）。AI軸を含む単一の型に統一。
- **`status=unknown` の扱い**: 本機能は事業レベルの集計スコア（axisIdentify 等）の表示で、支出先個別の `status` は出さない。将来サイドパネルで支出先一覧まで出す場合は「unknown=辞書未収録であって実在性ではない」点に注意（`docs/tasks/20260624_0746_*`）。
- **年度整合**: サンキーの表示年度とスコアの `year` を一致させる（サンキーが2024/2025どちらを見ているかに追従）。
- **キャッシュ**: B層 loader の年度別メモリキャッシュにより、pid単体APIも全件再読込なしで応答。

## 6. レイヤー整合（CLAUDE.md）
