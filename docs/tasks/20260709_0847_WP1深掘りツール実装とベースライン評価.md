### hy3:free に関する所見

- 品質スコア判定（軸E）では「代替にならない」と結論した hy3:free（`20260709_0617`）が、**ツール呼び出し型のフィルタ構築・深掘りタスクでは 11ターン中10成功**と実用的に動いた。数値スコアの較正が要らず、ツール応答からの転記と JSON Schema 準拠が主となるタスク特性の違いによるものと考えられる。無料運用の現況では十分な成績
- ただし単発実行での評価。回答品質の精査（正規表現の網羅性・説明の正確さ）と再現性は今後ハーネスで継続観測する
- 有料モデルとの比較（`SANKEY_AI_CHAT_MODEL` 上書きで実行可）はクレジット購入後の宿題

## 追記（2026-07-09）: Gemini API 直結の試行と無料枠の限界

LLM エンドポイントを環境変数化（`SANKEY_AI_CHAT_BASE_URL` / `SANKEY_AI_CHAT_API_KEY`）し、Google の OpenAI 互換エンドポイント経由で `gemini-3.5-flash`（無料枠）での再計測を試行した。結果:

- **無料枠のレート制限が構造的に足りない**: gemini-3.5-flash free tier は **5 RPM / 20 RPD**。チャット1ターン = LLM 2〜3往復のため、フルスイート（12ターン ≒ 25〜40リクエスト）は日次上限を1回で超える。取得できたのは6ターン分で、以降は 429（RESOURCE_EXHAUSTED）
- **部分結果の定性比較（gemini-3.5-flash vs hy3:free、同一シナリオ）**:

| シナリオ | hy3:free | gemini-3.5-flash |
|---------|----------|------------------|
| filter-renewable | 6件・toolCalls 8・26.8秒（類義語5種をORで展開） | 5件・toolCalls 2・5.7秒（「再エネ\|再生可能エネルギー」のみ） |
| filter-defense | 94件・toolCalls 2・14.6秒 | 94件・toolCalls 2・3.7秒 |
| drill-project-detail | ○ 12.2秒 | ○ 24.0秒（構成の整った詳しい回答） |
| drill-subcontract | ○ 10.1秒 | ○ 18.8秒（同一の正しい結論） |

  gemini は**収束が速い**（toolCalls が一貫して少ない）が、類義語展開は hy3 の方が積極的（6件 vs 5件の差はここ）。深掘りの結論はどちらも正確。**hy3:free で機能面の問題はない**という WP1 の結論は維持
- **改修（本追記に伴い実装）**: ①LLM エンドポイントの環境変数化、②429/5xx/choices欠落への1回リトライ（上流提案の待機秒を尊重・上限30秒）、③ハーネスに `--pace`（ターン間待機）と model 列
- 本格的な head-to-head は「クレジット購入」または「20 RPD 内での分割実行（1日6ターン×2日）」のいずれかで実施

## 発見事項（今後の改善候補）

1. **OpenRouter の一過性障害でリトライなし**: `choices[0].message` 欠落（HTTP 200）で即 502。hy3:free のプロバイダ（Novita）固有の癖の可能性もある。1回だけの自動リトライを入れる価値あり（v1 実装既存の挙動で WP1 のスコープ外。次の改修時に）
2. **`get_quality_scores` への pid 引き渡し経路**: 13件マッチ中10件しかスコア比較されなかった。run_sankey_query の summary が上位10件しか返さないため。「全件比較して」への対応は summary の pid リスト拡張か、フィルタ条件を直接受ける quality ツールへの改修が候補
3. レイテンシはフィルタ構築系で 15〜27秒。WP2-2（ストリーミング/進行イベント）の動機を定量的に裏付けた
