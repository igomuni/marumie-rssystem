# エージェント視点の「欲しい能力」と、モデル間レポート品質格差の考察

日時: 2026-07-03
ステータス: 考察メモ（実験①を実行したエージェント自身の一人称の所感 + ユーザー観察の突き合わせ）
関連: [20260702_2201_ロードマップ案](20260702_2201_ロードマップ案_AIエージェント活用と品質スコア統合.md)（特に A-1 プレイブック）

## 経緯

実験①（`20260702_1552`）を実行したエージェント（Claude Fable 5、本セッション）に「どんなことができるといいか」を尋ねた回答と、ユーザーがローカル（VSCode + Claude Opus 4.8）で同種の探索レポートを実走した際の観察を記録する。

## 1. 利用者エージェントとして欲しかった能力3つ

実験①で実際に探索した際に「手が届かない」と感じた順:

### 1-1. 事業概要の中身まで検索できること（最優先）

現状の検索は事業**名**のみが対象。概要文の中でだけ言及されるテーマ（例: 名前は「〇〇推進費」だが中身は蓄電池）は見つけられない。実験①で対象を特定できたのは事業名が説明的だった幸運による。`project-details` の purpose / overview / currentIssues を対象にした全文検索が1本あると探索の入口が大きく広がる。

- 実装イメージ: `/api/search/projects` に `scope=details` 等のパラメータを追加、または別エンドポイント。project-details JSON は既に loader 化されており、単純なテキスト部分一致から始められる

### 1-2. 数値の出所（プロベナンス）を辿れること

レポートに「127.6億円が『その他』」と書くとき、エージェントは自分の集計ミス（Multi-block 合算漏れ等）への不安を常に持つ。API応答の金額に「元データのどの報告行に由来するか」の参照が付けば、(a) レポートに正確な出典を書ける、(b) エージェントが自己検証できる。データ正確性最優先という本プロジェクトの思想と最も相性がよい拡張。

### 1-3. 年度間比較が1クエリでできること

「この条件で2024→2025はどう変わったか」は定番の問いだが、現状は2回呼んで自前で突き合わせる。エージェントは突き合わせ時にこそ間違えるため、同一 `SankeyQuery` の2年度比較モード（差分・増減率つき）があるとレポートの信頼性が上がる。

### 不要と考える能力

書き込み系は不要。読み取り専用という現在の境界は公開データを扱うサイトとして正しい。予算組み替えシミュレーションも「エージェント側で計算し仮想値と明示する」方式（`20260702_1537`）が筋がよい理由はここにある。

## 2. モデル間の品質格差 — 「データアクセス量」では説明できない

### 観察された事実

| | Fable 5（本セッション、実験①） | Opus 4.8（ローカル、ユーザー実走） |
|---|---|---|
| アクセス可能なデータ | **APIのみ**（設計上の制約として自ら課した） | API + **生データテキスト・事業CSV・リポジトリ全体**（ローカルなので制約なし） |
| 対象システムの知識 | このセッションで設計・実装した | **クエリAPI実装の作業者本人**（マージ後の確認実走） |
| レポート品質（ユーザー評価） | 高い | まずまず（Fable 5 に劣る） |

**注目すべきは、Opus 4.8 側がデータアクセスでも「作った本人」という点でも優位だったのに、品質で劣ったこと。** アクセスできる情報量の差では説明できない。

### 解釈

1. **文脈の質 > データの量**: 本セッションのエージェントは CLAUDE.md・過去の設計文書・実験の経緯（「その他」問題、スコア=形式品質、直接/再委託の分離）を長い対話の中で獲得していた。レポートの質を作ったのは生データへの近さではなく、**データの罠と「何を報せるべきか」の観点**だった
2. **生データへのアクセスはむしろ危険もある**: 29MBのJSONやCSVを直接読めるエージェントは、名寄せ前の生値を拾って集計を誤る・「その他」を重複計上する等、**キュレーション済みAPIなら踏まない罠を踏める**。APIの絞り込み・metadata.notes・HATEOASリンクは帯域の問題ではなく**探索品質の土台**である、という設計方針の裏づけになる
3. **モデルの地力の差はある**が、それは観点の適用力・文章化の差として現れており、観点そのものは文書で移転可能な部分が大きい

### 帰結: プレイブック（ロードマップ A-1）の優先度は高い

この観察は「プレイブックがモデル格差を埋める最も費用対効果の高い手段」という仮説を強く支持する。プレイブックに含めるべきもの:

- **探索手順**: search → sankey/query summary → 深掘り（details / quality / subcontracts / recipients）→ webView の標準ルート
- **データの罠の一覧**: 「その他」vs「その他の支出先」、直接/再委託の二重計上禁止、スコア=形式品質、予算年度=事業年度-1、Multi-block 合算
- **レポート様式**: 全体像テーブル → 発見（各発見に数値の根拠）→ 留意事項、模範例として実験①レポートを参照
- **禁じ手**: 生データファイルの直接読み込み（ローカルエージェント向け。「APIを使え、生JSONを開くな」を明文化する — Opus 4.8 の実走環境ではこれが可能だった）
- **自己検証の作法**: summary の合計と明細の突き合わせ、件数の相互確認

検証方法も提案できる: **同じ依頼文を「プレイブックあり/なし」で Opus 4.8（または他モデル）に与え、レポートを比較する**。プレイブックの効果測定自体が実験②の1シナリオになる。
