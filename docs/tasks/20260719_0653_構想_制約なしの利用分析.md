# 構想 — 制約なしで利用状況を分析するなら

作成日: 2026-07-19（JST）
性格: 構想メモ（実装計画ではない）。コスト・インフラ・トラフィック量の制約を外した場合の
理想形を記録し、将来の段階的実装の「北極星」とする。
前提として外さない制約: プライバシー原則（個人識別なし・同意なき本文収集なし・BYOK キー不可視）。
これは制約ではなく設計価値。

関連: `20260707_0801_利用分析基盤の検討.md`（現実版）、`20260705_1628_レポート駆動の自己改善ループ.md`

## 0. 一言でいうと

「ページビューの集計」ではなく「**市民が予算データに何を問うたか、答えられたか**」を
一級のデータとして蓄積・分析し、サイト自身が毎週賢くなる閉ループを回す。
そして行政支出の透明化サイトとして、**自分自身の利用も透明化して公開する**。

## 1. 収集層 — カウンタではなく構造化イベントストリーム

- 全イベント（API呼び出し・UI操作・チャット往復）を**個票のまま**カラムナDB
  （ClickHouse / Tinybird / BigQuery 級）へ流す。カウンタ畳み込みは読み出し時にやれば
  よく、事前集約で情報を捨てない
- **セッション再構成**: URL=状態 の設計を活かし、pushState 遷移を全部イベント化して
  「初期状態 → フィルタ操作 → ノード選択 → 深掘り → 離脱 or 成果」の**探索の軌跡**を
  そのまま復元できるようにする（匿名セッションID・24時間で失効）
- チャットは1往復ごとに: 要求分類（フィルタ要求/データ質問/解釈不能）・使用ツール列・
  outcome・モデル・レイテンシ・トークン量。本文は次節の同意層でのみ

## 2. 同意層 — 会話コーパス（ここが本当の金脈）

「会話を研究・改善のために提供する」**明示オプトイン**を用意し、同意された会話だけ
全文で蓄積する:

- **市民が予算データに投げる問いの分類学（taxonomy）**が得られる。どんな語彙で、
  どんな関心で、何を知りたがるか — これはツール設計・タグ体系・データ拡充の要求仕様
  そのものであり、たぶん行政側にも存在しない知識
- 同意済み会話は**評価コーパス**に転用: 実会話での回帰テスト（モデル更新・プロンプト
  変更のたびに過去の実要求で成立率を再測定）、モデル別ツール呼び出し成功率の実測表
  （BYOK のモデル推奨リストを実データで自動生成）
- 「よくある問いギャラリー」として优れた問答を（匿名化して）公開に還流

## 3. 分析層 — 未充足需要のクローズドループ（自己改善の完全自動化）

現実版で最重要とした「0件語」を、制約なし版では**ループ全体**に拡張する:

1. 失敗シグナルの収集: 0件検索・解釈不能応答（GIVE_UP）・ツールエラー・往復上限到達・
   適用されなかった提案チップ
2. **LLM埋め込みでクラスタリング** → 「要求されたが応えられなかったもの」を週次で
   自動グルーピング（例:「自治体別の内訳を求める要求が今週23件」）
3. 分析エージェントが週次レポートを自動起草 → バックログ issue の自動起票まで
4. 対応リリース後、**同じ要求クラスタの成立率を before/after で自動再測定**
   （改善が効いたかをデータが自分で答える）

これは 20260705 の「レポート駆動の自己改善ループ」の観測・分析工程を人手ゼロにする話。

## 4. 活用層 — エージェントが第一級の消費者

- 人間用ダッシュボードより先に、**分析エージェントがセマンティック層（メトリクス定義）
  越しに SQL を直接叩ける**環境を用意。定例レポート・異常検知（「今日、特定事業への
  アクセスが急増 → referrer を辿ると◯◯の報道」を自動で特定）まで任せる
- 遊び心の本命: **利用フローを自サイトのサンキー図で描く**。「問い → 機能 → 深掘り →
  成果/離脱」の資金流ならぬ関心流。予算の流れを可視化するサイトが、自分への関心の
  流れを同じ文法で可視化する

## 5. 公開層 — 利用統計そのものを公共データにする

このプロジェクトならではの選択: **どの省庁・事業・支出先が市民の関心を集めているか**は、
それ自体が公共性のあるシグナル。k-匿名を通した関心ランキング・未充足需要の要約を
公開ページ（/stats 等）として出す。行政支出の透明化を掲げるサイトが自分の利用を
透明化するのは、思想的な一貫性であり信頼の担保にもなる。

## 6. 現実版との橋渡し（段階的に理想へ寄せる順序）

| 段階 | 内容 | 現実版からの差分 |
|---|---|---|
| 1 | 現実版 T1/T2（KVカウンタ）— 発動待ち | — |
| 2 | カウンタ → 構造化イベント（Tinybird 無料枠等でも可能） | 個票化・セッションID |
| 3 | 会話コーパスのオプトイン + 評価コーパス転用 | 同意UI・保存基盤 |
| 4 | 失敗クラスタリングの週次自動レポート | 埋め込み・分析エージェント定例化 |
| 5 | /stats 公開・関心流サンキー | 公開判断・ポリシー |

どの段階も前段のデータが「次を作る価値があるか」を答えてくれる構造にしてある
（= 利用分析の使い方として自己整合的）。

## 7. 運用コスト試算（月1万セッション規模・段階2〜4稼働時の概算）

| 費目 | 概算/月 | 備考 |
|---|---|---|
| イベント基盤 | $0〜25 | 月100万イベント×~500B ≈ 0.5GB。Tinybird無料枠〜小プラン / BigQueryなら$5未満 |
| 埋め込み（失敗クラスタリング） | $1未満 | 対象は失敗シグナルのみ（数千件/月）。埋め込み単価は誤差レベル |
| 週次分析エージェント | $5〜60 | 1回 20〜50万トークン級。Sonnet級で$1〜3/回、Opus級で$5〜15/回 |
| 自動 issue→PR エージェント | $20〜80 | 1本 $2〜10 × 月4〜8本想定 |
| 評価コーパス回帰（変更時） | $1〜5/回 | 実会話50〜100本の再生。安価モデルで足りる |
| **合計** | **$30〜150** | インフラは誤差、支配項は LLM 実行費 |

**本当の運用コストは金銭ではなく人間のレビュー帯域**。週次レポートの読解・自動PRの
レビュー・「AIの提案を却下する判断」が律速で、ここを無人化しない（次節）のが健全な設計。

## 8. このプロセスは現時点でどのぐらい一般化しているか（2026年初時点の認識）

**部品は全部コモディティ化済み、ループ全体の閉環は最先端でも実験段階**、が正直な評価。

- 一般化済みの部品: プロダクト分析基盤（PostHog/Amplitude）、LLM可観測性
  （Langfuse/LangSmith 系）、評価ハーネス（promptfoo/Braintrust 系）、フィードバックの
  埋め込みクラスタリング（Enterpret/Unwrap 系の「フィードバック・インテリジェンス」）、
  エラー監視からの自動 issue 起票（Sentry 系）
- 急速に一般化中: **issue → 自動PR**（GitHub Copilot coding agent・Claude Code の
  GitHub Actions 連携・Devin 系）。スコープが明確な修正なら実用水準。本リポジトリでも
  部品は稼働済み（CodeRabbit レビュー・/resolve-reviews・usage-log が実需要の証拠に
  なった実績 = 再委託フィルタ）
- まだ一般化していない: **「利用データ → 未充足需要クラスタ → issue → PR → リリース →
  同一要求クラスタでの成立率 before/after 自動再測定」の全周閉ループ**。特に最後の
  「効果測定が次の判断に自動接続する」部分をプロダクト分析と評価基盤にまたがって
  実装している例は稀。ここが本構想の差別化点であり、逆に言えば前例の薄い部分
- 業界の合意事項として残るのは「人間のレビューゲートを外さない」こと。無人化の対象は
  観測・分析・起案までで、採否の判断とマージは人間に残す

## 9. 副次効果（UI/UX・シナリオ信憑性への波及）

利用分析は AI 運用のためだけでなく:

- **ユーザーシナリオの実証**: ペルソナ・利用モード（20260705 の5モード）が想定ではなく
  実測分布で裏付く/棄却される。「誰も使っていない想定シナリオ」の検出は UI 簡素化の根拠
- **UX の摩擦点の特定**: 探索の軌跡（1節）から「どこで離脱するか・どの深掘りが無視
  されるか」が見える。ビジネスロジック（フィルタ語彙・TopN 既定値等）の改善は
  未充足需要クラスタから直接要求仕様が出る

## 10. 能動的フィードバック — 履歴・発見メモ・バグレポート（2026-07-19 追記）

受動テレメトリの弱点（低トラフィックでは k-匿名を通るシグナルが出ない）を、
**ユーザーが自分のために残すもの**で補う観点。行動そのものが同意なので匿名化の
制約がなく、シグナル品質は受動収集より圧倒的に高い。

### この サイトでの実現形（アカウントレス・ローカルファースト）

| 機能 | ユーザー価値（主目的） | 運営が得るシグナル（副産物） | 実装の軽さ |
|---|---|---|---|
| 探索履歴 | 「さっき見たあの図に戻る」 | （共有された場合のみ）関心の軌跡 | **URL=状態 の設計で激安**。訪問した SankeyQuery を IndexedDB に積むだけ。BYOK で前例あり |
| 発見メモ | 図の状態+一言メモを保存・共有 | 共有メモ = 市民版レポートコーパス（docs/reports の一般ユーザー版） | URL + テキストのローカル保存。共有はURLに載せるか明示送信 |
| バグ/データ報告 | 「この事業のデータおかしい」をその場で報告 | **文脈付きの未充足需要・品質問題**（URL=再現状態が自動添付できる） | 薄い `/api/feedback` → GitHub Issue 自動起票。3節のループの明示入力版 |

- ローカルファースト（IndexedDB）なら Cookie もアカウントも不要でプライバシー文言と整合。
  「送信」はユーザーの明示操作のみ = 同意問題が構造的に消える
- バグ報告に URL（=完全な再現状態）が自動で付くのは本サイト特有の強み。
  スクリーンショットやセッションリプレイの代わりが URL 1本で済む

### 現在の業界構成（CRM的なものは何で賄われているか・2026年初時点）

- **正統な CRM（Salesforce/HubSpot/Intercom）はアイデンティティが背骨**。アカウントの
  ある SaaS 向けで、匿名公開サイトには不適合（本サイトには当てはまらない）
- 匿名公開サイト・開発者系プロダクトの実際の構成は「CRM」ではなく部品の組み合わせ:
  - フィードバック収集: Canny / Featurebase / GitHub Discussions（要望の投票・追跡）
  - バグ報告: Sentry User Feedback・Marker.io・BugHerd 系（スクショ・コンソールログ・
    リプレイ自動添付 → issue tracker 連携）
  - アプリ内サーベイ・簡易フィードバック: PostHog surveys 等の分析基盤付属機能
  - 保存ビュー・履歴: 製品ネイティブ実装（SaaS の saved views）か、近年は
    **local-first**（ブラウザ保存・アカウント不要）が公開ツール系のトレンド
- つまり「履歴・メモ・報告」を1つの既製品で賄う定番は存在せず、
  **local-first 保存 + 薄い報告API → issue tracker 連携**が匿名サイトの現代的な最小構成。
  本サイトは URL=状態 と IndexedDB の前例があるため、この構成との相性が例外的に良い

## 11. アイデンティティが加わると何が開くか — EC型「事業=商品」分析（2026-07-19 追記）

EC アナロジーの写像: 商品=事業（pid）、閲覧=図での注目・詳細表示、カート=ウォッチリスト、
購入に相当する深い関与=レポート生成・共有、レコメンド=「この事業を見た人はこの事業も」。

### 実は段階があり、アイデンティティ必須なのは最後だけ

| 分析 | 必要なもの | 備考 |
|---|---|---|
| 事業別人気ランキング（PV的） | **何も要らない**（イベント集計のみ） | 5節の /stats 公開そのもの |
| 「この事業を見た人はこの事業も」（協調フィルタ） | **匿名セッション**で十分 | 同一セッション内の共起だけで作れる。アカウント不要 |
| マイ事業リスト・閲覧履歴の個人化 | local-first で可 | 10節の延長。端末単位なら identity 不要 |
| クロスデバイス継続・更新通知（年度データ更新・ウォッチ事業の変化を知らせる）・記名の公開メモ | **ここで初めてアカウントが必要** | 通知は連絡先という identity の本体が要る |

### 透明化サイト特有の緊張関係（アイデンティティの代償）

- 「誰がどの事業を監視しているか」は EC の購買履歴より**センシティブになりうる**
  （記者・公益通報者・利害関係者の閲覧行動）。匿名で覗けることは公共データサイトの
  機能であり、アカウント必須化は萎縮効果と正面衝突する
- 現実解は**任意・後付けのアイデンティティ**: 既定は匿名 + local-first、通知や共有の
  クロスデバイスが欲しい人だけ軽い登録（メールのみ/パスキー）。EC 型分析のうち
  高価値な部分（ランキング・共起レコメンド）は匿名側で成立するため、アカウントは
  「分析のため」ではなく「通知・継続のため」と位置づけるのが筋が良い
