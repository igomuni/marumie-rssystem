# Globe Phase 4B — サッカーボール展開方式による領域割り当て

## 目的

Globe可視化の開発者が、BFS flood fillの経路ブロック問題（面数の過不足、浮島）に悩まされず、**機械的なループで全府省庁の面を正確に割り当てられる**ようにするため。

## 現方式の問題

### BFS flood fillの構造的欠陥

直近のCSV検証結果（`globe-assignment-check.mjs`）:

| 府省庁 | target | actual | diff |
|--------|--------|--------|------|
| 経済産業省 | 25,820 | 23,699 | **-2,121** |
| 特許庁 | 245 | 2,219 | **+1,974** |

BFSは「seedから隣接面を順に取る」方式のため：
- 先に成長した府省庁がBFS経路を塞ぐ
- 塞がれた未割り当て面が「最大府省庁が吸収」のフォールバックで意図しない府省庁に流入
- connectivity fixの断片再割り当てが連鎖的な面数ズレを引き起こす

**根本原因**: BFSは「どの面をどの府省庁に割り当てるか」を局所的な隣接関係で決めるため、大域的な面積配分を保証できない。

## サッカーボール展開方式

### 着想

サッカーボール（切頂二十面体）は正二十面体の12頂点を切り落として作る。結果として：
- 12個の五角形（元の頂点周り）
- 20個の六角形（元の面の中央）
- **合計32面** が球面をタイル状に覆う

これを「府省庁33エントリ（32大陸 + 1海）に近い数の区画」として利用する発想。

### icosphereとの関係

現在のicosphereは正二十面体をレベル7まで再帰4分割して327,680面を生成している。この分割構造には正二十面体の**12頂点**と**20面**が階層的に保存されている。

```
正二十面体: 12頂点, 20面, 30辺
  ↓ 切頂（頂点を切る）
切頂二十面体: 12五角形 + 20六角形 = 32面
  ↓ icosphereと対応づけ
icosphere level 7: 各元面(20面)が 4^7 = 16,384三角形に分割
  → 20元面 × 16,384 = 327,680面
```

### 核心アイデア: 「元面ID」による機械的割り当て

icosphereの各三角形面は、再帰分割の履歴を辿ることで**元の正二十面体の20面のどれに属するか**を特定できる。

```
面インデックス f の元面ID = floor(f / 4^7) = floor(f / 16384)
  → 0〜19 の値（20面のいずれか）
```

これは現在のicosphere生成コードの構造から自然に導出できる。`generateIcosphere`の各サブディビジョンステップで、面配列は元面ごとに4倍ずつ拡張される：

```
初期: [面0の3頂点, 面1の3頂点, ..., 面19の3頂点]  (20面)
Lv1:  [面0→4面, 面1→4面, ..., 面19→4面]  (80面)
Lv7:  [面0→16384面, 面1→16384面, ..., 面19→16384面]  (327,680面)
```

つまり**面インデックスの上位ビットが元面IDをそのまま表す**。ソートもBFSも不要。

### アルゴリズム設計

#### Phase 0: 元面への府省庁マッピング

20個の元面（各16,384三角形）に対して、府省庁を割り当てる。

大陸府省庁32エントリに対して元面20個なので、1:1対応ではない。
以下の方式で対応：

1. 各元面の重心（元面の16,384三角形の重心の平均）を計算
2. 元面を大陸中心からの距離でソート
3. 支出額降順の府省庁リストを上から順に、元面に割り当てる
   - 大きい府省庁は**複数の元面**を取る（例: 経産省 25,820面 → 2元面）
   - 小さい府省庁は**1つの元面を共有**する（同一元面内で後述のサブ分割）

#### Phase 1: 海と大陸の分離

現在のPhase Aと同じ考え方だが、**元面単位**で行う：

1. 20個の元面の重心を計算
2. 海のseed（大陸の対極）に最も近い元面群を「海」にマーク
3. 海の元面数 = `ceil(oceanFraction × 20)` ≈ `ceil(0.657 × 20)` = 14面
4. 残り6面が「大陸」

ただし、元面20個 × 16,384面 = 327,680面 に対し、海は215,264面（≈13.14元面分）。
元面単位では粗すぎるため、**境界の元面1つ**は海と大陸を内部で分割する必要がある。

→ これは「端数処理面」として、1元面(16,384三角形)内でBFSまたはカーソル分割する。

#### Phase 2: 大陸内の元面→府省庁割り当て

大陸の元面（6面 × 16,384 = 98,304面程度）に府省庁を割り当てる。

**方式: 元面を府省庁単位に分割**

1. 各大陸元面内の三角形を、元面重心からの角度でソート（螺旋順）
2. 府省庁を支出額降順に並べる
3. カーソル方式で順に割り当て（Phase 3の緯度帯方式と同じ原理）
   - ただし割り当て単位が「元面内」に閉じている

**元面内カーソルの利点**:
- 各元面は16,384面の閉じた領域 → BFS経路ブロックが発生しない
- カーソルは面番号順に走査 → O(N)で完了
- 面数の過不足が構造的に発生しない（カーソルは正確にtarget面ずつ進む）

#### Phase 3: 複数元面にまたがる府省庁の統合

経産省のような大規模府省庁は複数の元面にまたがる。
この場合、隣接する元面を選ぶことで領域が連結になる。

元面の隣接関係は正二十面体の位相から固定：
- 各元面は3辺を持ち、3つの元面と隣接
- 20面の隣接グラフは正二十面体の双対（正十二面体）

大規模府省庁に割り当てる元面は、隣接する元面を優先的に選ぶ。

#### Phase 4: 境界滑らか化

現在と同じローカルswap方式を最終段階で適用。元面境界のジャギーを軽減する。

### 20元面 vs 32面（切頂二十面体）の選択

ユーザーの「サッカーボールの五角形・六角形」は切頂二十面体の32面を指すが、icosphereの自然な構造は**正二十面体の20面**。

| 方式 | 区画数 | icosphereとの整合性 | 実装コスト |
|------|--------|-------------------|-----------|
| 正二十面体20面 | 20 | **完全一致**（面インデックスから即座に導出） | 低 |
| 切頂二十面体32面 | 32 | 不一致（頂点周りの三角形を手動で集約必要） | 高 |

**推奨: 正二十面体20面方式**

理由:
- `floor(f / 16384)` で元面IDが即座に得られる（追加のデータ構造不要）
- 府省庁33エントリに対し20面は少なめだが、元面内サブ分割で対応可能
- 切頂二十面体32面は幾何学的に美しいが、icosphereの面インデックスとの対応付けが非自明

### データの性質との適合性

| 課題 | BFS方式 | サッカーボール方式 |
|------|---------|-------------------|
| 面数の正確性 | BFS経路ブロックで±2000のズレ | カーソル走査で誤差±1面 |
| 浮島 | seedの位置と成長順に依存 | 元面単位で閉じるため構造的に不発生 |
| 小規模府省庁 | 3面の府省庁がBFSで消滅する | 元面内カーソルで確実に配置 |
| 計算量 | O(F × M)のseed探索 + O(F)のBFS × M | O(F)のカーソル走査（1パス） |
| 形状 | seedからの円形成長 | 元面の三角形セクター形状 |

### 残る検討事項

**20元面への府省庁マッピングの最適化**

32大陸府省庁を20元面に割り当てる際、以下のトレードオフがある：

- 大規模府省庁（経産省: 25,820面 ≈ 1.58元面分）→ 2元面割り当て、余りは隣接元面の別府省庁が使用
- 小規模府省庁（公正取引委員会: 3面）→ 他の小規模府省庁と1元面を共有

1元面 = 16,384面に対して、大陸全体の面数は約112,416面（≈6.86元面分）。
大陸元面が6〜7面しかないため、32府省庁を6〜7区画に収める必要がある。

→ 各元面内で**複数の府省庁をカーソル分割**する（緯度帯方式の元面内版）

**元面内の走査順序**

元面内の16,384面をどの順序で走査するかが形状に影響する：
- **面インデックス順**: サブディビジョンの再帰構造に従う。フラクタル的なZ字パターン
- **重心からの角度順**: 螺旋状の走査。より自然な形状になる可能性
- **重心Y座標順**: 緯度帯方式の局所版。水平ストライプ

検証スクリプトで各方式を試行して比較する必要がある。
