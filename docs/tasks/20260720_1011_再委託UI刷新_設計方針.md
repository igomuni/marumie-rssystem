# 再委託UI刷新 設計方針（Claude案）

GPT壁打ち（`docs/tasks/20260720_1902_再委託UI刷新GPT壁打ち.md`）と同じ題材について、
実装コード（`app/sankey-svg/page.tsx` 5,207行 / `app/subcontracts/[projectId]/page.tsx` 2,376行）と
スクリーンショット4枚（`docs/.ignore/20260720_subconstract_ui_renew/`）を踏まえた設計方針。

## GPT案への評価サマリ

| GPTの主張 | 判断 | 理由 |
|-----------|------|------|
| メイン画面を唯一のデザイン基準にする | ✅ 同意 | 現状の再委託画面は確かに「業務システム」に見える |
| サイドパネル＝詳細ページでなくインスペクター | ✅ 同意 | メインの左パネル（SidePanelChrome系）の作法に揃える |
| 色は意味付け（事業/直接/再委託/別財源）のみに限定 | ✅ 同意 | カード背景色の乱立が「別アプリ感」の主因 |
| タブ順を「流れ」優先に | ✅ 同意 | 一般利用者はブロック構造より金の流れを見たい |
| **同じ画面のまま中央だけ再委託フローへ変形** | ❌ **反対** | 下記「一番の相違点」参照 |
| 表示切替を Sankey/Block/Tree のラジオに | ⚠️ 保留 | Tree追加は需要未確認。2択トグルのまま様式だけ統一で十分 |

## 一番の相違点：in-place変形はやらない

GPT案の理想は「Sankeyをクリック → 右ペイン更新 → 中央だけ再委託フローへ変形」だが、私は**別ルートのまま「遷移の連続感」で解決する**方針を採る。

反対理由：

1. **ズームレベルが違う**。メインSankeyは全体俯瞰（147兆円・5,003事業）、再委託フローは1事業のミクロ構造。Google Mapsの比喩を正確に使うなら、店をクリックして地図がそのままなのは「同じズームの同じ地図」だからで、再委託はむしろ「ストリートビューに入る」操作に近い。ズームコンテキストが変わる操作は画面が変わってよい — 大事なのは**戻ったときに元の状態が完全復元されること**。
2. **実装リスクが非対称**。メインpageは既に5,207行のモノリスで、探索履歴・発見メモ・AIチャット・URL状態同期を抱えている。ここに第2の可視化モードを埋め込むと状態管理が乗算的に複雑化する。別ルート維持なら両モノリスは独立のまま進化できる。
3. **URLの価値**。`/subcontracts/{pid}?year=` は深いリンク・共有・ブラウザ履歴をタダで手に入れている。in-place変形はこれを自前で再実装することになる。

代わりに「連続感」を3つの手段で作る：

- **同タブ遷移に変更**（現状は `target="_blank"`）。ブラウザバックでメインに戻り、URLパラメータで選択状態・フィルタが復元される（メイン側は既にURL状態同期を持つ）。
- **メインのサイドパネルに「再委託」セクションを追加**（後述）。大半のユーザーはここで用が足り、遷移自体が不要になる。
- **共有クロームの一致**。ヘッダ・サイドパネル・タグ・フォントが同一なら、遷移しても「同じアプリの別ズーム」と知覚される。

## 設計方針

### 1. 統一の実現手段は「プロンプトで寄せる」ではなく「共有コンポーネント」

見た目を手作業（またはAIへの指示）で似せても再び乖離する。統一が保たれる構造を作る：

- `client/components/` に共有プリミティブを抽出する：
  - `TagChip`（事業/直接/再委託/別財源/一般 などの意味タグ。現メインの緑「事業」チップが原器）
  - `StatRow`（予算/支出の2カラム数値表示。億円表記＋1円単位の併記様式）
  - `SectionHeader`（▶折りたたみ見出し。メイン左パネルの「事業概要」「予算・執行」様式）
  - デザイントークン（フォントサイズ・余白・角丸・色）を1モジュールに集約し、両ページが import する
- 再委託ページ側はこれらを**消費するだけ**にする。ここが今回の刷新の実体。

### 2. 色の意味体系（全画面共通）

```text
緑     = 事業・予算側 #4db870（メインSankeyの実トークン TYPE_COLORS と同一）
赤     = 直接支出 #d94545（メインの recipient 赤と同一）
アンバー = 再委託 #d9952b（現行オレンジ #e07040 から変更 — 下記注記）
紫     = 別財源 #7b5ea7（現行の青から変更。青は「リンク」と衝突するため）
グレー  = 構造・集計・非強調
```

> **色覚検証の結果（dataviz validator）**: 現行の 直接 #d94545 × 再委託 #e07040 ペアは
> 通常視覚 ΔE 9.0（FAIL・15未満）、色覚多様性 ΔE 6.8 で分離不足。再委託を #d9952b に
> 振ると全チェック PASS（通常 ΔE 18.3）。いずれの場合もタグ文字を常時併記し、
> 色単独での判別に依存させない。

- カード・ノードの**背景は白**、意味色は左ボーダー／タグ／リンク線のみに使う（現状のオレンジ塗りカードが業務システム感の主因）。
- 「移替」「参考」はグレー系に落とす（意味色を増やさない）。

### 3. メイン画面サイドパネルに「再委託」セクション

事業ノード選択時、既存の「事業概要」「品質スコア」「予算・執行」の下に追加：

```text
▶ 再委託                      フロー ↗   ← 品質スコアの「一覧 ↗」と同パターンの省スペースリンク
  ブロック 16  支出先 103  階層 2
  直接 4   再委託 12   別財源 3   ← TagChipで表示
```

用語は「最大2層」でなく**「階層 2」**に統一する（既存表現の見直し。ユーザーフィードバック 2026-07-20）。

- データは選択時に遅延フェッチ（メインのgraph JSONを太らせない）。
- 「フローを見る」で `/subcontracts/{pid}` へ**同タブ**遷移。
- これにより GPT案の「検索→事業→サイドバー→再委託」の動線は、遷移なしで9割達成される。

### 4. 再委託詳細ページの再構成

- **レイアウトをメインと同じ「左＝インスペクター、中央＝図」に反転**（現行は右パネル）。左右が入れ替わっているだけで別アプリに見える。
- **デフォルト表示をフロー図（Sankey風）に変更**（現行はブロック図が先）。トグルは `フロー | ブロック` の2択維持。
- **タブ順**: `流れ → 支出先 → ブロック → 間接経費`（GPT案の「概要」タブは作らない。概要はタブでなくパネル上部の常設ヘッダが担う — メインと同じ構造）。
- 左パネル構成（上から）:

```text
ダム建設事業                    ← 事業名・RSリンク
予算 3,029億  支出 2,129億      ← StatRow
[事業] PID:4119 国土交通省      ← TagChip行
ブロック16・支出先103・階層2     ← サマリチップ
[予算・執行(18)|ブロック(16)|支出先(103)]  ← メイン画面の「省庁/事業/支出先」と同型のタブ
  タブ内に事業に含まれる全対象をリストアップ
  （ブロックタブはタグフィルタ付き: すべて/直接/再委託/別財源）
```

パネル下部はセクション積みではなく、メイン画面の「省庁/事業/支出先」3タブ構成と同型の
**「予算・執行/ブロック/支出先」タブ＋全件リスト**とする
（当初案の「流れ」中心セクション構成はメインUIとズレるためユーザーフィードバックで変更。2026-07-20）。

各タブのリスト項目（ユーザー指定・2026-07-20）:

| タブ | リスト項目（2行表示: 主情報＋金額 / メタ情報） |
|------|----------------------------------------------|
| 予算・執行 | 予算種別/歳出予算項目、備考、予算額、翌年度要求額 |
| ブロック | ブロック番号、支出先ブロック名、合計支出額、支出先の数、事業を行う上での役割 |
| 支出先 | ブロック番号、支出先名、契約概要（契約名）、費目、使途、金額 |

### データ実在確認（2026-07-20・全項目マップ可能。パイプライン新設不要）

`/data-query` で PID:4119 を実データ検証した結果、**指定した全項目が既存データに実在**する。

| UI項目 | データソース | フィールド |
|--------|-------------|-----------|
| **予算・執行タブ** | sankey graph の project-budget ノード `budgetBreakdown[]`（4119=18件） | |
| 予算種別 | 〃 | `budgetType` |
| 歳出予算項目 | 〃 | `item` / `subItem`（＋`account`/`accountCategory`） |
| 備考 | 〃 | `note` |
| 予算額 | 〃 | `amount` |
| 翌年度要求額 | 〃 | `nextYearRequestAmount` |
| **ブロックタブ** | `subcontracts-YYYY.json` の `blocks[]`（4119=16件） | |
| ブロック番号 | 〃 | `blockId`（**実データは A/J/P/G… の英字コード**。モックの `B-1` 等は要差替え） |
| 支出先ブロック名 | 〃 | `blockName` |
| 合計支出額 | 〃 | `totalAmount` |
| 支出先の数 | 〃 | `recipientCount` |
| 役割 | 〃 | `role`（例: 「工事の実施及び工事に係る調査・設計・用地取得」） |
| 直接/再委託/別財源 | 〃 | `originKind` = `direct` / `subcontract` / `separate-origin-broad` / `separate-origin-strong` |
| **支出先タブ** | `subcontracts-YYYY.json` の `blocks[].recipients[]` | |
| ブロック番号 | 親 `blocks[].blockId` | |
| 支出先名 | `recipients[]` | `name`（＋`corporateNumber`） |
| 契約概要（契約名） | 〃 | `contractSummaries[]` |
| 費目 | 〃 | `expenses[].category` |
| 使途 | 〃 | `expenses[].purpose` |
| 金額 | 〃 | `amount`（内訳は `expenses[].amount`） |

注意点:

- **予算・執行タブは sankey graph 側、ブロック/支出先タブは subcontracts 側**と、データソースが分かれる。
  再委託ページで予算・執行を出すには両ソースを読むか、`budgetBreakdown` を subcontracts 生成時に取り込む。
- 予算・執行の `fiscalYear` は表示年度より1年遅れ（2025グラフ内が2024実績）。既存の仕様どおり。
- 別財源は `originKind` の `separate-origin-broad/strong` で判別（モックの「別財源」タグに対応）。

- 「ブロック番号」を支出先タブにも併記し、どのブロックの支出先かを紐付ける。
- 実装ではRS Systemの該当項目にマップ（役割・契約概要・費目・使途は既存データ項目を確認して割当）。

- **フロー図の列構成**: `予算・執行 → 事業 → ブロック（第1層） → 再委託ブロック（第2層）…`。
  別財源はブロックへ紫リボンで流入する形で表現。
- **ブロック内支出先の深堀りは要検討**。暫定案 a =「ブロック選択→インスペクターに上位を表示」
  （図が安定・コスト最小）。対抗案 b = 選択ブロック右に支出先列を一時展開、c = ノードのその場展開。
  モックは a で作成、b/c は実データの件数分布を見て判断。

- **図中ノードクリック → 左パネルが当該ブロック/支出先のインスペクターに切り替わる**（戻るは事業名クリックかパンくず）。これがGPT案「サイドバー＝インスペクター」の実装形。
- フロー図はメインSankeyと同じ操作系（ズーム・パン・ホバーハイライト・クリック選択）に揃える。右下のズームUIは既に同型なので維持。

### 5. やらないこと（スコープ外の明示）

- メインSankeyキャンバス内での再委託フロー描画（in-place変形）
- Tree等の第3表示モード追加（需要が出てから）
- ブロック図の廃止（階層構造の確認用途で残す。ただし白背景カード＋意味色ボーダーに塗り替え）

## 段階実装計画

| Phase | 内容 | 効果 |
|-------|------|------|
| 1 | デザイントークン＋共有プリミティブ抽出、再委託ページの配色・フォント・カード様式をメインに統一 | 「別アプリ感」の解消（最小コストで最大効果） |
| 2 | メインサイドパネルに再委託サマリセクション＋同タブ遷移化 | 動線の連続化。遷移せず用が足りるケースを吸収 |

Phase 1〜2 だけでも体感は大きく変わる。Phase 3〜4 は 2,376行の詳細ページの分割リファクタを伴うため、独立PRに切る。

## GPT案の「Codex向けプロンプト」について

方針文としては良いが、「プロンプトで寄せる」アプローチ自体に持続性がない。
デザイン原則の文章より先に**共有コンポーネントとトークンを物理的に用意し、それを使わせる**方が、AI・人間どちらが実装しても乖離しない。原則文はその共有モジュールのdocコメントに書くのが正しい置き場所。
