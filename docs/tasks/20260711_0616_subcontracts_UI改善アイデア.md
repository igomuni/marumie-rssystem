# /subcontracts UI 改善アイデア — 現状の問題構造と段階的な改善案

日時: 2026-07-11 06:16 (JST)
ステータス: 検討（アイデア出し。実装スコープはユーザーと合意後に確定）
前提調査: Explore サブエージェントによる実装調査（page.tsx 920行 + [projectId]/page.tsx 1,638行 + sankey-svg 側UIパターンの突き合わせ。行番号は 2026-07-11 時点）

## 目的

**/subcontracts の利用者が、サンキー図と同じ操作感覚で再委託構造を読める状態にするため。** 現状はラベルはみ出し・場当たり的なホバー・状態依存で挙動が変わるサイドパネルにより、「同じサイトの機能」としての信頼感を損ねている。

## 1. 不満とコード上の原因の対応表

| 不満 | コード上の原因（調査で特定） |
|------|---------------------------|
| ラベルの文字はみ出し | 詳細ページのSVGラベルが**文字数ハードコードの手動切り詰め**（`truncateChars(name, 18)` 等）。カード幅 `lb.w`（可変）・全角半角・フォントサイズと非連動。名前と金額が別 text 要素で重なり検知なし |
| ホバーがとりあえず感 | ①一覧はネイティブ `title` のみ ②詳細はノード基準の固定吹き出し（foreignObject）で、サンキーの流儀（マウス追従・220ms遅延・操作後抑制）と別物 ③ルートノードは SVG `<title>` とカスタムツールチップが**二重表示** ④下方向のはみ出しフォールバックなし |
| サイドパネルの挙動が状態で違う | ノード再クリック=解除／リスト行クリック=選択+**recipientsタブへ強制遷移**／ルートクリック=解除+**flowタブへ強制遷移**、と操作ごとに副作用が異なる。解除手段も「同じノードをもう一度」「ルートを押す」「タブ内の✕」の3系統が混在 |
| サンキーとの非一貫性 | パネル: 幅390固定・リサイズ不可・折りたたみ不可（サンキーは200-800ドラッグ+折りたたみ）／フォント: px直書きで文字サイズ調整に非対応（サンキーは `scaleFont` で比例拡縮）／`formatYen` が**負値非対応の別実装**／アクセント色が一覧 `#2563eb` vs 詳細 `#6366f1` とページ内でも不統一／**URL同期が year のみ**（サンキーは URL=状態が設計原則。選択・ズーム・フィルタ・ページが共有/リロードで消える） |

## 2. 改善アイデア（3層構造）

### 層A: クイックウィン（各1PR未満・独立に着手可・見た目の信頼感が即改善）

1. **formatYen の一本化**: `subcontract-layout.ts` の負値非対応版を廃止し `sankey-svg-constants` 版へ。一覧の差分列の自前符号処理も簡素化
2. **ツールチップのサンキー流儀化**: マウス追従 + 220ms遅延 + 操作後抑制をサンキーから移植。SVG `<title>` との二重表示を解消。下方向フォールバック追加
3. **ラベル切り詰めの幅連動化**: 文字数ハードコードをやめ、(a) カード幅から文字数を算出（全角0.5補正）、または (b) foreignObject + CSS `text-overflow: ellipsis` へ置換（推奨は b。CSSに任せる方が今後の変更に強い）。名前と金額の重なりは金額側を優先幅として名前の許容幅を残りから計算
4. **アクセント色の統一**: 一覧・詳細・サンキーで青系アクセントを1色に（サンキーの選択色に合わせる）

### 層B: 一貫性の構造改修（サンキーとの共通化。1PRずつ）

5. **サイドパネルの chrome 共通化**: サンキー側のパネル実装（幅定数・ドラッグリサイズ・折りたたみタブ・fixed配置・影・ビューポートクランプ）を共通コンポーネント（例: `client/components/SidePanelChrome.tsx`）に抽出し、サンキー・subcontracts・AIチャットパネルの3者で共有。※懸念4（page.tsx肥大）の緩和にも寄与
6. **フォントスケール適用**: `scaleFont`/`baseFontPx` を subcontracts にも適用し、px直書きを置換。サンキーのフォントサイズ調整UIを詳細ページにも露出（既存 FontSizeControls の流用）
7. **URL=状態化**: 選択ブロック・アクティブタブ・ズーム/パン・一覧の検索/フィルタ/ソート/ページを URL パラメータへ（サンキーの pushState/popstate 復元パターンを踏襲）。共有リンク・リロード耐性・「AIが図のURLを提示する」将来の統一操作面としても価値

### 層C: 操作モデルの見直し（要ユーザー決定・デザイン判断）

8. **サイドパネルの状態遷移を1つの規則に統一**。案:
   - **案C1（選択とタブを分離・推奨）**: クリック=選択のみ（タブは動かさない）。選択解除は Esc / パネルの✕ / 空白クリックの3つに統一。タブはユーザーが最後に選んだものを維持
   - 案C2（常に自動遷移）: どの経路で選択してもrecipientsタブへ。解除で flow へ戻る（現状の「時々自動遷移」を「常に」へ倒す）
   - どちらでも「同じ操作は常に同じ結果」になる。C1 は自分の文脈が壊れない点で、C2 は次に見るべきものへ誘導する点で優れる
9. **一覧23列テーブルの見直し（任意・大きめ）**: 列リサイズ+横スクロールの現状を、既定は要約列（8列程度）+「列の表示切替」または行展開に。現状維持も選択肢（列幅 localStorage 済みでヘビーユーザーには使える）
