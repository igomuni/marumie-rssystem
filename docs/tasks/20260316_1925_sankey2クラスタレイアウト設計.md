# sankey2 クラスタレイアウト設計

## 背景

### 現行グリッドレイアウトの問題

現行のPhase 2グリッドレイアウト（6列×7行、府省庁セル単位）は、**予算額の偏りに対して非効率**。

| 府省庁 | 予算額 | 比率 | 累計 |
|--------|--------|------|------|
| 厚生労働省 | 92.83兆円 | 61.4% | 61.4% |
| 経済産業省 | 12.32兆円 | 8.2% | 69.6% |
| 国土交通省 | 11.13兆円 | 7.4% | 76.9% |
| 上位3省庁 | 116.28兆円 | 76.9% | — |
| 残り34府省庁 | 34.84兆円 | 23.1% | — |

**問題点**:
- 厚生労働省セルが全体の61%を占有し、他セルがスカスカ
- 府省庁単位のグリッドでは、各セルに ministry→project-budget→project-spending→recipient の4タイプが入り、小規模府省庁でも400px幅を消費
- 仮想空間 2,800×8,261px のうち、有効面積は一部のセルに集中

### 球体配置検討で得た知見

Globe Phase 4の検討で、面積配分の偏りは以下の通り:
- 厚生労働省: 61.4%（球体の半分以上を1つの国が占めるのと同じ問題）
- 上位8府省庁で92.8%をカバー
- 残り29府省庁は合計7.2%

→ **府省庁単位のセル分割ではなく、ノードtype単位の面（treemap）配置が自然**

## 提案: typeクラスタ × Squarified Treemapレイアウト

### コンセプト

各typeクラスタを**同じ面積の矩形**として配置し、クラスタ内は **Squarified Treemap** で縦横に敷き詰める。

#### フロー保存則

```
グループA（予算側）              グループB（支出側）
面積 = 151.12兆円               面積 = Σ支出額

┌────────┐ ┌────────┐ ┌────────┐   ┌────────┐ ┌────────┐
│        │ │        │ │        │   │        │ │        │
│  全体  │→│ 府省庁 │→│事業予算│ → │事業支出│→│ 支出先 │
│        │ │        │ │        │   │        │ │        │
│  面積A │ │ 面積A  │ │ 面積A  │   │ 面積B  │ │ 面積B  │
│        │ │        │ │        │   │        │ │        │
└────────┘ └────────┘ └────────┘   └────────┘ └────────┘

全体 = Σ府省庁 = Σ事業予算   （予算額の保存）
Σ事業支出 = Σ支出先          （支出額の保存）
```

### アルゴリズム: 2段階Squarified Treemap

#### ステップ1: クラスタ矩形の決定

5つのクラスタは同じ高さ `H` で左→右に並べる。各クラスタの幅は固定（フロー間隔を確保）。

```
クラスタ配置:
  total:            x=0,    width=W
  ministry:         x=W+G,  width=W
  project-budget:   x=2(W+G), width=W
  project-spending: x=3(W+G), width=W
  recipient:        x=4(W+G), width=W

  W = クラスタ幅（全クラスタ共通）
  G = クラスタ間のギャップ（エッジ描画スペース）
  H = クラスタ高さ（全クラスタ共通）
```

**重要**: 面積保存を実現するため、グループAとグループBで `H` は共通、`W` が異なる可能性がある。ただし視覚的整合性のため `W` も共通にし、予算額と支出額の差は無視する（差は小さい）。

#### ステップ2: クラスタ内のtreemap配置

各クラスタで Squarified Treemap を実行:

1. **totalクラスタ**: 1ノードのみ → クラスタ全体を1矩形
2. **ministryクラスタ**: 37ノードを金額降順で squarified treemap
3. **project-budgetクラスタ**: 2段階
   - 第1層: 府省庁37グループを squarified treemap → 府省庁サブ矩形を得る
   - 第2層: 各府省庁サブ矩形内で、事業ノードを金額降順 squarified treemap
4. **project-spendingクラスタ**: 同上
5. **recipientクラスタ**: 2段階（府省庁→支出先）

#### Squarified Treemap アルゴリズム

Bruls et al. (2000) の squarified treemap:

```
function squarify(items, rect):
  items を面積降順ソート
  残り = items
  while 残り.length > 0:
    row = []
    row に 残り[0] を追加
    while 残り.length > 1 && worstAspectRatio(row + [残り[1]], rect) <= worstAspectRatio(row, rect):
      row に 残り[1] を追加
    row を rect の短辺側に配置
    rect = rect の残り領域
```

- **目的**: 各矩形のアスペクト比を1:1（正方形）に近づける
- **特性**: 金額が大きいノードほど正方形に近い形状を得やすい

### エッジ描画

#### Treemap間のエッジ接続

隣接クラスタ間のエッジは、各ノード矩形の**右辺中央 → 左辺中央**を結ぶ Bezier 曲線。

```
クラスタN              クラスタN+1
┌──────┐              ┌──────┐
│      ├──── Bezier ──▶│      │
│ nodeA│              │ nodeB│
│      ├──── Bezier ──▶│      │
└──────┘              └──────┘
```

- エッジの出発点: `source.x + source.width`, `source.y + source.height/2`
- エッジの到着点: `target.x`, `target.y + target.height/2`
- 制御点オフセット: クラスタ間ギャップ `G` の 40%

#### 2段階treemapでのエッジ位置

事業ノードのX座標はクラスタ内で異なる（treemapの配置結果次第）。エッジの出発/到着X座標はノードの実際の矩形位置から決定する。

→ 同一府省庁のノード同士はtreemapの位置が近いため、エッジは短く水平に近い傾向。

### サイズ試算

#### パラメータ案

```
W (クラスタ幅) = 600px
G (クラスタ間ギャップ) = 200px
H (クラスタ高さ) = 600px
```

- 仮想空間幅: `5 * 600 + 4 * 200 = 3,800px`
- 仮想空間高さ: `600px`
- 仮想空間面積: `2,280,000 px²`

→ 現行 (2,800 × 8,261 = 23.1M px²) の **1/10**。

#### ノード最小サイズ

面積 = `W * H * (nodeAmount / clusterTotal)`

例: recipientクラスタ（25,892ノード）で最小支出先（1円）:
- 面積 = `600 * 600 * (1 / 151.12兆)` ≈ 0 px²
- → 事実上不可視。MIN_AREA閾値（例: 1px²）以下は座標のみ保持

例: 厚生労働省（92.83兆 = 61.4%）:
- 面積 = `600 * 600 * 0.614` = 221,040 px² → ~470 × 470px相当

### 実装計画

#### 変更対象

| ファイル | 変更内容 |
|---------|---------|
| `scripts/compute-sankey2-layout.ts` | グリッド → treemapクラスタアルゴリズムに書き換え |

#### アルゴリズム概要

```
1. グラフデータ読み込み
2. 支出先の府省庁帰属計算（現行と同じ）
3. クラスタ定義:
   clusters = [total, ministry, project-budget, project-spending, recipient]
4. 各クラスタについて:
   a. 対象ノードを抽出
   b. 府省庁グループでサブツリー化（ministry/totalは直接treemap）
   c. squarified treemap で座標決定
     - 第1層: 府省庁グループの面積比例で分割
     - 第2層: 各府省庁矩形内でノードを treemap 配置
5. エッジパス計算:
   - source矩形の右辺中央 → target矩形の左辺中央
   - Bezier制御点 = ギャップの40%オフセット
6. JSON出力
```

### Zoomレベルに応じたレンダリング最適化

#### 現行Phase 3の手法と課題

現行のグリッドレイアウトでは **type別Zoom閾値 + ビューポートカリング** でDOM数を削減している:

```
zoom < 0.3  → total + ministry のみ (38ノード)
zoom < 0.8  → + project-budget/spending (最大11,366ノード)
zoom ≥ 0.8  → + recipient (最大37,258ノード)
```

**問題**: type単位の粗いLODで、閾値を超えると一気に数千〜数万ノードが描画対象になる。

#### Treemapレイアウトの利点: 面積ベースLOD

Treemapでは各ノードが**金額比例の面積**を持つため、「スクリーン上で1px²未満のノードは描画不要」という**面積ベースのLOD**が自然に適用できる。

```
描画条件: node.width * node.height * zoom² ≥ MIN_SCREEN_AREA (1px²)
```

これにより type 閾値は不要。zoom に応じてノードが**連続的に**出現する。

#### 実測: Zoom別の描画ノード数（W=H=600px、MIN_SCREEN_AREA=1px²）

| zoom | ministry | project-budget | project-spending | recipient | 合計 | 関連エッジ推定 |
|------|----------|---------------|-----------------|-----------|------|-------------|
| 0.1 | 25 | 244 | 247 | 202 | **718** | ~1,500 |
| 0.3 | 30 | 880 | 900 | 701 | **2,511** | ~5,000 |
| 0.5 | 32 | 1,329 | 1,369 | 1,322 | **4,052** | ~8,000 |
| 1.0 | 33 | 2,082 | 2,091 | 3,019 | **7,225** | ~14,000 |
| 2.0 | 36 | 2,998 | 2,947 | 5,882 | **11,863** | ~24,000 |
| 5.0 | 37 | 4,042 | 3,914 | 11,810 | **19,803** | ~40,000 |

**重要な知見**:
- zoom=0.1（全体俯瞰）で **718ノード** — 現行Phase 3の38ノードよりリッチだが十分軽い
- zoom=1.0 で **7,225ノード** — SVG描画のスイートスポット（~15,000 DOM要素）
- zoom=5.0 でも **19,803ノード** — 全量37,258の53%に抑制

#### 金額カバー率

面積ベースLODでは小額ノードが省略されるが、金額カバー率は高い:

| zoom | 描画ノード数 | 金額カバー率 |
|------|------------|------------|
| 0.1 | 718 | ~96% |
| 0.3 | 2,511 | ~99% |
| 0.5 | 4,052 | ~99.4% |
| 1.0 | 7,225 | ~99.8% |

→ zoom=0.1の全体俯瞰でも**金額の96%をカバー**。省略されるのは小額の末端ノードのみ。

#### 実装: 3段階の最適化戦略

```
レイヤー1: 面積ベースLOD（レイアウトデータ側）
  → ノードに事前計算の面積(px²)を付与
  → 描画側で zoom² * area ≥ 1 をチェック
  → type閾値は廃止

レイヤー2: ビューポートカリング（現行と同じ）
  → ビューポート外のノード/エッジを除外
  → treemapは空間的にコンパクト(3,800×600px)なので効果は控えめ

レイヤー3: エッジの段階描画
  → zoom < 0.3: エッジ非表示（ノード面積のみで構造把握）
  → zoom 0.3〜1.0: 上位エッジのみ（value降順でN件）
  → zoom > 1.0: 表示ノード間の全エッジ
```

#### ビューポートカリングの効果比較

| レイアウト | 仮想空間サイズ | ビューポートカリング効果 |
|-----------|-------------|---------------------|
| グリッド(Phase2) | 2,800×8,261px | 高（縦に長いため画面外多い） |
| Treemapクラスタ | 3,800×600px | 低〜中（横長でフィット時はほぼ全域表示） |

→ treemapは空間がコンパクトなため、**面積ベースLODが主要な削減手段**となり、ビューポートカリングは補助的。

#### エッジ描画の最適化

74,390エッジは最大のボトルネック。ノードよりDOMコストが高い（polyline/pathはrect/textより重い）。

**方式A: value閾値**
```
描画条件: edge.value / clusterTotal ≥ threshold(zoom)
zoom=0.1 → threshold=0.001 (上位~500エッジ)
zoom=0.5 → threshold=0.0001 (上位~5,000エッジ)
zoom=1.0 → 全エッジ（表示ノード間のみ）
```

**方式B: 表示ノード連動（現行方式の発展）**
```
描画条件: source, targetの両方が面積LODで描画対象
→ 自動的に大きいノード間のエッジが優先される
```

→ **方式B推奨**: ノードの面積LODと自動連動するため追加ロジック不要。

#### パフォーマンスバジェット

目標: 60fps でのZoom/Pan操作

| DOM要素数 | SVG性能 | 備考 |
|----------|---------|------|
| ~1,000 | 60fps 安定 | zoom=0.1 での全体俯瞰 |
| ~5,000 | 60fps 可能 | zoom=0.3 |
| ~15,000 | 30-60fps | zoom=1.0 での通常操作 |
| ~30,000 | 15-30fps | zoom=2.0、操作感低下 |
| ~50,000+ | <15fps | Canvas切り替え検討ライン |

→ 面積ベースLODにより、**全zoom範囲でSVG描画が現実的**。Canvas切り替えは不要。

#### 現行Phase 3との実装差分

| 項目 | 現行（グリッド） | 提案（treemapクラスタ） |
|------|-----------------|---------------------|
| ノードLOD | type別zoom閾値 | 面積ベース (`area * zoom² ≥ 1`) |
| ビューポートカリング | 必須（縦8,261px） | 補助的（高さ600px） |
