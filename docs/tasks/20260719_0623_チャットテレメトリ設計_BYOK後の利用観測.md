## 1. 問題: BYOK 化で何が見えなくなったか

Phase 0（dev 専用 usage-log）の主目的は「**result なしで終わった要求 = 未充足需要**」の
観測だった。BYOK ではエージェントループがブラウザで回るため、サーバは何も知らない。

| 観測対象 | サーバモード | BYOKモード（現状） |
|---|---|---|
| チャット発生・outcome（result/text/error）・toolCalls・レイテンシ | dev の usage-log のみ | **見えない** |
| ツール由来の検索語・0件ヒット | API を通る | **API を通る（実は見える）** |
| 会話本文（userText） | dev のみ | 送らない（UI で約束済み） |

重要な観察: BYOK でも**ツール実行は自サイトの公開 API を通る**（利用分析検討 0節の
事実2）。つまり「未充足需要の中核シグナル = 0件検索語」はサーバ側だけで取れる。
見えないのは「チャットというセッションの存在・結末」だけ。

## 2. 提案: 2段構成（T1 = サーバ側カウンタが本体、T2 = 薄いビーコン）

### T1: API ルートのサーバ側カウンタ（Phase 1 の実装。チャット非依存・効果最大）

全公開 API ルートに `recordUsage(kind, dims)` フックを入れ、KV にカウンタとして
畳み込む（**生ログ・個票は一切持たない**。利用分析検討 3節の方式）:

```text
INCR usage:{YYYY-MM-DD}:{kind}:{dim=value}
例: usage:2026-07-19:api_search:target=spending
    usage:2026-07-19:api_search:hits0          ← 0件ヒットの件数
    usage:2026-07-19:api_search:client=byok-chat
```

- **0件検索語だけは語も保持**（`usage:{period}:zero-q:{語}` のカウンタ。集計出力時に
  k-匿名閾値 N=5 未満は出さない）。語はチャット会話の「本文」ではなく LLM が組み立てた
  検索キーワードで、既存のプライバシー文言（「検索キーワードのみ送られます」）と整合
- **client 次元**: BYOK チャットのツール fetch に `client=byok-chat` パラメータを付与
  （client-tool-executor の fetchApi に1行）。人間の直接利用・外部エージェント・
  チャット由来を区別できる（外部エージェントには従来どおり自己申告規約）
- 実装は fire-and-forget（await しない・失敗してもAPI応答に影響させない）
- **env（KV接続情報）未設定なら完全 no-op**。ローカルは従来の dev usage-log を継続

### T2: チャット結果ビーコン（会話本文なし・オプトアウト可）

チャット1往復の完了時に `POST /api/telemetry/chat` へ最小イベントを送る:

```json
{ "mode": "byok", "model": "google/gemini-3.5-flash", "outcome": "result",
  "toolCalls": 2, "latencyMs": 8400 }
```

- **含めない**: 会話本文・検索語・APIキー・結果の中身。containsは outcome
  （result / text / error の3値）と機械的メトリクスのみ
- サーバ側は T1 と同じカウンタに畳み込む（個票を持たない）
- **設定ビューにトグル**（既定オン・オフ可）。文言例:
  「匿名の利用統計（結果の種別・ツール回数など。**会話内容は含まれません**）を
  送信します。オフにできます」
- サーバモードのチャットは従来どおりサーバ側で記録できるため、ビーコンは
  BYOK モードのみ送る

### 効果の整理

- T1 だけで: 未充足需要（0件語）・機能の利用分布・チャット由来 vs 直接利用の比率が取れる
- T2 を足すと: BYOK チャットの成立率（result/text/error）・モデル分布・体感速度が取れる
- どちらも「会話の本文が当サイトへ送られることはない」の約束を破らない

## 3. 蓄積先の選定（未決事項1への推奨）

Vercel に永続ストレージがないため外部 KV が必須。**Upstash Redis（無料枠）を推奨**:

| 項目 | Upstash 無料枠 | 本件の見込み |
|---|---|---|
| コマンド数 | 500K/月 | 1リクエスト=数INCR。現トラフィックなら余裕 |
| 容量 | 256MB | カウンタ方式では実質無関係 |
| 接続 | REST API（サーバレス向き） | fetch だけで叩ける（SDK不要も可） |
| **登録** | **メール登録のみ・クレジットカード不要** | R2 で問題になったカード要件がない |

代替: Vercel KV（実体は Upstash のマーケットプレイス統合。請求が Vercel 経由になる
だけで実質同じ）。Vercel Web Analytics のカスタムイベントは Hobby では使えないため不採用。

## 4. 活用（Phase 3 の入口だけ用意）

`GET /api/usage-summary?token=<env値>` — カウンタを期間指定で機械可読に返す管理用
エンドポイント（トークンは env、未設定なら 404）。分析エージェントが
「利用分析レポート」を書くための唯一の入口（利用分析検討 4節）。初版は素朴に
日別キーの SCAN + 整形のみ。

## 5. 実装タスク分解

1. `app/lib/api/usage-kv.ts`: KV カウンタ書き込み（env 未設定 no-op・fire-and-forget・
   k-匿名は読み出し側で適用）
2. 公開 API ルート 10本に `recordUsage` フック（search 3本は hits0 と q 記録を含む）
3. client-tool-executor の fetchApi に `client=byok-chat` 付与
4. `POST /api/telemetry/chat`（T2 受け口・値検証・カウンタ畳み込みのみ）
5. page.tsx: BYOK 完了時のビーコン送信 + 設定トグル（IndexedDB の ByokSettings に
   `telemetry?: boolean` を追加）+ 設定ビュー文言
6. `GET /api/usage-summary`（トークン制・k-匿名適用）
7. ポリシー文言: 設定ビュー + フッター等への収集項目の明記（公開ページのどこに
   置くかは実装時に相談）

規模感: 1日弱。3・5 以外はクライアントに触らない。

## 6. 前提条件と判断点（ユーザー確認）

1. **Upstash アカウント作成**（メール登録・カード不要）と、Vercel への env 設定
   （`UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN`）が必要。
   → 作成しない場合、T1/T2 とも蓄積先がなく成立しない（コードは no-op で入れておき、
   env を設定した時点から収集開始、という段階導入は可能）
2. k-匿名閾値は N=5、時刻丸めは日単位（カウンタキー粒度）でよいか
3. T2 の既定（オン + オプトアウト可）でよいか（オプトイン既定だとデータがほぼ集まらない
   見込み。本文を含まない設計とセットで既定オンを提案）
