### ボトルネック候補

#### 1. Zoom/Pan時のuseMemo再計算（高リスク）

`visibleNodes`/`visibleEdges` の `useMemo` が `transform` に依存。
Wheel/Pan毎に37,258ノード＋89,034エッジを線形走査してフィルタリング。

```text
依存配列: [data, transform, containerSize, ministryFilter, minAmount, maxAmount, labelFilter]
```

Zoom/Panは高頻度（15-60FPS）で発生するため、毎フレーム再計算のコストが累積する。

#### 2. エッジの無条件描画（高リスク）

エッジはノードベースでしかフィルタされていない。
低〜中Zoom（0.1〜1.0）で両端ノードがvisibleなエッジが大量描画される。
エッジ自体の金額・画面サイズによるLODがない。

#### 3. requestAnimationFrame未使用（中リスク）

Wheel/Pan のイベントハンドラが直接 `setTransform` を呼び出し。
高頻度イベント（60+ events/sec）が毎回React再レンダーを引き起こす。

#### 4. Minimap再描画（中リスク）

Minimap Canvas が `transform` 変更のたびに全体再描画。
ビューポート矩形のみ変更でも全ノード描画をやり直す。

## 8-1: パフォーマンス計測

### 目的
最適化の優先順位を決定するためにボトルネックを定量化する。

### 計測方法

**ブラウザ Performance API**:
- `performance.mark()` / `performance.measure()` でZoom/Pan時のフレーム時間を計測
- 目標: 16.7ms/frame（60FPS）

**計測ポイント**:
1. `visibleNodes`/`visibleEdges` useMemo の実行時間
2. SVG DOM更新（ノード/エッジの追加・削除）の時間
3. Minimap Canvas描画の時間
4. 全体のフレーム時間（requestAnimationFrame間隔）

**計測シナリオ**:
- フィルタなしの初期表示（最大負荷）
- Zoom 0.1 → 5.0 の連続ズーム
- Pan操作（2秒間のドラッグ）
- 府省庁フィルタ変更

### 実装方針

- 開発時のみ有効なFPSカウンターをUI右下に表示
- `performance.now()` でuseMemo実行時間をconsole.logに出力
- 計測結果をもとに8-2〜8-5の優先度を確定

## 8-2: エッジバンドリング

### 目的
低Zoom時のエッジノイズを軽減し、描画数を削減する。

### 設計

**クラスタ間バンドリング**:
- 同一 `(sourceMinistry, targetCluster)` ペアのエッジを1本の太い帯に束ねる
- 帯の幅 = 束ねた金額合計の対数比例

**LOD切り替え**:
```text
zoom < 0.3  → バンドルエッジのみ描画（~37本 × 5クラスタ = ~185本）
zoom ≥ 0.3  → 個別エッジを描画（現行通り）
```

**データ構造**:
```typescript
interface LayoutEdge {
  // 既存フィールド...
  bundleId?: string;  // "ministry-厚生労働省→recipient" 等
}

// バンドルエッジ（事前計算）
interface BundleEdge {
  bundleId: string;
  sourceMinistry: string;
  targetCluster: string;
  totalValue: number;
  path: [number, number][];
  width: number;
  edgeCount: number;  // 束ねた本数
}
```

**実装方針**:
- `compute-sankey2-layout.ts` でバンドル情報を事前計算
- フロントエンドでは `zoom < 0.3` 時にバンドルエッジを描画、個別エッジを非表示

### 効果見込み
- 低Zoom時のエッジ数: 89,034 → ~185本（99.8%削減）

## 8-3: エッジ間引き

### 目的
中Zoom時にも不要な小額エッジをカリングし、描画負荷を軽減。

### 設計

**面積ベースLOD**（ノードと同じ思想）:
```typescript
const edgeScreenLength = Math.hypot(
  (edge.path[edge.path.length-1][0] - edge.path[0][0]) * transform.k,
  (edge.path[edge.path.length-1][1] - edge.path[0][1]) * transform.k
);
const edgeScreenArea = edge.width * transform.k * edgeScreenLength;
if (edgeScreenArea < MIN_EDGE_SCREEN_AREA) continue; // skip
```

**閾値**: `MIN_EDGE_SCREEN_AREA = 2`（~2px²未満のエッジを非表示）

### 効果見込み
- 中Zoom（0.3〜1.0）で表示エッジ数を50-70%削減

## 8-4: visibleNodes/Edgesの最適化

### 目的
Zoom/Pan毎のフル再計算を軽減する。

### 設計案A: requestAnimationFrameバッチ化

```typescript
// 現在: wheel毎に直接setTransform
svg.addEventListener('wheel', (e) => {
  setTransform(prev => ({ ...prev, k: newK, x: newX, y: newY }));
});

// 改善: rAFでバッチ化
const pendingTransform = useRef<Transform | null>(null);
const rafId = useRef(0);

svg.addEventListener('wheel', (e) => {
  pendingTransform.current = computeNewTransform(e);
  if (!rafId.current) {
    rafId.current = requestAnimationFrame(() => {
      setTransform(pendingTransform.current!);
      rafId.current = 0;
    });
  }
});
```

**効果**: 高頻度wheelイベント（100+ events/sec）を60FPSに制限。

### 設計案B: transform分離

`transform` の変更を `visibleNodes`/`visibleEdges` の再計算から分離:

```text
現在: transform変更 → useMemo再計算（37,258ノード走査） → 再レンダー
改善: transform変更 → SVG transform属性のみ更新 → useMemoはスロットル実行
```

- SVGの `<g transform="...">` でZoom/Panを反映（DOM操作なし）
- `visibleNodes`/`visibleEdges` は200ms間隔でデバウンス再計算
- ビューポートカリングのみスロットル内で更新

**効果**: Zoom/Pan中のReact再レンダー回数を5FPSに抑制。

### 設計案C: 差分更新

```text
前回visibleNodeIds: Set<string>
今回visibleNodeIds: Set<string>
追加: 今回 - 前回
削除: 前回 - 今回
```

- `useRef` で前回結果を保持
- 差分のみDOMに反映

**効果**: ノード/エッジの追加・削除が少ないPan操作で大幅改善。

### 推奨
- まず **案A（rAFバッチ化）** を実装（最小変更で効果大）
- 計測結果次第で案Bまたは案Cを追加

## 8-5: Canvas描画への移行検討

### 判断基準
8-1〜8-4の最適化後にフィルタなし状態で60FPS未達の場合のみ検討。

### 移行ステップ（必要な場合）

1. **エッジ層のみCanvas化**
   - エッジは視覚的にシンプル（polyline）→ Canvas 2D で描画容易
   - ノード矩形・ラベルはSVG維持（イベント処理・テキスト描画の恩恵）

2. **ハイブリッド構成**
   ```text
   <div style="position: relative">
     <canvas>       ← エッジ描画（背面）
     <svg>          ← ノード矩形 + ラベル（前面）
   </div>
   ```

3. **ヒットテスト不要**: エッジはクリック対象でないため、Canvas化のデメリットなし
