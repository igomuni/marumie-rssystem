# /sankey スライダー対応の振り返りと次のステップ

## 今回やったこと

### 実装内容

`feature/sankey-recipient-slider` ブランチ（main起点）で、Global View の支出先列にスライダーUIを追加した。

- **データ層**: `totalFilteredRecipients`（フィルタ後の支出先総数）を `DataSelection` → `metadata.summary.totalFilteredSpendings` に貫通
- **UIコンポーネント**: `client/components/RecipientRangeSlider.tsx` を新規作成（HTML range + 左右ボタン + 位置ラベル）
- **配置**: 凡例バー右寄せ（`ml-auto`）でSankey図の上部に統合

### 判明した構造的制約

スライダー操作後の描画に体感できる遅延がある。原因は /sankey のアーキテクチャに起因する:

1. **毎回のAPIラウンドトリップ**: スライダー操作 → URL変更 → `useEffect` → `fetch('/api/sankey?...')` → サーバー応答待ち
2. **サーバー側のフル計算**: キャッシュミス時、~96MB の structured JSON から `selectData()` + `buildSankeyData()` を実行
3. **@nivo/sankey の全再構築**: ResponsiveSankey は SVG DOM を毎回全再構築する。差分更新やバーチャライゼーションの仕組みがない

この遅延はスライダー固有の問題ではなく、既存のノードクリックによるドリルダウンでも同様に発生している。nivo を使い続ける限りの構造的制約。

## 次のステップ: /sankey の Canvas 方式への移行

### 移行の動機

| 課題 | nivo (現状) | Canvas (移行後) |
|------|-----------|----------------|
| スライダー等のインタラクション速度 | API往復 + SVG全再構築 | クライアント内でフィルタ + Canvas再描画 |
| ノード数スケーラビリティ | DOM要素数に比例して重くなる | 描画コストはほぼ一定 |
| デザイン自由度 | nivo の設定範囲内 | ピクセル単位で自由に制御可能 |
| データ取得モデル | 毎回サーバーで動的生成 | 事前計算済みレイアウトをクライアントに保持 |

### /sankey2 との関係

/sankey2 は既に Canvas + 事前計算レイアウト方式で動作しており、ツールチップ・ホバーハイライト・パン/ズームを実装済み。/sankey の Canvas 移行は /sankey2 の実績をベースにできる。

ただし /sankey と /sankey2 では目的が異なる:

| 観点 | /sankey | /sankey2 |
|------|---------|----------|
| データ生成 | 動的（APIパラメータでビュー切り替え） | 静的（事前計算済み graph.json + layout.json） |
| ビューモード | Global / Ministry / Project / Spending の4モード | 単一のグラフ全体表示 + パン/ズーム |
| ドリルダウン | ノードクリックでビュー遷移 | BFSハイライト |

### 移行で検討すべきこと

1. **データモデルの選択**: /sankey の動的生成（サーバー側 `generateSankeyData`）を維持するか、/sankey2 のように事前計算に寄せるか
   - 動的生成を維持する場合: APIレスポンスの nodes/links を Canvas で描画するだけ（nivo の置き換え）
   - 事前計算に寄せる場合: 4ビューモード分のレイアウトを事前計算する必要がある

2. **描画レイヤーの共通化**: /sankey2 の `Sankey2View.tsx` の Canvas 描画ロジック（ノード描画・リンク描画・ヒットテスト・ツールチップ）を共通ライブラリとして切り出せるか

3. **nivo が提供していた機能の再実装**: ノードのホバー時の他ノード透過、リンクのホバーハイライト、ラベルの自動配置など。/sankey2 で部分的に実装済みだが、/sankey の全機能カバーには追加作業が必要

4. **段階的移行**: 一度に全ビューを移行するのではなく、Global View から Canvas 化し、動作確認後に他ビューに展開する方法が現実的
