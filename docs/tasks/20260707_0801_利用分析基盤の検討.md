# 利用分析基盤の検討 — 収集・蓄積・活用の設計

日時: 2026-07-07
ステータス: 検討（自己改善ループ[20260705_1628]の「観測」と提案システム[20260705_1720 §3 系統2]の「関心シグナル」を支える基盤設計）

## 0. 設計を規定する2つの事実

1. **URL=状態だが、サーバーには初回ロードしか届かない。** `/sankey-svg` の状態変更は `history.replaceState/pushState` で行われ、ネットワークリクエストを発生させない。つまりサーバーログで観測できるのは「共有されたURLを開いた瞬間」だけで、**ページ内の探索行動（フィルタ操作・ノード選択）はクライアント側の計測がなければ一切見えない**
2. **API はサーバーを通る。** `/api/sankey/query`・検索系・pid系はリクエストごとにサーバーに届くため、**エージェント経由の利用はサーバー側だけでほぼ完全に観測できる**。人間のブラウザ利用とエージェント利用で観測手段が非対称

→ 帰結: エージェント利用の分析は今すぐ安価に始められ、人間のUI利用の分析はクライアントビーコンを入れる判断が必要（Phase 2）。

## 1. 分析が答えるべき問い（ここから逆算して収集を決める）

| 問い | 使い道 |
|------|--------|
| どの利用モード（5モード）が実際に使われているか | リソース配分の大枠 |
| どのテーマ・省庁・事業・支出先が関心を集めるか | 提案システムの関心ベクトル |
| **0件クエリ・400エラーは何か** | **最重要**。「検索されたが見つからない語」= 未充足需要の直接リスト（タグ体系・同義語・データ拡充の要求仕様がここから出る） |
| 探索はどこで止まるか（深掘りメニューのどれが押され、どれが無視されるか） | UX改善・「良い質問」の質の評価 |
| 使われない機能・パラメータはどれか | 機能の死蔵検出・簡素化 |
| webView URL はどこから流入するか | 「良い質問」の伝播測定（referrer） |
| エージェント利用と人間利用の比率・パターン差 | プラットフォーム段階論（MCP/内蔵アシスタント）の判断材料 |

## 2. 収集層の設計

### 2-1. イベント分類（taxonomy）

```text
共通スキーマ: { ts(日時・時間丸め), kind, dims{...}, client }
kind:
  api_query      dims: SankeyQueryの主要次元(テーマ語・fm・金額帯)・detail・結果件数(0件フラグ)
  api_search     dims: 対象(projects|recipients|spending)・q(閾値処理後)・hits
  api_error      dims: エンドポイント・400のdetails種別
  page_load      dims: パス・初回URLパラメータの有無・referrer種別
  ui_filter      dims: 確定したフィルタ次元（Phase 2・debounce後のみ）
  ui_drill       dims: ノード種別選択・詳細ダイアログ・subcontracts遷移・深掘りメニュー選択
client: human-web | agent(自己申告) | unknown
```

- **エージェントの自己申告規約**: プレイブックに「API呼び出し時に `User-Agent` または `?client=agent` を名乗る」を追記（1行・強制不能だが十分機能する）
- **0件・エラーは正規イベント**として一級で収集する（§1の最重要項目）

### 2-2. プライバシー原則（収集の前に固定する）

- Cookie なし・個人識別なし・IP は保存しない（必要なら日次ソルトでハッシュし重複排除のみに使用）
- 自由入力語（`fnp`/`q`）は集計時に **k-匿名化**（出現 N 回未満の語は集計結果に出さない）。時刻は時間単位に丸める
- 収集項目・目的をポリシーとして公開ページに明記（公開判断のゲート項目に追加済みの方針を具体化）
- DNT/GPC を尊重。ローカル環境（`VERCEL` なし）では外部送信を行わない

## 3. 蓄積層 — 生ログを持たない「カウンタ集計」方式を推奨

Vercel には永続ストレージがないため、選択肢は (a) Vercel Web Analytics、(b) 外部DB、(c) ログドレイン、(d) **KVカウンタ**。推奨は **(a)+(d) の併用**:

- **(a) Vercel Web Analytics**: ページ単位のトラフィック・referrer は無設定で取れる。ただしクエリパラメータ・カスタム次元は不足
- **(d) KVカウンタ（Upstash Redis 等・無料枠）**: イベントを**その場で次元別カウンタに畳み込み、生ログを一切保存しない**。`INCR usage:{period}:{kind}:{dim}` の形。プライバシー最良（そもそも個票が存在しない）・コスト最小・実装が薄い
- 0件クエリの語リストだけは例外的にカウンタ＋語（k-匿名閾値以上のみ保持）
- ローカル実験側は KV 不要: 構造化行を `data/usage-log.jsonl`（gitignore）に追記するだけの dev 専用フック

## 4. 活用層 — エージェントが読める形にする（ここが要件の核心）

利用分析の消費者は人間ダッシュボードではなく**まず分析エージェント**（自己改善ループの「分析」工程）。したがって:

1. **`/api/usage-summary`（管理用・要認証 or ローカル限定）**: 期間を指定してカウンタ集計を機械可読で返す。エージェントが「利用分析レポート」を書くための唯一の入口
2. **レポートコーパスとの統合ビュー**: 再現性ブロック（SankeyQuery JSON）から抽出した「レポート由来イベント」を同じスキーマに変換し、Web利用と同一の分析にかける
3. **定例の利用分析レポート**: プレイブックに依頼文テンプレを追加——「usage-summary と docs/reports/ を読み、①関心分布 ②未充足需要（0件語） ③死蔵機能 ④提案システムへの関心ベクトル更新 を報告せよ」。出力はロードマップ triage の一次資料
4. （遊び心のある将来案）利用フロー自体をサンキー図で描く——「問い → 機能 → 深掘り → 離脱/レポート」。このサイトの文法で自分自身の利用を可視化する

## 5. 段階的導入

| Phase | 内容 | 前提 |
|-------|------|------|
| 0（今すぐ・ローカル） | レポートコーパス規約（docs/reports/ + 再現性ブロック）+ dev 専用 usage-log.jsonl + プレイブックに client 自己申告 | 実装ほぼゼロ |
| 1（公開時） | API ルートのサーバー側カウンタ（KV）+ Vercel Web Analytics + ポリシー掲示 | 公開判断・KV選定 |
| 2（公開後・判断制） | クライアントビーコン（ui_filter / ui_drill。debounce 後のみ・サンプリング可） | Phase 1 で人間利用が観測され、UI行動の解像度が必要と判断された場合のみ |
| 3 | /api/usage-summary + 定例利用分析レポートの運用開始 | Phase 1 のデータが1〜2ヶ月分 |

Phase 2 を「判断制」にする理由: クライアント計測は実装・プライバシー・ページ性能のコストが相対的に高く、**エージェント経由が主流になるなら不要になる可能性がある**。Phase 1 の観測で決めるのが、まさに利用分析の使い方として正しい。
