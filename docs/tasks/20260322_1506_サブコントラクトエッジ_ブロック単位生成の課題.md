# サブコントラクトエッジ ブロック単位生成の課題

## 目的

5-2 CSVのブロック接続からサブコントラクトエッジを生成する際に、ブロック内に複数支出先がある場合の「代表選択」問題を整理し、正確な表現を検討するため。

## 現状の問題

### 1. 代表選択による捏造

ブロック接続 A→B は「ブロックA → ブロックB」の関係であり、ブロック内の個別支出先間の対応はCSVに記録されていない。

現在のコードはブロック内の最大金額支出先を「代表」として選ぶが、これはCSVにない関係を作り出す。

**例: PID=3440（農業施設災害復旧等事業）**

5-2 CSVの接続:
```
B(都府県) → D(市町村営事業)
F(北海道) → D(市町村営事業)
```

ブロックBの支出先（その他除外・金額順）:
| 支出先 | 金額 |
|--------|------|
| 熊本県 | 58.47億 |
| 広島県 | 40.73億 |
| 大分県 | 37.20億 |
| ... | ... |

ブロックFの支出先: 北海道（1件のみ）

ブロックDの支出先: 朝倉市（12.88億）、さつま町、その他 等

現在の生成結果: `熊本県 → 朝倉市 12.88億` — CSVにこの関係はない。

### 2. Fan-In時の二重計上と情報欠落

複数のソースブロック（B, F）が同じターゲットブロック（D）に接続する場合:

- **二重計上**: ブロックDの朝倉市12.88億がB→DとF→Dの両方で使われる
- **情報欠落**: 二重計上防止のため一方をスキップすると、もう一方のソース情報が消失

PID=3440の場合、B→Dが先に処理されるため:
- `熊本県 → 朝倉市 12.88億` が生成（捏造）
- `北海道 → 朝倉市` が二重計上防止でスキップ（情報欠落）

理想: `B(都府県)` と `F(北海道)` の**両方**が朝倉市への委託元として表示されるべき。

## CSVデータの構造的制約

5-2 CSVが提供する情報:
- **ブロック間の接続**: A→B→C→D のような構造
- **ブロック名**: 「都府県」「市町村」「北海道」等

5-1 CSVが提供する情報:
- **ブロック番号ごとの支出先一覧と金額**: ブロックBに属する支出先は熊本県、広島県...

**提供されない情報**:
- ブロック内のどの支出先がどのターゲット支出先に紐づくか
- Fan-In時に各ソースブロックからの配分比率

## ブロック名の性質

5-2 CSVのブロック名は2種類:

| 種類 | 例 | 支出先ノード存在 |
|------|-----|-----------------|
| 固有名 | 北海道、一般社団法人行政情報システム研究所 | あり |
| カテゴリ名 | 都府県、市町村、農業委員会、地方農政局等 | なし |

カテゴリ名のブロックは支出先ノードが存在しないため、recipient→recipientのサブコントラクトエッジのソースにできない。

## 設計の選択肢

### A. ブロックノードの導入

ブロック名をrecipientノードとは別の「ブロックノード」として新設し、ブロック単位のエッジを生成する。

```
[ブロックB: 都府県] →(subcontract)→ [recipient: 朝倉市]
[recipient: 北海道] →(subcontract)→ [recipient: 朝倉市]
```

- 利点: CSVの構造に忠実。カテゴリ名もノードとして表現できる
- 課題: 新しいノードタイプの追加。UIの表示ロジック変更が必要

### B. 固有名ブロックのみエッジ生成

ブロック名が既存の支出先ノードに一致する場合のみエッジを生成。カテゴリ名ブロックはスキップ。

```
[recipient: 北海道] →(subcontract)→ [recipient: 朝倉市]
（都府県はスキップ）
```

- 利点: 既存のノード構造を変更不要。捏造エッジなし
- 課題: カテゴリ名ブロック経由の委託が全て非表示になる。再委託エッジの大幅減少

### C. ブロック情報をエッジのメタデータに保持

recipient→recipientエッジは生成せず、パネル表示のsubcontractChainsのみで委託構造を表現。

- 利点: 捏造なし。CSVのブロック構造がそのまま表示される
- 課題: Sankey図上にサブコントラクトの線が表示されない

### D. Fan-In対応付きの固有名マッチ

選択肢BにFan-In対応を加える:
- 固有名ブロック（北海道等）はエッジ生成
- Fan-In時: 同じターゲット支出先への複数ソースを全て保持
- 金額: ターゲット支出先の金額はFan-Inソース数で均等按分、またはそのまま保持しメタデータで「按分元ブロック数」を記録

## 現在のコードの状態

- structured.json依存は完全に排除済み（5-2/5-1 CSVのみで生成）
- 「その他」はソース代表候補から除外済み
- 二重計上防止は実装済み（ただしFan-Inの片方が消失する問題あり）
- ブロック名マッチではなくブロック番号ベースで支出先取得済み

## パネル表示（subcontractChains）

パネル表示用の`subcontractChains`データは影響を受けない:
- `blockChain`: ブロック接続構造（A→B→C→D）はそのまま保持
- `directBlocks`: 直接支出ブロック情報もそのまま
- これらは5-2 CSVのブロック構造をそのまま反映しており、代表選択の問題がない

## CSVデータ設計の考察: なぜFan-Inが発生するか

### PID=3440のブロック構造が示す設計意図

```
担当組織 → A(地方農政局等) → B(都府県) → D(市町村営事業)
担当組織 → F(北海道)       → D(市町村営事業)
```

北海道は`B(都府県)`に含めず`F(北海道)`として独立ブロックにしている。
これは単に「都道府県」をひとまとめにせず、北海道を分離した結果。

つまりCSVの設計者は「都府県（=北海道以外）」と「北海道」を意図的に分けているが、
委託先のブロックD（市町村営事業）は分けていない。
結果として、BとFの両方がDに接続するFan-In構造になる。

### ブロックDの金額問題

ブロックDの支出先（朝倉市12.88億等）は**ブロックD全体の金額**であり、
BとFのどちらから流れた金額かは区別されていない。

CSVのブロック設計上、BとFを分離するなら、ターゲット側も:
```
B(都府県)  → D1(都府県配下の市町村営事業)
F(北海道)  → D2(北海道配下の市町村営事業)
```
のように分離すれば、金額の帰属が明確になる。

しかし現状のCSVはターゲットブロックを共有しており、個別の配分情報がないため、
ソースブロックごとの金額は原理的に特定できない。

### 理想的なデータ構造

本来は都道府県ごとにブロックがあり、その配下に属する市区町村の支出先がある構造が望ましい:

```
担当組織 → A(地方農政局等)
  A → B1(北海道)    → D1(北海道配下の市町村)
  A → B2(熊本県)    → D2(熊本県配下の市町村: 朝倉市等)
  A → B3(広島県)    → D3(広島県配下の市町村)
  ...
```

これならFan-Inが発生せず、都道府県→市区町村の対応も明確になる。
しかし現状のCSVはこの粒度で設計されていないため、
サブコントラクトエッジの正確な生成には限界がある。
