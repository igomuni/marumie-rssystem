# 再委託チェーン表現の限界と次のステップ

## 結論

**支出先ノード（recipient）のみでは再委託チェーンを正確に表現できない。**

CSVの再委託構造は「支出先ブロック」単位で定義されているが、現在のSankey図は支出先ノード間のエッジしか持たない。ブロックに複数の支出先が含まれる場合、どの支出先がどの委託先に対応するかはCSVに記録されておらず、原理的に特定できない。

## 立証プロセス

### Phase 1: 代表選択ロジックの問題発見

ブロック名が支出先ノードに一致しない場合（カテゴリ名ブロック: 都府県、市町村等）、ブロック内の最大金額支出先を「代表」として選ぶロジックが存在した。

- `熊本県 → 朝倉市 12.88億` — CSVにこの関係はない（PID=3440）
- `中野市 → 朝倉市農業委員会 0.02億` — CSVにこの関係はない（PID=5874）

これらはCSVにない関係を捏造するものであり、排除した。

### Phase 2: 問題の本質の特定

代表選択を排除した後、「なぜブロック名で一致できないのか」を分析した結果:

- ブロック名が**カテゴリ名**（都府県、市町村等）の場合 → 支出先ノードに一致しない
- ブロック名が**固有名**でも、ブロック内に複数の支出先がある場合 → ブロック名の支出先しかソースにできず、他の支出先は無視される

### Phase 3: データによる検証

5-2 CSV（ブロック接続）と5-1 CSV（ブロック内支出先）を突合し、ソースブロックの支出先数別に接続を分類:

| ソースブロックの支出先数 | 接続数 | 割合 | エッジ生成 |
|------------------------|--------|------|-----------|
| 0件 | 263 | 3.8% | 不可 |
| **1件** | **4,758** | **68.9%** | **正確に可能** — 唯一の支出先がソースとして一意に決まる |
| **2件以上** | **1,888** | **27.3%** | **不可能** — どの支出先がソースか特定不能 |

**68.9%の接続は支出先ノードだけで正確に表現可能だが、27.3%は原理的に不可能。**

### 構造的な理由

```text
5-2 CSV（ブロック間接続）:
  B(都府県) → D(市町村営事業)

5-1 CSV（ブロック内支出先）:
  ブロックB: 熊本県(58億), 広島県(41億), 大分県(37億), ...
  ブロックD: 朝倉市(13億), さつま町(5億), ...
```

CSVが提供するのは「ブロックB → ブロックD」の関係のみ。ブロックB内のどの県がブロックD内のどの市町村に委託したかは**記録されていない**。支出先ノード間のエッジ（`熊本県 → 朝倉市`）を作ることは**データの捏造**になる。

## 解決の方向性: ブロックノード層の導入

`事業(支出)` → `支出先` の2層構造に、`支出ブロック`を中間層として追加する:

```text
事業(支出) → 支出ブロック → 支出先
                │
                │ block→block（再委託）
                ↓
              支出ブロック → 支出先
```

ブロック自体がノードとして存在すれば、支出先数に関係なくブロック間のエッジを正確に表現できる。

詳細な調査データ: [20260322_1743_ブロックノード層導入の調査.md](20260322_1743_ブロックノード層導入の調査.md)

## 暫定対応

ブロックノード導入まで、現行コードは以下の方針:

1. ブロック名が支出先ノードに一致する場合のみサブコントラクトエッジを生成
2. カテゴリ名ブロックはスキップ（捏造防止）
3. サブコントラクトチェーンのブロック構造はパネル表示（subcontractChains.blockChain）で引き続き参照可能
