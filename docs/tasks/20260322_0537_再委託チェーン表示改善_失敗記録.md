# 再委託チェーン表示改善 — 失敗記録

## 目的

/sankey2 で再委託チェーンの構造を正しく表示するため、以下を実施しようとした:

1. 間接支出先への不正な直接エッジを除去
2. 支出先ノードの金額を直接支出のみに変更
3. 執行額の表示
4. 再委託チェーンのハイライト表示

## 何が起きたか

### Phase 1: データ修正（成功）

`generate-sankey2-data.ts` で `isDirectFromGov === false` のエッジを除去し、支出先ノードの金額を直接支出のみに変更。`executedAmount` フィールドも追加。

**結果**: データは正しくなった。PID=18（政府共通ネットワーク）のエッジが20本→1本（行政情報システム研究所のみ）に。間接支出29,438件を正しく除外。

### Phase 2: ハイライト表示（失敗の連鎖）

データ修正により、以前はBFS深さ1で見えていた支出先（富士通等）がBFS深さ2以上にしか到達できなくなった。これを補うために複数のアプローチを試行したが、すべて失敗した。

#### アプローチ1: BFS深さ制限の緩和（subcontractエッジのみ追加探索）

**変更**: `bfsHighlight` を修正し、`maxDepth` を超えた後もsubcontractエッジを双方向で辿るようにした（`SUBCONTRACT_EXTRA_DEPTH = 3`）。

**結果**: subcontractエッジを**双方向**で辿ったため、富士通に到達した後、富士通に再委託している**他の全プロジェクト**のエッジも逆方向で辿られ、グラフ全体にハイライトが広がった。

**教訓**: subcontractエッジのネットワークは双方向で密結合しており、双方向BFSでは爆発する。

#### アプローチ2: 順方向のみのBFS拡張

**変更**: `maxDepth` 超過後はsubcontractエッジの**順方向（source→target）のみ**辿るよう修正。逆方向は `maxDepth` 内のみ。

**結果**: 順方向に制限したが、依然として多数のノードがハイライトされた。特に問題だったのは:
- 富士通、NTTデータ等の大手受託者はレイアウト上で遠方に配置されている
- ハイライト対象ノードはLOD（Level of Detail）とビューポートカリングをスキップするため、画面上で視認できないほど小さいノードも強制レンダリング
- 結果、見えないノードへのエッジが画面を横断し「左上のなにもないところにエッジが繋がっている」状態に

**教訓**: Treemapレイアウトでは同種ノード（recipient）はクラスタ内で金額順に配置されるため、あるプロジェクトの直接支出先とその再委託先は空間的に離れている。BFSでハイライト対象を増やすと、長距離のエッジが大量に発生する。

#### アプローチ3: エッジのopacityのみ変更

**変更**: BFSは変更せず、ハイライト済みノードから出るsubcontractエッジの `opacity` を `0.02`（ほぼ不可視）→ `0.35`（半透明）に引き上げ。`MemoEdgeLine` に `isSubcontractFromHighlighted` propを追加。

**結果**: エッジは見えるようになったが、接続先のノード自体はハイライトされないため中途半端。修正前（間接エッジ除去前）のほうが全支出先がBFS深さ1でハイライトされていたため、UXが悪化。

**教訓**: エッジだけ見せてもノードがハイライトされないなら意味がない。

#### アプローチ4: projectIdsによるフィルタリング

**変更**:
- `generate-sankey2-data.ts`: subcontractエッジに `projectIds` フィールドを追加（どの事業IDに関連するか記録）
- `compute-sankey2-layout.ts`: `projectIds` をレイアウトデータにパススルー
- `types.ts`: `LayoutEdge` に `projectIds` 追加
- `extendSubcontractForward`: BFS後にprojectIdでフィルタしてsubcontractチェーンを順方向に追加

**結果**: PID=18で50ノードに絞り込めた（以前の4,737ノードから大幅改善）。しかし根本的な問題は解決しない:
- 50ノードでも遠方への長距離エッジが発生
- 支出先クラスタ全体の左上方向へエッジが伸びる視覚的ノイズ
- ユーザーから「支出ブロック全体の左上への委託エッジは必要ですか」との指摘

## 根本原因の分析

### Treemapレイアウトとsubcontractチェーンの相性問題

現在のレイアウトは金額ベースのTreemapで、ノードは以下のように配置される:
- total → ministry → project-budget → project-spending → recipient の5列
- 各列内は金額の大きい順

このレイアウトでは:
1. 富士通（2,187億）は支出先クラスタの上位に配置
2. 行政情報システム研究所（44億）は中位に配置
3. PID=18の小規模な再委託先（数億以下）は下位に配置

subcontractエッジは同一列内の離れたノード間を結ぶため、必然的に長距離エッジになる。これはSankey図（左から右への流れ）とは本質的に異なる表示パターン。

### 修正の方向性が間違っていた

「BFSでハイライト対象を増やす」アプローチは、Treemapレイアウトでは長距離エッジという副作用を避けられない。再委託チェーンの可視化は、グラフ上のハイライトではなく、別の表示方法（パネル表示、フォーカスビュー等）が適している。

## 今後への示唆

1. **レイアウトの制約を先に理解する**: 表示改善はレイアウト特性に依存する。Treemapで同一列内の長距離エッジは避けられない。
2. **データ修正と表示改善は分離する**: データの正確性（間接エッジ除去）は正しい方向だが、それに伴うUX変化を補うための表示改善が不十分だった。
3. **段階的に検証する**: 複数の変更（データ修正+BFS拡張+UI追加）を一度に行ったため、問題の切り分けが困難になった。
4. **再委託チェーンは別の可視化手法が必要**: Treemapレイアウト上のBFSハイライトではなく、選択ノード周辺のサブグラフ表示や、パネル内のツリー表示などを検討すべき。
