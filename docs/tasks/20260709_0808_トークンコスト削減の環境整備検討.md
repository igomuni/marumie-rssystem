## 2. P1: データファイル読込の禁止と `/data-query` スキル

### 問題

`public/data/sankey-svg-2024-graph.json`（96MB）や `project-quality-recipients-*.json` を Read で開くと、行クランプが効いても1行が巨大なため数万トークン級になる。調査タスクで「まずデータを見る」と自然に発生する事故で、**このリポジトリ最大の単発コスト**。

### 対策A: settings.json に deny を追加（機械的な防止）

```json
{
  "permissions": {
    "deny": [
      "Read(./public/data/**/*.json)",
      "Read(./public/data/**/*.json.gz)",
      "Read(./data/**)",
      "Read(./docs/プロンプトログ*.ignore.md)"
    ]
  }
}
```

- `public/data/dictionaries/*.csv` は小さく参照価値があるため deny から除外する（パターンを `*.json` 系に限定している理由）
- deny は共有 `settings.json` に置く（Sonnet/Opus どのセッションでも効く）

### 対策B: `/data-query` スキル新設（代替手段の提供）

deny だけでは agent が回り道（`cat` 等）を試してかえってトークンを浪費する。**「代わりにこうやる」を1枚で渡す**スキル `.claude/commands/data-query.md` を新設する。内容（3KB 以内）:

```text
1. 第一選択: dev サーバーの API を使う（/api/sankey/query, /api/search/*, /api/quality 等
   → 仕様は docs/api-guide.md、探索の作法は docs/agent-playbook.md）
2. API にない集計は jq / node --eval のワンライナーで:
   - 構造確認:   jq 'keys' / jq '.nodes[0]' file.json   （全体を出力しない）
   - 件数・合計: jq '[.edges[].value] | add'
   - gz直読み:   gunzip -c file.json.gz | jq -c '.nodes[:3]'
3. houjin.db は sqlite3 の SELECT のみ（スキーマは .schema で確認）
4. 出力は必ず head / jq の絞り込みでクランプしてから読む
5. Read での直接読込は禁止（settings.json の deny で拒否される）
```

- 期待効果: 事故防止に加え、正規ルート（query API）への誘導が agent-playbook・AIチャットのツール群と同じ思想で揃う

## 3. P2: `docs/tasks/INDEX.md` — 1行索引

### 問題

task doc は53本・6.7MB。過去の検討を探すとき、現状は `ls -lt` + ファイル名からの推測 + 数本の全文 Read になる（本セッションでも発生: レポート特定に3本読んだ）。1本 6〜25KB なので、探索1回で 2〜5万トークン。

### 対策

1. `docs/tasks/INDEX.md` を新設: 1ファイル1行（`- YYYYMMDD_HHMM_ファイル名 — 30字要約・状態`）。53本 × 1行 ≒ 4KB で全履歴を一望できる
2. `.claude/CLAUDE.md` の「回答の出力先」ルールに1文追加: **task doc を新規作成したら INDEX.md に1行追記する**（作成フローに組み込み、別途メンテ不要にする）
3. 同ルールに読込規律を追加: **過去の検討を探すときは INDEX.md を先に読み、対象を特定してから本文を読む**（`ls` からの推測読みをやめる）

- 初版53行の生成は1回だけの作業（各ファイルの先頭数行から要約を起こす。実装WPとして小）
- 将来 `docs/reports/` 分離（20260705_1544 §3-5）を行う場合も同じ索引方式を適用する

## 4. P3: プロンプトログの読込制限

現行ルールは「直接修正禁止・**読み取りは可**」だが、5本合計 500KB超・最大249KBで、無邪気な Read 1回が数万トークンになる。ルールを次に改める（`.claude/CLAUDE.md`）:

```text
- docs/プロンプトログ*.ignore.md: ユーザーの作業ログ。修正・編集禁止（現行どおり）。
  読み取りはユーザーが明示的に指示した場合のみとし、その場合も grep で該当箇所を
  特定してから offset/limit 付きで部分読みする（全文 Read 禁止）。
```

P1 対策Aの deny にも含めた（deny があれば grep/部分読みへ自然に誘導される）。

## 5. P4: スキルへの「読み方の規律」追記（各1〜3行）

スキル本体は軽量なので構成は変えず、誘導文だけ最適化する:

- `/sankey`（sankey.md）: 「タスク1: 上記から関連ファイルを読み込み**全体像を把握**」→ **「指示に関係するファイルだけを読む。page.tsx（3,000行超）は必ず Grep で該当箇所を特定してから部分読みする」** に変更。全体把握の一括読みは page.tsx だけで数万トークンになる
- `/pipeline`（pipeline.md）: 「生成JSONの中身確認は `/data-query` の作法で行う（Read しない）」を1行追加
- `/plan`・`/review`: 「過去の検討参照は docs/tasks/INDEX.md から」を1行追加
- `docs/api-guide.md`・`docs/agent-playbook.md` の関係を CLAUDE.md に1行明記: **API仕様の正典は api-guide、探索の作法は playbook**（両方を毎回読む必要はない、の交通整理）

## 6. P5: サブエージェント/Explore の使い分け指針（`.claude/CLAUDE.md` に3行）

```text
- 広域探索（多ファイル横断の調査・命名規則の掃討）は Explore サブエージェントに委譲し、
  本会話には結論だけ持ち帰る（ファイルダンプを本会話コンテキストに載せない）
- 逆に、既に文脈がある作業のためにサブエージェントを新規起動しない
  （コールドスタートは CLAUDE.md・関連ファイルの再読込コストが掛かる）
```

## 7. P6: 実装モデルのティアリング運用（ルールではなく運用）

トークン**量**の削減（P1〜P5）と独立に、トークン**単価**を下げる:

- ロードマップ（20260709_0759）で Sonnet 向けとした WP（深掘りツール・評価ハーネス・解釈宣言UI・ツール露出・ローカル利用ログ）は **Sonnet セッションで実施**する。プレスクリプティブ計画（ファイル・手順・受け入れ基準固定）はこのための投資
- 設計裁量のある WP のみ Opus/Fable を使う。「Opus で全部やる」は品質面の利点よりコスト面の損が大きい構成にしてある
- Sonnet セッションの失敗コストを下げるのが P1〜P4（ガードレールと索引はモデルが軽いほど効く）

## 8. 適用しない（検討済み・見送り）

- **CLAUDE.md のさらなる圧縮**: 5.2KB は既に十分小さく、削ると規律（PR draft・レイヤー規約）の失敗コストの方が高い
- **スキルの統合・削減**: 9本とも小さくオンデマンド。統合の効果なし
- **フックによる Read 阻止（PreToolUse）**: settings の deny で足りる。フックは保守コストが上回る
- **docs/*.md 恒久ガイドの要約版作成**: 8〜14KB は適正レンジ。二重管理の腐敗リスクの方が大きい
