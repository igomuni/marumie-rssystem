## 反映方針 — 3案

### 案A: 一括同期 PR（推奨）

過去の同期 PR と同じスタイル。`team-mirai-volunteer/sync/20260517` ブランチを切って、marumie main の対象範囲を丸ごとコピーし、単一の PR にする。

- **利点**: 過去フローと一致、レビューが1回で済む、データ整合性が壊れにくい
- **欠点**: 差分が大きい（+7,795）ので個別のレビュー粒度が粗くなる
- **適合度**: ◎（過去 3 回の同期 PR と同様）

### 案B: 機能単位で分割PR

- sankey-svg 系（#199, #200, #203, #204, #206, #207, #208, #209）
- subcontracts 系（#201, #202）
- docs/LICENSE 系（#198）

で 3〜4 PR に分割。

- **利点**: レビュー粒度が細かい、機能ごとの ON/OFF が可能
- **欠点**: コンフリクト解消が複雑化、データ JSON の依存（subcontracts と sankey-svg は別ファイル）も整理が必要
- **適合度**: △（過去フローから逸脱）

### 案C: 最小差分（コア機能のみ）

ユーザー向けに視認性のある機能（#199, #203, #206, #209）だけ先行で反映。subcontracts 全面刷新（#201, #202）は別途。

- **利点**: 反映が早い
- **欠点**: marumie と rs-vis の差が広がり、次回同期が困難に
- **適合度**: ✕（推奨しない）

## 推奨フロー（案A前提）

```bash
# 1. rs-vis 側で同期ブランチを切る
cd /Users/igomuni/MyGitHub/rs-vis
git fetch marumie
git checkout -b sync/20260517 main

# 2. marumie の対象範囲をマージ（または cherry-pick）
#    過去フローを踏襲するなら squash 系で1コミットに丸める
git merge --squash marumie/main
# もしくは: 範囲指定
# git cherry-pick 4953539..3fc5635

# 3. データファイル展開チェック
npm run prebuild   # .gz → .json 展開
npm run build      # ビルド検証
npm run lint
npx tsc --noEmit

# 4. Playwright で /sankey-svg と /subcontracts を主要観点で確認
npx playwright test tests/e2e/sankey-svg.spec.ts

# 5. PR 作成
gh pr create --base main --head sync/20260517 \
  --title "feat: marumie-rssystem の最新状態を反映（2026-05-17時点）" \
  --body-file ...
```
