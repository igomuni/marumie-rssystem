# sankey2 再委託エッジが一部しか生成されない原因調査

## 結論

**`targetBlockName` は5-2 CSVのブロック名（集約名）であり、5-1 CSVの `spendingName`（個別支出先名）とは命名体系が異なる。** これが根本原因。

- 件数マッチ率: **47.1%**（2,107 / 4,473）
- 金額マッチ率: **29.4%**（4.49兆 / 15.28兆円）
- 金額ベースで **10.79兆円分のエッジが欠落**

## 現在のマッチングロジック

```
scripts/generate-sankey2-data.ts L349-351:
  const targetName = flow.targetBlockName;
  const targetNode = recipientNodeIndex.get(targetName);  // ← spendingName完全一致
```

recipientノードのlabel = 5-1 CSVの `spendingName`（個別企業名）。
outflowの `targetBlockName` = 5-2 CSVのブロック名（集約名）。

## アンマッチの分類（2,366件）

| パターン | 件数 | 割合 | 例 |
|---------|------|------|-----|
| 「ほか」付き | 1,681 | 71.0% | `関彰商事株式会社ほか` |
| 括弧付き | 130 | 5.5% | `地方公共団体(56市町村等)` |
| 総称名 | 171 | 7.2% | `民間事業者`、`組合、大学法人、民間事業者等` |
| その他 | 384 | 16.2% | `市町村営事業`、`都道府県営事業` |

## 「ほか」パターンの詳細分析

「ほか」を除去して再マッチすると:
- **1,523件がマッチ** → 代表名が5-1のspendingNameと一致
- **158件は依然不一致** → `ほか29医療機関等`、`ほか4者` 等の変種

つまり「ほか」除去だけで、アンマッチの **64.4%** を回収できる。

## 重要な発見: `recipients` サブフィールド

outflowの `recipients` フィールド（`SpendingBlockFlow.recipients`）には**ブロック内の個別支出先名**が格納されており:
- 4,455件のoutflow（99.6%）に `recipients` あり
- **個別recipient名の5-1マッチ率: 100%**

つまり `flow.recipients[].name` を使えば、`targetBlockName` を介さず正確にマッチできる。

## 具体例: 「ほか」ブロックの中身

`一般社団法人全国石油協会` → `関彰商事株式会社ほか`（targetBlockName、不一致）
- ブロック内recipients: `関彰商事株式会社`、`出光リテール販売株式会社` ... → 全て5-1 spendingNameに存在

## inflows について

- **inflows は全spendings で 0件**（データが存在しない）
- inflows からのエッジ生成は不要（データがないため効果なし）

## sourceBlockName の不一致

outflowの `sourceBlockName` と `spending.spendingName` が一致しないケースも628件ある:
- `株式会社博報堂` vs `株式会社博報堂ほか(委託先)`
- `東京都` vs `東京都ほか46道府県`

ただし現在のコードはsource側は `spending.spendingName` でルックアップしているため、この不一致は影響しない。

## 推奨修正方針

**`flow.recipients[].name` を使って個別エッジを生成する:**

```
// 現在（targetBlockNameで1本のエッジ）
targetNode = recipientNodeIndex.get(flow.targetBlockName)

// 修正案（recipients個別にエッジ生成）
for (const r of flow.recipients) {
  targetNode = recipientNodeIndex.get(r.name)  // 100%マッチ
  // r.amount で個別金額も取得可能
}
```

メリット:
1. マッチ率が29.4% → ほぼ100%に改善
2. 10.79兆円分の再委託エッジが回復
3. 個別支出先単位の金額精度も向上（ブロック集約額ではなく個別額を使用）

注意点:
- エッジ数が大幅に増加する（ブロック1本 → recipients個別に展開）
- recipients が空の18件は別途ハンドリング必要
