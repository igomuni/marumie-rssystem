# /sankey2 Phase 7: 委託チェーン可視化 設計

## 背景

Phase 6までにブラウザ履歴対応・ラベル段階表示を実装。
37,258ノードの探索・フィルタ体験は充実したが、支出先ノードの**再委託構造**が不可視のまま。

RS2024のStructuredJSONには委託チェーン情報が既に存在する:
- `SpendingProject.isDirectFromGov`: 政府からの直接支出か
- `SpendingProject.sourceChainPath`: 委託経路（例: `"博報堂 → EYストラテジー"`）
- `SpendingRecord.outflows`: 再委託先ブロックフロー情報

Phase 7ではこれらの既存データをsankey2パイプラインに接続し、委託チェーンを可視化する。

---

## 既存データ構造の整理

### StructuredJSON（types/structured.ts）

```typescript
// 支出先ブロック間のフロー（再委託関係）
interface SpendingBlockFlow {
  projectId: number;
  sourceBlockNumber: string;   // 支出元ブロック（例: "A"）
  sourceBlockName: string;     // 支出元名（例: "株式会社博報堂"）
  targetBlockNumber: string;   // 支出先ブロック（例: "B"）
  targetBlockName: string;     // 支出先名
  flowType: string;            // フロー種別（例: "再委託"）
  amount: number;
  isDirectFromGov: boolean;
}

// 支出先レコード（SpendingRecord）
interface SpendingRecord {
  outflows?: SpendingBlockFlow[];  // この支出先からの再委託先
  inflows?: SpendingBlockFlow[];   // この支出先への流入元
}

// 支出プロジェクト（SpendingProject）
interface SpendingProject {
  isDirectFromGov?: boolean;     // 政府直接支出か
  sourceChainPath?: string;      // 間接の場合の委託経路
  blockNumber: string;           // ブロック番号
}
```

### 現行sankey2データ（Sankey2Node）

```typescript
// scripts/generate-sankey2-data.ts
interface Sankey2Node {
  id: string;         // "recipient-支出先名"
  label: string;
  type: 'total' | 'ministry' | 'project-budget' | 'project-spending' | 'recipient';
  amount: number;
  projectId?: number;
  ministry?: string;
  // ← 委託チェーン情報なし
}
```

## 7-1: データパイプライン拡張

### 目的
支出先ノードに委託チェーン情報を付与し、レイアウトデータに渡す。

### Sankey2Node 拡張

```typescript
interface Sankey2Node {
  // 既存フィールド...
  isIndirect?: boolean;       // 間接支出（委託経由）を含むか
  chainPaths?: string[];      // ユニークな委託経路リスト
}
```

### 実装方針

1. **rs2024-structured.json.gz を読み込み**
   - `generate-sankey2-data.ts` は5-1 CSVからの支出先ノード生成に加え、structured.json.gzを読み込み
   - `spendings[].projects[]`から`isDirectFromGov`と`sourceChainPath`を取得

2. **支出先ノードへの付与**
   - 各recipientの全projectをスキャンし:
     - いずれかの`isDirectFromGov === false` → `isIndirect = true`
     - `sourceChainPath`をSetで収集 → `chainPaths`（ユニーク化）

3. **LayoutNode 拡張**（`client/components/Sankey2/types.ts`）
   ```typescript
   interface LayoutNode {
     // 既存フィールド...
     isIndirect?: boolean;
     chainPaths?: string[];
   }
   ```

4. **`compute-sankey2-layout.ts`**: Sankey2Nodeの新フィールドをLayoutNodeにパススルー

### データ量の影響

- 新フィールドは `recipient` タイプのノード（25,892件）のみに付与
- `chainPaths` は多くの支出先で空配列 → gzip後の影響は軽微

## 7-2: サイドパネル拡張

### 目的
支出先ノード選択時に委託チェーン情報をサイドパネルに表示。

### UI設計

**ヘッダー部分**:
- `isIndirect === true` の場合、typeバッジの横に「委託経由」バッジ追加
  - 背景色: `#f59e0b`（amber-500）
  - テキスト: "委託経由"

**委託経路セクション**（`chainPaths` が存在する場合のみ表示）:
```text
┌─ 委託経路 ──────────────────────┐
│ 政府 → 博報堂 → EYストラテジー    │
│ 政府 → 博報堂 → 三菱総研          │
└────────────────────────────────┘
```

- 各経路を行として表示
- 経路の各ステップを `→` で区切り
- 最終ノード（自身）をハイライト表示

### 実装方針

- `Sankey2View.tsx` の `DetailPanel` コンポーネントを拡張
- `node.type === 'recipient' && node.isIndirect` で条件分岐
- 「政府 →」は先頭に自動付与（sourceChainPathは政府を含まないため）

## 7-3: 委託チェーンBFS探索

### 目的
再委託先→再々委託先をエッジとして可視化し、BFS探索で深堀り可能にする。

### 現状の課題
- recipientクラスタ内のノードはすべて孤立（ノード間エッジなし）
- 再委託関係が視覚的にたどれない

### 設計

**再委託エッジの生成**:
1. `generate-sankey2-data.ts` で `outflows.recipients[].name` から recipient→recipient エッジを生成
2. `flowType`フィルタで委託・外注・請負系のみ対象（補助金交付等は除外）
3. エッジには `edgeType: 'subcontract'` フラグを付与
4. 通常の project-spending→recipient エッジと区別

**Sankey2Edge 拡張**:
```typescript
interface Sankey2Edge {
  source: string;
  target: string;
  value: number;
  edgeType?: 'direct' | 'subcontract';  // 新規
}
```

**LayoutEdge / フロントエンド**:
- `edgeType === 'subcontract'` のエッジは破線または別色で描画
- 色: `#f59e0b`（amber-500、直接支出の赤と区別）
- 既存のBFS探索はそのまま動作（edgeIndexにsubcontractエッジも含まれるため）

### 注意点

- **ブロック名 vs 支出先名**: outflowsの`targetBlockName`は「〜ほか」等で5-1 CSVの`spendingName`と不一致が多い
  - 解決: `flow.recipients[].name`を使用（個別支出先名で100%マッチ）
  - `recipients`がない場合のみ`targetBlockName`でフォールバック
- **循環エッジ**: A→B→A のような循環は除外
- **パフォーマンス**: 再委託エッジ数は通常エッジ（74,390本）より少ない見込み

## 実装順序

1. **7-1: データパイプライン拡張**（structured.json読み込み、isIndirect/chainPaths付与）
2. **7-2: サイドパネル拡張**（委託バッジ、経路表示）
3. **7-3: 委託チェーンBFS探索**（再委託エッジ生成、色分け描画）

各タスクは独立してリリース可能（7-1完了後に7-2、7-3は並行可能）。

---

## パフォーマンス考慮

- `chainPaths` は文字列配列で軽量、gzip後の影響は数KB程度
- 再委託エッジはrecipientクラスタ内のみ → BFS探索のスコープは限定的
- 既存のBFS深度制限（3ホップ）がそのまま適用される
