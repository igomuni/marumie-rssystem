# 実験②プロトコル — プレイブック有無によるレポート品質の比較測定

日時: 2026-07-03
位置づけ: ロードマップ A-2。[考察メモ](20260703_0431_エージェント視点の欲しい能力と品質格差の考察.md) の仮説「**文脈の質 > データの量**（プレイブックがモデル格差を埋める）」の検証。
役割分担: プロトコル設計・判定 = Fable 5（本セッション）／被験者 = ローカルの Opus 4.8（VSCode）

## 仮説

同一モデル・同一依頼文でも、`docs/agent-playbook.md` を先に読ませた条件（B）は読ませない条件（A）より、(1) データの罠の回避、(2) レポート様式の完成度、(3) 探索効率で上回る。

## 実施手順（ユーザー操作）

1. `npm run dev` でローカルサイトを起動しておく
2. **条件A（プレイブックなし）**: VSCode で**新規セッション**を開き、下記の依頼文Aを貼り付けて実行。生成されたレポートを `docs/.ignore/20260703_実験2_条件A_防衛.md` などに保存
3. **条件B（プレイブックあり）**: **別の新規セッション**（条件Aの文脈を持ち込まない）で依頼文Bを実行。同様に保存
4. 2本のレポートをこのセッション（Fable）に渡す → 下記の評価基準で判定し、結果を記録する

重要: 条件A→Bの**順序で実施**し、同一セッションで両条件を実行しないこと（文脈の持ち込みが測定を汚染する）。

## 依頼文（コピペ用）

### 条件A（プレイブックなし）

```text
防衛関連の大型事業（予算100億円以上）を調べて、資金の流れとデータ品質の懸念も含めて
Markdownレポートにまとめてください。ローカルサイト http://localhost:3000 のAPIが使えます。
API仕様は docs/api-guide.md を参照してください。
制約: docs/agent-playbook.md と docs/tasks/20260702_1552_*.md（過去のレポート）は
読まないでください（実験のため）。
レポートは docs/.ignore/ に保存してください。
```

### 条件B（プレイブックあり）

```text
防衛関連の大型事業（予算100億円以上）を調べて、資金の流れとデータ品質の懸念も含めて
Markdownレポートにまとめてください。ローカルサイト http://localhost:3000 のAPIが使えます。
まず docs/agent-playbook.md を読み、その手順・様式に従って作業してください。
レポートは docs/.ignore/ に保存してください。
```

- テーマは「防衛」（実験①の再エネ・ユーザー実走の省エネと重複しない未使用ドメイン）
- 依頼文の差分は「プレイブック参照指示の有無」のみ。他の文言は一致させる

## 評価基準（Fable が判定に使用）

### 1. データの罠の回避（各 ○/×）

| # | 観点 | 確認方法 |
|---|------|---------|
| T1 | 「その他」と「その他の支出先」を混同していない | 両語の使い分け・注記の有無 |
| T2 | 直接受注と再委託受注を合算していない | 合算値の記載有無 |
| T3 | 年度のズレ（2024データ=2023予算実績）を明記 | 留意事項の記載 |
| T4 | 「国の全予算の約27%」の範囲限定を認識 | 「国の予算全体」と語っていないか |
| T5 | 品質スコア=形式品質の限界を正しく解釈 | スコア高=透明と誤読していないか |
| T6 | 金額の単位換算が正確（1円単位→億円） | 数値のスポットチェック |

### 2. 禁じ手の回避（各 ○/×）

| # | 観点 |
|---|------|
| P1 | 生データファイル（public/data/*.json 等）を直接読んでいない |
| P2 | 全件系APIの不要な取得をしていない（quality-scores は pids= で絞ったか） |

### 3. レポートの完成度（各 0-2点）

| # | 観点 |
|---|------|
| R1 | webView URL（サンキー図への接続）の記載 |
| R2 | 全体像テーブル（事業数・予算/支出合計・事業一覧） |
| R3 | 発見に数値の根拠が付いているか（羅列でなく主張があるか） |
| R4 | 留意事項セクションの有無と正確さ |
| R5 | 自己検証の痕跡（合計の突き合わせ等） |

### 4. 探索効率

- API呼び出し回数（セッションログから概算）: 目安10〜15回、30回超は発散
- 探索の手戻り（同じ情報の再取得・無関係なAPIの試行）

## 判定上の注意（バイアス開示）

- 判定者（Fable）はプレイブックの著者であり、条件Bに有利なバイアスを持ちうる。判定は上記の**事前固定した基準のみ**で行い、基準外の印象点は加えない
- レポートがどちらの条件か判定者に伏せる（ブラインド）ことが望ましいが、レポート自体がプレイブック様式に従うため実質判別可能。この限界は結果に付記する

## 結果の使い道

- B が明確に優位 → プレイブックを恒久運用とし、公開フェーズの llms.txt 設計（F）に反映する
- 差が小さい → 「モデルの地力の差」が支配的と判断し、プレイブックは罠回避（T系）の保険として最小限維持
- A が生データを読んでいた場合（P1×）→ 禁じ手明文化の価値の直接証拠として記録
