# 再委託チェーンのデータ化：制約と設計方針

## 目的

開発者が再委託チェーン（国→支出先→再委託先→...）のデータ化において、CSVから特定できること/できないことを正確に把握し、サンキー図用データと再委託データの分離方針を決定するため。

## データソース

| CSV | 行数 | 内容 |
|-----|------|------|
| 5-1（支出情報） | 194,133行 | ブロックごとの支出先名・金額・契約方式等 |
| 5-2（ブロックのつながり） | 22,662行 | ブロック間の資金フロー、担当組織からの直接支出フラグ |
| 5-3（費目・使途） | 33,987行 | ブロック×支出先ごとの費目・使途・金額の内訳 |
| 5-4（国庫債務負担行為） | 9,291行 | 長期契約の契約先・契約額 |

## 再委託チェーンの構造

5-2 CSVにおける間接フロー（`担当組織からの支出≠TRUE`、かつ`支出元/支出先ブロック`が存在）: **6,909件**

```text
国（担当組織）
  → ブロックA（直接支出先: 支出先1, 支出先2, ...）
       → ブロックB（再委託先: 支出先3, 支出先4, ...）
            → ブロックC（再々委託先: ...）
```

5-2 CSVはブロック単位のフローを記述するが、5-1 CSVの支出先はブロック内に複数名いるため、**どの支出先からどの支出先への再委託か**が特定できないケースがある。

## できること / できないこと

### できること

| 項目 | 詳細 |
|------|------|
| ブロック間の資金フローの全体像 | 5-2 CSVで完全に把握可能 |
| 直接支出の判定 | `担当組織からの支出=TRUE` で正確に判定 |
| 支出先1名ブロック間のチェーン特定 | 元・先ブロック両方が1名 → 「誰から誰へ」が確定 |

### できないこと（構造的制約）

| 項目 | 詳細 |
|------|------|
| 複数名ブロックからの再委託元の特定 | ブロック内に支出先が複数いる場合、どの支出先が再委託したか不明 |
| 複数名ブロックへの再委託先の特定 | 先ブロック内の複数支出先のうち、どれが再委託を受けたか不明 |
| 支出先レベルの金額按分 | ブロック単位の金額フローはあるが、支出先単位の按分根拠なし |

### 特定可能性の定量分析

**件数ベース**（間接フロー 6,909件中）:

| 区分 | 件数 | 割合 |
|------|------|------|
| 両端1名（完全特定可能） | 2,343 | 33.9% |
| いずれか複数名（曖昧） | 4,254 | 61.6% |
| いずれか0名 | 312 | 4.5% |

**金額ベース**（間接支出 28.58兆円中）:

| 区分 | 金額 | 割合 |
|------|------|------|
| 両端1名（完全特定可能） | 2.81兆円 | 9.8% |
| いずれか複数名（曖昧） | 25.72兆円 | **90.0%** |

金額ベースでは再委託チェーンの **90%が曖昧**。大規模事業ほどブロック内に複数支出先がいるため、金額の大きい再委託ほど特定できない。

### 同一支出先が複数ブロックに登場する問題

同一事業内で同じ支出先名が複数ブロックに存在するケース: **5,483件**

例:
- PID=11: 日本電気株式会社 → Block A, D, E
- PID=11: 株式会社NTTデータ → Block A, B, D

これは、同一企業が直接契約（Block A）と再委託先（Block D）の両方に登場するケース。支出先名だけではフロー上の位置が決まらず、ブロック情報が不可欠。

## 設計方針：サンキー図データと再委託データの分離

### 分離すべき理由

1. **データの粒度が異なる**: サンキー図は「事業→支出先」の金額フロー（5-1 CSV）、再委託は「ブロック→ブロック」の資金チェーン（5-2 CSV）
2. **曖昧さの扱いが異なる**: サンキー図は金額を正確に描画する必要があるが、再委託チェーンはブロック単位の概観でも価値がある
3. **同一支出先の意味が異なる**: サンキー図では支出先は1つのノードだが、再委託チェーンではブロックごとに異なる役割（直接契約 vs 再委託先）を持つ
4. **表示の枠組みが異なる**: サンキー図は「国→省庁→事業→支出先」のフロー、再委託チェーンは「事業×ブロック」を基準にした別の切り口

### 提案するデータ構造

#### 1. サンキー図用データ（`sankey3-graph.json`）

対象: 5-1 CSV + 5-2 CSVの直接支出判定
粒度: 事業 → 支出先（直接支出のみ）

```text
nodes: total, ministry, project-budget, project-spending, recipient
edges: ministry→project, project→recipient（直接支出先のみ）
```

再委託チェーンは含まない。直接支出先（129.73兆円）のフローに集中。

#### 2. 再委託チェーンデータ（`subcontract-chains.json`）

対象: 5-2 CSV（ブロック間フロー） + 5-1 CSV（ブロック内支出先）
粒度: 事業 × ブロック

```ts
interface SubcontractChain {
  projectId: number;
  projectName: string;
  ministry: string;
  blocks: BlockNode[];      // 事業内の全ブロック
  flows: BlockFlow[];       // ブロック間の資金フロー
}

interface BlockNode {
  blockId: string;           // "A", "B", "C", ...
  blockName: string;         // "株式会社NTTデータほか"
  isDirect: boolean;         // 担当組織からの直接支出か
  totalAmount: number;       // ブロックの合計支出額
  recipients: BlockRecipient[];  // ブロック内の支出先一覧
}

interface BlockRecipient {
  name: string;
  corporateNumber?: string;  // 法人番号
  amount: number;
  role?: string;             // 事業を行う上での役割（5-1 CSV）
}

interface BlockFlow {
  sourceBlock: string;       // 支出元ブロック
  targetBlock: string;       // 支出先ブロック
  note?: string;             // 資金の流れの補足情報（5-2 CSV）
}
```

この構造では:
- ブロック単位のフローを忠実に再現（曖昧な按分をしない）
- 同一支出先が複数ブロックにいてもブロックごとに区別可能
- UIで「事業→ブロック一覧→ブロック間フロー」を表示可能

### 5-3, 5-4 CSVの活用

| CSV | サンキー図データ | 再委託チェーンデータ |
|-----|----------------|-------------------|
| 5-3（費目・使途） | 不要 | ブロック×支出先の費目内訳として付加可能 |
| 5-4（国庫債務負担行為） | 不要 | 長期契約情報としてブロック×支出先に付加可能 |

5-3, 5-4 は再委託チェーンデータの `BlockRecipient` に追加属性として紐づける形が自然。サンキー図の金額フローには影響しない。

## まとめ

| 観点 | サンキー図 | 再委託チェーン |
|------|-----------|--------------|
| 主要CSV | 5-1 + 5-2（直接判定のみ） | 5-2 + 5-1 + 5-3 + 5-4 |
| 粒度 | 事業 → 支出先名 | 事業 × ブロック |
| 金額 | 正確（支出先単位） | ブロック単位（支出先按分は不可） |
| 曖昧さ | なし | ブロック内複数名の場合あり（金額の90%） |
| 同一支出先 | 1ノードに集約可能 | ブロックごとに区別が必要 |
