# 静的データの外部 offload（Cloudflare R2）設計と帯域試算

作成日: 2026-06-13（JST）
目的: 乱用・帯域枯渇の回避（データは公開のまま）。大きな静的データを Vercel から外部ストレージへ offload し、帯域上限・可用性リスクを解消する。
関連: `docs/tasks/20260612_0628_悪意あるユーザー対策_DoS設計.md`

> 数値（Vercel Hobby の含有量等）は変動するため、最終確認は公式（<https://vercel.com/docs/limits>）で行うこと。

---

## 1. 前提データ（実測サイズ）

ブラウザがクライアント側で直接 fetch するファイル（= 配信帯域を消費するもの）:

| ファイル | raw | gzip | 取得元ページ |
|---|---|---|---|
| `sankey-svg-2024-graph.json` | 13.5 MB | **1.34 MB** | /sankey-svg |
| `sankey-svg-2025-graph.json` | 14.3 MB | **1.43 MB** | /sankey-svg |
| `subcontracts-2024.json` | 29.9 MB | **4.38 MB** | /subcontracts |
| `subcontracts-2025.json` | 30.5 MB | **4.45 MB** | /subcontracts |
| `project-quality-scores-2024.json` | 4.47 MB | **0.58 MB** | /subcontracts/[id] |
| `project-quality-scores-2025.json` | 4.56 MB | **0.60 MB** | /subcontracts/[id] |
| `mof-budget-overview-2023.json` | 4.4 KB | — | /mof-budget-overview |

サーバ側（API ルート）だけが読む大物（recipient-index 31MB, rs-project-details 13-14MB, project-quality-recipients 28-29MB）は**ブラウザ配信帯域には乗らない**（`/data/*.json.gz` を直 DL された場合のみ）。

圧縮率はおおむね **raw の 1/7〜1/10**（JSON なので高圧縮）。

## 2. 帯域試算（Vercel Hobby ≈ 100 GB/月 と仮定）

「1ロード = そのページが取る主ファイル1本」とした概算の月間上限。

### ケースA: 圧縮配信される場合（gzip 相当が wire を流れる）

| ファイル | wire | 月間ロード上限（100GB ÷ wire） |
|---|---|---|
| sankey graph | 1.34 MB | **≈ 74,600 回** |
| subcontracts | 4.38 MB | **≈ 22,800 回** |
| quality-scores | 0.58 MB | ≈ 172,000 回 |

### ケースB: 非圧縮で配信される場合（raw が wire を流れる＝最悪値）

| ファイル | wire | 月間ロード上限 |
|---|---|---|
| sankey graph | 13.5 MB | **≈ 7,400 回** |
| subcontracts | 30 MB | **≈ 3,400 回** |
| quality-scores | 4.5 MB | ≈ 22,200 回 |

**差は約10倍**。どちらに転ぶかは「配信時に gzip/brotli が効くか」に依存する。

### Vercel の圧縮について（重要・要確認）

- Vercel は `Accept-Encoding` 付きリクエストに対しテキスト系応答を自動で gzip/brotli 圧縮する。効けば**課金はケースA（圧縮後サイズ）**に近づく。
- ただし**非常に大きいファイルは自動圧縮がスキップされる閾値**が存在する可能性がある（プラットフォーム実装依存）。13〜30MB クラスが圧縮対象外だとケースB に転落する。→ **実機で `content-encoding` レスポンスヘッダを確認すべき**。
- 確実に圧縮を効かせたいなら、後述の通り**事前 gzip 済みバイトを `Content-Encoding: gzip` で配信**するのが安全（自動圧縮に依存しない）。

### ケースC: Cloudflare R2 へ offload した場合

- **egress（転送）課金なし**＝Vercel の帯域上限と無関係。配信量による「停止」リスクが消える。
- R2 の関連リミット（要確認・概算）:
  - Class B（読み取り/GET）オペレーション: 約 1,000万/月 まで無料 → 本規模では実質無制限。
  - ストレージ: 10 GB まで無料。配布データ合計 ≈ 11 MB（gz）なので余裕。
- 実質、**配信量の上限がボトルネックでなくなる**。残る関心はレイテンシ（次節）と運用。

## 3. ロード遅延（レイテンシ）のリスクと対策

外部 offload は帯域を解決するが、**初回ロードのレイテンシ要因**が増える。主なリスク:

| # | リスク | 内容 | 対策 |
|---|---|---|---|
| R1 | 別オリジン接続コスト | データ用ドメインへ別途 DNS 解決 + TLS ハンドシェイク（初回 +50〜150ms） | `<link rel="preconnect">` / `dns-prefetch` を `app/layout.tsx` に追加 |
| R2 | **r2.dev は本番不可** | R2 の公開 `*.r2.dev` ドメインは**レート制限あり・CDN キャッシュされない** | **独自ドメインを Cloudflare ゾーンに接続**し Cache Rules でキャッシュ |
| R3 | CDN キャッシュミス（コールド） | 低トラフィックだとエッジ PoP のキャッシュが揮発し、オリジン(R2)から再取得＝数MBの取得が遅延 | 長期 `immutable` キャッシュ + Cache Rules で TTL を長く。トラフィック少なくても warm を維持 |
| R4 | 非圧縮配信での肥大 | R2 は自動圧縮しない。raw 13〜30MB をそのまま配ると体感が大幅悪化（特にモバイル） | **事前 gzip 済みを `Content-Encoding: gzip` + `Content-Type: application/json` で格納**（ブラウザ透過解凍） |
| R5 | CORS プリフライト | カスタムヘッダ付き cross-origin fetch は OPTIONS 往復が増える | fetch はシンプルに（カスタムヘッダ無し）保ち preflight を回避。バケットの CORS は許可オリジンを明示 |
| R6 | 可用性の二重化 | Vercel が生きていても R2/Cloudflare 障害でデータ取得失敗 | fetch のエラーハンドリング/リトライ・ローディング表示を堅牢化 |
| R7 | 地理 | 現状 Vercel `hnd1`(東京)で日本ユーザーに近い。R2+CF も東京 PoP あり | R2 のロケーションヒントを APAC、CF キャッシュで近接配信 |

要点: **「独自ドメイン + Cache Rules + 事前 gzip + Content-Encoding」をセットで満たせば、warm 時は Vercel 直配信と遜色ない**。逆に r2.dev 直・非圧縮だと遅くなりうる。

## 4. 実装プラン

### コード側（実装可能）

1. `NEXT_PUBLIC_DATA_BASE_URL` を導入。未設定なら同一オリジン `/data`（ローカル `npm run dev` 維持）、本番は R2 独自ドメイン。
2. `dataUrl(path)` ヘルパを `app/lib/` に新設し、`/data/*.json` を叩く箇所（`app/sankey-svg/page.tsx` ほか）を一括で差し替え。
3. パイプライン末尾に `scripts/upload-data-to-r2.sh`（S3 互換 CLI / rclone / wrangler）。**gzip 済みバイトを `Content-Encoding: gzip`・`Content-Type: application/json`・`Cache-Control: public, max-age=31536000, immutable` で put**。
4. `app/layout.tsx` にデータドメインへの `preconnect` を追加（R1 対策）。

### インフラ側（要手作業）

- R2 バケット作成 → **独自ドメインを Cloudflare に接続**（r2.dev は使わない）→ Cache Rules でキャッシュ → CORS で自サイトオリジン許可 → アップロード用 API トークン発行。

## 5. 結論

- **帯域の根本解決は R2 offload**（egress 無料でケースの 100GB 上限から解放）。
- ただし **R4（事前 gzip 配信）と R2(独自ドメイン+キャッシュ)** を満たさないと、帯域は解決してもレイテンシが悪化する。両方セットで実施する。
- まず実機で **Vercel が現状そもそも圧縮配信しているか（`content-encoding` ヘッダ）** を確認すれば、「そのまま圧縮を効かせるだけで当面しのげる（ケースA）」のか「offload が必要」かの判断ができる。
