## 0. 解こうとしている問題（1文で）

**「この支出先は誰か」にデータが一意に答えられないため、集計を間違え（e-Tax調査での NTTデータ別番号分裂が実例）、かつ利用者がその誤りに気づく手段（UI上の法人番号表示）が無い。**

設計書の「目的」はこれを概ね捉えているが、以下の通り**性質の異なる2系統が1つの再設計に束ねられており、実行順と切り出しを明確にしないと迷走リスクがある**。

---

## 1. 仕分け結果

| 設計書の要素 | 分類 | 判定 |
|---|---|---|
| /quality 明細に法人番号実値（方針4） | **問題の可視化**（低リスク・射影変更のみ） | ✅ 最優先。誤記載を「見える」ようにする |
| Sankeyノードに法人番号集合＋サイドパネル表示（方針2・3） | **問題の可視化**（中リスク） | ✅ 優先。ただしJSONサイズ懸念あり（後述F4） |
| チェックディジット検証・houjin.db突合・名寄せ辞書（方針1前半） | **問題そのものの解決**（高リスク・数値が動く） | ✅ 本丸。ただしキー安定性の分析が欠落（F2） |
| 親子・グループ会社のグループ計（方針1後半） | **別の問題**（分析の利便機能） | ⚠️ 切り出し推奨（F1） |
| houjin.db正式名を代表名に採用 | **別の判断**（UX変更・全画面に波及） | ⚠️ 未決のまま分離してよい |

**核心の指摘: 「可視化」（方針2/3/4）は「名寄せ改善」（方針1）に依存しない。** 元データが持つ法人番号をそのまま表に出すだけで実装でき、しかも誤記載の実態を画面で確認できるため、**名寄せ辞書（人手確定）の作業台になる**。順序は「可視化 → 解決」が正しい。設計書はこの依存方向を明示しておらず、一括実装と読める。

---

## 2. 指摘事項（重要度順）

### F1: グループ計はスコープ外に切り出すべき（スコープクリープ）

- グループ計（親子会社の合算別掲）は「識別の誤り」ではなく「分析の利便」であり、**本設計が解く問題ではない**。e-Tax事故の原因は誤記載分裂であって、グループ非表示ではない。
- さらに設計書は「支店・支社は既に `parentName` を持つ。これを拡張し」とするが、**`parentName` は同一法人内の支店・支社（例: ○○株式会社△△支店 → ○○株式会社）の解決であり、別法人の親子関係（例: NTTデータSMS → NTTデータ）とは概念が異なる**。前者は名寄せの一部、後者は資本関係の知識が必要。
- **houjin.db は親子関係を一切持たない**（corporate_number/name/住所のみ）。グループ計を入れるには外部の資本関係データか手作業辞書が丸ごと必要で、隠れコストが大きい。
- **推奨**: 方針1から削除し「`groupKey` フィールドを型に予約するだけ・データ投入は将来課題」へ格下げ、または対象外セクションへ移動。

### F2: `buildRecipientKey` 変更の影響範囲（キー安定性）が未分析

- チェックディジット検証の追加や誤記載突合は、**既存キーの付け替え**を意味する（無効化された番号 → `name:` キーへ移動、突合された番号 → 代表キーへ統合）。
- `/api/recipients/[key]` の key はエージェントのレポート・プレイブック・既存リンクに**既に流通している**。キーが変わると旧URLが404になる。
- 検証観点に「totals の整合」はあるが「キーの互換性」が無い。**推奨**: 「旧キーでの参照に対するリダイレクト/`mergedInto` 応答」など互換方針を設計に1項追加し、検証観点にも「旧キー参照が案内付きで解決すること」を加える。

### F3: 成功条件が事故と接続されていない

- 検証観点は良いが、「何をもって解決とするか」が明文化されていない。**推奨**: 成功条件を「`houjin_quality` の6分類それぞれに処置（自動補完/辞書突合/フラグ表示/放置）が定義され、NTTデータ事例が API 応答上『統合』または『明示フラグ』になること」と定義する。649件（同名別番号）・557件（補完候補）・1,868件（表記揺れ）という母数が既にあるので、処置率で進捗を測れる。

### F4: Sankeyノードへの法人番号「集合」はサイズ影響の見積りが必要

- グラフJSONは全支出先（26,823件）を含む。各ノードに `{corporateNumber, name, amount}[]` を持たせる案は肥大化リスク（現状ですら .gz 11MB/展開96MB 制約下）。
- **推奨**: グラフ本体には「代表法人番号＋内包件数」の2フィールドのみ持たせ、内訳リストはサイドパネル選択時に `/api/recipients` 系から引く構成を第一候補に。設計書の「ノードに集合を保持」は代替案に格下げ。

### F5: `houjin_quality` 6CSVの生成スクリプトが `scripts/` に存在しない

- 設計は「この分析を土台に確定作業へ進める」とするが、CSVは2026-02-23生成のアドホック産物で再現手段が無い。houjin.db 更新（updated_at 2026-02-20）にも追随できない。
- **推奨**: 実装の最初のタスクを「突合分析の `scripts/` 恒常化（6分類の再生成）」と明記する。これが無いと辞書確定作業の入口が塞がる。

### F6（軽微）: 「複数ヒットは住所で絞る」はRS側に住所が無ければ機能しない

- 設計書自身が「RSデータに住所が無い行は補完不能」と書いており整合しているが、`06` の557件のうち一意ヒットが何件かを実装前に数えれば、この施策の効果量が先に分かる（1クエリで済む）。効果が小さければ優先度を落とせる。

## 3. 推奨する実行順（迷走防止の背骨）

```text
Phase 1（可視化・数値を動かさない）:
  /quality 明細に cn 実値 → サイドパネルに代表法人番号＋内包件数
  ＝ 誤記載が画面で見える。名寄せ判断の材料が揃う
Phase 2（分析の恒常化）:
  houjin_quality 突合の scripts/ 化（6分類の再生成・houjin.db照合）
Phase 3（解決・数値が動く）:
  チェックディジット検証 → 一意ヒット自動補完 → 誤記載突合辞書
  ＝ キー互換方針（F2）とセットでのみ実施
切り出し: グループ計（F1）・代表名のhoujin正式名化 は別タスク
```

Phase 1〜2 は現行の集計数値を一切変えないため安全に先行でき、Phase 3 の辞書作業は Phase 1 の画面と Phase 2 の分類CSVを道具として使う。この依存関係が「やりたいこと（表示）」と「懸念（名寄せ）」を混ぜずに進める骨組みになる。

---

## 4. 総評

設計書は現状把握（3経路で法人番号の扱いが違う・元データは番号を持つのに射影で捨てている）と houjin.db 3観点の切り分け（Alias で解けるのは表記揺れのみ、同名別番号は辞書必須）が正確で、問題の同定は正しい。弱点は**実行順の未定義（可視化と解決の依存方向）・グループ計の混入・キー互換の欠落**の3点に集約される。F1〜F3 を設計書に反映すれば実装に進める品質になる。
