```

  │    ├─ Cブロック「民間企業」
```

### カテゴリ1: 政府機関（辞書の誤判定）

地方整備局・本省等が `valid=False` になっている根本原因は、`recipient_dictionary.csv` の validity 判定が**法人番号DBとの名称一致**に依存していること。

```
中国地方整備局  CN=2000012100001  db_name=国土交通省  name_cn_match=False → valid=False
国土交通本省    CN=2000012100001  db_name=国土交通省  name_cn_match=False → valid=False
```

地方整備局は法人番号上は「国土交通省」の一部として登録されており、「中国地方整備局」という名称は法人DBに存在しない。そのため名称照合に失敗し invalid 扱いになっている。

しかし `public/data/dictionaries/field_office_names.csv` にはこれらが正しく収録されており、**実在する政府機関であることは確認済み**。辞書の設計ミスによる偽陽性 invalid である。

### カテゴリ2: 個人(X)（用地取得匿名化）

道路用地取得に伴う土地所有者への補償費の支払い。個人情報保護の観点から「個人(イ)」「個人(ロ)」...「個人(ヲ)」のように日本語アルファベットで匿名化される。

- すべてブロックD「個人等」に含まれる
- 法人番号は当然存在しない（個人の土地所有者）
- `個人`（完全一致）は不透明辞書で level=2 に分類されているが、`個人(イ)` はマッチしない
- 辞書には `valid=False` として登録済み（匿名化としての認識はある）

**解釈の問題**: これは「データ品質が悪い」のではなく、「用地取得における正当な匿名化手続き」。道路事業固有の支出パターンであり、スコアリングで減点対象とすべきか再考が必要。

### カテゴリ3: JV・設計共同体（辞書未対応）

公共工事では、大型案件で複数の建設会社・コンサルが一時的に共同体を結成して受注する。

- **特定建設工事共同企業体**: 建設工事向け（例: `東京外環中央JCT北側Aランプシールド工事清水・竹中土木特定建設工事共同企業体`）
- **設計共同体**: コンサルタント業務向け（例: `道路新産業開発機構・長大設計共同体`）

構成企業はそれぞれ法人として存在するが、JV・共同体そのものは法人DBに一般に登録されない一時的な契約体。名称がプロジェクトごとに異なるため辞書収録が困難。

### カテゴリ4: その他（正当invalid）

各ブロックに2件ずつ計12件。不透明支出先辞書にも正しくマッチ（`その他` exact, level=1）。これは真の問題。

## 根本原因の整理

```
辞書の validity 判定 = 法人番号DBとの名称一致
    ↓ 問題点
地方整備局は法人DB上「国土交通省」として登録
  → 「中国地方整備局」は名称不一致でinvalid
    ↓ 実態
field_office_names.csv に実在が確認されているが、
この情報がrecipient_dictionary.csvのvalidフラグに反映されていない
```

辞書辞書化パイプラインの各段階:
1. `public/data/dictionaries/field_office_names.csv` → 地方整備局は正しく収録 ✓
2. `build-recipient-dictionary.py` → 法人番号DB照合でvalid=Falseを付与 ✗
3. `recipient_dictionary.csv` → valid=Falseで出力
4. `score-project-quality.py` → invalid として軸1スコア算出

辞書カテゴリ（field_office_names, ministry_names 等）に収録されていれば、法人番号照合に依存せず **valid=True** とする仕組みが必要。

## 示唆されること

### 道路事業特有の「正当な支出先パターン」が辞書でカバーされていない

1. **地方整備局モデル**: 本省から地方整備局に資金を配分し、地方整備局が民間・個人等に直接発注する構造が道路直轄事業の標準形
2. **用地取得個人**: 道路用地取得の補償費支払いでは匿名化が義務。これをdataの品質低下として扱うのは不適切
3. **JV・共同体**: 大型工事の入札では共同体が標準的。名称が都度異なるため辞書化は困難（パターンマッチングが必要）

### 軸1スコアの設計上の限界

現行の軸1は「辞書に登録済みかつvalid」なものだけを評価する。辞書未整備・誤判定の領域が大きい事業では、実際のデータ品質よりも大幅に低いスコアになる可能性がある。

道路事業(直轄・改築等)の場合、真に「不透明な支出先名」は `その他` の12件（8.7%）のみ。軸5は82.6点と正しく評価できているが、軸1の偽陽性invalidが総合スコアを約10点以上引き下げている。

## 今後の対応候補

1. **辞書パイプラインの修正**: `field_office_names.csv` 等の「辞書カテゴリ収録済み」を valid 判定に優先適用する仕組みを `build-recipient-dictionary.py` に追加
2. **個人(X)パターンの扱い見直し**: 軸1の評価対象から除外、または用地取得向けの特別カテゴリとして valid 扱いにする
3. **JV・共同体の正規表現マッチング**: `・` 区切りの共同体名をパターン認識して valid として扱う新しい辞書ルールの検討
