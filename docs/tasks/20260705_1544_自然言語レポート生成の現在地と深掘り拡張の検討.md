## 1. 現在地 — 「自然言語でレポートを頼む」で出来ていること

実レポート5本以上の実績から、次のループは**安定して機能している**と評価できる。

| 能力 | 実証 |
|------|------|
| 自然言語 → 絞り込み → 図の提示 | `/api/sankey/query` summary での反復調整 + `links.webView`。全レポートが図つき |
| テーマ横断の深掘りチェーン | search → query → details / quality(pids) / subcontracts / recipients の標準ルートが API だけで閉じる |
| 透明性の「質」の分類 | 「その他」67%の給付型（こども家庭庁）vs 企業名まで追える調達型（e-Tax）という対比が可能に |
| 集中構造の定量化 | e-Tax = NTTデータ親会社単独85%（ルート額を分母にした正しい按分）。二重計上の罠2種（直接/再委託、ルート/ベンダー）は playbook で体系化済み |
| 名寄せの信頼性 | #243 で法人番号キー + houjin.db 裏取り（誤記載統合62件・補完443件・チェックディジット検証・redirects）。横断集計は `/api/recipients/[法人番号]` 1コール |
| 年度間比較（手動） | e-Tax レポートが 2024/2025 の2回取得で KSK +221億円（+25%）を発見 |
| 計上のねじれの発見 | 国税システム群が「デジタル庁一括計上・予算0円」で府省庁フィルタから漏れることをレポート自身が特定 |
| モデル非依存の品質 | 実験②で playbook 効果を実証（10/10 vs 7/10）。playbook は Opus 側の実戦知見（名寄せ・二重計上）で自己増殖的に強化されており、**使う側が育てる文書**として機能し始めた |

要約すると「**レビューデータの内側で答えが出る問い**」——どこにいくら流れたか、誰に集中しているか、どこで追跡が途切れるか——には、実用レベルで到達している。

## 2. 参考URL（PDF）アプローチの見送りについて

ユーザー判断（参考URLを深掘り起点にするのは筋が悪い）に**同意**する。理由の整理:

- 参考URLの多くは PDF（概算要求資料・基金シート等）で、抽出コストがトークン・実装とも高く、表の構造化品質も不安定
- 事業ごとに形式がバラバラで、プレイブック化（再現性の担保）ができない
- キャッシュ・検証が利かず「数値はAPI応答から転記」という現行の信頼性規約を壊す

**代替の最小運用**: エージェントは参考URLを**読まずに**、レポート末尾に「人間向けの外部参照リスト」として転記するだけに留める（`project-details` に URL は既にある。ゼロコストで人間の深掘り導線にはなる）。

## 3. 深掘りのために有ると良さそうなもの（優先順）

### 3-1. 年度間比較の一級化（最優先・データ内で完結）

e-Tax レポートは2年度を手動で2回引いて突き合わせたが、突き合わせはエージェントが最も間違えやすい工程。`/api/sankey/query` に **`compareYears=2024,2025`** 相当のモードを足し、同一 `SankeyQuery` の両年度 summary + 差分（事業別増減・新規/消滅・支出先別増減）を1応答で返す。

- 価値の実証済み: KSK +221億の類の発見はレポートの核になる
- 実装は薄い: 両年度の graph は loader にキャッシュ済み。summarize を2回呼んで pid/支出先キーで突合するだけ
- 「予算年度=事業年度-1」の注記を差分応答の notes に必ず同梱（年度ズレの罠×2年度分）

### 3-2. 事業概要の全文検索（探索の入口拡大）

事業**名**検索だけでは、名前が抽象的な事業（「〇〇推進費」）と**計上のねじれ**（国税システム群がデジタル庁計上）を取りこぼす。`/api/search/projects` に `scope=details`（purpose / overview / currentIssues を対象に含める）を追加する。

- e-Tax 探索は事業名の複雑な正規表現（8分岐）で凌いだが、概要検索なら「KSK」「国税」で素直に届く
- project-details は loader 化済みで、単純部分一致から始められる（形態素・embedding は不要。まず部分一致で実験）

### 3-3. gBizINFO 接続（外部接続で唯一筋が良いもの・実験扱い）

PDF と違い、gBizINFO は**法人番号キーの構造化 REST API**で、#243 の名寄せ強化により接続点（検証済み法人番号）が整った。`links.external.gbizinfo` は既にあるが、エージェントが読むには API キー登録が必要。

- 得られる文脈: 資本金・従業員数・所在地・**国からの補助金交付実績**（横断）
- 使い道: 「ベンダー集中」レポートに企業規模の文脈（例: 事業費85%を受注する企業の規模感）、執行団体の壁の先の一部（補助金交付先として gBizINFO 側に載る場合がある）
- ロートークン: 1法人=1コール・構造化JSON。プレイブックに「上位N社のみ引く」の上限を書けば発散しない
- 位置づけ: ローカル実験の追加ツールとして試し、価値が出たら recipient API 応答へのキャッシュ埋め込みを検討（外部依存を本体に入れるかは公開判断と同様に慎重に）

### 3-4. 集中度・比較の定型指標を summary に同梱（軽量）

e-Tax でエージェントが手計算した「上位1社シェア」「上位3社シェア」を `/api/sankey/query` summary の recipients に追加（`topShare1` / `topShare3`）。ベンダー集中の一次スクリーニング（「集中度の高い事業を探して」という自然言語要求）が summary だけで回るようになる。

### 3-5. レポートライブラリの規約化（仕組み・実装なし）

レポートが5本を超え、**過去レポートの相互参照が新レポートを強化する**現象が既に起きている（e-Tax がこども家庭庁レポートと対比）。規約を2つ足すだけで資産化が進む:

1. 置き場所の分離: 調査レポートは `docs/reports/`（設計メモの docs/tasks と分ける）+ プレイブックから索引参照
2. **再現性ブロック**: レポート末尾に使用した `SankeyQuery` JSON と主要APIコールを記載する（実験②の改訂で回数記録は必須化済み。JSON まで残せば年度更新時に同一クエリで再実行し「差分レポート」を機械的に作れる——3-1 と接続する将来価値）

### 見送り（今回は着手しない）

- 参考URL/PDF 深掘り（§2）
- 全事業 embedding 検索（3-2 の部分一致で足りるかを先に確認）
- プロベナンス（元CSV行への参照）: 価値は高いが実装が重い。名寄せ再設計（#243）の rawCorporateNumber 保持が部分的に同じ方向を向いており、必要になった時に再検討

## 4. 推奨する着手順

1. **3-1 年度間比較**（実装小・実証済み価値・エージェントの誤り源を除去）
2. **3-2 概要全文検索**（入口の取りこぼし対策。デジタル庁一括計上の教訓に直接応える）
3. **3-4 集中度指標**（3-1 と同じ summary 改修のついでに）
4. **3-5 レポートライブラリ規約**（ドキュメントのみ）
5. **3-3 gBizINFO**（外部依存のため、1〜4 が回った後の実験として）
