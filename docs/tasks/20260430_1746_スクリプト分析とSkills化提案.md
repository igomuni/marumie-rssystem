#### 2. 分析・調査スクリプト（18個）- 手動実行

**分析スクリプト** （データ問題の深掘り調査）:
- `analyze-invalid-recipients.py` → 不正な支出先の特定
- `analyze-label-coverage.py` → ラベル付けカバレッジ分析
- `analyze-mof-*.ts` （5種） → MOF予算分析（特別会計、歳入、繰入など）
- `analyze-rs-*.ts` （3種） → RS システム分析（MOF比較、特別会計、年度別）
- `analyze-other-recipients.ts` → 「その他」ノード分析
- など

**調査スクリプト** （特定データの品質検査）:
- `investigate-houjin-quality.py` → 法人番号照合品質調査
- `investigate-unlabeled-with-houjin.py` → 未ラベル・法人番号ありの調査

**用途**: 一度限りの調査 → レポート生成。Claude に分析依頼を掛けるときの手段として有用。

## Skills化の提案

### Tier 1: 高優先度（車輪の再発明を削減）

#### 1-1. **sankey2-update Skill** - Sankey2の完全パイプライン

**目的**: `generate-sankey2` + `compute-sankey2-layout` の両方を順序立てて実行

```markdown
前提条件:
- 5-1, 5-2 CSVが data/year_2024/ に存在

実行ステップ:
1. npm run generate-sankey2 → sankey2-graph.json (~25MB)
2. npm run compute-sankey2-layout → sankey2-layout.json (~45MB)
3. npm run compress-data → .gz ファイル圧縮
4. ファイルサイズ検証
```

**効果**: 現在は手動で3ステップ実行する必要 → 1ステップに簡略化

**所要時間**: 20行程度の Skill markdown ファイル

---

#### 1-2. **sankey-svg-update Skill** - SVG Sankey データ更新

**目的**: `generate-sankey-svg` をラップ（年度選択対応）

```markdown
パラメータ: year (2024 | 2025 | ...)
実行:
1. npm run generate-sankey-svg-${year}
2. ファイルサイズ検証 (目安: ~XX MB)
3. git diff で変更確認
```

**効果**: `/sankey-svg` ページの定期更新を自動化

---

#### 1-3. **quality-score-update Skill** - 品質スコア更新

**目的**: `score-project-quality` をラップ

```markdown
実行:
1. npm run score-quality (→ quality.json 生成)
2. 変更箇所を確認・コミット
```

**効果**: 支出先品質スコアの定期更新を自動化

---

### Tier 2: 中優先度（新規ツール作成）

#### 2-1. **year-migration Skill** - 新年度への全体移行

**目的**: 年度切り替え時（例: 2024→2025）に全パイプラインを自動実行

```markdown
パラメータ: from_year (2024), to_year (2025)

実行順序（順序重要）:
1. ZIP ファイル確認 (data/download/RS_${to_year}/)
2. CSV 正規化 (normalize --year ${to_year})
3. 構造化JSON生成 (generate-structured)
4. Sankey2 データ生成 (generate-sankey2)
5. Sankey2 レイアウト計算 (compute-sankey2-layout)
6. Sankey-SVG データ生成 (generate-sankey-svg)
7. 品質スコア計算 (score-quality --year)
8. 再委託データ生成 (generate-subcontracts)
9. 全ファイル圧縮 (compress-data)
10. Git コミット
```

**效果**: 10個のステップを 1 コマンドで実行。エラーハンドリング・確認ダイアログ付き。

**所要時間**: 実装 2-3 時間（複雑さ中）

---

#### 2-2. **data-integrity-check Skill / CLI** - 生成データの検証

**新規実装対象**: Python スクリプト

```markdown
検証項目:
1. ファイルサイズ（目安値と比較）
2. JSON 構文チェック
3. ユニーク数（府省庁、事業、支出先）
4. 合計金額（期待値との照合）
5. ハッシュ値記録（前回との差分検知）
6. 行数・レコード数確認

出力: HTML レポート or CLI 表示
```

**効果**: パイプライン実行後の品質保証自動化

**所要時間**: 新規実装 1-2 時間

---

#### 2-3. **analysis-runner CLI** - 分析スクリプトのインタラクティブランナー

**新規実装対象**: Python または TypeScript スクリプト

```markdown
機能:
1. 利用可能な分析スクリプト一覧表示
   - Category 別リスト (analyze-mof-*, analyze-invalid-*, など)
   - 説明・出力形式

2. ユーザーが実行したいスクリプトを選択
   - 対話型メニュー（fzf 風）

3. パラメータ入力（必要に応じて）
   - 年度選択、フィルタ条件など

4. 実行 & 結果表示
   - 出力ファイルの自動判定
   - CSV なら head 表示、JSON なら jq フォーマット
```

**効果**: 
- 分析スクリプトの発見性向上
- ユーザーが「どんな分析ができるか」を自動表示
- Claude に分析依頼を掛けるハードルを下げる

**所要時間**: 新規実装 2-3 時間

## 参考: 分析スクリプトの活用パターン

**現在の課題**: 分析スクリプト 18 個があるが、「どんな分析ができるか」が不明瞭

**提案**: analysis-runner CLI を作成すれば：
```bash
$ npm run analysis
# → インタラクティブメニュー
# 1. Analyze MOF special accounts
# 2. Analyze invalid recipients
# 3. Investigate houjin quality
# ...
```

これにより Claude も「分析スクリプトが使える」ことを認識できる。
