## 目的

**記者・研究者・エージェントが「広報にいくら使われている？」「システム改修を受注しているのは誰？」という使途起点の問いに、データがあるのに答えられない状態を解消するため。**

使途テキストは `project-quality-recipients-{year}.json` に**支出行単位**で既に存在する（`role`=事業を行う上での役割、`cc`=契約概要、金額 `a2`、再委託深度 `d`、約7.5万行/年）。欠けているのは検索・集計の入口だけ。

## 設計判断1: 専用インデックスのパイプラインは新設しない

元検討（20260705_1720 §1-1）は `scripts/generate-spending-search-index.ts` の新設を想定していたが、調査の結果**不要**と判断する:

- データ源はロード可能な JSON（展開後 30MB/年・gz 3.3MB）で、既存の quality-recipients API が同ファイルをメモリキャッシュ済み。graph（96MB）をキャッシュする既存 loader 群と同水準
- 7.5万行 × 2フィールドの正規化部分一致は in-memory 線形走査で十分速い（正規化済みテキストをローダで事前計算しキャッシュする）
- パイプライン新設は「生成物が増える・年度更新手順が増える・Git 管理対象が増える」コストがあり、検索が遅いと観測されてから導入すれば足りる

よって実装は「**ローダ抽出 + Pure 検索関数 + API route + チャットツール**」の既存パターン（WP1-1 と同型）に収まる。

## 全体構成

| Layer | ファイル | 役割 |
|-------|---------|------|
| Domain | `app/lib/api/quality-recipients-loader.ts`（新規） | 既存 route 内のロード処理を抽出（.json/.gz フォールバック・年度別キャッシュ）。検索用の正規化済みテキストも初回構築してキャッシュ |
| Domain | `app/lib/search/spending-search.ts`（新規・Pure） | role/cc の正規化部分一致（既存 `normalizeQuery` 流用）+ 集計 |
| API | `app/api/search/spending/route.ts`（新規） | `GET /api/search/spending?q=&year=&limit=&offset=` |
| AI | `sankey-chat-agent.ts` | 新ツール `search_spending`（クランプ済み応答） |
| 既存変更 | `app/api/quality-scores/recipients/route.ts` | ロード処理を新ローダへ委譲（応答不変） |

## API 仕様（案）

`GET /api/search/spending?q=広報&year=2024&limit=20&offset=0`

応答:

- `metadata`: 既存 search 系と同じ流儀（query・totalHits・limit/offset・notes）
- `aggregate`（マッチ全体の集計。ページングに依存しない）:
  - `hitCount`: マッチ行数
  - `projectCount`: マッチ行が属する事業数
  - `amountDirect` / `amountSubcontract`: マッチ行の金額合計を**直接支出（d=0）と再委託（d>0）に分けて**集計
  - `topProjects`: 事業別合計の上位10件（pid・事業名・府省庁・金額）
- `items`（ページング対象のマッチ行）: pid・事業名・支出先名・法人番号・金額・深度・`matchedIn`（'role' | 'cc'）・マッチフィールドの抜粋（前後含め120字程度）・links（事業詳細・sankeyView）

### データの罠への対処（notes に明記）

1. **直接/再委託の二重計上**: quality-recipients は再委託行（d>0）を含む。直接行と単純合算すると資金の通過分が二重に数えられるため、`aggregate` は必ず分離して返し、notes で「合計はamountDirectを基本とし、amountSubcontractは内訳の重複を含みうる」旨を明示する（agent-playbook §2 と同じ規律）
2. 金額は1円単位（既存規約どおり）
3. 検索対象は role / cc のみ。`chain`（ブロック名連結）は経路情報でありテキストとしてはノイズが多いため対象外（設計判断2）

### バリデーション・ガード

- `q` 必須・2文字以上（1文字は総当たりに近くなるため 400）。regex は v1 では受けない（部分一致のみ）
- 公開ガード: 既存 `/api/search/projects` と同じ扱い（現状の search 系と有効化条件を揃える。専用ガードは追加しない）

## チャットツール `search_spending`

- 引数: `{ q: string }`。応答は items 上位10行 + aggregate（既存クランプ規律 4,000字目安）
- ツール description: 「支出の使途テキスト（役割・契約概要）を横断検索する。『広報にいくら使われている？』『◯◯を受注しているのは誰？』のような使途起点の質問に使う。金額集計は直接支出と再委託を分けて読むこと」
- システムプロンプト更新: 「フィルタで表現できない条件」の説明で、使途を「フィルタ化はできないが **search_spending で調べることはできる**」に更新（現在は一律「できない」と案内している）

## レイヤー整合チェック

| ルール | 本設計での扱い |
|--------|---------------|
| `scripts/` は CSV 処理のみ | 触れない（パイプライン新設なしの判断による） |
| `app/lib/` は Pure・HTTP/React 禁止 | ローダは既存 loader 群と同格（fs+キャッシュ）。検索関数は Pure |
| `app/api/` はハンドラのみ | route はパラメータ解釈・応答整形のみ |
| 既存応答の互換 | quality-scores/recipients はローダ委譲のみで応答不変 |
| 「その他」の扱い | 支出行の名称はそのまま返す（名寄せ・除外はしない。集計は行単位） |

## 検証方針

1. tsc / lint、既存 quality-scores/recipients API の応答不変（回帰）
2. curl 実測: 「広報」「システム改修」「コールセンター」で件数・金額・上位事業が妥当か、応答時間が実用域（数百ms以内）か
3. チャット: 「広報にいくら使われてる？」で aggregate の数値が直接/再委託を区別して回答されるか（ハーネスにシナリオ1件追加）

## 未決事項（ユーザー判断）

1. `aggregate.topProjects` の上位件数（案: 10件）と items の既定 limit（案: 20件）
2. チャットツール露出を同一PRに含めるか、API先行で分けるか（案: 同一PR。WP3-1/3-2 と同じ進め方）
3. 検索応答の抜粋に `role`/`cc` 全文を返すか抜粋120字か（案: 抜粋。全文は行あたり最大数百字ありトークン効率が悪い）
