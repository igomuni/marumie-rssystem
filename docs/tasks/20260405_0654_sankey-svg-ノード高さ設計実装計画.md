# /sankey-svg ノード高さ設計見直し 実装計画

> 作成日: 2026-04-05  
> 更新日: 2026-04-05（設計方針確定）  
> 対象Issue: igomuni/marumie-rssystem#119  
> 対象ファイル: `app/lib/sankey-svg-filter.ts`

---

## 目的

ユーザーが /sankey-svg を見たときに、「同じ金額 = 同じ高さ」という一貫したスケールでノードが表示され、図の読み取りと解釈が正確にできるようにする。また、予算・支出データの欠損事業（予算=0・支出=0）をサイドパネルから確認できるようにする。

---

## 現状の問題点

[docs/tasks/20260402_1153_sankey-svgノード高さ設計考察.md](20260402_1153_sankey-svgノード高さ設計考察.md) から抽出した、修正が必要な箇所。

### 問題1: `__agg-project-budget` の高さが予算データでない

```text
現在: value = otherProjectWindowTotal（ウィンドウ内支出合計）
あるべき: 集約対象事業の予算額合計
```

`filterTopN` の L147:
```typescript
nodes.push({ id: '__agg-project-budget', ..., value: otherProjectWindowTotal, ... });
```

集約対象事業（`otherProjects`）の予算ノードは `nodeById` に存在するため、それらの `value` を合計できる。

### 問題2: 集約ノードの `layoutCap` が複雑で不安定

`projectLayoutCap` の計算:
```typescript
const projectLayoutCap = Math.max(1, minTopProjectWindowValue > 0 ? minTopProjectWindowValue * topProject : otherProjectTailTotal);
```

- 「最小TopNプロジェクトのウィンドウ支出 × TopN件数」という推計値
- TopNが少ないと過大・支出がまばらだと過小になり得る
- `skipLinkOverride:true` を集約ノードにも適用することで、この推計キャップが不要になる

### 問題3: 支出先ノードの高さが全エッジ合計（ウィンドウ外含む）

```text
現在 (L154): value = allRecipientAmounts.get(rid)（全省庁・全事業からの流入合計）
あるべき: ウィンドウ内リンクの合計（= windowRecipients の値）
```

支出先ノードはウィンドウ内のリンクだけを受け取るが、`allRecipientAmounts` はオフセット前・ウィンドウ外事業も含む全エッジ合計になっている。これによりノード高さとリンク幅が不一致になる。

---

## データの確認

graph データ (`sankey3-graph.json`) 上の値：

- `project-budget` の `value` = 予算額（CSV由来）。`value=0` のノードは 5,664件中 1,007件
- `project-spending` の `value` = 支出額（CSV由来）
- 予算合計 ≈ 151.1兆、支出合計 ≈ 129.7兆（予算 > 支出 が多いが逆転事業も存在）

---

## 設計方針

### 「同じ金額 = 同じ高さ」原則

`ky`（縦スケール係数）は全列共通の単一値で決まるため、**列をまたいで同じ金額でも列の合計が異なると見た目の高さが変わる**。

- 予算列（151兆）と支出列（130兆）で列合計が異なる → 同じ1兆円でも高さが違う
- これは図の読み取りにおいて誤解を招く

**結論: 全列をウィンドウ内支出で統一する**

`project-budget` ノードの高さも「予算額」ではなく「ウィンドウ内支出額（`projectWindowValue`）」を使う。予算額という情報はツールチップ・サイドパネルで表示することで補完する。

### 列ごとのノード高さ基準（確定）

| ノード種別 | 高さ基準 | `skipLinkOverride` | `layoutCap` |
|---|---|---|---|
| total | ウィンドウ内支出先合計（現状維持） | false | — |
| ministry（TopN） | ウィンドウ内支出合計（現状維持） | false | — |
| `__agg-ministry` | ウィンドウ内支出合計（現状維持） | false | — |
| project-budget（TopN） | **ウィンドウ内支出額**（変更） | **true**（現状維持） | ~~ウィンドウ内支出額~~ → 不要 |
| `__agg-project-budget` | ウィンドウ内支出合計（現状維持） | **true**（新規） | ~~otherProjectWindowTotal~~ → 不要 |
| project-spending（TopN） | ウィンドウ内支出額（現状維持） | true | — |
| `__agg-project-spending` | ウィンドウ内支出合計（現状維持） | **true**（新規） | ~~projectLayoutCap~~ → 廃止 |
| recipient（window内） | **ウィンドウ内受取額のみ**（変更） | **true**（新規） | — |
| `__agg-recipient` | tail受取額 + 非TopN事業tail流（現状維持） | — | 最小ウィンドウ受取額 × topRecipient（現状維持） |

> `project-budget` の `rawValue` に予算額を保持し、ツールチップ・サイドパネルで表示する。

### キャップ設計の変更

全列をウィンドウ内支出で統一することで列間スケール差がなくなり、`__agg-project-budget` と `__agg-project-spending` の `layoutCap` が不要になる。

| キャップ | 変更前 | 変更後 |
|---|---|---|
| `__agg-project-budget.layoutCap` | `otherProjectWindowTotal` | **廃止**（value 自体がウィンドウ内支出合計） |
| `__agg-project-spending.layoutCap` | `projectLayoutCap`（複雑な推計） | **廃止**（`skipLinkOverride:true` で直接保護） |
| `__agg-recipient.layoutCap` | `minWindowRecipientValue × topRecipient` | **維持**（件数が多すぎて比例制御困難） |

### ウィンドウ外リンクの扱い

- **支出先ノードの高さ**: `allRecipientAmounts`（全流入合計）→ ウィンドウ内受取額のみに変更
  - `skipLinkOverride:true` を付与してレイアウトエンジンの上書きを防止
  - リボン幅とノード高さが一致するようになる

## 変更箇所（`app/lib/sankey-svg-filter.ts`）

### 変更1: `project-budget`（TopN）の value をウィンドウ内支出額に変更

```typescript
// 変更前
{ ...budgetNode, skipLinkOverride: true, layoutCap: hasHidden ? wv : undefined }

// 変更後
{ ...budgetNode, rawValue: budgetNode.value, value: wv, skipLinkOverride: true }
// rawValue に予算額を保持（ツールチップ・サイドパネル表示用）
// layoutCap は不要（value が最初からウィンドウ内支出額）
```

### 変更2: `__agg-project-budget` に `skipLinkOverride:true` を追加、`layoutCap` を廃止

```typescript
// 変更前
{ value: otherProjectWindowTotal, layoutCap: projectLayoutCap }

// 変更後
{ value: otherProjectWindowTotal, skipLinkOverride: true }
// value はすでにウィンドウ内支出合計なのでキャップ不要
```

### 変更3: `__agg-project-spending` に `skipLinkOverride:true` を追加、`projectLayoutCap` を廃止

`projectLayoutCap` の複雑な推計計算ごと削除する。

```typescript
// 変更前
const projectLayoutCap = Math.max(1, ...);
{ value: otherProjectWindowTotal, layoutCap: projectLayoutCap }

// 変更後
{ value: otherProjectWindowTotal, skipLinkOverride: true }
```

### 変更4: 支出先ノードの value をウィンドウ内受取額に変更

`projectWindowValue` の計算ループ（L43-L47）と同時に `recipientWindowValue` も計算する:

```typescript
// L43-L47 のループを拡張
const recipientWindowValue = new Map<string, number>();
for (const e of allEdges) {
  if (windowRecipientIds.has(e.target)) {
    projectWindowValue.set(e.source, (projectWindowValue.get(e.source) || 0) + e.value);
    recipientWindowValue.set(e.target, (recipientWindowValue.get(e.target) || 0) + e.value);
  }
}
```

```typescript
// 変更前（L154）
value: allRecipientAmounts.get(rid) || rNode.value

// 変更後
value: recipientWindowValue.get(rid) || 0,
skipLinkOverride: true
```

## 副作用と確認事項

| 影響 | 内容 |
|---|---|
| `project-budget` の高さが変わる | 予算額 → ウィンドウ内支出額に揃うため、予算 > 支出の事業はノードが小さくなる。ただしすべての列が同じ支出スケールに統一されるため比較は正確になる |
| `project-budget` の tooltip に予算額が表示される | `rawValue` に予算額を保持するため、ホバー時・サイドパネルで予算額を確認可能 |
| 支出先ノード高さの縮小 | ウィンドウ外からの流入が除外されるため高さが減少するケースが多い。リンク幅とノード高さの整合性が向上する |
| `projectLayoutCap` 削除 | 推計キャップが消えるため `__agg-project-spending` の高さがより実態に近くなる |
| `ky`（縦スケール）の安定化 | 全列が支出ベースに統一されるため、列合計の差が小さくなり `ky` が安定する |

## 0円ノードの可視化設計（サイドパネルアプローチ）

### 背景データ（`sankey3-graph.json`）

| 区分 | 件数 | 備考 |
|---|---|---|
| 予算・支出ともにあり | 4,392件 | 正常 |
| 予算のみ（支出=0） | 264件 | 合計 ≈ 1.44兆円 |
| 支出のみ（予算=0） | 225件（※） | 合計 ≈ 0.66兆円 |
| 両方0 | 783件 | データなし |

※ デジタル庁が 154件と突出（ITシステム移管の影響）

### 設計方針

Sankeyレイアウトへの組み込みは矛盾が多いため、**サイドパネルを通じた可視化**で対応する。

#### 予算総計・府省庁ノード選択時のサイドパネル

現在：inEdges/outEdges のリスト表示のみ

**変更後：以下のセクションを追加**

1. **事業数サマリ**（ノードヘッダー直下）
   - 「事業数: N件（うち支出データなし: X件 / 予算データなし: Y件）」
   - 府省庁ノードでは自省庁に所属する全事業を対象
   - 予算総計ノードでは全5,664事業を対象

2. **府省庁グループ別事業リスト**（予算総計ノード選択時）
   - 府省庁ごとに折りたたみ可能なグループ
   - 各グループに「N件（うちデータ欠損: X件）」のカウント表示
   - 事業リストからクリックで対象ノードへ遷移（`handleConnectionClick` を使用）

3. **データ欠損事業リスト**（府省庁ノード選択時）
   - 通常の流入/流出リストの下に「データ欠損事業」セクションを追加
   - 「予算=0の事業（N件）」「支出=0の事業（N件）」を折りたたみで表示
   - 各事業をクリックすると当該ノードに遷移（対象ノードが非表示の場合はピン留め）

#### 0円ノードへの遷移（アクセス導線）

サイドパネルの事業リストから `project-budget-{pid}` または `project-spending-{pid}` をクリック → 既存の `handleConnectionClick` が動作。

- ノードがレイアウトに存在する場合：`focusOnNeighborhood` で遷移
- ノードが非表示（TopN外）の場合：`pinnedProjectId` でピン留め → レイアウト再計算後にフォーカス

ただし value=0 のノードはピン留めしても `projectWindowValue=0` のため TopN 選出に引っかからない。**ピン留め対象を project-spending に限定するとこの問題は起きない**（支出のあるノードはウィンドウ操作で表示可能）。予算=0/支出ありのケースも `projectWindowValue > 0` であれば通常どおりピン留めで表示できる。

#### `graphData` へのメタデータ追加（必要な場合）

現在 `graphData.nodes` には全ノードが含まれているため、サイドパネルで府省庁別・欠損別の集計は**クライアント側で計算可能**。`graphData` の変更は不要。
