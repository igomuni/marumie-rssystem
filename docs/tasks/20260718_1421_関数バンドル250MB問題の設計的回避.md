# Vercel 関数バンドル 250MB 問題の設計的回避

作成日: 2026-07-18（JST）
関連:

- `docs/tasks/20260715_0731_R2offload実施対応案.md` — 帯域 offload の検討（優先度低下、発動待ち）
- PR #259（コミット 9209923） — AIチャット関数の 270MB 超過を `outputFileTracingExcludes` で対症療法済み

## 1. 背景と問題の構造

2026-07-10、`/api/ai/sankey-chat` のデプロイが「関数サイズ 270.15MB > 上限 250MB」で失敗した。
直接の解消は `next.config.ts` の `outputFileTracingExcludes`（当該ルートから `public/data/**` を除外）だが、これは**対症療法**であり、データの年度追加や新 API 追加で他ルートでも再燃しうる構造が残っている。

Next.js のファイルトレーシングは各 API ルートの import グラフを静的解析し、`fs.readFileSync(path.join(process.cwd(), 'public', 'data', ...))` パターンを検出すると該当ファイルを関数バンドルに同梱する。問題を生む設計要因は3つ:

| # | 要因 | 内容 |
|---|---|---|
| F1 | 全ローダを束ねる import | AIチャットが全ローダを import → 全データのトレースが1関数に集約（直接原因） |
| F2 | パスの `${year}` テンプレート | トレーサーがファイル名を特定できず**ディレクトリ単位で同梱** |
| F3 | `public/data` の兼務 | クライアント配信用とサーバ読込用が同居。サーバローダが触ると展開後 raw（96MB級含む）が全部巻き込まれる |

## 2. 設計ルール（回避策・推奨順）

### R1: サーバ専用データを `public/` から分離する

サーバ API しか読まないデータ（recipient-index、rs-project-details、project-quality-recipients 等）を `data/server/` など**非公開ディレクトリ**へ移す。

- トレーシングの巻き込み範囲がルートごとに最小化される。
- 副次効果: bulk `.gz` の URL 直ダウンロード経路（帯域ベクタ、offload 検討 6-2）が自然に塞がる。

### R2: サーバは `.gz` を正として実行時に gunzip する（効果最大）

現状、`quality-scores-loader.ts` / `quality-recipients-loader.ts` / `project-details-loader.ts` の3本は既に `.gz` フォールバック実装済み（raw が無ければ `zlib.gunzipSync` で読む）。これを「フォールバック」から「正」に昇格させる:

- `prebuild` の `.gz` → `.json` 展開を**クライアント配信ファイルだけに限定**。
- サーバローダは常に `.gz` を読む（raw 分岐を削除、または raw をローカル開発の高速化用に残す）。
- 未対応の `sankey-graph-loader.ts` / `recipient-index-loader.ts` / `subcontracts-loader.ts` / `mof-overview/route.ts` にも同じ読み方を適用。

効果: 関数同梱データが **raw 96MB級 → gz 十数MB** に縮小（1桁下のオーダー）。年度追加程度では 250MB に再接近しない。gunzip コストは初回コールドスタートのみ（メモリキャッシュ構造なら影響軽微）。

### R3: ルートは自分が使うローダだけ import する

バレルファイルや「全ローダのレジストリ」を作らない。handler 内の dynamic import でも**トレーサーは追跡する**ため、分離は「参照しないこと」でのみ担保できる。AIチャット型の再発防止ルール。

### R4: トレース検証をビルドフローに組み込む

ローカル `npm run build` 後に `.next/server/app/api/**/*.nft.json`（トレース結果）を検査し、「どの関数が `public/data` / `data/server` の何を同梱しているか」を出力するスクリプトを用意する（PR #259 時の検証手法のスクリプト化）。`/quality-check` 系のフローに追加し、無自覚な巻き込みをデプロイ前に検出する。

## 3. 期待効果と offload 検討への含意

- R1 + R2 だけで関数サイズ問題は**外部インフラなしに構造解消**できる。250MB 対策としての外部ストレージ化（R2/GitHub Pages）は不要。
- 帯域問題は前回結論どおり観測ベース（Vercel Usage 監視、逼迫時に Pages 案を発動）。
- したがって、**今実施する価値が高いのは本ドキュメントのバンドル設計整理**であり、offload は発動待ちバックログ。

## 4. 実装タスク分解（着手時）

1. `data/server/` 新設、サーバ専用 `.gz` を移動（パイプライン出力先・`.gitignore`/Git 管理対象の調整含む）
2. ローダ 5〜6 本のパス変更と `.gz` 読み一本化（R1・R2）
3. `prebuild` 展開スクリプトをクライアント配信ファイル限定に変更
4. `outputFileTracingExcludes`（対症療法）の要否を再評価 — R1/R2 後は原則不要になる見込み。AIチャット有効化時の見直し宿題（WP5-1）もここで解消
5. トレース検証スクリプト（R4）作成、`/quality-check` フロー追記
6. ローカル build でルートごとの同梱ファイルを確認 → デプロイ検証

規模感: 半日〜1日。リスクは prebuild フロー変更に伴うローカル開発体験の変化（raw 直読みの高速パスを残すかで調整可能）。

## 5. 実装結果（2026-07-18 実施・同日完了）

ブランチ `refactor/function-bundle-gz-tracing` で実装。設計からの差分と実装中の発見:

### 5-1. 重要な発見: excludes は includes 適用後に掛かる（Next 15 実装制約）

当初「`outputFileTracingExcludes` で `public/data/**` を全関数除外し、`outputFileTracingIncludes`
で `.gz` だけ同一ツリーから同梱し直す」構成を試みたが、**include した `.gz` まで消えた**。
Next の `collect-build-traces.js` は「トレース結果 + includes の結合集合」に対して excludes を
最後に適用するため、**除外ツリー内のファイルを include で残すことは構造的に不可能**。

→ 解決として R1（別ツリー分離）を「Git 上の物理移動」ではなく「**prebuild での同期**」で実現:
`scripts/decompress-data.sh` が `public/data/*.json.gz`（+ `.gz` を持たない小容量 mof raw）を
`data/server/`（gitignore 済み・生成物）へコピーし、includes は `./data/server/**` を指す。
パイプラインの出力先・Git 管理は無変更で済む。

### 5-2. 実装内容

| 変更 | ファイル |
|---|---|
| 読み込みヘルパ新設（`public/data` raw → 同 `.gz` → `data/server` の順で探索。パス構築を一元化しトレーサーの推測同梱を防ぐ） | `app/lib/api/data-file.ts` |
| ローダ 6本 + mof ルートをヘルパに統一（raw 専用だった sankey-graph / recipient-index / subcontracts / mof も `.gz` 読み可能に） | `app/lib/api/*-loader.ts`, `app/api/sankey/mof-overview/route.ts` |
| 全関数から `public/data` 除外・`data/server` 同梱 | `next.config.ts` |
| 展開をクライアント配信 6ファイル限定に + `data/server` 同期 | `scripts/decompress-data.sh` |
| トレース検査スクリプト（`npm run check-traces`） | `scripts/check-function-traces.mjs` |
| `/quality-check` に手順4（対象変更時の check-traces）追加 | `.claude/commands/quality-check.md` |

### 5-3. 検証結果

- `npm run build && npm run check-traces`: 全26関数が `data/server`（15ファイル・約34MB）のみ同梱、`public/data` 同梱 0件。
- ヘルパ単体: gz フォールバック・raw・欠損 null・必須版エラーの4経路を疑似ツリーで確認。
- 本番サーバ（`next start`）で API 疎通: sankey/query・quality-scores・quality-recipients（search/spending）・project-details・subcontracts・recipient-index（search/recipients）・mof-overview・highlights すべて 200。
- lint エラー0（警告は既存のみ）・tsc OK・vitest 71件パス。

### 5-4. 追加インシデント: パス構築の静的解析不能で「プロジェクトルート全体」が同梱（初回 Preview で 383MB 超過）

初回実装の Preview デプロイが 383.6MB で失敗。原因は2つのパス構築パターンで、
トレーサー（nft）が引数を静的解析できないと**既知プレフィックスのディレクトリ全体**を依存に含めること:

1. `data-file.ts` の `path.join(process.cwd(), ...dir, fileName)`（配列スプレッド）
   → プレフィックスが `process.cwd()` のみ = **プロジェクトルート全体**（ローカル実測で
   houjin.db 1GB・`.git` パック・CSV 群を含む 2.8GB がトレースに載っていた）
2. `usage-log.ts` の `path.join(process.cwd(), 'data')` → **`data/` 全体**（約2.1GB）

初回の検査スクリプトは public/data と data/server の有無しか見ていなかったため見逃した。

対処:

- **`path.join` の引数は文字列リテラルで書く**規約を data-file.ts / usage-log.ts に明記
  （リテラルなら nft のグロブが該当ディレクトリに限定され、config の excludes/includes 制御下に入る）
- usage-log の出力先を `data/usage/`（専用サブディレクトリ）に変更しトレース範囲を限定
- 検査スクリプトを**全ファイル分類 + サイズ集計**に強化: 禁止ルート（public/data・
  data/(server・usage以外)・.git）の検出と、関数あたり合計 200MB 超で失敗する閾値を追加。
  再ビルド後、全26関数が約36MB（data/server 35.2MB + コード）で違反なしを確認。

### 5-5. 副次効果と残メモ

- 旧 `outputFileTracingExcludes`（AIチャット限定の対症療法）は全関数一律の構成に置換され、WP5-1 の「チャット有効化時に除外を見直す」宿題は解消（有効化しても `.gz` 約34MBのみで上限に触れない）。
- サーバ専用 raw（recipient-index 31MB 等）が Vercel の静的配信からも消える（prebuild で展開しなくなったため）。`/data/*.json.gz` の直DLは従来どおり可能（offload 検討 6-2 のバックログのまま）。
- 年度追加時の増分は `.gz` 約13MB/年度 × 全関数で、250MB まで1桁の余裕。
